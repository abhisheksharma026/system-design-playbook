01Key Terms

Configuration
A set of parameters, constants, or settings that control a system's behavior without changing its code. Config lives in files (JSON, YAML, TOML), environment variables, databases, or dedicated services. Every production system has config — from the database connection string to the color of a button. Kubernetes stores all cluster state as configuration in etcd. Your laptop's /etc/hosts file is configuration. A neural network's hyperparameters are configuration. The line between "code" and "config" is the line between "what the system does" and "how it does it in this environment."
Static Configuration
Config that is baked into the application at build or deploy time and requires a redeploy to change. Think of a React app's API_BASE_URL injected during npm run build, or a Go binary compiled with -ldflags embedding a version string. Static config is simple and predictable — you know exactly what's running because it shipped with the code. The downside: changing a single feature flag means a full CI/CD pipeline cycle (build → test → deploy → rollout). Netflix uses static config for values that must be identical across all instances of a microservice release.
Dynamic Configuration
Config that lives outside the application binary and can be changed at runtime without redeploying. The app reads config from an external source (file on disk, database, config service like etcd/Consul/ZooKeeper) and picks up changes either by periodic polling, filesystem watches, or push notifications. Dynamic config powers feature flags, A/B tests, rate limits, and circuit breaker thresholds. The tradeoff: more flexibility but more risk — a bad dynamic config change propagates to all instances instantly with no CI gate.
Feature Flag (Feature Toggle)
A boolean or multi-variant configuration value that enables or disables a feature at runtime without deploying new code. Martin Fowler categorizes them into release toggles (ship dark, enable later), experiment toggles (A/B tests), ops toggles (kill switches), and permission toggles (per-user access). LaunchDarkly, Unleash, and Flagsmith are dedicated feature flag platforms. GitHub uses feature flags to gradually roll out new UI to employees → beta users → 1% → 10% → 100% of users.
Environment Variable
A key-value pair set in the operating system's process environment, accessible to the running program via process.env (Node.js), os.environ (Python), or os.Getenv (Go). The Twelve-Factor App methodology (Factor III) mandates storing all environment-varying config in env vars — not in code, not in config files checked into git. Heroku, Docker, and Kubernetes all use env vars as the primary config injection mechanism. The limitation: env vars are flat strings with no nesting, no types, and no built-in change notification.
Config Precedence (Override Chain)
The order in which multiple config sources are consulted, where later sources override earlier ones. A typical chain: hardcoded defaults → config file → environment variables → command-line flags → runtime overrides. Libraries like Viper (Go), python-decouple, and dotenv implement this layered resolution. The chain enables setting safe defaults that developers can override locally without touching shared config files.
Secrets Manager
A dedicated service for storing, rotating, and auditing access to sensitive configuration values — API keys, database passwords, TLS certificates, encryption keys. HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager, and Azure Key Vault all provide encryption at rest, fine-grained IAM policies, automatic rotation, and audit logs. The fundamental rule: secrets must never appear in source code, config files committed to git, or container images. In 2019, a researcher found over 100,000 repos on GitHub with hardcoded AWS keys.

02Why Configuration Matters

Configuration is the most common cause of production outages that aren't code bugs. A misconfigured timeout, a wrong database endpoint, a feature flag flipped to the wrong value — these take down systems faster than any software defect because they bypass testing pipelines.

Facebook, October 2021: A configuration change to the backbone routers caused a cascading failure that took Facebook, Instagram, and WhatsApp offline for ~6 hours. The config change withdrew BGP route advertisements, making Facebook's DNS servers unreachable. This wasn't a code bug — it was a config update that passed automated validation but had a devastating real-world effect.

Google, December 2020: A misconfigured authentication quota caused a ~47-minute global outage of Gmail, YouTube, and Google Cloud. The config update reduced the authentication token quota below the level needed to handle current traffic.

Cloudflare, June 2019: A single bad regex in a WAF rule (which is configuration, not code) caused CPU exhaustion across all edge servers globally for ~27 minutes.

The pattern is clear: configuration changes are deployments. They should go through the same review, testing, and gradual rollout process as code changes. The industry is slowly learning this — Google's SRE book calls it "treating configuration as code."

The Hybrid Approach (Most Common)

In practice, real systems layer both. Static config defines the structure and defaults, dynamic config overrides specific values at runtime. A typical Express/Flask app reads a config.json baked into the Docker image (static defaults), then overlays environment variables (per-environment overrides), then connects to a config service like Consul for dynamic values (feature flags, rate limits). The precedence chain ensures that a missing dynamic value falls back to the static default gracefully.

04Config Formats Deep Dive

JSON

{
  "database": {
    "host": "db.prod.internal",
    "port": 5432,
    "pool_size": 20,
    "ssl": true
  },
  "feature_flags": {
    "new_checkout": false,
    "dark_mode": true
  },
  "rate_limit": {
    "requests_per_minute": 1000
  }
}

Strengths: Universal parser support in every language. Strict syntax catches typos (missing comma = parse error). Native types: strings, numbers, booleans, arrays, objects. Weaknesses: No comments (a constant pain when documenting config). No multi-line strings. Trailing comma is invalid. Verbose for deeply nested structures.

YAML

# Database configuration
database:
  host: db.prod.internal
  port: 5432
  pool_size: 20
  ssl: true

# Feature flags - updated by product team
feature_flags:
  new_checkout: false
  dark_mode: true

rate_limit:
  requests_per_minute: 1000

Strengths: Comments. Human-readable. Less punctuation. Multi-line strings with | and >. Anchors and aliases (&default / *default) for DRY config. Weaknesses: Indentation-sensitive (a single space error silently changes structure). Implicit type coercion is dangerous — YAML parses no as boolean false, 3.14 as float, and country codes like NO (Norway) as false. The "Norway problem" has caused real production bugs. YAML 1.2 fixed some of these, but many parsers still use YAML 1.1 rules.

TOML

[database]
host = "db.prod.internal"
port = 5432
pool_size = 20
ssl = true

[feature_flags]
new_checkout = false
dark_mode = true

[rate_limit]
requests_per_minute = 1000

Strengths: Explicit typing (strings always quoted, booleans are only true/false). Comments. No indentation sensitivity. First-class date/time support. Used by Rust's Cargo, Python's pyproject.toml, and Hugo. Weaknesses: Less widely supported than JSON/YAML. Deeply nested tables become awkward syntactically.

.env Files

DATABASE_HOST=db.prod.internal
DATABASE_PORT=5432
DATABASE_POOL_SIZE=20
FEATURE_NEW_CHECKOUT=false
SECRET_API_KEY=sk-abc123  # NEVER commit this to git

Flat key-value pairs. No nesting. No types (everything is a string). Loaded into the process environment via dotenv libraries. The convention: .env for local dev (gitignored), .env.example committed as a template. In production, the real env vars are injected by the orchestrator (Kubernetes ConfigMap, Docker Compose environment block, Heroku config vars).

05Environment Variables & the Twelve-Factor App

The Twelve-Factor App methodology (Factor III) is the single most influential idea in modern configuration management. Its rule is simple: store config in the environment, not in code.

The core insight: anything that changes between deploys (staging vs. production, region A vs. region B) is config. Anything that stays the same across deploys is code. Config belongs in the environment — not in a checked-in file, not in a build constant, not in a database that could be down when the app starts.

The litmus test: Could the codebase be made open source at any moment without compromising any credentials? If yes, your config is properly separated. If no, you have config (probably secrets) embedded in code.

How Env Vars Flow in Production

Docker: docker run -e DATABASE_URL=postgres://... myapp or via --env-file .env. The container's PID 1 process inherits these env vars.

Kubernetes: ConfigMaps for non-sensitive config, Secrets for sensitive values. Both can be mounted as env vars or as files in the pod's filesystem. Secrets are Base64-encoded (not encrypted!) by default — you need a KMS plugin or external operator (Sealed Secrets, External Secrets Operator) for real encryption.

CI/CD: GitHub Actions secrets, GitLab CI variables, CircleCI contexts — all inject env vars into the build/deploy process. These are never visible in logs or build artifacts.

06Feature Flags

Feature flags are the most powerful form of dynamic configuration. They decouple deployment (shipping code to production) from release (making the feature visible to users). This is a fundamental operational capability — it means you can deploy on Friday afternoon, ship the feature on Monday morning, and have a kill switch if anything goes wrong.

Gradual Rollout Pattern

// Pseudocode: percentage-based rollout
function isFeatureEnabled(userId, flagName) {
  const flag = configService.getFlag(flagName);
  if (flag.type === 'boolean') return flag.value;
  if (flag.type === 'percentage') {
    // Deterministic: same user always gets same result
    const hash = murmur3(userId + flagName) % 100;
    return hash < flag.percentage;
  }
  if (flag.type === 'allowlist') {
    return flag.userIds.includes(userId);
  }
  return false; // default off
}

The key property: the hash is deterministic. User "alice" at 10% rollout will either always see the feature or never see it — she won't flip between states on every page load. As the percentage increases from 10% → 50% → 100%, users already included stay included.

Flag debt: Stale feature flags are a real maintenance burden. A codebase with 500 flags where 300 are permanently "on" is a minefield of dead branches, untested code paths, and confusing conditionals. Best practice: every flag has an expiration date. When the flag is fully rolled out or the experiment concludes, the flag and its conditional code are removed in a cleanup PR.

07Secrets Management

Secrets are configuration values that provide access to protected resources — database passwords, API keys, TLS private keys, OAuth client secrets, encryption keys. They require fundamentally different handling from regular config.

The Rules

1. Never in source control. Not in code, not in config files, not in comments, not in commit history. Use .gitignore and pre-commit hooks (e.g., git-secrets, trufflehog) to catch accidental commits. If a secret is committed, consider it compromised — rotate it immediately, don't just delete and force-push.

2. Never in logs. Middleware should redact known secret patterns. A common mistake: logging the full request body that contains an API key in a header.

3. Encrypted at rest. Kubernetes Secrets are only Base64-encoded by default — this is encoding, not encryption. Enable etcd encryption at rest or use Sealed Secrets / External Secrets Operator backed by Vault or cloud KMS.

4. Rotated regularly. If a secret never changes, its blast radius on compromise is infinite. Automatic rotation (AWS Secrets Manager supports Lambda-based rotation) limits exposure to the rotation window.

5. Least-privilege access. A frontend service should never have the database root password. Issue per-service credentials with the minimum required permissions. Vault's dynamic secrets generate short-lived, per-request database credentials that automatically expire.

08Configuration in Distributed Systems

When you have 500 microservices across 10,000 containers, config management becomes a distributed systems problem. The core challenges: consistency (all instances see the same config), propagation speed (how fast does a change reach every instance), and availability (what happens when the config store is down).

Centralized Config Services

etcd: Distributed key-value store using Raft consensus. Kubernetes stores all cluster state here — pod specs, ConfigMaps, Secrets, service endpoints. Linearizable writes, watch API for real-time change notifications. Typical cluster: 3 or 5 nodes for fault tolerance. Write throughput: ~10K writes/sec. Read throughput: ~100K reads/sec (served locally from any replica). Used when you need strong consistency over partition tolerance.

ZooKeeper: The original distributed coordination service (Yahoo, 2010). Hierarchical namespace (like a filesystem), ephemeral nodes for presence detection, sequential nodes for distributed locks. Used by Kafka (pre-KRaft), HBase, Solr. Operational complexity is higher than etcd — JVM tuning, session management, and the "herd effect" on watched nodes.

Consul: HashiCorp's service mesh + config store. Key-value storage with HTTP API, DNS interface for service discovery, health checking, and multi-datacenter support. Unique feature: "watches" that trigger handler scripts when config values change. Used when you want config + service discovery + health checking in one system.

Spring Cloud Config: Git-backed config server for Java/Spring microservices. Config files live in a Git repo; the config server serves them over HTTP. Change the git repo → webhook triggers refresh → apps pick up new config. Good for teams that want config versioning and PR-based review, but it adds a single point of failure (the config server) and has higher latency than a KV store.

Demo 1: Config Precedence in Action

# Create a config file with defaults
$ cat > config.json <<'EOF'
{"port": 3000, "debug": false, "db_host": "localhost"}

# Python script that demonstrates precedence
$ cat > app.py <<'PYEOF'
import json, os, sys

# Layer 1: Hardcoded defaults
config = {"port": 8080, "debug": False, "db_host": "127.0.0.1", "log_level": "info"}

# Layer 2: Config file overrides defaults
with open("config.json") as f:
    file_config = json.load(f)
    config.update(file_config)

# Layer 3: Env vars override file (if set)
for key in config:
    env_val = os.environ.get(key.upper())
    if env_val is not None:
        # Coerce types
        if isinstance(config[key], bool):
            config[key] = env_val.lower() in ("true","1","yes")
        elif isinstance(config[key], int):
            config[key] = int(env_val)
        else:
            config[key] = env_val

print("Final config:")
for k, v in config.items():
    print(f"  {k} = {v!r}")
PYEOF

# Run with no env vars — file overrides defaults
$ python3 app.py
# Expected:

  debug = False          # from config.json

12Mini-Projects

Simple: Feature Flag Gate

Goal: Build a Python HTTP server that reads feature flags from a JSON file. One endpoint (/feature) returns different content based on a flag. Change the flag file while the server is running — observe hot-reload.

# Save as flag_server.py
import json, os, time
from http.server import HTTPServer, BaseHTTPRequestHandler

FLAG_FILE = 'flags.json'
flags = {}
last_load = 0

def load_flags():
    global flags, last_load
    mtime = os.path.getmtime(FLAG_FILE)
    if mtime != last_load:
        with open(FLAG_FILE) as f:
            flags = json.load(f)
        last_load = mtime
        print(f"[{time.strftime('%H:%M:%S')}] Flags reloaded: {flags}")

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        load_flags()
        if self.path == '/feature':
            if flags.get('new_ui', False):
                body = '{"ui": "v2", "message": "Welcome to the new experience!"}'
            else:
                body = '{"ui": "v1", "message": "Classic view"}'
            self.send_response(200)
            self.send_header('Content-Type', 'application/json')
            self.end_headers()
            self.wfile.write(body.encode())
        else:
            self.send_response(200)
            self.send_header('Content-Type', 'application/json')
            self.end_headers()
            self.wfile.write(json.dumps(flags).encode())
    def log_message(self, *a): pass

# Create initial flags
if not os.path.exists(FLAG_FILE):
    json.dump({"new_ui": False, "dark_mode": True}, open(FLAG_FILE, 'w'))

load_flags()
print("Flag server on :8080 | GET / (all flags) | GET /feature (gated content)")
HTTPServer(('', 8080), Handler).serve_forever()
# Terminal 1: python3 flag_server.py
# Terminal 2: curl http://localhost:8080/feature
→ {"ui": "v1", "message": "Classic view"}

# Flip the flag:
python3 -c "import json; json.dump({'new_ui':True,'dark_mode':True},open('flags.json','w'))"

# Hit the endpoint again (no server restart!):
curl http://localhost:8080/feature
→ {"ui": "v2", "message": "Welcome to the new experience!"}

Advanced: Config Service with Watchers

Build a tiny config server that stores key-value pairs in memory, exposes a REST API (GET/PUT /config/:key), and supports a GET /watch/:key endpoint using long polling — when the key changes, all watchers receive the new value immediately. This mirrors how etcd's watch API works. Add a TTL mechanism: keys can optionally expire after N seconds (like a session or a temporary override). Build a client that watches a key and prints changes in real time.

13Production Examples

Kubernetes: ConfigMaps & Secrets

Kubernetes separates configuration from container images using two primitives: ConfigMaps for non-sensitive data and Secrets for sensitive data. Both can be consumed as environment variables or as files mounted into the pod's filesystem.

The live-update behavior: When a ConfigMap is mounted as a volume, Kubernetes periodically syncs the mounted files with the ConfigMap's current state (kubelet sync period, default ~1 minute). The application can use inotify/fswatch to detect the change and reload. This is dynamic configuration without a redeploy — but the application must be written to handle mid-flight config changes gracefully (e.g., don't crash if the config format changes unexpectedly).

Immutable ConfigMaps (Kubernetes 1.21+): Setting immutable: true on a ConfigMap prevents accidental changes and improves performance by disabling the kubelet's periodic sync. For truly static config, this is the safest option — changes require creating a new ConfigMap with a new name and updating the pod spec.

Netflix: Archaius & Dynamic Properties

Netflix's Archaius is a Java configuration library that implements a layered config model with dynamic property support. Config sources are stacked in priority order: defaults → config file → database → environment → runtime override. Each source can be polled independently — the database source polls every 30 seconds, while the runtime override takes effect instantly.

The killer feature: Dynamic property callbacks. You register a callback on a config key, and Archaius invokes it whenever the value changes. This lets Netflix engineers change circuit breaker thresholds, timeout values, and feature flags in production without redeploying any service. A single Archaius config change can propagate to thousands of instances within 30 seconds.

Safety mechanism: Netflix's config changes go through a "canary" process — the change is applied to a small percentage of instances first, and if error rates spike, the change is automatically rolled back. This is the dynamic config equivalent of a canary deployment.

Google: Borg & Global Config Distribution

Google's internal config system (described in SRE book Chapter 14) treats configuration as a form of code. Config changes go through the same review process as code changes — they have owners, reviewers, automated tests, and a staged rollout process. Config is stored in a version-controlled repository and distributed to production machines via a global config distribution service.

The principle: "Configuration is the most common cause of serious production problems at Google. The most insidious configuration problems are not syntax errors — they are logical errors where the config is valid but wrong." Google's response: config validation that goes beyond syntax checks to test the config against a model of the system's expected behavior.

14Edge Cases & War Stories

The Partial Update Problem: You update a config file with 5 fields. The file write is not atomic — the app reads the file between the 3rd and 4th field being written. It sees a half-written config with an inconsistent state (e.g., new database host but old database port). Fix: Write to a temporary file, then rename() — rename is atomic on POSIX filesystems within the same partition. Kubernetes ConfigMap volume mounts use this pattern internally (symlink swap).

Config Drift Between Instances: Instance A polls the config service and gets the new value. Instance B's poll hasn't fired yet. For the next 30 seconds, A and B behave differently — A serves the new feature, B serves the old one. If a load balancer round-robins between them, users experience inconsistent behavior on every other request. Fix: Use push-based propagation (etcd watch) instead of polling, or accept the drift window and ensure the feature is safe during mixed-state operation.

The Config Service is Down: Your app reads all config from etcd/Consul at startup. The config service is temporarily unavailable. Options: (1) Fail to start — simple, safe, but fragile. (2) Use a cached copy from the last successful fetch — more resilient, but the cache could be stale. (3) Fall back to static defaults baked into the binary — safest, but the defaults might not be suitable for production. Best practice: Cache the last known good config locally. Start with the cache. Log a warning. Retry connection in the background. This is what the Consul agent does — it maintains a local snapshot of the service catalog.

Secret Rotation Breaks Running Connections: You rotate the database password in Vault. The new password propagates to all app instances. But Instance A has a long-running database connection pool using the old password. When that pool needs to create a new connection (e.g., the old connection times out), it fails because it's using the stale credential. Fix: Application-level secret refresh: when a connection fails with an auth error, re-fetch the credential from Vault and retry before reporting the error to the caller.

Env Var Leakage in Logs/Errors: A Python traceback prints the full stack including variables in scope. If os.environ['SECRET_KEY'] was recently accessed, it can appear in the traceback. Many error tracking services (Sentry, Bugsnag) capture environment variables by default. Fix: Configure error reporters to redact sensitive keys. Use a secret management wrapper that never stores the raw value in a variable longer than needed.

15Interview Q&A

Q1: What's the difference between static and dynamic configuration? When would you use each?

Static config is baked into the artifact at build/deploy time — config files committed to the repo, env vars set in the deployment manifest. Changes require a redeploy. Use it for values that are fundamental to the application's identity in an environment: API base URLs, database schemas, service names, TLS certificate paths.

Dynamic config lives in an external store and can be changed at runtime. Use it for values that need to change faster than your deploy cycle: feature flags, rate limit thresholds, circuit breaker timeouts, A/B test parameters.

Follow-up nuance: The boundary isn't binary — it's a spectrum of change frequency and risk. A database connection string changes maybe once a year (static). A rate limit threshold might change during a traffic spike (dynamic). A feature flag for a gradual rollout changes daily for a week then becomes permanent (starts dynamic, becomes static, then gets removed). Design your config system to handle all three gracefully.

Q2: How would you safely roll out a configuration change to 10,000 servers?

Same way you roll out a code change: progressively. Step 1: Apply to a canary group (1% of instances). Step 2: Monitor error rates, latency, and business metrics for 5–10 minutes. Step 3: If healthy, expand to 10%, then 50%, then 100%. Step 4: If anything looks wrong at any stage, roll back the config change instantly.

Implementation: Your config service should support targeting — "apply this value to instances with label canary=true." This is what LaunchDarkly calls "segments" and what Netflix Archaius calls "deployment context." The config propagation system should support both the new and old values simultaneously during rollout.

Deep follow-up: What if the config change is non-reversible? Example: you change a database migration flag that triggers a schema change. Once the schema is changed, you can't just revert the flag. This is where config changes must be treated as deployments — with pre-flight validation, staged rollout, and a tested rollback plan (including the database migration rollback).

Q3: An application reads its config from a centralized config service. The config service goes down. What happens?

It depends on two design decisions: (1) when does the app read config, and (2) does it cache.

Startup-only read, no cache: If the app reads config only at boot and the config service is down, the app fails to start. Existing running instances are fine (they already have their config in memory), but any new instances, restarts, or scaling events fail. This is a hard dependency.

Startup + periodic refresh, with cache: Running instances continue with their last fetched config. New instances start with a locally cached copy (written to disk on each successful fetch). The app logs a warning and retries in the background. When the config service recovers, apps pick up the latest values. The risk: if the config service was down during a critical config change, some instances run with stale config.

Best practice (as described in DDIA): The config service should be treated as a hard dependency only at first boot of a fresh instance with no cache. After that, it should be a soft dependency — nice to have, not required for continued operation. This mirrors how DNS works: DNS being down doesn't break existing connections, only new lookups.

Q4: Why shouldn't you store secrets in environment variables?

This is a nuanced question — the Twelve-Factor App says do store config in env vars, and for many teams, env vars for secrets are fine. But at scale, there are real concerns:

1. Process environment is readable: On Linux, any process running as the same user can read /proc/PID/environ. If an attacker gets code execution (even limited), they can dump all secrets from the process environment.

2. Inherited by child processes: If your app spawns a subprocess (e.g., shelling out to a CLI tool), that subprocess inherits all env vars — including secrets it doesn't need. This violates least privilege.

3. No audit trail: Env vars don't have access logging. You can't answer "who accessed the database password at 3:47 AM?" A secrets manager like Vault provides an audit log of every secret access.

4. No rotation: Changing an env var requires a process restart. A secrets manager can rotate credentials and push the new value to the app without downtime.

Pragmatic answer: For small teams and non-critical systems, env vars for secrets are fine with proper access controls. For regulated industries, multi-team organizations, or systems handling financial/medical data, a dedicated secrets manager is necessary.

Q5: How do you validate configuration before it reaches production?

Layer 1 — Syntax: JSON/YAML linters in CI. Catches parse errors before merge. This is trivial but essential — a missing comma in a JSON config has caused more than one outage.

Layer 2 — Schema: JSON Schema, YAML schema validation, or typed config libraries (TypeScript's zod, Python's pydantic). Catches wrong types (string where number expected), missing required fields, values outside allowed ranges.

Layer 3 — Semantic: Application-specific validation. Does this database host actually resolve? Is this port number in the valid range? Is this rate limit value sane (not 0, not 10 billion)? This runs as part of the app's startup — if validation fails, the app refuses to start and the deploy is rolled back.

Layer 4 — Integration: Staging environment with production-like traffic. Apply the config change to staging first, run smoke tests, verify behavior. Only then promote to production.

Google's approach: Their config system runs the config through a "hermetic build" that simulates the system's behavior with the new config. If the simulation shows unexpected state transitions, the change is blocked.

Q6: You notice that after a config change, some users see the new behavior and some see the old behavior, alternating on each request. What's happening?

This is config drift between instances. The load balancer is round-robining requests across multiple app instances. Some instances have picked up the new config, others haven't yet. Each request hits a different instance, so the user alternates between old and new behavior.

Root cause analysis: (1) Polling-based config propagation with staggered poll timers — Instance A polled 2 seconds ago (has new config), Instance B won't poll for another 28 seconds (still has old config). (2) A failed config fetch on some instances — they're stuck on the old cached value while others have the new one.

Fix approaches: (1) Use push-based config propagation (etcd watch) so all instances get the change simultaneously. (2) If using polling, reduce the poll interval. (3) Add a "config version" header to responses so you can see which config version served each request. (4) For user-visible features, use feature flags with sticky assignment (hash-based) so each user consistently sees one version regardless of which instance serves them.

Round 1: Core Architecture

Interviewer: "Walk me through the high-level design."

Three components: a Flag Management Service (CRUD API + admin UI), a Flag Distribution Layer (how flags get to SDKs), and Evaluation SDKs embedded in each microservice.

Flag Management Service: REST API backed by a PostgreSQL database. Each flag has a name, type (boolean/percentage/segment/experiment), targeting rules, owner team, and expiration date. Changes are audited (who changed what, when). The admin UI lets product managers create flags and set rollout percentages without engineering involvement.

Flag Distribution Layer: When a flag is changed, the management service publishes an event to a Kafka topic. A distribution service consumes these events and pushes updates to all connected SDKs via SSE (or long polling as fallback). Each SDK maintains an in-memory cache of all flag definitions relevant to its service. If the distribution layer is down, the SDK uses its last known cache.

Evaluation SDK: A lightweight library (one per language: Go, Java, Python, JS) that evaluates flag rules locally. The SDK takes a user context (user ID, attributes like plan, country, device) and the flag definition, and returns the evaluated value. All evaluation happens in-process — no network call per flag check. This is critical for latency: a flag check must be <1ms, not a network round-trip.

Round 2: Evaluation Logic

Interviewer: "How does percentage-based rollout work? How do you ensure consistency?"

The SDK hashes user_id + flag_name using a deterministic hash (MurmurHash3) and takes the result modulo 100. If the result is less than the rollout percentage, the user is in. This guarantees: (1) the same user always gets the same evaluation for a given flag, (2) different flags distribute independently because flag_name is part of the hash input, and (3) increasing the rollout from 10% to 20% adds users without ejecting the ones already in.

Segment targeting: Before percentage rollout, the SDK evaluates targeting rules such as country = US AND plan = premium. Rules run in priority order, so you can express combinations like "100% of employees, 50% of premium US users, 0% of everyone else." In practice this is a short-circuit rule engine that has to stay deterministic across every language SDK.

Round 3: Edge Cases

Interviewer: "What happens during a network partition between the SDK and the flag distribution layer?"

The SDK operates in degraded mode: it continues evaluating flags using its local cache. New flag changes won't be reflected until connectivity is restored. The SDK should log a warning and expose a health metric ("time since last successful sync") so monitoring can detect the drift.

Critical design decision: Should the SDK fail open (feature enabled if no data) or fail closed (feature disabled if no data)? This should be configurable per flag. For a new, risky feature: fail closed (if you can't check the flag, don't show it). For an established, critical feature: fail open (if you can't check, assume it's on).

Boot-time concern: If a brand-new service instance starts and cannot reach the flag distribution layer, it has no cache to fall back on. Solution: bake a "default flag snapshot" into the Docker image during CI/CD. This snapshot is updated on each deploy and serves as the cold-start fallback. It may be hours stale, but it's better than having zero flag definitions.

Interviewer: "How do you handle A/B experiment metrics tracking?"

The SDK emits an "exposure event" every time it evaluates an experiment flag: {user_id, flag_name, variant, timestamp}. These events are batched and sent asynchronously to an analytics pipeline (Kafka → data warehouse). The data science team joins exposure events with outcome events (conversion, revenue, engagement) to compute experiment results. The key: exposure events must be deduped per user-session to avoid inflating sample sizes. The experiment is statistically valid only if exposure logging is reliable.

19References & Further Reading

Use these sources to study the actual mechanics behind config systems: precedence rules, watch semantics, consensus-backed control planes, rollout safety, and cache invalidation during live config change.

[1] The Twelve-Factor App — Factor III: Config (Wiggins, 2011)

Adam Wiggins. 2011.
https://12factor.net/config

• Defines "config" as anything that varies between deploys (staging, production, developer environments) — database handles, credentials, per-deploy values like canonical hostname. Explicitly excludes internal application config that does not vary between deploys.

• Prescribes environment variables as the config mechanism because they are language- and OS-agnostic, unlikely to be accidentally committed to source control, and granular (each var is orthogonal, never grouped into "environments" like "staging" or "production").

• The litmus test for correct config separation: the codebase could be open-sourced at any moment without compromising credentials. This single sentence has driven more config architecture decisions than any other principle in cloud-native development.

• Explicitly rejects "config files that are not checked into revision control" (e.g., Java property files in config/) because they tend to be scattered, in different formats, and language-specific. Env vars are a universal primitive.

• The methodology emerged from observing patterns across hundreds of thousands of Heroku deployments (2007–2011), codifying what worked and what consistently caused outages.

[2] ZooKeeper: Wait-free Coordination for Internet-scale Systems (Hunt et al., 2010)

Patrick Hunt, Mahadev Konar, Flavio P. Junqueira, Benjamin Reed. USENIX ATC, June 2010.
https://www.usenix.org/legacy/event/atc10/tech/full_papers/Hunt.pdf

• Introduces a coordination kernel rather than a fixed set of primitives — the API exposes hierarchical znodes (like a filesystem) with create, delete, getData, setData, getChildren, and exists operations. Locks, leader election, and config distribution are built on top by clients, not baked into the service.

• Wait-free reads: any server in the ensemble can serve reads locally without contacting the leader, achieving 10:1 to 100:1 read:write ratios in Yahoo's production deployment. The tradeoff: reads are not linearizable (a client might read stale data), but writes are linearizable (total order via ZAB atomic broadcast).

• Watch mechanism: a client can set a one-time watch on a znode. When the znode changes, the server sends a notification to the client before any subsequent operation on that znode is visible. This enables reactive config propagation — when a config znode changes, all watching clients are notified before any new value can be read.

• Benchmarked at ~21K create operations/sec (3 servers), ~70K reads/sec with saturated read load. Write throughput is limited by the atomic broadcast protocol (ZAB), while read throughput scales linearly with the number of servers because reads are served locally.

• Ephemeral znodes (deleted automatically when the creating client's session expires) enable presence detection: a service registers an ephemeral node at startup; if it crashes, the node disappears, and watchers are notified. This is the foundation of ZooKeeper-based service discovery.

[3] In Search of an Understandable Consensus Algorithm (Ongaro & Ousterhout, 2014)

Diego Ongaro, John Ousterhout. USENIX ATC, June 2014.
https://raft.github.io/raft.pdf

• Raft decomposes consensus into three independent subproblems: leader election (randomized timeouts, typically 150–300ms, prevent split votes), log replication (leader appends entries, followers replicate; a log entry is committed once a majority of servers have written it), and safety (the Election Restriction ensures a leader always has the most up-to-date log).

• etcd's implementation of Raft powers Kubernetes' entire config store. Every ConfigMap, Secret, pod spec, and service definition is a Raft log entry replicated across 3–5 etcd nodes. A Kubernetes config change is literally a Raft proposal that must achieve majority consensus before taking effect.

• Cluster membership changes use a joint consensus protocol: old and new configurations overlap during the transition, requiring majorities in both old and new configs for any decision. This prevents split-brain during cluster expansion/contraction.

• Log compaction via snapshotting: the Raft log grows indefinitely as config changes accumulate. Periodically, the state machine's current state is snapshotted and the log is truncated. etcd defaults to snapshots every 10,000 applied entries.

• User study results: 33 students at Stanford and UC Berkeley scored significantly better on Raft than Paxos after studying both (median quiz score 25.7 vs 20.8 out of 60). This understandability advantage is the reason etcd, Consul, CockroachDB, and TiKV chose Raft over Paxos.

[4] Site Reliability Engineering — Chapter 14: Managing Machine Configuration (Google, 2016)

Betsy Beyer, Chris Jones, Jennifer Petoff, Niall Richard Murphy. O'Reilly, 2016.
https://sre.google/sre-book/configuration-management/

• The chapter treats config as an operational change stream, not just a file format problem. The key lesson is that a syntactically valid change can still be catastrophically wrong because it changes system behavior everywhere at once.

• "Configuration as code" in this chapter is a workflow discipline: review, test, staged rollout, and rollback for config changes, not only for binaries. That is the operational control loop the rest of the page is arguing for.

• The most technical part is the rollout model: emergency changes need a fast path, but the default path still has to preserve validation and blast-radius control. The point is to speed up change safely, not to bypass safety.

• The hermetic-build recommendation matters because it closes the gap between what was tested and what is actually deployed. If environment-specific interpolation happens later, you have already lost that guarantee.

• The "N+1 problem" is the cluster-level failure mode: a change that looks fine on one machine can still break the fleet when every machine responds to it at once, for example by reopening connections or rebalancing load simultaneously.

[5] Chubby: The Lock Service for Loosely-Coupled Distributed Systems (Burrows, 2006)

Mike Burrows. OSDI, November 2006.
https://research.google.com/archive/chubby-osdi06.pdf

• Chubby is Google's internal coordination service — the predecessor and inspiration for ZooKeeper. It provides a small file-based namespace where each "file" is a small (up to 256KB) key-value pair. Originally built for distributed lock management, it evolved into Google's primary config distribution mechanism.

• Event-driven config: clients cache Chubby file contents and register callbacks. When a file changes, Chubby invalidates caches and delivers events, so config updates propagate without polling. This cache-invalidation model was directly adopted by ZooKeeper's watch mechanism.

• Design insight: Burrows observes that developers initially intended to use Chubby as a lock service but overwhelmingly used it as a config/naming store. Writes to Chubby are infrequent (minutes to hours apart), but reads are extremely frequent. This led to the design decision of caching aggressively and invalidating on write — the exact read-heavy pattern that ZooKeeper later formalized.

• The paper documents Chubby's Paxos-based replication across 5 replicas (typically 2 in the same datacenter, 3 in nearby datacenters). A master lease of ~12 seconds ensures a single leader; if the master fails, a new election happens within seconds. DNS-based discovery lets clients find the current master.

• Discusses the "thundering herd" problem when the Chubby leader changes: thousands of clients simultaneously reconnect, potentially overwhelming the new leader. The mitigation: random back-off on reconnection and grace periods during leader transitions.
