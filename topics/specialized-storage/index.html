Specialized Storage Paradigms

Not every problem fits in a relational table or a key-value pair. Some data is inherently temporal, spatial, graph-shaped, or just enormous binary blobs. Picking the right specialized storage paradigm can turn an impossible query into a trivial one.

Key Terms

Blob Storage

A storage service optimized for holding large, unstructured binary objects — images, videos, database backups, ML model artifacts, log archives. Unlike a KV store (which optimizes for small, fast reads), blob storage optimizes for large object throughput and durability at low cost per GB. The API is minimal: PUT(key, bytes), GET(key) → bytes, DELETE(key). Internally, objects are split into chunks, replicated across multiple disks/datacenters, and erasure-coded for durability. AWS S3 guarantees 99.999999999% (eleven 9s) durability. GCS and Azure Blob Storage offer similar guarantees. In system design interviews, blob storage is the default answer for "where do I put images/videos/files?" — never store large binaries in a relational database.

Time Series Database (TSDB)

A database optimized for data that arrives as a stream of (timestamp, value) pairs — CPU utilization at 10-second intervals, stock prices per second, IoT sensor readings, application metrics. TSDBs exploit the temporal structure: timestamps are nearly sequential (so delta-of-delta compression shrinks them dramatically), values change slowly between consecutive points (XOR compression reduces them to a few bits), and queries are almost always time-range scans ("give me the last 24 hours of CPU for server X"). Products: InfluxDB, Prometheus, TimescaleDB (PostgreSQL extension), ClickHouse, VictoriaMetrics. Facebook's Gorilla achieved 12x compression using these techniques. [1]

Graph Database

A database where the primary data model is nodes (entities) and edges (relationships between entities), each with properties. Optimized for traversing deep relationship chains — "find all friends of friends who also like jazz and live within 50km" — which would require expensive recursive JOINs in SQL but is a natural traversal in a graph. The query language is typically Cypher (Neo4j) or Gremlin (Apache TinkerPop). Use cases: social networks, recommendation engines, fraud detection (follow the money through transaction chains), knowledge graphs, identity resolution. Products: Neo4j, Amazon Neptune, JanusGraph, Dgraph.

Spatial Database

A database (or extension to an existing DB) that can store, index, and query geographic data — points (latitude/longitude), lines, polygons, regions. The key capability: spatial queries like "find all restaurants within 2km of my location" or "does this delivery zone polygon contain this address point?" Spatial indexing structures like quadtrees, R-trees, and geohashes make these queries efficient — without them, you'd need to compute distances to every point in the dataset. Products: PostGIS (PostgreSQL extension, the industry standard), MongoDB (2dsphere indexes), Redis (GEO commands), Elasticsearch (geo_point). Uber's H3 hexagonal grid system is a modern approach to spatial indexing.

Quadtree

A tree data structure for indexing 2D spatial data. Each node represents a rectangular region. When a region contains more than N points (the capacity), it splits into four equal quadrants (children). This recursively partitions space: dense areas (cities) get deeply subdivided, sparse areas (oceans) stay as single large nodes. Lookup is O(log₄ N) — traverse from root, pick the quadrant containing the query point, repeat until a leaf. Range queries ("find all points within this rectangle") prune entire subtrees that don't intersect the query rectangle.

Blob Storage — S3 & GCS

Blob storage solves a specific problem: storing and retrieving large, immutable objects cheaply and durably. It's not a database — you can't query the contents. You can only put, get, and delete by key (the object path).

When to Use Blob Storage

Always use blob storage for: user-uploaded images/videos, static website assets (CSS/JS/fonts via CDN), database backups, log archives, ML training datasets, large file downloads. Never put large binaries in PostgreSQL — a 10MB image stored as a BYTEA column bloats the table, slows VACUUM, wastes buffer pool RAM, and makes backups enormous. Store the image in S3, store the S3 URL in PostgreSQL. This pattern (relational DB for metadata, blob store for content) is universal.

Time Series Databases

Time series data has three properties that specialized TSDBs exploit: (1) writes are append-only (you never update a past data point), (2) timestamps are nearly evenly spaced (delta compression works beautifully), and (3) queries are almost always time-range scans ("last hour," "last 7 days") rather than random lookups.

How TSDB Compression Works

Delta-of-delta for timestamps + XOR for values (Gorilla algorithm)
Raw data (16 bytes per point):
  t=1711900000  v=72.5    ← 8 bytes timestamp + 8 bytes float64
  t=1711900010  v=72.3
  t=1711900020  v=72.3
  t=1711900030  v=72.8

Step 1: Delta-of-delta timestamps
  delta:      10,     10,     10
  delta-of-delta: 0,  0,  0     ← all zeros! Encode as 1 bit each.
  If interval is perfectly regular → each timestamp costs 1 bit (not 64!).

Step 2: XOR for values
  72.5 XOR 72.3 = small diff   ← only a few bits differ
  72.3 XOR 72.3 = 0           ← identical! Encode as 1 bit (zero flag).
  72.3 XOR 72.8 = small diff

Result: 16 bytes/point → ~1.37 bytes/point average (12× compression)
This is Facebook's Gorilla algorithm. Prometheus and VictoriaMetrics use variants.
When to Use a TSDB vs PostgreSQL

Use a TSDB when: You're ingesting millions of data points per second (monitoring, IoT), queries are time-range aggregations (avg CPU last hour, 95th percentile latency last week), and you need built-in downsampling and retention policies. Use PostgreSQL (or TimescaleDB) when: You need JOINs between time series and relational data (join metrics with user metadata), your volume is moderate (<100K points/sec), or your team already knows SQL. TimescaleDB is a good middle ground — full SQL with auto-partitioning for time series.

03

Graph Databases

Graph databases model data as nodes (entities) connected by edges (relationships). Each node and edge can have arbitrary properties. The killer feature: traversals — following chains of relationships — are O(1) per hop (follow a pointer), whereas SQL JOINs for the same traversal are O(N) per level (scan/index lookup on the joined table).

Graph query: "Find friends of friends who like jazz" — Cypher (Neo4j)
-- In SQL, this requires 3 recursive JOINs:
SELECT fof.name
FROM users u
JOIN friendships f1 ON u.id = f1.user_id
JOIN friendships f2 ON f1.friend_id = f2.user_id
JOIN users fof ON f2.friend_id = fof.id
JOIN user_interests ui ON fof.id = ui.user_id
WHERE u.name = 'Alice' AND ui.interest = 'jazz';
-- Gets exponentially slower with depth. 6 degrees? Practically impossible.

-- In Cypher (Neo4j), the same query is a natural traversal:
MATCH (alice:User {name: 'Alice'})-[:FRIENDS_WITH*2]-(fof:User)
      -[:LIKES]->(genre:Interest {name: 'jazz'})
RETURN fof.name;
-- *2 means "exactly 2 hops." Change to *1..6 for "up to 6 degrees."
-- Each hop follows a pointer — O(1) per hop, regardless of graph size.

When NOT to Use a Graph Database

Don't use a graph DB if: (1) Your queries are primarily key-value lookups or simple filters (use PostgreSQL or DynamoDB). (2) You need ACID transactions across large batches of data (graph DBs have varying transaction support). (3) Your relationships are simple and shallow (1-2 levels of JOINs in SQL are fine). (4) You need full-text search or aggregation-heavy analytics (use Elasticsearch or ClickHouse). Graph DBs shine specifically when you need deep, variable-length path traversals — if you're not doing that, they add complexity without benefit.

Spatial Databases & Quadtrees

Spatial databases store and query geographic data — points, lines, and polygons on the Earth's surface. The fundamental query is: "find all entities near this location" — which is O(N) without a spatial index and O(log N) with one.

Quadtree — How It Works

QUADTREE SUBDIVISION (capacity = 4 points per node) │ ├── Root: entire map area (e.g., 0,0 to 1000,1000) │ Contains 7 restaurants → exceeds capacity (4) → SPLIT into 4 quadrants │ ├── NW quadrant (0-500, 500-1000): 1 restaurant → leaf (under capacity) ├── NE quadrant (500-1000, 500-1000): 2 restaurants → leaf ├── SW quadrant (0-500, 0-500): 0 restaurants → leaf (empty) └── SE quadrant (500-1000, 0-500): 4 restaurants → at capacity, stays as leaf If a 5th restaurant is added to SE → SE splits into 4 sub-quadrants. QUERY: "Find all restaurants within 2km of (700, 200)" │ ├── Start at root. Which quadrants overlap a circle of radius 2km around (700,200)? ├── NW? No overlap → prune entire subtree (0 points checked) ├── NE? Partial overlap → descend and check 2 points ├── SW? No overlap → prune └── SE? Full overlap → check all 4 points Total: checked 6 out of 7 points. With 10M points, pruning saves 99%+ of work.

The Polyglot Persistence Pattern

Most real systems use multiple storage paradigms together: PostgreSQL for user accounts and orders (relational), Redis for sessions and rate limiting (KV), S3 for user-uploaded images (blob), Prometheus for infrastructure metrics (TSDB), and maybe Neo4j for a recommendation graph. This is called polyglot persistence — choosing the right storage engine for each data shape, rather than forcing everything into one database. Reference: DDIA Ch. 2 discusses this as a natural consequence of the "one-size-fits-all database" being a myth.

Hands-On: Terminal Demos

Demo 1: Spatial Queries with SQLite

Haversine distance query — works anywhere Python is installed (macOS/Linux/WSL)
$ python3 -c "
import sqlite3, math

db = sqlite3.connect(':memory:')
db.execute('CREATE TABLE places (name TEXT, lat REAL, lng REAL)')

# Insert some Berlin landmarks
places = [
    ('Brandenburger Tor', 52.5163, 13.3777),
    ('Fernsehturm', 52.5208, 13.4094),
    ('Reichstag', 52.5186, 13.3761),
    ('East Side Gallery', 52.5052, 13.4396),
    ('Schloss Charlottenburg', 52.5209, 13.2959),
]
db.executemany('INSERT INTO places VALUES (?,?,?)', places)

# Haversine distance function (registered as SQLite UDF)
def haversine(lat1, lon1, lat2, lon2):
    R = 6371  # Earth radius in km
    dlat = math.radians(lat2 - lat1)
    dlon = math.radians(lon2 - lon1)
    a = math.sin(dlat/2)**2 + math.cos(math.radians(lat1)) * math.cos(math.radians(lat2)) * math.sin(dlon/2)**2
    return R * 2 * math.atan2(math.sqrt(a), math.sqrt(1-a))

db.create_function('haversine', 4, haversine)

# Query: places within 3km of Alexanderplatz (52.5219, 13.4132)
my_lat, my_lng = 52.5219, 13.4132
results = db.execute('''
    SELECT name, ROUND(haversine(lat, lng, ?, ?), 2) AS dist_km
    FROM places
    WHERE haversine(lat, lng, ?, ?) < 3
    ORDER BY dist_km
''', [my_lat, my_lng, my_lat, my_lng]).fetchall()

print('Places within 3km of Alexanderplatz:')
for name, dist in results:
    print(f'  {dist:.2f} km — {name}')
"

Mini-Project: Build a Quadtree

quadtree.py — spatial index from scratch (macOS/Linux/WSL, Python 3.7+)
import random, time

class Point:
    def __init__(self, x, y, data=None):
        self.x, self.y, self.data = x, y, data

class Rect:
    def __init__(self, cx, cy, hw, hh):
        self.cx, self.cy, self.hw, self.hh = cx, cy, hw, hh  # center + half-width/height
    def contains(self, p):
        return abs(p.x - self.cx) <= self.hw and abs(p.y - self.cy) <= self.hh
    def intersects(self, r):
        return not (r.cx - r.hw > self.cx + self.hw or r.cx + r.hw < self.cx - self.hw
                 or r.cy - r.hh > self.cy + self.hh or r.cy + r.hh < self.cy - self.hh)

class QuadTree:
    def __init__(self, boundary, capacity=4):
        self.boundary = boundary
        self.capacity = capacity
        self.points = []
        self.divided = False
        self.nw = self.ne = self.sw = self.se = None

    def subdivide(self):
        cx, cy, hw, hh = self.boundary.cx, self.boundary.cy, self.boundary.hw / 2, self.boundary.hh / 2
        self.nw = QuadTree(Rect(cx - hw, cy + hh, hw, hh), self.capacity)
        self.ne = QuadTree(Rect(cx + hw, cy + hh, hw, hh), self.capacity)
        self.sw = QuadTree(Rect(cx - hw, cy - hh, hw, hh), self.capacity)
        self.se = QuadTree(Rect(cx + hw, cy - hh, hw, hh), self.capacity)
        self.divided = True

    def insert(self, point):
        if not self.boundary.contains(point): return False
        if len(self.points) < self.capacity and not self.divided:
            self.points.append(point); return True
        if not self.divided: self.subdivide()
        return self.nw.insert(point) or self.ne.insert(point) or self.sw.insert(point) or self.se.insert(point)

    def query(self, range_rect, found=None):
        if found is None: found = []
        if not self.boundary.intersects(range_rect): return found  # PRUNE
        for p in self.points:
            if range_rect.contains(p): found.append(p)
        if self.divided:
            self.nw.query(range_rect, found); self.ne.query(range_rect, found)
            self.sw.query(range_rect, found); self.se.query(range_rect, found)
        return found

# ── Benchmark: quadtree vs brute-force ──
N = 100_000
points = [Point(random.uniform(0, 1000), random.uniform(0, 1000), f"p{i}") for i in range(N)]
qt = QuadTree(Rect(500, 500, 500, 500), capacity=8)
for p in points: qt.insert(p)

search = Rect(450, 450, 50, 50)  # 100×100 area in center

t0 = time.perf_counter()
qt_found = qt.query(search)
qt_ms = (time.perf_counter() - t0) * 1000

t0 = time.perf_counter()
bf_found = [p for p in points if search.contains(p)]
bf_ms = (time.perf_counter() - t0) * 1000

print(f"Quadtree:    {len(qt_found)} points found in {qt_ms:.1f}ms")
print(f"Brute-force: {len(bf_found)} points found in {bf_ms:.1f}ms")
print(f"Speedup:     {bf_ms/qt_ms:.0f}×")
Expected output (approximate)
Quadtree:    1012 points found in 0.8ms
Brute-force: 1012 points found in 8.5ms
Speedup:     10×

# With 10M points, the speedup would be 100×+
# Quadtree prunes ~90% of points without checking them.

Production Example 1: Gorilla (Facebook TSDB)

Gorilla is Facebook's in-memory TSDB, published at VLDB 2015. [1] It was built because HBase-backed monitoring was too slow for real-time anomaly detection across Facebook's infrastructure.

Key Design Decisions

In-memory first: All recent data (last 26 hours) lives in RAM. Queries never hit disk for recent data — reducing read latency by 73× and improving query throughput by 14× compared to the HBase-backed system.

12× compression: Using delta-of-delta timestamps + XOR'd floating point values, Gorilla compresses each data point from 16 bytes to an average of 1.37 bytes. This is what makes in-memory storage practical at Facebook's scale.

Availability over accuracy: Gorilla prioritizes always being available for writes (monitoring can't have gaps). If a region fails, the other region keeps ingesting. Small amounts of data loss during failure are acceptable — missing a few data points doesn't affect aggregate analysis.

Production Example 2: Uber H3 (Spatial Indexing)

Uber's H3 is a hexagonal hierarchical spatial indexing system. It divides the Earth's surface into hexagonal cells at 16 resolution levels (from continent-size to ~0.9m² cells).

Why Hexagons?

Uniform neighbors: Every hexagon has exactly 6 neighbors, all at equal distances from the center. Squares have 8 neighbors at two different distances (4 edge, 4 corner). This uniformity makes distance calculations and aggregations more accurate.

Consistent area: At a given resolution, all hexagons have approximately the same area (small distortion due to projecting a sphere to hexagons). Geohash rectangles have wildly different areas near the poles vs equator.

Use cases at Uber: Pricing (surge pricing per H3 cell), ETA estimation (average speed per H3 cell at each time of day), demand forecasting (predict ride requests per cell), marketplace matching (find nearby drivers within adjacent H3 cells).

Edge Cases & Gotchas

1. S3 Eventual Consistency (Mostly Resolved)

Historically, S3 provided eventual consistency for overwrite PUTs and DELETEs — you could write a new version and immediately read the old one. Since December 2020, S3 delivers strong read-after-write consistency for all operations at no extra cost. But other blob stores (or older S3 documentation) may still have eventual consistency — always verify.

2. TSDB High Cardinality Explosion

If you create a time series for every unique user_id (millions of distinct series), the TSDB's inverted index explodes in memory. Prometheus is particularly vulnerable — its in-memory index grows proportionally to the number of unique label combinations. Fix: avoid high-cardinality labels (user_id, request_id) in metrics. Use logs (Elasticsearch, Loki) for per-user data instead.

3. Graph DB Performance Cliffs

Graph traversals are O(1) per hop, but if each node has 10,000 edges and you traverse 4 hops, you're touching 10,000⁴ = 10¹⁶ paths. Without filters and limits, graph queries can explode combinatorially. Always bound traversal depth and filter early.

4. Quadtree Degeneration

If all points cluster in one small area (e.g., all restaurants in Manhattan), the quadtree subdivides that area into thousands of tiny cells while the rest of the world is one big empty node. The tree becomes very deep in one corner and trivially shallow everywhere else. Fix: use capacity limits per node and accept some imprecision, or use an R-tree which groups nearby objects regardless of quadrant boundaries.

5. Storing Images in PostgreSQL

BYTEA columns in PostgreSQL technically work for images, but each row becomes megabytes. This means: table scans read the entire image for every row checked, WAL files grow enormously (every image write is logged), VACUUM must process multi-MB dead rows, and the buffer pool fills with image data instead of useful index pages. Always use blob storage (S3) for files and store the URL in PostgreSQL.

What If? — Interactive Exploration

Tradeoff Cards

PostgreSQL + PostGIS vs dedicated graph DB

PostGIS: Full SQL, ACID transactions, JOINs with relational data, excellent for spatial + relational queries ("restaurants in this area with avg rating > 4.5"). Struggles with deep graph traversals (friend-of-friend chains > 3 hops).
Neo4j: Blazing fast traversals (O(1) per hop), Cypher query language, but no SQL, limited ACID, separate system to manage.
Decision: If your primary access is proximity search + relational filters → PostGIS. If it's deep relationship traversal → Neo4j.

Prometheus vs TimescaleDB for metrics

Prometheus: Pull-based, native K8s integration, excellent alerting (Alertmanager), PromQL, but limited to ~2 weeks of local storage, no JOINs.
TimescaleDB: Full SQL, JOINs with relational data, unlimited retention, but push-based (needs adapter), no native K8s service discovery.
Common pattern: Prometheus for real-time alerting (short retention) → remote-write to VictoriaMetrics or TimescaleDB for long-term analysis.

Geohash vs H3 vs S2 for spatial indexing

Geohash: Simple string prefix, works on any B-tree index. Rectangles vary in size by latitude. Easy to implement.
H3: Uniform hexagons, consistent neighbors, better for aggregation. Requires the H3 library.
S2: Google's choice. Arbitrary precision, excellent covering queries. Complex but handles all edge cases (poles, anti-meridian).

S3 storage tiers for cost optimization

Standard: Frequent access, ~$0.023/GB/month. Use for active data.
Infrequent Access: ~$0.0125/GB/month, but retrieval costs more. Use for backups accessed monthly.
Glacier Deep Archive: ~$0.00099/GB/month (100× cheaper!), but retrieval takes 12+ hours. Use for compliance archives.
Lifecycle policies automate transitions: Standard → IA after 30 days → Glacier after 90 days.

Interview Questions & Answers

Q1: When would you use a time series database vs PostgreSQL for metrics?

TSDB when: You're ingesting >100K points/sec, queries are purely time-range aggregations (avg, p99 over windows), and you need built-in downsampling/retention policies. TSDBs compress 10-12× better than PostgreSQL for timestamped data because they exploit temporal locality (delta-of-delta timestamps, XOR'd values). PostgreSQL when: Volume is moderate, you need JOINs (correlate metrics with user events), or your team doesn't want to operate another database. TimescaleDB (PostgreSQL extension) is the bridge — TSDB compression + partitioning with full SQL. For K8s monitoring specifically, Prometheus is the standard because of its pull model, service discovery, and Alertmanager integration.

Q2: How would you design the storage layer for a photo-sharing app?

Photos → S3 (blob storage, 11-nines durability, CDN-friendly). Metadata → PostgreSQL (user_id, caption, tags, location, upload_time, s3_url). Thumbnails → S3 + CDN (generate on upload via serverless function, cache at edge). Feed → Redis (pre-computed feed as a sorted set of photo IDs). Search by location → PostGIS (store lat/lng in the photos table, spatial index for "photos near me"). Search by tags → Elasticsearch (inverted index on tags for full-text search). This is polyglot persistence: 5 storage systems, each optimal for its data shape. Never store the actual image bytes in PostgreSQL — the S3 URL in the photos table is the bridge.

Q3: Explain how a quadtree helps with "find nearby" queries.

Without a spatial index, "find all restaurants within 2km" requires computing the Haversine distance to every restaurant in the database — O(N). A quadtree recursively partitions 2D space into quadrants. For a proximity query, you only descend into quadrants that overlap with the search radius and prune entire subtrees that don't overlap. With 10M restaurants, a quadtree typically checks only a few hundred points instead of 10M — a 10,000× speedup. The lookup complexity is O(log₄ N + M) where M is the number of results. In practice, PostGIS uses R-trees (better for disk-based storage), Redis uses sorted sets with geohashes, and Uber uses H3 hexagonal cells.

Q4: When is a graph database the right choice, and when is SQL good enough?

Graph DB when: your core queries involve variable-depth traversals (friend-of-friend at depth 2-6, shortest path, pattern matching on relationship chains). Each hop in a graph DB is O(1) — follow a pointer. In SQL, each hop is a JOIN — O(N log N) with indexes. At depth 4+, SQL becomes prohibitively expensive. SQL is good enough when: relationships are shallow (1-2 JOINs), the data model is primarily tabular with occasional FK lookups, or you need ACID transactions across related records (graph DB transaction support varies). The turning point: if you're writing recursive CTEs or self-JOINs 3+ levels deep, it's time to evaluate a graph DB.

Interviewer Follow-ups

"How do you efficiently find businesses within a radius? You have 200M businesses."

Step 1: Geohash the user's location. Convert (lat, lng) to a geohash string (e.g., "u33dc" at 5 characters ≈ 4.9km × 4.9km cell). Step 2: Find neighboring geohash cells. The search radius may span multiple cells — compute the 8 neighboring geohash prefixes. Step 3: Range query. In the spatial index (which maps geohash prefixes to business IDs), do a prefix scan for all 9 geohash prefixes. This returns candidate businesses. Step 4: Post-filter. Compute exact Haversine distance for each candidate and filter out those beyond the radius. The key insight (DDIA Ch. 3): geohash turns a 2D spatial query into a 1D range query on a B-tree — no special spatial index type needed. This is how DynamoDB handles spatial queries (geohash as the sort key).

"Businesses rarely change location. How does this affect your design?"

The read-to-write ratio is extremely high — maybe 100K reads/sec vs 100 writes/sec (new businesses, closures, updates). This means: (1) The spatial index can be heavily cached — geohash → business_ids mappings change rarely, so cache them in Redis with a long TTL (hours). (2) The index can be pre-computed and replicated — build a static geohash index periodically (every hour) and replicate it to all API servers as an in-memory lookup table. No need for a live spatial database on the read path. (3) Write path is separate — when a business updates its location, update the PostgreSQL source of truth and asynchronously rebuild/invalidate the affected geohash cells in the index. This is the classic CQRS (Command Query Responsibility Segregation) pattern — reads and writes go through different paths optimized for different access patterns. Reference: Alex Xu, System Design Interview Vol. 2, Chapter 1 ("Proximity Service").

"What about geohash boundary problems? Two restaurants 10 meters apart could be in different geohash cells."

This is the classic geohash edge case. A query at the edge of a geohash cell might miss businesses just across the boundary. The fix: always query the target cell AND its 8 neighboring cells (9 cells total). This guarantees coverage regardless of where the user is within a cell. The post-filter step (compute exact Haversine distance) removes false positives from neighboring cells. The cost: you query ~3× more data than strictly needed (9 cells vs ~3 that actually overlap the radius), but geohash lookups are fast enough that this is fine. H3 hexagons handle this more elegantly — each hexagon has exactly 6 uniform neighbors, and the ring-based expansion (h3.k_ring(cell, 1)) gives consistent coverage.

What the Interviewer Is Testing

(1) Spatial indexing knowledge — can you explain geohash, quadtree, or R-tree and why O(N) distance calculation doesn't scale? (2) Read/write asymmetry — businesses rarely move, so the index can be heavily cached or pre-computed. (3) Boundary handling — the geohash edge case shows attention to detail. (4) Polyglot storage — spatial index + relational DB for details + cache for hot queries. (5) Scale reasoning — 200M businesses with geohash prefix indexing is tractable; 100K QPS with caching is achievable.

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Blob storage (S3/GCS): for large binaries. 11-nines durability, ~$0.023/GB/month. Never store images in PostgreSQL.
2. TSDBs exploit temporal structure: delta-of-delta timestamps + XOR values = 10-12× compression. Gorilla: 16 bytes → 1.37 bytes/point.
3. Prometheus for K8s monitoring (pull model, PromQL, Alertmanager). InfluxDB/TimescaleDB for general time series.
4. Graph DBs: O(1) per traversal hop vs SQL's O(N log N) per JOIN. Use for deep relationship chains (4+ hops). Don't use for simple lookups.
5. Quadtree: recursive 2D space partitioning. O(log₄ N) lookup. Dense areas subdivide deeply, sparse areas stay large. Great for "find nearby."
6. Geohash: encode lat/lng into a string. Nearby points share a prefix. Works on any B-tree index. Boundary problem: always query 9 cells.
7. PostGIS (R-tree) for spatial + relational queries. Redis GEO for simple proximity. H3/S2 for aggregation and analytics.
8. Polyglot persistence: use multiple storage paradigms together. PostgreSQL + Redis + S3 + Prometheus is a common stack.
9. TSDB pitfall: high-cardinality labels (user_id in metrics) explode the inverted index. Use logs for per-entity data.
10. In interviews: "Design a proximity service" tests geohash knowledge, read/write asymmetry, and caching of spatial results.

Key Intuition

The right storage paradigm is determined by the shape of your data and your dominant access pattern. Time-ordered streams → TSDB. Connected entities → graph. Geographic points → spatial index. Large blobs → object storage. Structured rows with relationships → relational. The mistake is forcing the wrong shape into the wrong storage: images in PostgreSQL, graph traversals in SQL, time series without compression. Each paradigm exists because someone hit the limits of a general-purpose database for a specific data shape.

References & Further Reading

[1] Pelkonen et al. — "Gorilla: A Fast, Scalable, In-Memory Time Series Database" (VLDB 2015)

Tuomas Pelkonen, Scott Franklin, Justin Teller, et al. Facebook. PVLDB Vol. 8, No. 12, August 2015.
https://www.vldb.org/pvldb/vol8/p1816-teller.pdf

• Delta-of-delta timestamp compression: if data arrives at a regular interval (e.g., every 10s), the delta between timestamps is constant (10) and the delta-of-delta is 0 — encoded as a single bit. ~96% of timestamps in Facebook's workload compressed to 1 bit each.

• XOR value compression: consecutive values are XOR'd. ~51% of all values were identical to the previous point (XOR = 0, stored as 1 bit). ~30% had small differences (encoded in ~26.6 bits on average). Overall: 16 bytes/point → 1.37 bytes/point average (11.7× compression).

• In-memory storage with disk backup: Gorilla acts as a write-through cache for the most recent 26 hours. Data is also written to HBase for long-term storage, but queries for recent data (the 85% case) never touch disk — reducing query latency by 73× and throughput by 14× vs HBase.

• Availability-first design: data is written to two Gorilla instances in different regions. If one region fails, the other continues ingesting. Small data loss during failover is acceptable because monitoring data is best-effort — a few missing points don't affect trend analysis.

• Data is organized as a TimeSeries Map (TSmap): a hash map from time series key (string) to a compressed stream of (timestamp, value) pairs. The TSmap supports both O(1) lookup by key and efficient full scans for dashboards that aggregate across all series.

[2] Kleppmann — "Designing Data-Intensive Applications" (2017), Ch. 2 & 3

Martin Kleppmann. O'Reilly Media, 2017.
https://dataintensive.net/

• Chapter 2 presents the document, relational, and graph models as three ways to represent relationships. The graph model is introduced specifically for data with many-to-many relationships and variable-depth traversals, where the relational model's recursive JOINs become unwieldy.

• Chapter 3 contrasts B-tree (update-in-place, read-optimized) vs LSM-tree (append-only, write-optimized) storage engines. This distinction is critical for understanding why TSDBs use LSM-trees: time series are append-only by nature, and LSM-trees' sequential writes match perfectly. B-trees are better for random reads (relational workloads).

• Discusses column-oriented storage (relevant to ClickHouse and analytics workloads): instead of storing each row contiguously, store each column contiguously. Aggregation queries (SUM, AVG over one column across millions of rows) read only the needed column from disk, achieving much higher throughput than row-oriented storage.

• Section 3.3 covers multi-dimensional indexes: R-trees, space-filling curves (Z-order/Hilbert), and the concept of mapping multi-dimensional data to a single dimension for B-tree indexing — the theoretical foundation for geohash and S2 cells.

[3] Uber Engineering / H3 — Hexagonal Hierarchical Spatial Index

Uber Engineering and the H3 project.
https://h3geo.org/docs/

• H3 models the earth as a hierarchy of hexagonal cells plus a small number of pentagons. Each increase in resolution subdivides a parent cell into finer children, which is why one cell ID can participate in both point lookup and multi-resolution rollups.

• The system's value is in the index representation: a location is encoded as a compact cell ID, then neighborhood queries become set operations such as gridDisk rather than raw great-circle scans across every point.

• The hierarchy makes aggregation practical. You can count rides, deliveries, or events at resolution 9 for street-level heatmaps, then collapse the same data to coarser cells for city or regional views without rebuilding the index from scratch.

• The docs are also explicit about the awkward edges: pentagon cells, distortion near icosahedron boundaries, and the fact that "nearby in geometry" is only approximately "nearby in index space." Those details matter when spatial systems cross cell boundaries or try to rank candidates by exact distance.
