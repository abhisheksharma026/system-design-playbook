Key Terms

Formal definitions from the source material. Memorize these — they're the vocabulary of every system design conversation.

IP

Stands for Internet Protocol. This network protocol outlines how almost all machine-to-machine communications should happen in the world. Other protocols like TCP, UDP and HTTP are built on top of IP.

TCP

Network protocol built on top of the Internet Protocol (IP). Allows for ordered, reliable data delivery between machines over the public internet by creating a connection.

TCP is usually implemented in the kernel, which exposes sockets to applications that they can use to stream data through an open connection.

HTTP

The HyperText Transfer Protocol is a very common network protocol implemented on top of TCP. Clients make HTTP requests, and servers respond with a response.

Requests typically have the following schema:

HTTP Request Schema
host:    string    (example: algoexpert.io)
port:    integer   (example: 80 or 443)
method:  string    (example: GET, PUT, POST, DELETE, OPTIONS or PATCH)
headers: pair list (example: "Content-Type" => "application/json")
body:    opaque sequence of bytes

Responses typically have the following schema:

HTTP Response Schema
status code: integer   (example: 200, 401)
headers:     pair list (example: "Content-Length" => 1238)
body:        opaque sequence of bytes

IP Packet

Sometimes more broadly referred to as just a (network) packet, an IP packet is effectively the smallest unit used to describe data being sent over IP, aside from bytes. An IP packet consists of:

An IP header, which contains the source and destination IP addresses as well as other information related to the network.

A payload, which is just the data being sent over the network.

01

What Is a Network Protocol?

A network protocol is an agreed-upon set of rules that define how two machines communicate. That's it. It specifies: (1) the format of messages (what the bytes mean), (2) the order of messages (who speaks first, how replies work), and (3) the actions taken on send/receive/timeout (retransmit? close connection? ignore?).

Without protocols, machines are just blasting random bytes at each other. Protocols give structure to chaos.

Core Intuition

Think of human languages. English is a "protocol" — it defines vocabulary, grammar, and turn-taking rules. If I speak English and you speak Mandarin, we can't communicate. Protocols are the same: both sides must agree on the same rules. HTTP is one language. TCP is another. They work at different levels, like how phonetics (TCP) is different from semantics (HTTP).

Why Multiple Protocols?

Because different layers of the network solve different problems. IP solves routing — "how do I get a packet from machine A to machine B across the planet?" TCP solves reliability — "how do I make sure all the data arrives, in order, without corruption?" HTTP solves application semantics — "what does 'give me the homepage' look like as a structured message?" Each protocol layers on top of the one below it, adding more capabilities. This layering follows the end-to-end principle [4] — keep lower layers simple, push application-specific correctness to the endpoints.

02

The Protocol Stack — Big Picture

Protocols are layered. Each layer only talks to the layer directly above and below it. Data flows down the stack on the sender, across the wire, and up the stack on the receiver.

TCP/IP 4-layer model with encapsulation — each layer wraps the layer above
Encapsulation — The Key Concept

When you send an HTTP request, the Application layer creates HTTP data. TCP wraps it with a TCP header (ports, sequence numbers). IP wraps that with an IP header (source/destination IP). Ethernet wraps that with a frame header (MAC addresses). On the wire, it's all just bytes. The receiving machine unwraps each layer in reverse, peeling off headers until the app gets pure HTTP data. This is called encapsulation/decapsulation.

03

IP — The Internet Protocol

IP is the foundation of the entire internet. Every single piece of data — whether it's a Google search, a Netflix stream, or a WhatsApp message — travels inside IP packets. IP's job is simple but critical: get a packet from source IP address to destination IP address. [1]

What IP Does

Addressing: Every device on the internet gets an IP address. IPv4 uses 32-bit addresses (4 billion possible, mostly exhausted). IPv6 uses 128-bit addresses (practically unlimited).

Routing: Routers read the destination IP in the header and make a "next hop" decision — forward the packet one step closer to its destination. This happens at every router along the path, like a relay race.

Fragmentation: If a packet is too large for a network segment's MTU (Maximum Transmission Unit, typically 1500 bytes for Ethernet), IP can split it into smaller fragments that are reassembled at the destination.

What IP Does NOT Do

IP is a best-effort, connectionless protocol. This means:

No guaranteed delivery — packets can be dropped (router buffer full, link failure). IP doesn't know and doesn't care.

No ordering — packets can arrive out of order. Packet 3 might arrive before packet 1 if it took a different route.

No duplication detection — the same packet can be delivered multiple times.

No error correction — if a bit flips in transit, the payload is corrupted. (IP has a header checksum, but not a payload checksum.)

Why This Matters

IP alone is unreliable. That's by design — it keeps the network layer simple and fast. If you need reliability (and most apps do), you add TCP on top. If you need speed more than reliability (gaming, video calls), you use UDP on top. IP is the "postal system" — it routes letters, but doesn't guarantee delivery. TCP is "registered mail." UDP is "throw it in the general direction."

IP Packets — Anatomy in Detail

An IP packet is the fundamental unit of data on the internet. Every piece of information that crosses the network is wrapped inside one (or more) IP packets. Let's look at exactly what's inside.

IPv4 Packet Structure

An IP packet has two parts: the header (20–60 bytes of metadata) and the payload (the actual data, up to ~65,535 bytes total packet size, but typically ~1,500 bytes due to Ethernet MTU).

Structured IPv4 header layout: each row is 32 bits wide, with fields grouped by bit range before the payload begins.

Key Fields Explained

FieldSizeWhat It Does
Version4 bitsIPv4 = 4, IPv6 = 6. Tells the router which format to expect.
IHL4 bitsInternet Header Length, in 32-bit words. Minimum = 5 (20 bytes). Max = 15 (60 bytes with options).
Total Length16 bitsEntire packet size (header + payload) in bytes. Max = 65,535. Typically ~1,500 (Ethernet MTU).
TTL8 bitsTime To Live. Decremented by 1 at each router. When it hits 0, the packet is dropped. Prevents infinite loops. Default: 64 or 128.
Protocol8 bitsWhat's inside the payload. 6 = TCP, 17 = UDP, 1 = ICMP (ping).
Source IP32 bitsThe sender's IP address.
Destination IP32 bitsThe receiver's IP address. Routers use this to make forwarding decisions.
Flags / Fragment Offset16 bitsControls fragmentation. DF = "Don't Fragment" (drop instead). MF = "More Fragments" (reassemble at dest).
Header Checksum16 bitsError-detection for the header only (not the payload). Recalculated at every router (because TTL changes).
Critical Detail

IP's header checksum only covers the header, not the payload. If a bit flips in your actual data during transit, IP won't detect it. That's why TCP has its own checksum covering the payload. If you use UDP and care about data integrity, your application must verify it (or rely on link-layer CRC).

05

TCP — Transmission Control Protocol

TCP takes unreliable IP and builds reliable, ordered, error-checked delivery on top. It's the workhorse of the internet — HTTP, HTTPS, SSH, FTP, SMTP, database connections all run over TCP. [2]

What TCP Provides

Connection-oriented: Before exchanging data, client and server perform a 3-way handshake (SYN → SYN-ACK → ACK) to establish a connection. Both sides allocate resources (buffers, sequence numbers).

Reliable delivery: Every byte is acknowledged. If the sender doesn't get an ACK within a timeout, it retransmits. No data silently disappears.

Ordered delivery: Each byte has a sequence number. Even if packets arrive out of order, TCP reassembles them correctly before passing to the application.

Flow control: The receiver advertises a "window size" — how much data it can buffer. The sender won't overwhelm a slow receiver.

Congestion control: TCP dynamically adjusts its sending rate based on network conditions (packet loss, latency). Algorithms: Slow Start, Congestion Avoidance, Fast Retransmit, Fast Recovery. Modern variants: CUBIC (Linux default), BBR (Google).

The 3-Way Handshake

Three-step TCP setup: synchronize sequence numbers in both directions, then start sending application data.

The 4-Way Teardown

Four-step TCP close: one side closes its send stream first, the other acknowledges, then closes its side, then receives the final ACK.
Why 3 Handshakes, Not 2?

Two-way would establish one direction only. The third packet (ACK) is critical: it confirms to the server that the client received the server's sequence number. Without it, the server wouldn't know if its SYN-ACK was received or lost. Both sides need to confirm they can both send and receive — that requires 3 messages minimum.

06

TCP Headers — Every Field Explained

The TCP header sits inside the IP packet's payload. It's 20 bytes minimum (up to 60 with options). Every field has a purpose — here's the complete anatomy.

Structured TCP header layout: ports, sequence tracking, control flags, flow-control window, and options are grouped by their on-wire bit layout.
FieldSizePurpose
Source Port16 bitsThe sender's port. Ephemeral ports (49152–65535) for clients. Known ports for servers.
Dest Port16 bitsThe receiver's port. Identifies which process gets the data. 80=HTTP, 443=HTTPS.
Sequence Number32 bitsThe byte offset in the stream. If seq=1000 and payload=500 bytes, the next segment starts at seq=1500. Enables ordering.
ACK Number32 bitsThe next byte the sender expects to receive. "I've got everything up to byte X, send me X next." Enables reliability.
Flags9 bitsSYN=start connection, ACK=acknowledge, FIN=close, RST=abort, PSH=push to app immediately.
Window Size16 bitsHow many bytes the receiver can accept right now. Flow control — prevents sender from overwhelming receiver.
Checksum16 bitsCovers the TCP header AND payload. Catches bit-flip corruption that IP's checksum misses.
OptionsvariableMSS (max segment size), window scaling, timestamps, SACK (selective ACK). Negotiated in handshake.
How Sequence Numbers Enable Reliability

Imagine sending a 10KB file. TCP breaks it into segments: seq=0 (1460 bytes), seq=1460 (1460 bytes), seq=2920 (1460 bytes)... The receiver sends ACK=1460, then ACK=2920, etc. If the sender doesn't get ACK=2920, it knows that segment was lost and retransmits it. If segments arrive out of order (seq=2920 before seq=1460), the receiver buffers them and waits, then delivers everything in order to the application.

07

UDP — User Datagram Protocol

UDP is TCP's simpler, faster sibling. It sits at the same Transport layer but provides almost nothing beyond IP: no connection setup, no reliability, no ordering, no flow control. Just ports + checksum + your data.

UDP Header (8 bytes total — that's it)

Structured UDP header layout: just four 16-bit fields, then the application payload. No sequence numbers, no ACKs, no window management.

TCP vs UDP — Head-to-Head

PropertyTCPUDP
ConnectionConnection-oriented (handshake)Connectionless (fire and forget)
ReliabilityGuaranteed delivery via ACKs + retransmissionNo guarantees. Packets may be lost.
OrderingData delivered in order (sequence numbers)No ordering. Packets arrive in any order.
Flow ControlWindow-based. Receiver controls send rate.None. Sender can overwhelm receiver.
Congestion ControlBuilt-in (Slow Start, CUBIC, BBR)None. Can flood the network.
Header Size20–60 bytes8 bytes
LatencyHigher (handshake, ACKs, retransmits)Lower (send immediately)
Use CasesHTTP, SSH, FTP, SMTP, databasesDNS, video streaming, VoIP, gaming, DHCP
When to Use UDP

UDP makes sense when: (1) Speed matters more than completeness — in a video call, a dropped frame is better than a delayed one. Retransmitting old video is useless. (2) Messages are small and self-contained — DNS queries fit in a single packet. (3) You'll build your own reliability — game engines and QUIC implement custom reliability on top of UDP. (4) Broadcast/multicast — UDP supports sending to multiple recipients. TCP doesn't.

HTTP — The Application Protocol

HTTP (HyperText Transfer Protocol) sits at the top of the stack — Layer 7 — and defines how clients request resources and servers respond. It's built on top of TCP (HTTP/1.1, HTTP/2) or QUIC/UDP (HTTP/3).

HTTP Request — Complete Anatomy

Host: api.example.com              ← Required in HTTP/1.1
Content-Type: application/json     ← Format of the body
Content-Length: 47                 ← Size of body in bytes
Authorization: Bearer eyJhbG...    ← Auth token
Accept: application/json           ← What formats I accept back
User-Agent: curl/8.1.2             ← Who's calling

                                    ← Empty line separates headers from body

HTTP/3 and QUIC — The Big Shift

HTTP/3 runs on QUIC, a transport protocol built on UDP. Why? TCP's head-of-line blocking: if one packet is lost, every stream sharing that TCP connection can stall behind it. QUIC gives each stream independent loss recovery. It also integrates TLS 1.3 directly, reducing connection setup to 1 RTT (or 0-RTT for resumed connections). Large operators such as Meta have published substantial QUIC adoption, while Cloudflare's published traffic data shows HTTP/3 growing quickly but HTTP/2 still carrying a larger share across much of the web. [3] [6] [7]

Server Push Reality Check

Server Push exists in the protocol history, but it never became a mainstream browser optimization. Chrome removed HTTP/2 Server Push support in Chrome 106, so treat it as historical context, not a default performance technique. [8]

Memory Map — Encapsulation in RAM

Here's what a single HTTP request looks like as it's wrapped layer by layer into a complete network packet ready to go on the wire.

Encapsulation in RAM: each lower layer wraps the bytes from the layer above with its own header before transmission.
The Total Overhead

For a tiny 16-byte JSON body ({"name":"Alice"}), the total on the wire is: Ethernet 14B + IP 20B + TCP 20B + TLS ~40B + HTTP headers ~200B + 16B body + Ethernet CRC 4B ≈ 314 bytes. Your 16 bytes of actual data is only ~5% of the packet. This overhead is why protocols like HTTP/2 compress headers and why batching small requests matters.

Mini-Project: Build a Protocol Inspector

Theory and terminal demos are great — but building something cements it. Here are two runnable mini-projects. Copy, run, and experiment.

Project 1: Python TCP Echo Server + Header Logger

Build a raw TCP server from scratch using Python's socket module — no frameworks. This shows you exactly what happens at the socket layer: binding, listening, accepting connections, reading bytes.

tcp_inspector.py — save this file and run it
import socket
import threading

def handle_client(conn, addr):
    """Handle one client connection — log everything we receive."""
    print(f"\n{'='*60}")
    print(f"[TCP] Connection from {addr[0]}:{addr[1]}")
    print(f"[TCP] Server socket: {conn.getsockname()}")
    print(f"{'='*60}")

    data = conn.recv(4096)  # Read up to 4KB from the TCP stream
    print(f"\n[RAW BYTES] ({len(data)} bytes):")
    print(data)

    # Try to decode as text (HTTP requests are text)
    print(f"\n[DECODED]:")
    print(data.decode('utf-8', errors='replace'))

    # Parse the HTTP request line
    lines = data.decode().split('\r\n')
    if lines:
        print(f"\n[PARSED]")
        print(f"  Request Line: {lines[0]}")
        for line in lines[1:]:
            if ':' in line:
                key, val = line.split(':', 1)
                print(f"  Header: {key.strip()} = {val.strip()}")

    # Send back a proper HTTP response
    body = '{"status":"ok","message":"Protocol inspector received your request!"}'
    response = (
        f"HTTP/1.1 200 OK\r\n"
        f"Content-Type: application/json\r\n"
        f"Content-Length: {len(body)}\r\n"
        f"X-Server: tcp-inspector/1.0\r\n"
        f"\r\n"
        f"{body}"
    )
    conn.sendall(response.encode())
    conn.close()
    print(f"\n[TCP] Connection closed.")

# Create a TCP socket (AF_INET = IPv4, SOCK_STREAM = TCP)
server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind(('127.0.0.1', 7777))
server.listen(5)
print("[TCP Inspector] Listening on 127.0.0.1:7777")
print("[TCP Inspector] Try: curl -v http://localhost:7777/anything")
print("[TCP Inspector] Or open http://localhost:7777 in your browser\n")

while True:
    conn, addr = server.accept()
    threading.Thread(target=handle_client, args=(conn, addr)).start()
Run it, then hit it with curl
# Terminal 1: Start the inspector
$ python3 tcp_inspector.py

# Terminal 2: Send different types of requests and watch the server log

# Simple GET
$ curl http://localhost:7777/users

# POST with JSON body
$ curl -X POST http://localhost:7777/api/data \
    -H "Content-Type: application/json" \
    -d '{"sensor":"temp","value":23.5}'

# PUT with custom headers
$ curl -X PUT http://localhost:7777/config \
    -H "Authorization: Bearer my-secret-token" \
    -H "X-Request-Id: abc-123" \
    -d '{"theme":"dark"}'

# Open in browser — you'll see the browser sends a LOT of headers
# (User-Agent, Accept-Language, Accept-Encoding, Connection, etc.)
What You'll Learn

The server logs the raw bytes received over TCP before any framework parses them. You'll see the literal HTTP request text: the request line (GET /users HTTP/1.1), each header as a key-value pair, and the body. This is what Nginx, Apache, and Express see before they turn it into a nice req object. The socket.AF_INET + SOCK_STREAM call maps directly to "IPv4 + TCP" from our protocol stack.

Project 2: UDP vs TCP Reliability Test

Send 100 messages over TCP and UDP side by side. Watch TCP deliver all 100 reliably. Watch UDP potentially drop some (especially under load).

udp_vs_tcp.py — reliability comparison
import socket, threading, time

received_tcp = []
received_udp = []

def tcp_server():
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    s.bind(('127.0.0.1', 9010)); s.listen(1)
    conn, _ = s.accept()
    while True:
        data = conn.recv(1024)
        if not data: break
        received_tcp.append(data.decode())
    conn.close(); s.close()

def udp_server():
    s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    s.bind(('127.0.0.1', 9011))
    s.settimeout(2.0)
    try:
        while True:
            data, _ = s.recvfrom(1024)
            received_udp.append(data.decode())
    except socket.timeout: pass
    s.close()

# Start servers
threading.Thread(target=tcp_server, daemon=True).start()
threading.Thread(target=udp_server, daemon=True).start()
time.sleep(0.3)

# TCP client: send 100 numbered messages
tcp_c = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
tcp_c.connect(('127.0.0.1', 9010))
for i in range(100):
    tcp_c.sendall(f"msg-{i}|".encode())
tcp_c.close()

# UDP client: send 100 numbered messages
udp_c = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
for i in range(100):
    udp_c.sendto(f"msg-{i}".encode(), ('127.0.0.1', 9011))
udp_c.close()

time.sleep(3)

# Results
tcp_msgs = '|'.join(received_tcp).split('|')
tcp_msgs = [m for m in tcp_msgs if m]
print(f"\n{'='*50}")
print(f"TCP: Received {len(tcp_msgs)}/100 messages  {'✓ ALL' if len(tcp_msgs)==100 else '✗ LOST'}")
print(f"UDP: Received {len(received_udp)}/100 messages {'✓ ALL' if len(received_udp)==100 else '(some may be lost)'}")
print(f"{'='*50}\n")

Production Example 2: Discord Voice Chat

Discord is a perfect example of TCP and UDP working together in the same application.

TCP for control plane (setup, text, signaling — must be reliable). UDP for data plane (audio, video, game state — must be fast). This TCP+UDP split is standard in real-time applications: Zoom, Google Meet, Twitch, online gaming, and VoIP all follow the same pattern.

Edge Cases & Gotchas

1. TCP Head-of-Line Blocking

In HTTP/2 over TCP, all streams share one TCP connection. If one packet is lost, TCP stalls all streams until the retransmit succeeds — even streams whose data arrived fine. This is why HTTP/3 switched to QUIC (UDP): each stream has independent loss recovery.

2. TCP TIME_WAIT Exhaustion

After closing a TCP connection, the socket enters TIME_WAIT state for 2×MSL (~60 seconds). High-traffic servers making many short-lived connections can exhaust available ports. Fix: connection pooling, SO_REUSEADDR, or longer-lived connections (keep-alive, HTTP/2 multiplexing).

3. MTU / Fragmentation Problems

If a packet exceeds the path MTU (typically 1500 bytes) and the DF (Don't Fragment) flag is set, routers will drop the packet and send back an ICMP "Fragmentation Needed" message. If ICMP is blocked by firewalls (common!), the sender never learns the packet was dropped — a "black hole." Fix: Path MTU Discovery, or clamp MSS in TCP to stay under 1500.

4. UDP Amplification Attacks (DDoS)

UDP has no handshake, so the source IP can be spoofed. Attacker sends tiny DNS queries with the victim's IP as source. DNS servers send back large responses to the victim. A 60-byte query can produce a 3,000-byte response — 50× amplification. This is how some of the largest DDoS attacks work.

5. Nagle's Algorithm + Delayed ACK Deadlock

Nagle's algorithm buffers small TCP writes to combine them. Delayed ACK waits ~200ms before ACKing in hopes of piggybacking. Together, they can cause a 200ms latency on small messages. Fix: TCP_NODELAY socket option (disables Nagle). Critical for low-latency applications like games and trading systems.

6. TLS Version Mismatch

Client supports only TLS 1.3, server only TLS 1.2 → handshake fails. Or server has an expired/untrusted certificate → client rejects connection. These are application-visible failures that look like "connection timeout" or "SSL error" but are actually protocol negotiation failures.

Production War Story

Cloudflare documented a production incident where Path MTU Discovery broke under ECMP routing and a slice of traffic started failing. It is a good reminder that network-layer issues such as MTU, fragmentation, and ICMP handling often show up as mysterious application failures. [9]

Actual System Design Interview

A Prompt You Could Actually Get

Design the network protocol choices for a global collaboration product. Document sync, presence updates, and large file upload live in the same product, but they should not all ride the network the same way. A good answer separates packet forwarding, transport behavior, encrypted session setup, and application semantics. [1] [2] [3] [5]

How I'd Structure the Answer

I would break the system into traffic classes. Control-plane API calls and file upload metadata can stay on HTTP semantics over a reliable transport, because request ordering and retries matter there. Latency-sensitive interactive updates need a stronger discussion: if I need browser compatibility and encrypted multiplexed streams, HTTP/3 over QUIC is a practical default because loss on one stream does not stall all streams the way HTTP/2 over TCP can under packet loss. I would also pin down the application routing key: edits for one document or one room often need to land on the same collaboration shard, so I would say whether I route by document_id, room_id, or user_id. Transport choice does not remove that ownership problem. I would also mention that packet size and path behavior matter in the real world: Path MTU issues, NAT idle timers, and retransmission behavior are often what turn a “correct” protocol choice into a user-visible outage. [2] [3] [5] [9]

Where the Interviewer Pushes

The classic push-back is: what breaks on lossy mobile networks? That is where I would explain head-of-line blocking on TCP, why QUIC keeps separate stream-level loss recovery, and why blindly sending larger packets can fail if a path cannot carry them. Another good push is: what does “HTTP/3” buy you if your application semantics are still wrong? I would answer that transport can reduce handshake cost and isolate stream loss, but it cannot fix bad retry policy, bad timeout policy, or unsafe application methods. If document-shard routing changes during failover, I would also call out stale route metadata: one client can still stream edits to the old owner while another has already switched, which becomes an ordering bug rather than a transport bug. Whiteboard answer: separate transport choice from shard ownership, then explain where lossy networks and stale routing create different classes of failure.
Q1: Explain the difference between TCP and UDP. When would you use each?

TCP is connection-oriented, reliable, ordered, with flow/congestion control. 20+ byte header. Uses handshake. Best for: HTTP/HTTPS (web), SSH, database connections, file transfers — anything where every byte must arrive correctly.

UDP is connectionless, unreliable, unordered. 8-byte header. Fire and forget. Best for: DNS lookups, video/voice streaming, online gaming, IoT telemetry — anything where speed trumps completeness or the application handles its own reliability.

The deeper answer: UDP isn't "worse" TCP. It's a different tradeoff. Modern protocols like QUIC build custom reliability on top of UDP that's better than TCP for their use case (independent stream recovery, 0-RTT connection setup). Choosing UDP gives you control over the reliability/latency tradeoff that TCP's one-size-fits-all approach doesn't.

Q2: What is an IP packet? What are the most important fields in the header?

An IP packet is the fundamental unit of data on the internet. It has a header (20–60 bytes) and a payload. The header contains routing information; the payload contains the data (typically a TCP segment or UDP datagram).

Most important fields: Source IP and Destination IP (the addressing that makes routing possible), TTL (prevents infinite loops — decremented at each hop, dropped at 0), Protocol (tells the receiver what's in the payload: 6=TCP, 17=UDP), and Total Length (packet size). The Flags/Fragment Offset fields handle fragmentation when packets exceed the link MTU.

Q3: Walk me through the TCP 3-way handshake. Why 3 steps and not 2?

SYN: Client sends SYN with its initial sequence number (ISN). SYN-ACK: Server responds with its own ISN and acknowledges the client's ISN+1. ACK: Client acknowledges the server's ISN+1.

Why not 2? A 2-way handshake only confirms one direction. The third step (ACK) confirms to the server that the client received the server's sequence number. Without it: if the SYN-ACK was lost, the server thinks a connection exists but the client doesn't. The server allocates resources for a ghost connection. The 3-way handshake ensures both sides confirm both send and receive paths work, and both sequence numbers are known. It also prevents old duplicate SYN packets from creating false connections (the client can reject stale SYN-ACKs because the sequence numbers won't match).

Q4: What is HTTP/3 and why does it use UDP instead of TCP?

HTTP/3 uses QUIC, a transport protocol built on UDP, instead of TCP. The primary reason: head-of-line blocking. In HTTP/2 over TCP, all multiplexed streams share one TCP connection. If one packet is lost, TCP's ordered delivery stalls all streams until the retransmit completes. QUIC gives each stream independent loss recovery — a loss in stream A doesn't affect stream B.

Additional benefits: 0-RTT connection setup (for resumed connections, QUIC can send data in the very first packet — TCP+TLS needs 2-3 round trips), connection migration (QUIC connections are identified by a Connection ID, not IP+port, so they survive network changes like switching from WiFi to cellular), and built-in encryption (TLS 1.3 is mandatory, unlike TCP where it's optional).

Q5: How does TCP guarantee reliable delivery?

Four mechanisms: (1) Sequence numbers: Every byte in the stream gets a sequence number. The receiver knows exactly which bytes it has and which are missing. (2) Acknowledgments: The receiver sends ACKs telling the sender "I've received everything up to byte X." (3) Retransmission: If the sender doesn't receive an ACK within an RTO (Retransmission Timeout), it resends the data. Modern TCP also uses Fast Retransmit — if it receives 3 duplicate ACKs for the same byte, it retransmits immediately without waiting for the timeout. (4) Checksums: The TCP checksum covers the entire segment (header + payload), detecting corruption.

Together, these ensure: no data loss, no corruption, and correct ordering. The cost: latency (retransmits take time), bandwidth (ACKs are overhead), and complexity (congestion control, window management).

Q6: What are HTTP status codes? Name the most important ones.

Status codes are 3-digit integers in the response indicating the result. 2xx = Success: 200 OK, 201 Created, 204 No Content. 3xx = Redirect: 301 Moved Permanently (SEO-safe), 302 Found (temporary), 304 Not Modified (use cached version). 4xx = Client Error: 400 Bad Request, 401 Unauthorized (auth required), 403 Forbidden (auth present but insufficient), 404 Not Found, 409 Conflict, 429 Too Many Requests (rate limited). 5xx = Server Error: 500 Internal Server Error, 502 Bad Gateway (upstream failed), 503 Service Unavailable (server overloaded/maintenance), 504 Gateway Timeout.

In interviews, knowing 429 and the difference between 401/403 shows depth. 429 is critical for rate limiting discussions. 502 vs 504 matters when designing load balancer behavior.

Q7: What happens at the network level when a TCP connection is idle for a long time?

By default, nothing — TCP has no built-in idle detection. The connection stays "open" even if both sides are silent. This creates problems: (1) If one side crashes without sending FIN, the other side doesn't know. It'll wait forever (or until it tries to send data and gets no ACK). (2) NAT gateways and firewalls have timeout tables. An idle connection's NAT entry may expire (typically 5-30 minutes), silently breaking the connection.

Solutions: TCP Keep-Alive — the OS sends tiny probe packets every N seconds (Linux default: 7200s = 2 hours, configurable). If no response after several probes, the connection is declared dead. Application-level heartbeats — WebSocket ping/pong, gRPC keepalive, HTTP/2 PING frames. These are more responsive than TCP keep-alive. Shorter timeouts — configure tcp_keepalive_time, tcp_keepalive_intvl, tcp_keepalive_probes in the kernel.

Key Takeaways Cheat Sheet

The 12 Things You Must Know

1. A protocol is an agreed-upon set of rules for communication format, order, and error handling.
2. IP is best-effort, connectionless. Routes packets from source to dest. No reliability guarantees.
3. An IP packet = header (20B: src IP, dst IP, TTL, protocol) + payload (TCP/UDP segment).
4. TCP adds reliability on top of IP: connections, sequence numbers, ACKs, retransmission, flow control.
5. TCP 3-way handshake: SYN → SYN-ACK → ACK. Both sides confirm bidirectional communication.
6. UDP is connectionless and unreliable. 8-byte header. Used when speed > completeness.
7. HTTP is an application-layer protocol on top of TCP. Request (method, path, headers, body) → Response (status code, headers, body).
8. HTTP methods: GET (read), POST (create), PUT (replace), PATCH (partial update), DELETE (remove).
9. Encapsulation: HTTP data → TCP segment → IP packet → Ethernet frame → wire. Reversed on receive.
10. HTTP/2 adds multiplexing over TCP. HTTP/3 uses QUIC (over UDP) to eliminate head-of-line blocking.
11. Real systems mix protocols: TCP for reliable control plane, UDP for fast data plane (voice, video, gaming).
12. Key debugging tools: tcpdump (packet capture), traceroute (IP routing), curl -v (HTTP), nc (raw TCP/UDP).

Key Intuition

Every protocol layer adds a header and a guarantee — but also a failure mode. TCP guarantees delivery but introduces head-of-line blocking. IP routes globally but drops packets without notice. Understanding which layer is responsible for which failure — and which layer to blame when things go wrong — is the core skill of network debugging.

References & Further Reading

Primary RFCs and papers cited throughout this document. Each is broken down by the concrete protocol behavior, deployment tradeoff, or failure mode to extract from it.

[1] RFC 791 — Internet Protocol (1981)

Postel, J. "Internet Protocol — DARPA Internet Program Protocol Specification." IETF, September 1981.
https://datatracker.ietf.org/doc/html/rfc791

• Defines the IPv4 datagram header, including version, IHL, total length, identification, flags, fragment offset, TTL, protocol number, header checksum, and source/destination addresses.

• Makes IP a best-effort, connectionless service: delivery, ordering, duplicate suppression, and retransmission are explicitly outside the protocol contract.

• Specifies fragmentation semantics through the MF/DF flags and fragment offsets, which is why MTU mismatches and PMTU failures show up as transport problems later.

• The protocol field is the demultiplexing hook that hands payloads upward to TCP, UDP, ICMP, or another upper-layer protocol.

[2] RFC 793 / RFC 9293 — Transmission Control Protocol

Postel, J. "Transmission Control Protocol." IETF, September 1981. Obsoleted by Eddy, W. RFC 9293, August 2022.
https://datatracker.ietf.org/doc/html/rfc9293 (modern consolidated version)

• Specifies TCP’s stateful connection machine, three-way handshake, teardown paths, byte-sequence space, acknowledgments, retransmission timers, and advertised-window flow control.

• Sequence numbers are over bytes, not packets, which is why retransmission and out-of-order repair can work on arbitrary segment boundaries.

• RFC 9293 consolidates the modern TCP baseline and references later extensions such as SACK, timestamps, and updated robustness/security behavior.

• The protocol is the canonical example of endpoint-managed reliability layered above an unreliable network substrate.

[3] RFC 9000 — QUIC: A UDP-Based Multiplexed and Secure Transport (2021)

Iyengar, J. (Fastly) & Thomson, M. (Mozilla). IETF Standards Track, May 2021.
https://datatracker.ietf.org/doc/html/rfc9000

• Defines QUIC as a UDP-based transport with multiple independent streams, per-stream flow control, and packet-level loss recovery decoupled from stream delivery.

• Integrates TLS 1.3 handshake state into the transport so encryption keys, transport parameters, and connection establishment are negotiated as one protocol.

• Uses opaque Connection IDs to support migration across address changes and to avoid binding connection identity to the current 4-tuple.

• Encrypts most transport metadata, which both reduces ossification and changes how debugging, observability, and middlebox interaction work compared with TCP.

[4] Saltzer, Reed & Clark — "End-to-End Arguments in System Design" (1984)

J.H. Saltzer, D.P. Reed, D.D. Clark. ACM Transactions on Computer Systems, Vol. 2, No. 4, Nov 1984, pp. 277–288.
https://web.mit.edu/saltzer/www/publications/endtoend/endtoend.pdf

• The paper’s precise claim is that lower layers may optimize reliability or integrity, but complete correctness checks still belong at the communicating endpoints.

• It justifies the split between simple packet forwarding in the network core and semantic correctness logic in transport/application protocols.

• This is why IP can stay best-effort while TCP, TLS, and application retries still sit above it.

• The argument generalizes to distributed systems: shared infrastructure can assist, but end-to-end semantics still must be enforced where full application context exists.

[5] RFC 2616 / RFC 9110 — HTTP Semantics

Fielding, R., et al. "HTTP Semantics." IETF RFC 9110, June 2022. (Replaces HTTP/1.1 spec RFC 2616.)
https://datatracker.ietf.org/doc/html/rfc9110

• Separates HTTP semantics from wire format so the same request model can be carried over HTTP/1.1, HTTP/2, or HTTP/3.

• Defines method properties such as safety and idempotency, which directly govern retry behavior, caching eligibility, and intermediary handling.

• Specifies status-code classes, representation metadata, conditional requests, validators, and content negotiation rules.

• This is the document that defines what an HTTP exchange means even when the framing changes underneath it.

https://blog.cloudflare.com/cloudflare-view-http3-usage/

Engineering at Meta. October 2020.
