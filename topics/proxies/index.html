Key Terms

Forward Proxy

An intermediary that sits in front of clients and makes requests to servers on their behalf. The server sees the proxy's IP address, not the client's — which is why forward proxies are used for anonymity, content filtering, and bypassing geo-restrictions. When people say "proxy" without qualification, they usually mean forward proxy. Real-world examples: corporate web filters, VPN services, Tor exit nodes, and browser-configured HTTP proxies.

Reverse Proxy

An intermediary that sits in front of servers and accepts requests from clients on the servers' behalf. The client thinks it's talking directly to the server, but it's actually talking to the proxy, which then forwards the request to the real backend. Used for load balancing, SSL termination, caching, compression, rate limiting, and hiding the backend architecture. Real-world examples: Nginx, HAProxy, Cloudflare, AWS ALB, Envoy.

Nginx

Pronounced "engine X" — an open-source, high-performance web server created by Igor Sysoev in 2004 to solve the C10K problem (handling 10,000+ concurrent connections). Uses an event-driven, non-blocking I/O architecture instead of the thread-per-connection model of Apache. Today it's one of the most widely deployed reverse proxies on the internet and a standard tool for TLS termination, HTTP routing, and static content delivery. [1]

01

What Is a Proxy?

A proxy is any server that sits between two communicating parties and relays traffic between them, potentially modifying or inspecting it along the way. The word comes from "act on behalf of" — a proxy acts on behalf of either the client (forward proxy) or the server (reverse proxy).

Core Intuition

Think of a personal assistant. A forward proxy is like your assistant — you tell them "book me a restaurant," and the restaurant only sees the assistant's name, not yours. Your identity is hidden. A reverse proxy is like the restaurant's maître d' — guests walk in the front door, the maître d' decides which table (which backend server) to seat them at, handles the greeting (SSL), and the guest never sees the kitchen (backend infrastructure).

The key distinction: who configured the proxy and who benefits from it? Forward proxies are configured by the client (or client's network admin) to benefit the client. Reverse proxies are configured by the server operator to benefit the server infrastructure.

02

Forward Proxy — Acting for the Client

A forward proxy intercepts outbound requests from clients and forwards them to the destination server. The server only sees the proxy's IP — the client's real IP is hidden.

What Forward Proxies Do

IP masking / anonymity: The destination server sees the proxy's IP, not the user's. VPNs are essentially encrypted forward proxies. Tor chains multiple forward proxies for stronger anonymity.

Content filtering: Corporate networks use forward proxies (like Squid or Zscaler) to block access to prohibited sites. The proxy inspects the URL/hostname and rejects requests to blacklisted domains.

Caching: A forward proxy can cache responses (like a shared browser cache for the whole network). If 100 employees request the same npm package, the proxy fetches it once and serves the cached copy 99 times. This was huge in the days of slow internet — ISPs ran transparent forward proxies for exactly this purpose.

Geo-restriction bypass: A forward proxy in another country lets you access content restricted to that region. The server thinks the request is coming from the proxy's location.

Forward proxy flow
Client (you)  ──→  Forward Proxy  ──→  Server (google.com)
192.168.1.42       45.33.32.156         142.250.80.46

Server sees: request from 45.33.32.156 (proxy's IP)
Server does NOT see: 192.168.1.42 (your real IP)
Client configured the proxy. Client benefits from anonymity.
03

Reverse Proxy — Acting for the Server

A reverse proxy sits in front of backend servers and accepts requests from clients on the servers' behalf. The client doesn't know (or care) that there's a proxy — it thinks it's talking directly to the service.

What Reverse Proxies Do

Load balancing: Distributes incoming requests across multiple backend servers (round-robin, least connections, IP hash, etc.). In most production systems, this is the job people mean when they say "reverse proxy."

SSL/TLS termination: The proxy handles the expensive TLS handshake and decryption. Backend servers receive plain HTTP — simpler, cheaper, and the CPU-intensive crypto is done once at the proxy instead of on every backend.

Caching: Cache responses for frequently-requested content at the proxy layer. Static assets, API responses with Cache-Control headers, and even full HTML pages can be served without hitting the backend.

Compression: Gzip/Brotli compression of responses at the proxy, reducing bandwidth without backend CPU cost.

Rate limiting & DDoS protection: The proxy can throttle abusive clients, block suspicious IPs, and absorb volumetric attacks before they reach the backend.

Request routing: Route requests to different backends based on URL path, hostname, headers, or cookies. /api/* goes to the API service, /static/* goes to the CDN, /admin/* goes to the admin service.

Reverse proxy flow
Client  ──→  Reverse Proxy (Nginx)  ──→  Backend Server 1
                                       ──→  Backend Server 2
                                       ──→  Backend Server 3

Client sees: api.example.com (the proxy's address)
Client does NOT see: 10.0.1.5, 10.0.1.6, 10.0.1.7 (backend IPs)
Server operator configured the proxy. Server benefits from LB, SSL, caching.

The proxy adds headers so the backend knows the real client IP:
X-Forwarded-For: 203.0.113.42      ← original client IP
X-Forwarded-Proto: https           ← original protocol (HTTPS)
X-Real-IP: 203.0.113.42            ← Nginx convention

Interview Tip

When an interviewer says "proxy" without specifying, ask: "forward proxy or reverse proxy?" This shows you understand the distinction. In system design interviews, reverse proxies come up far more often than forward proxies because they are a standard building block in web architectures. Mentioning a reverse proxy is table stakes; explaining the reasons — TLS termination, load balancing, routing, caching, and security controls — is the stronger answer.

05

What Reverse Proxies Actually Do — In Detail

1. SSL/TLS Termination

TLS handshakes are CPU-intensive. The reverse proxy handles that once, then forwards plain HTTP or internal TLS to backends depending on the trust model. This simplifies certificate management, offloads crypto from app servers, and gives you one place to manage ciphers, redirects, and certificate rotation.

2. Load Balancing

The proxy distributes incoming requests across a pool of backend servers. Common algorithms: round-robin, least connections, IP hash, and weighted balancing. We'll cover the load-balancing side in depth in the load balancers topic.

3. Request Routing (L7 routing)

Because the reverse proxy operates at Layer 7 (HTTP), it can inspect URLs, headers, and cookies to make routing decisions:

Path-based routing example (Nginx-style pseudoconfig)
# Route by URL path
/api/*        ──→  api-service:8080         (backend API servers)
/static/*     ──→  s3-bucket or CDN         (static file storage)
/admin/*      ──→  admin-service:8081       (restricted admin panel)
/ws/*         ──→  websocket-service:9090   (WebSocket upgrade)

# Route by hostname (virtual hosting)
api.example.com   ──→  api-service
app.example.com   ──→  frontend-service
docs.example.com  ──→  docs-service

4. Header Manipulation

The proxy can add, remove, or modify HTTP headers. It commonly adds X-Forwarded-For, X-Forwarded-Proto, and X-Request-Id. It can also strip or normalize headers that would otherwise leak backend details.

5. Response Caching

The proxy can cache backend responses and serve them directly for subsequent identical requests. This is how Nginx cache, Varnish, and CDN edge caches reduce backend load and improve tail latency for cacheable traffic.

Nginx — The Reverse Proxy You'll Use Most

Nginx, created in 2004 by Igor Sysoev, became popular because its event-driven architecture handles large numbers of concurrent connections efficiently. A single worker process can manage many connections using non-blocking I/O such as epoll on Linux or kqueue on BSD/macOS. [1]

Nginx Architecture

NGINX PROCESS MODEL │ ├── Master Process (root) │ Reads config, binds to ports (80, 443), manages workers │ Does NOT handle any traffic itself │ ├── Worker Process 1 (non-root, one per CPU core) │ Event loop: epoll/kqueue for non-blocking I/O │ Handles thousands of connections simultaneously │ Single-threaded — no locks, no context switches │ ├── Worker Process 2 │ (same as above — each worker is independent) │ ├── Cache Loader (startup only) │ Loads cached metadata from disk into shared memory │ └── Cache Manager (periodic) Prunes expired cache entries to stay within size limits KEY INSIGHT: WHY NGINX IS FAST Apache: 1 thread per connection → 10K connections = 10K threads = GBs of RAM Nginx: 1 event loop per core → 10K connections = a few workers = far less overhead This is the C10K solution — event-driven I/O multiplexing.

Mini-Project: Build a Reverse Proxy from Scratch

First, a simple example to feel what proxying means — then a more advanced one with header injection and load balancing.

Simple: The Smallest Possible Reverse Proxy

tiny_proxy.py — just forwarding (macOS/Linux/WSL, Python 3.7+)
import http.server, urllib.request

BACKEND = "http://httpbin.org"

class ProxyHandler(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        url = f"{BACKEND}{self.path}"
        try:
            resp = urllib.request.urlopen(url)
            self.send_response(resp.status)
            self.send_header("X-Proxied-By", "tiny-proxy/1.0")
            self.end_headers()
            self.wfile.write(resp.read())
        except Exception as e:
            self.send_response(502)
            self.end_headers()
            self.wfile.write(f"Proxy error: {e}".encode())

    def log_message(self, *a): pass

print("[Tiny Proxy] localhost:6060 → proxying to httpbin.org")
http.server.HTTPServer(("", 6060), ProxyHandler).serve_forever()
Test it
# Terminal 1
$ python3 tiny_proxy.py

# Terminal 2
$ curl -s http://localhost:6060/ip
{ "origin": "YOUR_SERVER_IP" }

$ curl -I http://localhost:6060/get
X-Proxied-By: tiny-proxy/1.0
Demo Reality Check

In this localhost demo, the proxy process and your client run on the same machine, so the upstream service will see the same public egress IP. What the demo proves is not anonymity but that the proxy is in the path and can observe, forward, and modify the request.

Advanced: Reverse Proxy with Load Balancing + Header Injection

lb_proxy.py — round-robin reverse proxy (macOS/Linux/WSL, Python 3.7+)
import http.server, urllib.request, itertools, time

BACKENDS = ["http://127.0.0.1:8081", "http://127.0.0.1:8082", "http://127.0.0.1:8083"]
pool = itertools.cycle(BACKENDS)

class LBProxy(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        backend = next(pool)
        url = f"{backend}{self.path}"
        t0 = time.perf_counter()
        try:
            req = urllib.request.Request(url, headers={
                "X-Forwarded-For": self.client_address[0],
                "X-Request-Id": f"req-{int(t0*1000)}",
            })
            resp = urllib.request.urlopen(req, timeout=5)
            elapsed = (time.perf_counter() - t0) * 1000
            self.send_response(resp.status)
            self.send_header("X-Backend", backend)
            self.send_header("X-Latency-Ms", f"{elapsed:.1f}")
            self.end_headers()
            self.wfile.write(resp.read())
        except:
            self.send_response(502); self.end_headers()
            self.wfile.write(f"Backend {backend} unreachable".encode())

    def log_message(self, fmt, *args):
        print(f"  [PROXY] {self.client_address[0]} → {fmt % args}")

print(f"[LB Proxy] localhost:7070 → round-robin across {BACKENDS}")
http.server.HTTPServer(("", 7070), LBProxy).serve_forever()
What You'll Learn

Hit localhost:7070 a few times and the X-Backend header cycles through :8081, :8082, :8083. Kill one backend and you will get a 502 Bad Gateway for that slot. That is the production pattern in miniature: route traffic, add forwarding context, and surface upstream failure cleanly.

Production Example 1: Cloudflare

Cloudflare runs a globally distributed reverse-proxy and CDN network. When you put a domain behind Cloudflare, requests first reach Cloudflare's edge, which can terminate TLS, apply security controls, cache responses, and then forward to the origin when needed. [3]

What Cloudflare Does as a Reverse Proxy

SSL termination: Cloudflare handles TLS for your domain while your origin can use a different internal TLS or origin setup.

DDoS mitigation: Volumetric attacks can be absorbed at the edge instead of directly reaching the origin.

CDN caching: Static assets are cached close to users, reducing origin load and latency.

WAF: Cloudflare can inspect HTTP requests for attack signatures before forwarding them.

Workers (edge compute): Application logic can run at the edge before the request reaches your origin.

Key Insight

Cloudflare is a good example of how a reverse proxy stops being "just a relay." Once it owns TLS termination, request routing, and cache lookup, it is already in a position to run logic at the edge.

Production Example 2: Envoy at Lyft

Envoy is a high-performance proxy originally built at Lyft for service-to-service communication. Unlike Nginx, which is commonly used at the edge, Envoy is also widely used as a sidecar in service-mesh architectures. [2]

The Sidecar Pattern

Envoy as a sidecar proxy
Traditional: Service A ──(HTTP)──→ Service B
Problem: Every service must implement retries, circuit breakers,
         TLS, metrics, tracing, load balancing... in every language.

With Envoy sidecar:
Service A ──→ Envoy (sidecar A) ══→ Envoy (sidecar B) ──→ Service B
                    ↑ handles:              ↑ handles:
                    · retries               · auth/authz
                    · circuit breaking      · rate limiting
                    · mTLS encryption       · mTLS decryption
                    · metrics               · metrics
                    · tracing               · access logging

Service code is simple — just HTTP calls to localhost.
All the networking complexity lives in Envoy.
Why Envoy Matters

Envoy centralizes retries, mTLS, circuit breaking, tracing, and policy enforcement into the proxy layer. That lets service teams keep application logic simpler while the platform team standardizes networking behavior across languages and runtimes.

Edge Cases & Gotchas

1. X-Forwarded-For Spoofing

Clients can set X-Forwarded-For themselves. If your backend trusts this header blindly for rate limiting or geo decisions, attackers can bypass controls. The fix is to trust only headers set by your known proxy chain.

2. Proxy Adding Latency

Every proxy hop adds work. For most web systems the benefits dominate, but if you stack CDN → API gateway → service mesh sidecar → backend, the overhead becomes real and should be measured.

3. WebSocket and Long-Lived Connections

Many proxy defaults assume short request-response cycles. WebSockets and streaming protocols need explicit upgrade support and longer idle timeouts.

4. The Single Point of Failure Problem

If your reverse proxy goes down, all traffic can fail because it sits in the critical path. Production deployments use multiple proxy nodes or managed load balancers in front of the proxy tier.

5. Large Request/Response Bodies

Many proxies buffer request or response bodies by default. Large uploads or downloads can therefore stress the proxy if buffering, limits, or streaming modes are not configured correctly.

Production War Story Pattern

Timeouts, buffering defaults, and header trust are common proxy-related failure modes because the application often looks broken even when the real problem is sitting in the proxy layer in front of it.

What If? — Interactive Exploration

Tradeoff Cards

Adding a reverse proxy layer

Improves: TLS offload, load distribution, caching, security, centralized observability.
Costs: More latency, more configuration, more moving parts.
When worth it: Almost always for public-facing production services.

Envoy sidecar vs shared proxy

Improves: Per-service observability and language-agnostic networking policy.
Costs: More resource overhead and operational complexity.
When worth it: Larger microservice estates.

Edge compute at the proxy

Improves: User latency, origin offload, request shaping near the client.
Costs: Runtime constraints, debugging complexity, platform coupling.
When worth it: Auth, routing, personalization, bot detection, lightweight logic.

Removing a proxy layer

Improves: Lower latency and simpler operations.
Costs: Lose TLS offload, caching, or routing features from that layer.
When worth it: Highly latency-sensitive systems with simpler network needs.

Toy Simulation: Proxy Overhead Measurement

proxy_overhead.py — measure the latency cost of proxying (macOS/Linux/WSL)
import urllib.request, time

DIRECT = "http://httpbin.org/get"
PROXIED = "http://localhost:6060/get"

def measure(url, n=10):
    times = []
    for _ in range(n):
        t0 = time.perf_counter()
        urllib.request.urlopen(url, timeout=5)
        times.append((time.perf_counter() - t0) * 1000)
    avg = sum(times) / len(times)
    p99 = sorted(times)[int(len(times) * 0.99)]
    return avg, p99

print("Measuring proxy overhead...")
d_avg, d_p99 = measure(DIRECT)
print(f"  Direct:  avg={d_avg:.0f}ms  p99={d_p99:.0f}ms")

Interview Questions & Answers

Actual System Design Interview

A Prompt You Could Actually Get

Put a reverse-proxy layer in front of a multi-service API. Explain what lives there, what absolutely should not, and how the backend still learns who the original client was. This is usually where an interview moves from “proxy definition” to practical production concerns like trust boundaries, retries, timeouts, and header propagation. [1] [2] [4]

How I'd Structure the Answer

I would say the reverse proxy is the shared network edge for concerns that should be enforced consistently: TLS termination, host and path routing, request buffering, compression, basic edge caching, auth integration, and observability. Then I would define the trust boundary clearly. If the backend reads X-Forwarded-For or Forwarded, it must trust only the proxy chain that is under our control; otherwise clients can spoof those headers. If the proxy also owns sticky routing, I would name the affinity key explicitly (cookie, user ID, source IP) and explain what happens when the backend pool changes; otherwise I would avoid making correctness depend on affinity and keep state outside the app servers. I would also explain that retries and timeouts belong at the proxy only if the request semantics are safe enough for them. [1] [2] [4]

Where the Interviewer Pushes

The push-back is usually operational: what causes 502 or 504 from the proxy, and how do you prove whether the problem is the proxy or the upstream? That is where I would talk about upstream connect timeout, request timeout, response timeout, circuit-breaking, and per-hop tracing headers. If the proxy fleet rolls out route or sticky-session changes asynchronously, I would also call out split behavior across proxies: one edge may still send a client to the old backend while another sends it to the new one. That is survivable for cache affinity, but dangerous if session state still lives on one app server. In a service-mesh setting, I would also mention that sidecars move cross-cutting traffic policy out of the application, but they do not remove the need to understand which hop owns retries, which hop owns TLS, and which hop is authoritative for client identity. [1] [2] [3] [4]

Production Case Study Lens

Think Cloudflare-style edge or Envoy-style gateway: TLS terminates once, policy is enforced once, and forwarded identity is trusted only from the proxy chain you operate. The interview value is not naming the product. It is showing the edge responsibilities, the trust boundary, and what breaks if sticky routing or config rollout drifts across the proxy fleet. Whiteboard answer: define what the proxy owns, what it forwards, and where sticky routing becomes a correctness risk rather than a convenience.
Q1: What's the difference between a forward proxy and a reverse proxy?

Forward proxy: sits in front of clients, acts on the client's behalf. Hides the client's IP from the server. Configured by the client or network admin. Used for anonymity, filtering, and outbound caching.

Reverse proxy: sits in front of servers, acts on the server's behalf. Hides backend architecture from the client. Configured by the server operator. Used for load balancing, TLS termination, routing, caching, and security.

The core question is: who is the proxy representing?

Q2: Why put a reverse proxy in front of application servers?

To centralize TLS termination, load balancing, request routing, caching, security controls, and request logging. It lets backends stay simpler while the proxy handles common network concerns once for the whole service.

Q3: How does Nginx achieve high concurrency compared to Apache?

Nginx uses an event-driven, non-blocking architecture rather than the classic thread-per-connection model. A small number of workers can therefore manage many open connections using OS event-notification mechanisms.

Q4: What is a service mesh and how do proxy sidecars work?

A service mesh is an infrastructure layer for service-to-service traffic. Proxy sidecars such as Envoy sit beside each service instance and handle concerns like mTLS, retries, tracing, and policy enforcement outside the application code.

Q5: A client complains about 502 Bad Gateway errors. How do you debug?

Start at the proxy: check upstream health, error logs, timeout settings, backend reachability, and whether the backend is returning malformed or incomplete responses. 502 usually means the proxy could not get a valid upstream response.

Q6: What headers does a reverse proxy typically add, and why do they matter?

X-Forwarded-For exposes the original client IP, X-Forwarded-Proto preserves the original scheme, and request IDs help tracing. These headers matter because the backend otherwise only sees the proxy hop in front of it.

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Forward proxy: in front of clients. Reverse proxy: in front of servers.
2. Reverse proxies are standard in production web systems.
3. Nginx became popular because event-driven I/O handles high concurrency efficiently.
4. TLS termination at the proxy simplifies certificate and crypto management.
5. L7 routing lets proxies dispatch requests by path, host, or headers.
6. Forwarding headers matter because backends otherwise lose client context.
7. Envoy sidecars power many service-mesh designs.
8. Every proxy hop adds some latency and operational complexity.
9. 502 and 504 errors often need proxy-level debugging, not just app debugging.
10. The proxy layer itself must be redundant because it sits in the critical path.

Reverse proxy traffic under load: the proxy absorbs client-facing work, removes failed backends from service, and pushes more load onto the remaining healthy servers.
Key Intuition

A reverse proxy is a single control point for traffic — which is both its power and its risk. It gives you one place to terminate TLS, route requests, cache responses, and shed load. But if that layer fails and is not redundant, everything behind it becomes unreachable.

References & Further Reading

Each source below is broken down by the mechanism, control point, or failure boundary you should extract from it.

[1] Sysoev + Alexeev — Nginx Architecture

The Architecture of Open Source Applications, Vol. 2.
https://aosabook.org/en/v2/nginx.html

• Describes the master / worker split: the master parses configuration, binds sockets, and manages lifecycle; workers handle accept/read/write on active connections.

• Explains the event-driven, non-blocking request model in which each worker multiplexes many sockets through epoll/kqueue rather than creating a thread per connection.

• Shows how Nginx models HTTP processing as a chain of phases and filters, which is why features like reverse proxying, compression, buffering, and caching compose cleanly.

• Covers accept mutexes, connection pools, and preallocated request structures that reduce lock contention and allocator overhead under high concurrency.

[2] Envoy Architecture Overview

Envoy Proxy documentation.
https://www.envoyproxy.io/docs/envoy/latest/intro/arch_overview/intro/arch_overview

• Defines Envoy’s core forwarding graph precisely: a listener owns one or more filter chains, each chain selects network or HTTP filters, and the request eventually resolves to an upstream cluster plus a concrete endpoint.

• The important technical split is L4 interception versus L7 request processing: TCP filters can proxy opaque streams, while HTTP connection managers decode requests so retries, routing, header mutation, and per-route policy happen above the socket layer.

• The xDS material is the real control-plane mechanism: CDS/LDS/RDS/EDS/SDS independently distribute clusters, listeners, routes, endpoints, and secrets so config can change incrementally without process restarts.

• Read Envoy as a stateful proxy runtime: upstream connection pools, circuit-breaker thresholds, timeout budgets, outlier detection, and retry policy all execute on the request path and directly change failure behavior.

[3] Cloudflare Documentation

Cloudflare docs and product documentation.
https://developers.cloudflare.com/fundamentals/

• The key architectural detail is the orange-cloud reverse-proxy path: authoritative DNS returns Cloudflare edge IPs, the nearest POP accepts the TCP/TLS connection, and the origin never talks directly to most clients.

• The docs are useful for the edge-to-origin control surface: origin certificates, connection reuse, cacheability rules, and request-header rewriting determine what context the origin actually receives.

• Read them as a proxy pipeline, not a product catalog: TLS termination, WAF evaluation, cache lookup, rate limiting, Workers execution, and origin fetch all happen as ordered stages on the same request path.

• Reverse proxies are trust boundaries. Client IP propagation, header rewriting, and origin trust rules all live here.

[4] RFC 7239 — Forwarded HTTP Extension

IETF RFC 7239.
https://datatracker.ietf.org/doc/html/rfc7239

• Standardizes the exact wire format for preserved client context: each proxy can append a Forwarded element with for=, by=, host=, and proto= parameters.

• The technical point is hop-by-hop provenance: the header is a list, not a singleton, so downstream services can reconstruct the proxy chain instead of trusting the most recent hop blindly.

• The RFC is also useful for parser edge cases that break real systems: IPv6 literals require brackets, identifiers may be obfuscated, and quoting rules matter if intermediaries normalize headers differently.

• This is the standard way for a backend to learn the real client IP and scheme, but it only works if the backend knows exactly which proxy hops it trusts.
