Key Terms

Client

A client is the process that initiates communication in order to obtain data, trigger work, or use a remote capability exposed by another system. In practice, a client might be a browser, mobile app, CLI tool, backend service, or even a database worker calling another service. The important point is that client is a role: the side that starts the interaction and waits for a response.

Server

A server is the process that listens for incoming requests, applies business logic, accesses data or other dependencies, and returns a response. A server is usually designed to be continuously available, to handle many clients over time, and to expose a stable interface such as HTTP, gRPC, or a database protocol. Like the client, server is also a role, not a guarantee about a physical machine.

Client—Server Model

The client-server model is an architectural pattern in which responsibilities are split between request initiators and request handlers. Clients focus on presentation, user interaction, or orchestration; servers centralize computation, data access, coordination, and shared system behavior. This separation makes systems easier to scale, secure, evolve, and operate because the two sides can change somewhat independently as long as they preserve the communication contract.

IP Address

An IP address is the network-level identifier used to route packets to a host or network interface. On the public internet, it is how other machines know where to send traffic; inside private networks, it plays the same routing role within that local environment. IPv4 addresses consist of four numbers separated by dots, like a.b.c.d, where each number is between 0 and 255. Special values include:

127.0.0.1 — Your own local machine. Also referred to as localhost.

192.168.x.y — Your private network. Your machine and all machines on your private wifi network will usually share the 192.168 prefix.

Port

A port identifies a specific network endpoint on a machine so the operating system knows which process should receive incoming traffic. The IP address gets data to the right machine; the port gets it to the right application on that machine. A port is an integer between 0 and 65,535 (216 total), and servers bind to ports so clients know where to connect.

Typically, ports 0–1023 are reserved for well-known system services, while higher ports are commonly used by application processes and ephemeral outbound connections. Certain ports have standard conventions:

PortService
22SSH (Secure Shell)
53DNS lookup
80HTTP
443HTTPS
5432PostgreSQL
6379Redis
3306MySQL

DNS

DNS, or Domain Name System, is the distributed naming system that translates human-readable domain names like example.com into IP addresses that clients can actually connect to. Before a browser can send an HTTP request to a host name, it usually performs DNS resolution through a resolver, which may answer from cache or recursively query other DNS servers. DNS is therefore a prerequisite step in most client-server interactions: name first, then network connection.

01

What Is the Client–Server Model?

The Client-Server model is a distributed computing architecture where the system is split into two distinct roles: a client that makes requests, and a server that processes those requests and sends back responses. That's the entire idea. Every time you open a browser, type a URL, and see a webpage — you just participated in a client-server interaction.

The client doesn't know how the server produces the answer. The server doesn't care why the client wants it. Each side has a well-defined contract (the protocol), and they communicate over a network.

Core Intuition

Think of a restaurant. You (the client) look at the menu and place an order. The kitchen (the server) prepares the food and sends it to your table. You don't walk into the kitchen. The kitchen doesn't eat the food. Clear separation of responsibility.

Why does this matter?

Without the client-server model, every computer would need to have every piece of data and every program locally. There'd be no Google, no Netflix, no APIs, no cloud computing. The model lets us centralize resources (data, computation, storage) on powerful servers while keeping lightweight clients (browsers, mobile apps, IoT devices) that simply request what they need. This separation aligns with the end-to-end principle [2] — keep the network simple, push intelligence to the edges.

Formal Definition

A client is a process (a running program) that initiates a communication session by sending a request. A server is a process that listens for incoming requests, performs some computation or data retrieval, and returns a response. They communicate over a network using an agreed-upon protocol (most commonly HTTP/HTTPS over TCP/IP). [1]

Key Distinction

"Client" and "Server" are roles, not physical machines. A single physical computer can act as both a client and a server simultaneously. Your laptop runs a browser (client) and could also run a local development server (server). In microservice architectures, a service is a server to external clients and a client to internal services it depends on.

How It Actually Works (Step by Step)

Let's trace exactly what happens when you type https://www.example.com/api/users into your browser and press Enter. Every single step, no magic.

Step 1: DNS Resolution

Your browser doesn't know where www.example.com lives. It asks the OS, which checks its local cache, then asks a DNS resolver (usually your ISP's or a public one like 8.8.8.8). Your machine makes a recursive query to that resolver, and the resolver typically performs the upstream lookup by walking root servers → TLD servers (.com) → authoritative nameservers until it gets back an IP like 93.184.216.34.

Step 2: TCP Connection (3-Way Handshake)

Your browser opens a TCP connection to 93.184.216.34:443 (port 443 for HTTPS): [3]

TCP 3-Way Handshake
Client → Server:  SYN           "Hey, I want to talk"
Server → Client:  SYN-ACK       "OK, I'm ready, are you?"
Client → Server:  ACK           "Yes, let's go"

Connection is now ESTABLISHED. Both sides agree on
sequence numbers, window sizes, and MSS (max segment size).

Step 3: TLS Handshake (for HTTPS)

Since we used https://, there's a TLS handshake on top of TCP. The server presents its SSL/TLS certificate. The client verifies it against trusted Certificate Authorities. They negotiate an encryption algorithm and exchange keys. All subsequent data is encrypted.

Step 4: HTTP Request

HTTP Request
GET /api/users HTTP/1.1
Host: www.example.com
Accept: application/json
User-Agent: Mozilla/5.0 ...
Authorization: Bearer eyJhbGciOi...
Connection: keep-alive

Step 5: Server Processing

The server receives the raw bytes, parses them into an HTTP request object, routes to the correct handler (e.g., UsersController.index()), validates the auth token, queries a database, serializes the result to JSON, and builds an HTTP response.

Step 6: HTTP Response

HTTP Response
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 245
Cache-Control: max-age=60

[
  {"id": 1, "name": "Alice"},
  {"id": 2, "name": "Bob"}
]

Step 7: Client Rendering

The browser receives the response, parses the JSON, and renders it. The TCP connection may stay open for subsequent requests (keep-alive) or be closed.

Latency Budget — Where Time Goes

DNS lookup: 1–50ms (cached) or 100–200ms (cold) · TCP handshake: 1 RTT (~10–100ms) · TLS handshake: 1–2 RTT (~20–200ms) · Server processing: 1ms–seconds · Data transfer: depends on size/bandwidth · Total TTFB for a cold transatlantic request: ~200–500ms.

The Network Stack — Where It Lives

The Client-Server model sits at the Application Layer of the network stack. Here's the OSI model mapped to reality:

LayerOSI NameProtocolWhat It Does
7ApplicationHTTP, gRPC, WebSocketWhere client-server logic lives. "GET /users" is an application-layer concept.
6PresentationTLS/SSL, JSON, ProtobufEncryption (TLS), data serialization (JSON → bytes).
5SessionTLS session resumption, RPC sessionsCoordinates longer-lived conversations between endpoints. In practice, this layer is often thin or folded into application/transport behavior.
4TransportTCP, UDPReliable delivery (TCP) or fast unreliable (UDP). Port numbers. Segmentation, retransmissions.
3NetworkIP (v4/v6)Routing packets across networks. IP addresses. Each router picks a "next hop."
2Data LinkEthernet, Wi-FiFrames data for local segment. MAC addresses. Error detection (CRC).

Interview Tip

In practice, most engineers think in the simpler TCP/IP 4-layer model: Application → Transport → Internet → Link. Each layer wraps the bytes from above with its own header, then the receiver peels those layers back off in reverse order.

Hands-On: Terminal Demos

Theory helps, but seeing bytes move through your own terminal helps more. These commands let you watch the client-server model on your local machine. Open two terminals and follow along.

Where command syntax differs, this section calls out both macOS and Linux. If only one command is shown, assume it works on both.

Exact output will vary by machine, network, DNS answers, TLS negotiation, and remote server configuration. Focus on the shape of the interaction, not the literal IPs or header values.

Demo 1: Raw TCP with Netcat (nc)

Netcat is the Swiss Army knife of networking. It lets you create raw TCP connections — no HTTP, no frills, just a client and a server talking over a port.

Terminal 1 — Start a Server (macOS / Linux)
# -l = listen mode (act as a SERVER)
# 8081 = the port to listen on
$ nc -l 8081

# The terminal now hangs — it's WAITING for a connection.
# This is exactly what a real server does: listen on a socket.
Terminal 2 — Connect as a Client (macOS / Linux)
# Connect to localhost (127.0.0.1) on port 8081
$ nc 127.0.0.1 8081

# Connection established! Type anything and press Enter:
Hello from the client!

# → This text appears in Terminal 1 (the server).
# Type in Terminal 1 and it appears in Terminal 2.
# You just built a bidirectional client-server connection.
What Just Happened

Terminal 1 created a TCP socket on 127.0.0.1:8081 and called listen(), so it became the server. Terminal 2 called connect(), so it became the client. The OS handled the TCP handshake under the hood. Every byte you type then moves as a TCP segment inside an IP packet across the loopback interface. Same mechanism as a website. Less ceremony.

Demo 2: DNS Lookup with dig

See how DNS resolution actually works — your machine acting as a client to a DNS server.

DNS Resolution (macOS / Linux)
# Query the DNS server for google.com's IP address
$ dig google.com

# Key parts of the output:

# 300 = TTL (time-to-live) in seconds — cache this for 5 minutes

;; Query time: 12 msec
;; SERVER: 192.168.1.1#53    ← your router's DNS server, port 53

# Your machine (client) sent a UDP packet to 192.168.1.1:53 (server)
# The DNS server responded with the IP. Client-server in action.

Build a fake HTTP server that responds to a browser — proving that servers are just programs that follow a protocol.

# Now open your browser and go to: http://localhost:8080

# 1. nc listened on port 8080 (server)
# 2. Browser connected to 127.0.0.1:8080 (client)

Production Example 1: Netflix

Netflix is a masterclass in client-server architecture at extreme scale.

The Client

Netflix has clients for every platform: iOS, Android, Smart TVs, game consoles, Roku, Fire TV, and web. Each is purpose-built but speaks the same API contract to Netflix's backend.

The Server(s)

Netflix runs thousands of microservices in AWS. But from the client's perspective, it's one server — the Netflix API Gateway (originally Zuul, now custom). The gateway fans out to downstream services internally.

How a Play Request Works

Netflix Play Flow (Simplified)
1. Client → POST /api/playback/start
   Body: { titleId: "80100172", profileId: "abc123" }

2. API Gateway authenticates, routes to Playback Service

3. Playback Service checks:
   - Title available in user's region? (Licensing Service)
   - What quality can device handle? (Device Capabilities DB)
   - Closest CDN node? (Open Connect / CDN Service)

4. Response:
   {
     "manifestUrl": "https://cdn-fra001.netflix.com/...",
     "drmLicense": "...",
     "profiles": ["h264-main-L40", "hevc-main10-L51"]
   }

5. Client fetches video chunks directly from CDN
   (SECOND client-server interaction: client ↔ CDN)
Key Insight

Netflix uses two layers of client-server: (1) Client ↔ API servers for metadata, auth, recommendations. (2) Client ↔ CDN edge servers for video bytes. The API path optimizes for correctness; the CDN path optimizes for bandwidth and latency.

Production Example 2: Stripe Payments

Stripe processes hundreds of billions of dollars per year. The client-server model is at the heart of every transaction.

The Multi-Layer Client-Server Chain

Stripe Payment Flow
LAYER 1: Browser ↔ Your Server
Browser (client) → POST /checkout → Your Server (server)

LAYER 2: Your Server ↔ Stripe API
Your Server (now a client!) → POST api.stripe.com/v1/payment_intents
  Headers: { Authorization: "Bearer sk_live_..." }
  Body:    { amount: 4999, currency: "usd" }
  → Stripe API (server)

LAYER 3: Stripe ↔ Card Networks
Stripe (now a client!) → ISO 8583 auth message
  → Visa/Mastercard (server) → Issuing Bank (server)

RESPONSE CHAIN (reverse):
  Issuing Bank → Visa → Stripe → Your Server → Browser
  "Payment succeeded"
Critical Observation

Your server plays BOTH roles. It's a server when receiving the browser's request. It's a client when calling Stripe's API. This duality is fundamental — in microservices, every service is simultaneously a client and a server.

Why This Matters for System Design

The Stripe example shows a chain of client-server interactions, creating complexity around: error handling (Stripe times out?), idempotency (server crashes after charge but before responding to browser?), and consistency (4 systems must agree on the outcome).

Edge Cases & Gotchas

These break in production and interviewers love to ask about them.

1. The Server Is Down

Client gets connection refused or timeout. Fix: retries with exponential backoff + jitter, circuit breakers, and fallback responses (cached data, degraded experience).

2. The Network Is Unreliable

Packets drop, reorder, duplicate. TCP handles most of this, but application-level partial failures remain: the request arrived and was processed, but the response was lost. Client retries → operation happens twice. Fix: idempotency keys.

3. DNS Caching Gone Wrong

Client caches a DNS entry. Server's IP changes (migration, failover). Client talks to the old IP. Fix: respect TTL values.

4. Head-of-Line Blocking (HTTP/1.1)

Requests on the same connection are serialized. First request slow → everything waits. HTTP/2 fixes this with multiplexing.

5. Stateful vs. Stateless Servers

Stateless servers treat each request independently — dramatically easier to scale. Stateful servers remember things (sessions) but create problems when user's next request hits a different server. Fix: externalize state to Redis/DB, or use sticky sessions.

6. TLS Certificate Expiry

Expired cert → all clients refuse to connect. Has caused major outages. Fix: automated renewal (Let's Encrypt + cert-manager), expiration monitoring.

7. The "Thundering Herd"

Server goes down, 10K clients wait. Comes back → 10K clients retry simultaneously → instant overload. Fix: jittered retries + server-side rate limiting.

Production War Story

In 2017, Amazon S3 experienced a major outage caused by an operational error during routine maintenance that cascaded across thousands of dependent services. Lesson: when a foundational server fails, every client upstream fails. Design for failure.

Actual System Design Interview

A Prompt You Could Actually Get

Design the request path for a mobile checkout API. The interviewer is not asking for "browser sends HTTP request." They want to hear where naming, transport, retries, and server-side state boundaries live, and where the system can still fail on a flaky network. The clean frame is: HTTP for request semantics, TCP for byte delivery, and application logic for correctness. [1] [2] [3] [4]

How I'd Structure the Answer

I would start by naming the resource and the failure domain clearly: the client sends a checkout request to an API endpoint, DNS resolves the service name, TCP establishes a byte stream, TLS protects the channel, and HTTP carries the method, headers, and body. Then I would separate delivery from business correctness. TCP gives me ordered bytes, but it does not tell me whether the payment operation already committed. So if the client retries after a timeout, I need an application-level idempotency key and a server-side dedup record. If someone proposes sticky sessions, I would ask what the affinity key is and what happens when the server pool changes; correctness cannot depend on the next request landing on the same machine. I would also say out loud that the app tier should stay stateless so any healthy server can accept the retry, while durable state lives below it. [2] [3] [4]

Where the Interviewer Pushes

The push-back is usually: what if the server commits the payment but the response never reaches the phone? That is the point where I would say the network only gives me at-least-once retry behavior unless I build something stronger above it. If a later retry lands on a different app server, the dedup check still has to work because the correctness condition belongs with the operation, not with one TCP connection. I would also mention stale routing state explicitly: DNS caches, load balancers, and mobile reconnects can all send the retry somewhere new, so any design that needs one specific server to remember the checkout is brittle by construction. That is the end-to-end argument in practice: keep the network simple, but place the business guarantee at the application boundary that understands what “same checkout” means. [1] [2]

Production Case Study Lens

Think a Stripe-style mobile checkout path: retries are normal, the network is flaky, and the server-side idempotency record is what turns at-least-once delivery into business correctness. The key interview move is to show the request path end to end, then point out exactly why stickiness and one TCP connection are insufficient. [2] [4]

Whiteboard answer: walk the request path, then show where retries land and where the idempotency guarantee actually lives.
Q1: What happens when you type a URL into your browser and press Enter?

The classic question. Full answer covers: (1) URL parsing (scheme, host, path, query). (2) DNS resolution — browser cache → OS cache → resolver → root/TLD/authoritative nameservers. (3) TCP 3-way handshake. (4) TLS handshake (cert validation, key exchange, cipher negotiation). (5) HTTP request sent. (6) Server processing (LB → web server → app server → DB). (7) HTTP response. (8) Browser parses HTML, builds DOM, requests CSS/JS/images (each a new client-server interaction). (9) Rendering: DOM + CSSOM → render tree → layout → paint → composite.

Depth signals level. Juniors: steps 1-7. Seniors add: connection pooling, HTTP/2 multiplexing, CDN behavior, caching layers (memory, disk, service workers), CORS.

Q2: What's the difference between a client and a server? Can a machine be both?

Client = initiates request. Server = listens and responds. These are roles, not machines. A machine can absolutely be both. A Node.js backend receives browser requests (server) while calling PostgreSQL and Redis (client). In microservices, every service plays both roles.

Q3: Why is HTTP "stateless"? How do we add state?

HTTP is stateless: each request is independent. Server doesn't inherently remember previous requests. State is added via: (1) Cookies — server sends Set-Cookie, browser includes it in subsequent requests. (2) Session IDs — cookie holds a session ID, server looks up state in Redis/DB. (3) JWT tokens — self-contained tokens in Authorization header. (4) URL params (fragile, security risk). (5) Hidden form fields.

Tradeoff: stateless = easy to scale (any server handles any request) but harder to personalize. Stateful = richer experience but scaling/consistency challenges. If you add sticky sessions, define the affinity key and remember that pool changes can remap a user to a different server anyway, so correctness still needs a shared state store.

Q4: How would you handle 1 million concurrent connections?

Single server: Event-driven, non-blocking I/O (Node.js, Nginx) over thread-per-connection. One server can handle ~100K+ connections with epoll/kqueue.

Horizontal scaling: Load balancer in front of 10+ servers. Health checks route away from failures.

Connection management: Aggressive idle timeouts, HTTP/2 multiplexing, WebSocket only when push is genuinely needed.

Architecture: Separate stateless API servers (easy to scale) from stateful WebSocket servers. Externalize state. Multi-AZ deployment. CDN offloads static content.

Q5: Server processes request but response is lost. Now what?

Client times out, retries. If non-idempotent ("transfer $100"), money transfers twice. Solutions: (1) Idempotency keys — unique key per operation, server deduplicates. Stripe uses this. (2) At-most-once — don't retry, accept silent failures. (3) Exactly-once semantics via at-least-once + server-side dedup log.

Deep insight: exactly-once delivery is impossible at the network level. You simulate it with at-least-once + idempotent processing.

Q6: Long Polling vs WebSockets vs Server-Sent Events?

Long Polling: Client sends HTTP request, server holds it open until data arrives, responds, client immediately re-requests. Works everywhere but high overhead.

SSE: Single HTTP connection, server pushes text events. Unidirectional (server→client). Auto-reconnect, works with HTTP/2. Text only.

WebSocket: Full-duplex binary channel after HTTP upgrade. Lowest latency, bidirectional. More complex, harder to scale (persistent connections).

When: Chat → WebSocket. Live dashboards → SSE. Legacy compat → Long Polling.

Q7: Why do we need load balancers?

Single server has finite resources. Load balancer distributes requests across server pool. Algorithms: round-robin, least connections, weighted, IP hash (sticky sessions).

L4 vs L7: Layer 4 = TCP level (fast, can't inspect HTTP). Layer 7 = HTTP level (route by URL, headers, cookies — more flexible, slightly slower).

Also provides: health checking, SSL termination, DDoS protection.

12

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Client initiates, server responds. These are roles, not machines.
2. Communication happens over a network using a protocol (usually HTTP over TCP/IP).
3. DNS translates domains to IPs. IP + Port = Socket (endpoint).
4. TCP provides reliable delivery (3-way handshake). UDP is faster but unreliable.
5. HTTPS = HTTP + TLS encryption. TLS adds ~1-2 RTTs on first connection.
6. HTTP is stateless. State is added via cookies, sessions, or tokens.
7. A machine can be both client and server. Microservices live on this principle.
8. Stateless servers scale horizontally. Stateful servers need shared stores or sticky sessions.
9. Networks fail. Design for it: retries with backoff, idempotency keys, circuit breakers.
10. Real systems use multiple client-server layers (browser → your API → Stripe → Visa).

Key Intuition

Every client-server interaction is a chain of handoffs, and the system breaks at its weakest link. Under load, pressure cascades backward: the database slows down → server connections pile up → client sees timeouts. Most real-world outages trace back to an overwhelmed resource at the end of the chain — almost always the database.

References & Further Reading

Primary sources cited throughout this document. Each is broken down by the concrete protocol or architectural mechanism it contributes.

[1] RFC 2616 — Hypertext Transfer Protocol — HTTP/1.1

Fielding, R., et al. "Hypertext Transfer Protocol — HTTP/1.1." IETF, June 1999. (Obsoleted by RFCs 7230–7235.)
https://datatracker.ietf.org/doc/html/rfc2616

• Specifies HTTP/1.1 message syntax: request line, status line, headers, optional message body, and the rules for delimiting payload length via Content-Length or transfer codings.

• Makes persistent connections the default, which changed client-server performance materially by allowing multiple requests over one TCP connection.

• Defines method semantics and safety/idempotency expectations, which is why clients can retry GET differently from POST.

• Formalizes intermediary behavior such as proxies, caching directives, and hop-by-hop headers, which is central to real client-server deployments rather than just origin-server communication.

[2] Saltzer, Reed & Clark — "End-to-End Arguments in System Design" (1984)

J.H. Saltzer, D.P. Reed, D.D. Clark. ACM Transactions on Computer Systems, Vol. 2, No. 4, Nov 1984, pp. 277–288.
https://web.mit.edu/saltzer/www/publications/endtoend/endtoend.pdf

• States the end-to-end argument precisely: a function should live at a lower layer only if it can be completely and correctly implemented there; otherwise lower-layer support is merely an optimization.

• Uses file transfer and duplicate suppression to show that network-level reliability cannot eliminate the need for endpoint-level acknowledgment and integrity validation.

• Explains why client and server endpoints own correctness semantics while the network core remains minimal and general-purpose.

• The paper’s key systems tradeoff is architectural placement of functionality: reliability, encryption, and semantic validation belong at the ends because only the application has full context.

[3] RFC 793 — Transmission Control Protocol (1981)

Postel, J. "Transmission Control Protocol." IETF, September 1981. (Obsoleted by RFC 9293, August 2022.)
https://datatracker.ietf.org/doc/html/rfc793

• Defines TCP as a reliable ordered byte stream over IP, with sequencing performed over bytes rather than packets.

• Specifies the three-way handshake, connection teardown with FIN/RST, retransmission timers, flow control via the advertised window, and checksum-based segment integrity.

• Introduces the socket pair abstraction (source IP, source port, destination IP, destination port) that uniquely identifies a TCP connection.

• The protocol mechanisms in RFC 793 are exactly what make client-server request/response exchanges survivable over lossy, reordered packet networks.

[4] Fielding — "Architectural Styles and the Design of Network-based Software Architectures" (2000)

Roy Thomas Fielding. Doctoral Dissertation, University of California, Irvine, 2000.
https://ics.uci.edu/~fielding/pubs/dissertation/top.htm

• Chapter 3 formally identifies client-server as a constraint that separates user-interface concerns from data storage and processing concerns.

• The dissertation’s REST model adds further constraints on top of client-server: stateless interactions, cacheability, a uniform interface, layered systems, and optional code-on-demand.

• It explains why the uniform resource and method model reduces coupling: clients manipulate representations, not server internals.

• The document is technical evidence that “client-server” is not just topology; it is an architectural separation chosen to improve evolvability, cacheability, and independent deployment.
