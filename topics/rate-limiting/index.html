Rate Limiting

The traffic cop of distributed systems — how to protect services from abuse, ensure fair usage, and gracefully degrade under pressure using algorithms that fit in a few lines of code but scale to billions of requests.

01Key Terms

Rate Limiting
Controlling the number of requests a client can make to a service within a time window. It protects against abuse, ensures fair resource sharing, and prevents cascading failures. Rate limits can be enforced per IP address, per user/API key, per endpoint, per region, or globally. A typical API might allow 100 requests per minute per API key, 1000 per hour, and 10000 per day — these are tiered rate limits. GitHub's API allows 5,000 requests/hour for authenticated users, 60/hour for unauthenticated. Stripe allows 100 read requests/second in live mode.
Throttling
A broader term for slowing down or restricting traffic. Rate limiting is one form of throttling. Other forms include bandwidth throttling (limiting data transfer speed), CPU throttling (Linux cgroups), and connection throttling (limiting concurrent connections). In practice, "rate limiting" and "throttling" are often used interchangeably in API contexts, but throttling can also mean degrading service quality (returning lower-resolution images, skipping non-essential computation) rather than outright rejection.
DoS Attack (Denial of Service)
An attack where a malicious actor floods a system with requests to exhaust its resources (CPU, memory, network bandwidth, database connections), making it unavailable to legitimate users. A single-origin DoS can be mitigated with IP-based rate limiting. A DDoS (Distributed Denial of Service) uses thousands of compromised machines (botnets) to send traffic from many IPs simultaneously, making IP-based rate limiting insufficient — you need behavioral analysis, CAPTCHA challenges, and CDN-level protection (Cloudflare, AWS Shield). The 2016 Dyn DNS DDoS attack used Mirai botnet IoT devices to generate ~1.2 Tbps of traffic.
Token Bucket
The most widely used rate limiting algorithm. A bucket holds tokens; tokens are added at a fixed rate (refill rate) up to a maximum capacity (burst size). Each request consumes one token. If the bucket is empty, the request is rejected. The bucket naturally allows short bursts (up to its capacity) while enforcing an average rate over time. Used by AWS API Gateway, nginx, and most cloud providers. Two parameters: refill rate (sustained throughput) and bucket size (burst tolerance).
Leaky Bucket
Requests enter a queue (the bucket) and are processed at a constant rate (the leak). If the queue is full, new requests are rejected. Unlike token bucket, leaky bucket enforces a strictly constant output rate — no bursts at all. Think of it as a FIFO queue with a fixed drain rate. Used for traffic shaping on network devices (routers, switches) where you want to smooth bursty traffic into a steady stream before forwarding it to a downstream link.
Sliding Window
A rate limiting approach that tracks requests over a continuously moving time window rather than fixed intervals. The sliding window log variant stores timestamps of all recent requests and counts how many fall within the current window. The sliding window counter variant approximates this by weighting the previous fixed window's count with the current window's count based on the overlap. Sliding window avoids the "boundary burst" problem of fixed window counters, where a client can send 2× the limit by timing requests at the boundary of two fixed windows.
Redis
An in-memory key-value store frequently used as the backing store for distributed rate limiters. Its sub-millisecond latency, atomic operations (INCR, EXPIRE, Lua scripting), and built-in TTL on keys make it ideal for tracking request counts per client. A single Redis instance can handle ~100K+ operations/second. Redis Cluster or Redis Sentinel provides high availability. Alternatives include Memcached (simpler but no Lua) and DynamoDB (higher latency but fully managed).

02Why Rate Limit?

1. Prevent abuse & DoS: Without rate limiting, a single malicious client (or a buggy client in a retry loop) can exhaust your server's resources, starving all other users. Rate limiting is the first line of defense.

2. Ensure fairness: In a multi-tenant API, one heavy user shouldn't degrade service for everyone else. Rate limits enforce resource quotas per tenant. Stripe, Twilio, and every public API use this pattern.

3. Protect downstream dependencies: Your API server can handle 10K req/s, but your database can only handle 2K. Rate limiting at the API layer prevents overloading the database. This is the "circuit breaker" philosophy applied proactively.

4. Cost control: Cloud billing is per-request. A runaway client can generate a $50,000 bill overnight. Rate limiting caps exposure. AWS API Gateway lets you set throttle limits per API key specifically for this reason.

5. Revenue model: Tiered rate limits are a monetization strategy. Free tier: 100 req/day. Pro: 10,000 req/day. Enterprise: 1M req/day. The rate limiter enforces the business model in code.

Defense in depth: Don't rate limit at just one layer. 04Algorithms Deep Dive

1. Fixed Window Counter

Divide time into fixed intervals (e.g., every 60 seconds). Count requests per client in each interval. Reject if count exceeds the limit. Reset the counter at the start of each new interval.

// Fixed Window — simplest possible implementation
window_key = f"rate:{user_id}:{current_minute}"
count = redis.incr(window_key)
if count == 1:
    redis.expire(window_key, 60)  # auto-cleanup
if count > LIMIT:
    return 429  # Too Many Requests

The boundary problem: A client sends 100 requests at 12:00:59 and 100 more at 12:01:01. The fixed window counter sees 100 in the 12:00 window and 100 in the 12:01 window — both under the 100/min limit. But within a 2-second real window, the client sent 200 requests. This 2× amplification at boundaries makes fixed windows unsuitable for strict rate enforcement.

2. Sliding Window Log

Store the timestamp of every request in a sorted set. On each new request, remove timestamps older than the window, count remaining entries, and reject if over the limit.

// Sliding Window Log — accurate but memory-heavy
now = time.now()
window_start = now - 60 seconds
redis.zremrangebyscore(key, 0, window_start)    # purge old
redis.zadd(key, now, request_id)                # add current
count = redis.zcard(key)                        # count
if count > LIMIT:
    redis.zrem(key, request_id)                 # undo add
    return 429

Tradeoff: Perfectly accurate — no boundary problem. But memory usage is O(requests per window per user). If you allow 10,000 req/min per user and have 1M users, you're storing up to 10B timestamps. This doesn't scale for high-volume APIs.

3. Sliding Window Counter (Hybrid)

Approximation that gets ~99.9% accuracy with O(1) memory per user. Keep two fixed window counters: previous and current. Estimate the count as:

// Sliding Window Counter — best balance of accuracy and efficiency
prev_count = redis.get(f"rate:{user}:{prev_window}")  or 0
curr_count = redis.get(f"rate:{user}:{curr_window}")  or 0

// Weight previous window by overlap fraction
overlap = (window_size - elapsed_in_current) / window_size
estimated = curr_count + prev_count * overlap

if estimated > LIMIT:
    return 429

Cloudflare uses this approach for their Rate Limiting product. The approximation error is bounded and acceptable for virtually all use cases.

4. Token Bucket

The most popular algorithm for production API rate limiting. Two parameters: refill rate (tokens per second) and bucket capacity (maximum burst size).

// Token Bucket — allows bursts while enforcing average rate
function allow_request(user_id):
    bucket = get_bucket(user_id)
    now = time.now()

    // Refill tokens based on elapsed time
    elapsed = now - bucket.last_refill
    tokens_to_add = elapsed * REFILL_RATE
    bucket.tokens = min(CAPACITY, bucket.tokens + tokens_to_add)
    bucket.last_refill = now

    if bucket.tokens >= 1:
        bucket.tokens -= 1
        save_bucket(user_id, bucket)
        return ALLOW
    else:
        save_bucket(user_id, bucket)
        return REJECT  // 429

Why token bucket wins in practice:

Burst tolerance: A user who has been idle for 10 seconds has accumulated 10 tokens (at 1 token/sec refill). They can instantly send 10 requests. This matches real user behavior — humans do things in bursts, not at a constant rate.

Predictable average rate: Over any sufficiently long period, the average rate converges to the refill rate regardless of bursts. This gives you both flexibility and control.

Simple state: Only two values per user: tokens (float) and last_refill (timestamp). ~16 bytes. Compare with sliding window log's unbounded memory.

Used by: AWS API Gateway, Stripe, Google Cloud Endpoints, nginx (limit_req with burst parameter), and most API management platforms.

5. Leaky Bucket

Requests enter a FIFO queue. The queue drains at a constant rate. If the queue is full, new requests are rejected. Unlike token bucket, there is no burst — output is always perfectly smooth.

// Leaky Bucket — enforces constant output rate
function allow_request(user_id):
    bucket = get_bucket(user_id)
    now = time.now()

    // Drain requests that would have "leaked" since last check
    leaked = (now - bucket.last_check) * DRAIN_RATE
    bucket.queue_size = max(0, bucket.queue_size - leaked)
    bucket.last_check = now

    if bucket.queue_size < CAPACITY:
        bucket.queue_size += 1
        save_bucket(user_id, bucket)
        return ALLOW
    else:
        return REJECT

Best for: Traffic shaping at the network level (smoothing bursty traffic before forwarding to a capacity-limited downstream). Not ideal for user-facing APIs because it penalizes legitimate burst behavior (e.g., a page load that fires 10 API calls simultaneously).

05HTTP 429 & Response Headers

RFC 6585 (April 2012) introduced HTTP 429 as the standard status code for rate-limited responses. A well-behaved rate limiter returns informative headers:

HTTP/1.1 429 Too Many Requests
Content-Type: application/json
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1712190000

{"error": "rate_limit_exceeded", "message": "100 requests/minute exceeded. Retry after 30s."}
HeaderMeaningStandard?
Retry-AfterSeconds (or date) until client should retryYes (RFC 6585)
X-RateLimit-LimitMaximum requests allowed in the windowDe facto (draft RFC)
X-RateLimit-RemainingRequests remaining in current windowDe facto
X-RateLimit-ResetUnix timestamp when the window resetsDe facto

Client-side best practice: When receiving a 429, read the Retry-After header and wait exactly that long. If the header is missing, use exponential backoff with jitter: wait = min(base × 2^attempt + random(0, 1000ms), max_wait). Never retry immediately in a tight loop — that's exactly how a well-intentioned client becomes a DoS attacker.

06Distributed Rate Limiting

When your API runs on 50 servers behind a load balancer, where does the rate limit state live? If each server tracks counts independently, a user gets 50× the actual limit (100/server × 50 servers = 5,000 effective limit).

Centralized Counter (Redis)

All servers check/increment a shared counter in Redis. This is the most common approach. A Redis INCR + EXPIRE takes ~0.2ms. Even with 50 servers each doing 10K req/s, the Redis load is 500K ops/s — well within a single Redis instance's capacity.

// Atomic rate check in Redis using Lua script (no race conditions)
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)
end

if current > limit then
    return 0  -- rejected
end
return 1  -- allowed

Why Lua? Without Lua, the INCR and EXPIRE commands are two separate operations. If the server crashes between them, the key persists forever without a TTL (memory leak). The Lua script executes atomically on the Redis server — both operations succeed or neither does. This is the #1 gotcha in Redis rate limiting that catches people in production.

Local + Sync (Approximate)

Each server maintains a local counter and periodically syncs with a central store. Faster (no network call per request) but allows temporary overages. Useful when absolute accuracy isn't critical and latency is paramount. The local counters report to Redis every N seconds; Redis aggregates and returns "you've used X% of your quota."

Consistent Hashing (Sticky)

Route all requests from a given user to the same server (via consistent hashing on user ID). That server tracks the user's rate locally with no coordination needed. Simple and fast, but creates hot spots if one user sends massive traffic, and fails if that server goes down (all state is lost until the user is re-hashed to a new server).

09Hands-On: Terminal Demos

Demo 1: Rate-Limited HTTP Server

# Save as rate_server.py — a server with per-user rate limiting
$ cat > rate_server.py <<'EOF'
import time, json
from http.server import HTTPServer, BaseHTTPRequestHandler

LIMIT = 3         # max requests
WINDOW = 10       # per 10 seconds
accesses = {}     # user -> list of timestamps

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        user = self.headers.get('user', self.client_address[0])
        now = time.time()
        
        # Clean old timestamps
        if user not in accesses:
            accesses[user] = []
        accesses[user] = [t for t in accesses[user] if now - t < WINDOW]
        
        if len(accesses[user]) >= LIMIT:
            retry_after = int(WINDOW - (now - accesses[user][0]))
            self.send_response(429)
            self.send_header('Retry-After', str(retry_after))
            self.send_header('X-RateLimit-Limit', str(LIMIT))
            self.send_header('X-RateLimit-Remaining', '0')
            self.end_headers()
            self.wfile.write(b'Too many requests.\n')
            print(f"  REJECTED {user} ({len(accesses[user])}/{LIMIT})")
            return
        
        accesses[user].append(now)
        remaining = LIMIT - len(accesses[user])
        self.send_response(200)
        self.send_header('X-RateLimit-Limit', str(LIMIT))
        self.send_header('X-RateLimit-Remaining', str(remaining))
        self.end_headers()
        self.wfile.write(b'OK\n')
        print(f"  ALLOWED  {user} ({len(accesses[user])}/{LIMIT}, {remaining} remaining)")
    
    def log_message(self, *a): pass

print(f"Rate-limited server on :8080 ({LIMIT} req / {WINDOW}s per user)")
HTTPServer(('', 8080), Handler).serve_forever()
EOF

10Mini-Projects

Simple: Token Bucket Visualizer

Goal: Implement a token bucket in Python. Print the bucket state after each request. Experiment with different refill rates and capacities. Watch how idle time accumulates tokens and bursts drain them.

# token_bucket.py — interactive token bucket demo
import time

class TokenBucket:
    def __init__(self, capacity, refill_rate):
        self.capacity = capacity
        self.tokens = capacity  # start full
        self.refill_rate = refill_rate  # tokens per second
        self.last_refill = time.time()
    
    def allow(self):
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity, self.tokens + elapsed * self.refill_rate)
        self.last_refill = now
        
        if self.tokens >= 1:
            self.tokens -= 1
            return True, self.tokens
        return False, self.tokens

bucket = TokenBucket(capacity=5, refill_rate=1)  # 5 burst, 1/sec refill

print("Token Bucket: capacity=5, refill=1/sec")
print("Commands: 'r' = send request, 'w N' = wait N seconds, 'q' = quit\n")

while True:
    cmd = input("> ").strip()
    if cmd == 'q': break
    elif cmd == 'r':
        ok, tokens = bucket.allow()
        status = "ALLOWED" if ok else "REJECTED"
        print(f"  {status} | tokens remaining: {tokens:.1f}")
    elif cmd.startswith('w'):
        secs = float(cmd.split()[1])
        print(f"  Waiting {secs}s...")
        time.sleep(secs)
        print(f"  Tokens now: {min(bucket.capacity, bucket.tokens + secs * bucket.refill_rate):.1f}")

11Production Examples

Stripe: Tiered Token Bucket

Stripe's API uses a token bucket rate limiter with tiered limits: 100 read requests/second and 100 write requests/second in live mode, 25 each in test mode. They return standard rate limit headers (X-RateLimit-Limit, X-RateLimit-Remaining). When rate limited, the response includes a Retry-After header and a rate_limit error code.

Key design decision: Stripe rate-limits per API key, not per IP. This prevents a legitimate user behind a NAT (sharing an IP with thousands of others) from being unfairly rate limited. It also means a compromised API key can be throttled independently without affecting other users.

Cloudflare: Sliding Window at Edge

Cloudflare's Rate Limiting product runs at their edge network (~300 PoPs globally). It uses a sliding window counter algorithm that tracks request counts per client across a configurable time window (10s to 1 hour). The rate limit state is maintained locally at each PoP, with optional cross-PoP synchronization for global rate limits.

Scale: Cloudflare evaluates rate limit rules on every HTTP request they proxy — that's 50+ million requests per second. The rate limiter must add less than 1ms of latency. They achieve this by keeping all state in the edge node's memory with O(1) operations per request, using the sliding window counter's two-counter approximation rather than storing per-request timestamps.

GitHub API: Multi-Tier Rate Limits

GitHub uses three tiers of rate limiting: primary (per-user, 5000/hour authenticated, 60/hour unauthenticated), secondary (per-endpoint, protects expensive operations like search and GraphQL), and abuse detection (automated behavioral analysis that blocks clients exhibiting bot-like patterns regardless of rate).

Headers returned: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset (Unix epoch), X-RateLimit-Used, and X-RateLimit-Resource (which quota pool the request counts against). GitHub was one of the first major APIs to fully implement rate limit transparency headers.

12Edge Cases & War Stories

The Redis INCR + EXPIRE Race: You use INCR key then EXPIRE key 60 as two separate commands. Between them, the server crashes. The key now has a count of 1 with no TTL — it persists forever. Over time, thousands of these orphaned keys accumulate, consuming Redis memory. Eventually Redis hits maxmemory and starts evicting random keys, breaking rate limiting for everyone. Fix: Use a Lua script to make INCR + EXPIRE atomic, or use SET key 1 EX 60 NX for initialization followed by INCR for subsequent requests.

Clock Skew in Distributed Systems: Your API servers use wall-clock time for rate limit windows. Server A's clock is 2 seconds ahead of Server B's. A client's requests alternate between A and B. Server A thinks it's minute 42, Server B thinks it's minute 41. The client gets counted in different windows on different servers, effectively getting 2× the rate limit during the clock skew overlap. Fix: Use Redis server-side time (TIME command) as the source of truth, not application server clocks. Or use NTP with tight synchronization (<100ms).

Rate Limiting Behind CDN: Your rate limiter checks request.ip, but all requests come from Cloudflare's proxy IPs. You rate limit Cloudflare's IP instead of the actual client. Every user sharing that Cloudflare PoP is treated as a single client — one heavy user rate-limits everyone else at that PoP. Fix: Use the X-Forwarded-For or CF-Connecting-IP header to get the real client IP. But validate that the header actually comes from your trusted CDN (check the connecting IP against Cloudflare's published IP ranges) — otherwise attackers can spoof it.

Client Retry Storm: You return 429 without a Retry-After header. Clients use aggressive exponential backoff but no jitter. When the rate limit window resets, all previously-rejected clients retry simultaneously, creating a thundering herd that immediately triggers rate limiting again. The system never recovers because every window reset is met with a synchronized burst. Fix: Always include Retry-After. Add jitter on the client side. Stagger rate limit window starts per user (hash user_id to determine window offset).

Rate Limiting Your Own Internal Services: Service A calls Service B 10,000 times/second (normal for microservices). You add rate limiting to Service B at 1,000/sec per client. Service A is now rate-limited, causing cascading failures across your own infrastructure. Fix: Separate rate limit policies for internal vs. external traffic. Internal services get much higher limits (or no limits, relying instead on circuit breakers and backpressure). External/public traffic gets strict limits.

13Interview Q&A

Q1: Which rate limiting algorithm would you choose for a public REST API? Why?

Token bucket is the standard choice. It provides the best balance of accuracy, simplicity, and user experience. Burst tolerance is critical for APIs — real clients send requests in bursts (page load = 10 parallel API calls), and a strict fixed-rate limiter (leaky bucket) would reject legitimate traffic patterns.

Token bucket gives you two tunable knobs: refill rate (your sustained rate limit, e.g., 100/min) and bucket capacity (your burst limit, e.g., 20 concurrent). You can offer "100 requests per minute with bursts up to 20" — this is exactly how Stripe, AWS API Gateway, and most major APIs describe their limits.

Follow-up: If an interviewer asks "what about the sliding window counter?", that's also a great choice — especially if you need to match the exact "N requests per M seconds" semantics that users expect. The sliding window counter is what Cloudflare uses. In practice, both are excellent; token bucket is slightly more common because it maps naturally to "refill rate + burst size" which is intuitive for API consumers.

Q2: How would you implement rate limiting across 50 API servers?

Use a shared Redis instance (or cluster) as the central counter store. Each API server, before processing a request, sends an atomic Lua script to Redis that increments the counter and checks the limit. Redis Lua scripts execute atomically — no race conditions.

Latency concern: Each rate limit check adds a Redis round-trip (~0.2ms local, ~1ms cross-AZ). For 50 servers at 10K req/s each, Redis handles 500K ops/s (well within capacity). If even 0.2ms is too much, use a hybrid approach: local in-memory counters with periodic sync to Redis (less accurate, but ~0 latency).

Redis availability: If Redis is down, should you fail open (allow all requests) or fail closed (reject all)? For most APIs: fail open. A few seconds of un-rate-limited traffic is better than a total API outage. Log the Redis failure, alert on it, and have a circuit breaker that stops checking Redis after N failures (so you don't add latency for every request during the outage).

Q3: A user reports they're being rate-limited but they're well under the advertised limit. What could be wrong?

Top causes:

1. Shared IP/NAT: If rate limiting is per-IP, the user shares an IP with other clients (corporate NAT, VPN, university network). All clients behind that IP collectively hit the limit.

2. Fixed window boundary: The user sent 80 requests at 12:00:55 and 30 more at 12:01:03. The fixed window counter saw 80 + 30 = 110 in the 12:01 window. From the user's perspective, they sent 110 requests across 2 minutes (55 req/min) — well under the 100/min limit. But the fixed window counted all 110 in one window.

3. Wrong rate limit key: Rate limiting is per-API-key, but the user has two applications using the same key. Each app thinks it's under the limit, but combined they exceed it.

4. Retries counted against limit: The user's HTTP client automatically retries failed requests. Each retry counts as a new request against the rate limit, creating a feedback loop: rate limited → retry → counted again → more rate limited.

Debug approach: Check the rate limit key being used (IP? user? API key?), inspect the exact timestamps in Redis/logs, and verify whether retries are inflating the count.

Q4: How do you rate limit a multi-tenant SaaS where different customers have different plans?

Each tenant gets a rate limit configuration tied to their plan: free (100/hour), pro (10,000/hour), enterprise (1M/hour). The rate limiter reads the plan from the user's metadata (cached in Redis or fetched at auth time) and applies the corresponding limit.

Implementation: Store the rate limit config per plan in a config service. On each request: (1) authenticate → get user_id and plan, (2) look up plan's rate limit parameters, (3) check/increment the rate counter with those parameters. The rate limiter is parameterized, not hardcoded.

Nuance: Enterprise customers may need per-endpoint limits (e.g., 1000 reads/sec but only 50 writes/sec), per-resource limits (100 bulk exports/day), and separate rate limit pools per sub-organization. This requires a hierarchical rate limiting model where a single request is checked against multiple rate limit rules, and the most restrictive one wins.

Q5: What's the fixed window boundary burst problem, and how do you fix it?

With a fixed window of 60 seconds and a limit of 100 requests: a client sends 100 requests at second 59 (end of window 1) and 100 more at second 61 (start of window 2). Both pass the rate limit check (100 ≤ 100 in each window), but the server received 200 requests in a 2-second span — 2× the intended rate.

Fix 1 — Sliding window counter: Weight the previous window's count by the overlap fraction. At second 61, the previous window (59s elapsed = 98.3% overlap) contributes 100 × 0.983 = 98.3. Current window count is 100. Total estimate: 198.3 — exceeds the limit. Request rejected.

Fix 2 — Token bucket: The bucket has a refill rate of 100/60s ≈ 1.67 tokens/sec. After draining 100 tokens at second 59, the bucket is empty. At second 61 (2 seconds later), only 3.3 tokens have refilled. The client can only send 3 more requests — not 100. The burst is naturally constrained by the refill rate.

Interview tip: Mentioning this boundary problem and its fix demonstrates you understand rate limiting beyond the naive textbook implementation. Most candidates only describe fixed windows.

Q6: How does rate limiting interact with load balancers and auto-scaling?

Tension: auto-scaling adds servers when traffic increases. If rate limiting is per-server, each new server provides additional rate limit headroom — scaling up increases the effective limit. If rate limiting is centralized (Redis), the limit stays constant regardless of server count — correct behavior.

But: Centralized rate limiting means the Redis round-trip is in the critical path of every request. If auto-scaling adds 100 servers under a traffic spike, Redis suddenly handles 100× more rate limit checks. Redis itself can become the bottleneck.

Production pattern: Use centralized Redis for the authoritative rate limit, but cache the "remaining quota" locally on each server for ~1 second. Each server allows requests against its local cache and syncs with Redis every second. This reduces Redis traffic by 100–1000× while maintaining near-accurate limits (with up to 1-second drift). This is how AWS API Gateway's distributed rate limiter works internally.

Round 1: Core Design

Interviewer: "Walk me through your architecture."

Rate Limit Middleware: Sits in the API Gateway layer, evaluated before any request reaches the backend. On each request: extract the API key → look up the customer's plan config (cached locally, refreshed every 30s from config service) → evaluate rate limit rules → allow or reject with proper 429 response.

Algorithm choice: Token bucket for the per-customer global limit (allows bursts, O(1) memory), with separate sliding window counters per endpoint (prevents a customer from sending all their quota to a single expensive endpoint like search).

State store: Redis Cluster with 3 shards. Each customer's rate limit state is hashed to a shard by customer_id. At 1M req/s and ~0.2ms per Redis operation, we need ~200K ops/s per shard — well within Redis's ~100K+ ops/s capacity. Token bucket state is implemented as a Redis hash: HSET rate:{customer_id} tokens {float} last_refill {timestamp}, updated atomically via Lua script.

Fail-open policy: If Redis is unreachable, the gateway allows requests through (fail open) and logs the incident. A circuit breaker stops Redis checks after 3 consecutive failures, checking again every 5 seconds. This prevents Redis latency from becoming API latency.

Round 2: Multi-Tier Rate Limits

Interviewer: "A customer has a global limit of 10,000/min, but the /search endpoint should be limited to 100/min. How do you handle overlapping limits?"

Each request is checked against multiple rate limit rules in order: (1) global per-customer limit, (2) per-endpoint limit, (3) per-IP limit (DDoS protection). The request is rejected if any rule is exceeded. All counters are incremented atomically in a single Redis Lua call to avoid partial updates.

Rule evaluation order matters for performance: Check the cheapest, most likely to reject first (global limit is a single key lookup), then the endpoint limit (requires knowing the endpoint path), then the IP limit (rarely triggered for legitimate users). Short-circuit: if the first check rejects, skip the rest — no need to increment other counters for a rejected request.

Round 3: Edge Cases

Interviewer: "What happens if a customer upgrades from free to pro mid-billing cycle?"

The rate limit config is fetched from the plan config service, which is updated when the customer upgrades. The gateway's local config cache refreshes every 30 seconds, so within 30 seconds the new limit applies. The customer's existing counter values remain — upgrading doesn't reset usage, it just raises the ceiling. If the customer was at 95/100 on the free tier and upgrades to pro (10,000/min), their counter shows 95/10,000 — they immediately have headroom.

Downgrade scenario: More interesting. If a customer downgrades from pro (10,000/min) to free (100/min) and their current usage is 5,000 in this window, they're instantly over the free limit. The system should handle this gracefully: apply the new limit immediately but don't retroactively reject already-served requests. The customer simply can't send more requests until the window resets. Communicate this in the 429 response: "Your plan has been downgraded. Current usage: 5000. New limit: 100/min."

17References & Further Reading

Use these sources to study the actual mechanics: the HTTP contract for 429, the burst math behind token bucket, overload behavior under retries, and the coordination tradeoffs of distributed counters.

[1] RFC 6585 — Additional HTTP Status Codes (2012)

M. Nottingham, R. Fielding. IETF, April 2012.
https://datatracker.ietf.org/doc/html/rfc6585

• Defines HTTP 429 (Too Many Requests) as the standard response for rate-limited requests. Prior to this RFC, APIs used 403 Forbidden or 503 Service Unavailable — both semantically incorrect for rate limiting.

• Specifies that 429 responses SHOULD include a human-readable explanation and MAY include a Retry-After header indicating seconds until the client should retry. The RFC deliberately leaves the identification mechanism (IP, user, API key) and counting method (per-server, per-cluster) undefined — implementations choose their own.

• Security consideration: responding with 429 to every rate-limited request itself consumes resources. Under a DDoS, generating and sending 429 responses can amplify the attack. The RFC recommends that under extreme load, servers may simply drop connections rather than sending structured responses.

• Also introduced HTTP 428 (Precondition Required) for optimistic concurrency and HTTP 431 (Request Header Fields Too Large) for header-based attacks — both relevant to API security but less widely known than 429.

• The X-RateLimit-* headers are outside this RFC. If your API returns limit, remaining, and reset metadata, that is an application convention layered on top of the 429 contract, not a requirement defined here.

[2] Cloudflare — "How we built rate limiting capable of scaling to millions of domains" (2017)

Cloudflare Engineering. 2017.
https://blog.cloudflare.com/how-we-built-rate-limiting-capable-of-scaling-to-millions-of-domains/

• The important mechanism is the counter design: the edge path cannot afford per-request logs, so the limiter stores compact rule state and evaluates each request in O(1) time.

• It explains why locality changes the architecture. If every edge request had to round-trip to a central counter, the limiter would add more latency than the abuse it is trying to stop, so the design keeps decisions close to the request and accepts bounded approximation where needed.

• The write-up is also good on rule distribution. Customer-defined limits are configuration objects that must propagate safely to the edge without turning config rollout into a correctness bug.

• Read it for the production question interview answers often skip: where the counter lives, how quickly it updates, and how much approximation the edge path can tolerate before the product semantics change.

[3] RFC 2697 — A Single Rate Three Color Marker (1999)

J. Heinanen, R. Guerin. IETF, September 1999.
https://datatracker.ietf.org/doc/html/rfc2697

• This is a concrete token-bucket specification, not just a high-level description. It defines how committed information rate, committed burst size, and token replenishment interact to classify traffic under sustained load.

• The useful detail is the refill invariant: tokens accumulate at a configured rate until the bucket reaches its burst ceiling, so over any interval the admitted traffic is bounded by rate plus saved burst credit.

• It is also useful for separating policing from shaping. The marker decides whether traffic is in profile or out of profile; it does not smooth the stream the way a queued shaper does. That distinction maps cleanly to API rate limiters that reject rather than queue excess requests.

• Read it for the exact contract behind token bucket parameters. If you say “10 requests per second with burst 50,” this RFC gives the formal behavior you are approximating at the application layer.

[4] Google SRE Book — Chapter 21: Handling Overload (2016)

Betsy Beyer, Chris Jones, Jennifer Petoff, Niall Richard Murphy. O'Reilly, 2016.
https://sre.google/sre-book/handling-overload/

• Describes Google's approach to load shedding: each server tracks its own "criticality" metric (a function of CPU utilization, queue depth, and error rate). When a server is overloaded, it starts rejecting low-criticality requests first, preserving capacity for high-criticality ones.

• Introduces the concept of "adaptive throttling" at the client side: before sending a request, the client estimates the probability it will be rejected based on recent rejection rates. If the estimated probability exceeds a threshold, the client self-throttles (doesn't even send the request), reducing load on the server. The formula: P(reject) = max(0, (requests - K × accepts) / (requests + 1)).

• Discusses the danger of retry amplification: if 50% of requests are rate-limited and clients retry once, the server sees 1.5× the original load. If clients retry twice, the server sees 1.75×. With aggressive retries, the additional load from retries can exceed the original overload — a positive feedback loop. Google mandates exponential backoff with jitter for all internal service-to-service calls.

• Recommends per-request "importance" labels (critical, degraded, sheddable+, sheddable) that flow through the entire call graph. When shedding load, the system drops "sheddable" requests first, then "sheddable+", preserving "critical" requests (e.g., user-facing reads) until the very end.

• Key metric: "goodput" (successful requests per second), not raw throughput. A server doing 10K req/s but returning 8K errors has a goodput of 2K. Rate limiting should maximize goodput, not minimize rejections.

[5] Designing Data-Intensive Applications — Chapter 8: The Trouble with Distributed Systems (Kleppmann, 2017)

Martin Kleppmann. O'Reilly, 2017.
https://dataintensive.net

• Discusses the fundamental challenges of distributed rate limiting: clock synchronization (Section 8.3), where wall-clock disagreements between nodes cause rate limit windows to drift; and network delays (Section 8.2), where a request that was within the limit when sent may exceed the limit by the time it's processed.

• The "happens-before" relationship (Lamport clocks) is relevant for ordering rate limit events across distributed nodes: if two increments to the same counter happen concurrently on different nodes, which one "counts" first? In practice, rate limiters use eventual consistency and accept minor over-counting.

• Describes the tradeoff between strong consistency (centralized counter, always accurate, adds latency) and eventual consistency (local counters, allows temporary overages, no latency penalty) — the same CAP theorem tradeoff from the replication and sharding topic, applied specifically to rate limit state.

• The concept of "fencing tokens" (Section 8.4) applies to rate limit token buckets: if a token is "consumed" but the request fails and needs to be retried, should the token be returned? Most systems say no (simpler, avoids race conditions) — the retry consumes a new token.

• Motivates the design principle that rate limiting should be a "best effort" system — it's better to occasionally allow a few extra requests than to add significant latency to every request for perfect accuracy.
