Prerequisites & Key Terms

Relational Database

A relational database stores data in tables made of rows and columns. Tables point at each other through foreign keys, which is what makes joins possible. What matters is that you define the schema up front and the database enforces it. That model comes from Edgar F. Codd's 1970 work at IBM. [1] Common examples are PostgreSQL, MySQL, MariaDB, Oracle, SQL Server, SQLite, and CockroachDB.

SQL (Structured Query Language)

SQL is the language relational databases speak. You use it to define tables (CREATE TABLE), insert rows (INSERT), query data (SELECT ... FROM ... WHERE ... JOIN), update rows (UPDATE), and delete them (DELETE). There is a standard core, but every engine adds its own extra features. PostgreSQL has JSONB. MySQL has INSERT ... ON DUPLICATE KEY UPDATE. The shared core transfers well; the extensions do not.

ACID Transaction

A transaction is a bundle of operations that the database treats as one unit. ACID is the contract around that unit: Atomicity means all of it happens or none of it does, Consistency means constraints still hold at commit, Isolation means concurrent work does not leak half-finished state, and Durability means committed data survives a crash. This is why relational databases are still the default for money, inventory, billing, and account data.

Database Index

An index is an extra data structure, usually a B-tree, that lets the database find matching rows without reading the whole table. It is the same idea as the index at the back of a book: you jump to the pages that matter instead of scanning everything. The catch is that every write now has more work to do because the index has to stay in sync. PostgreSQL defaults to B-tree indexes, but it also supports hash, GiST, GIN, and BRIN for more specific access patterns.

Strong Consistency

After a write is acknowledged, every later read sees it, no matter which client asks or which replica serves the request. That is normal on a single database node. In a distributed system, it takes coordination, usually Paxos or Raft, and that coordination costs latency. The alternative is eventual consistency, where some reads can stay stale for a while after the write.

Eventual Consistency

With eventual consistency, replicas will converge if writes stop, but the system does not promise when. A read in that window may return older data. That trade-off is fine for feeds, analytics, and caches. It is not fine for balances, inventory, or anything else where stale reads create the wrong answer.

01

What Is a Relational Database?

A relational database keeps data in tables. Each row is a record. Each column has a type. Tables connect through foreign keys, which lets you join related data instead of copying it everywhere.

A simple relational schema
-- Two tables with a foreign key relationship

CREATE TABLE users (
    id        SERIAL PRIMARY KEY,
    username  VARCHAR(50) UNIQUE NOT NULL,
    email     VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE orders (
    id        SERIAL PRIMARY KEY,
    user_id   INTEGER REFERENCES users(id),  -- foreign key → users table
    amount    DECIMAL(10,2) NOT NULL,
    status    VARCHAR(20) DEFAULT 'pending',
    placed_at TIMESTAMP DEFAULT NOW()
);

-- The schema ENFORCES: every order must reference an existing user.
-- Try INSERT INTO orders (user_id, amount) VALUES (999, 50.00);
-- → ERROR: violates foreign key constraint (user 999 doesn't exist)
Core Intuition

A spreadsheet analogy gets you close, but only up to a point. A relational database is a spreadsheet that refuses bad data. It can reject nulls, duplicate values, or foreign keys that point nowhere. That enforcement layer is the real value. SQL is just how you talk to it.

02

ACID — The Transaction Contract

A transaction is the database saying, "I will treat this whole block as one thing." ACID spells out what that promise means.

ACID — WHAT EACH PROPERTY ACTUALLY MEANS │ ├── Atomicity │ All operations in a transaction succeed, or none do. │ If step 3 of 5 fails → steps 1 & 2 are rolled back. │ Mechanism: WAL (write-ahead log) records all changes before applying them. │ On crash: replay WAL to redo committed txns, undo uncommitted ones. │ ├── Consistency │ The database transitions from one valid state to another. │ All constraints (NOT NULL, UNIQUE, CHECK, foreign keys) hold after commit. │ If a transaction would violate a constraint → it's rejected entirely. │ ├── Isolation │ Concurrent transactions behave as if they ran one at a time. │ Reality: strict serialization is expensive. Databases offer LEVELS: │ Read Uncommitted → Read Committed → Repeatable Read → Serializable │ Each level trades isolation for concurrency (throughput). │ └── Durability Once committed, the data is written to non-volatile storage (disk). Mechanism: fsync() the WAL to disk BEFORE returning "commit OK." Even if the machine loses power 1ms after commit → data survives. Cost: fsync is the single biggest bottleneck in write-heavy workloads.
ACID in action: transferring money between accounts
-- Transfer $100 from Alice to Bob. MUST be atomic.
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE username = 'alice';
  UPDATE accounts SET balance = balance + 100 WHERE username = 'bob';
COMMIT;

-- If the server crashes between the two UPDATEs:
-- WITHOUT transactions: Alice lost $100, Bob got nothing. Money vanished.
-- WITH transactions (Atomicity): both UPDATEs are rolled back. No change.

-- If another transaction reads Alice's balance DURING the transfer:
-- WITH isolation (Read Committed): sees Alice's balance BEFORE the debit.
-- WITH isolation (Serializable): waits until the transfer commits or aborts.
03

Indexes — Making Queries Fast

Without an index, the database falls back to a sequential scan and reads the table row by row until it finds the matches. On a 10 million row table, that means touching 10 million rows. With a B-tree index, the same lookup usually turns into a short tree walk plus a few heap fetches.

Index TypeStructureBest ForPostgreSQL Syntax
B-treeBalanced tree, sorted keys, O(log N) lookupEquality and range queries (=, <, >, BETWEEN, ORDER BY). Default for most columns.CREATE INDEX ON orders(user_id);
HashHash table, O(1) lookupEquality-only queries (=). Faster than B-tree for exact match but no range support.CREATE INDEX ON users USING hash(email);
GINGeneralized Inverted IndexFull-text search, JSONB containment, array element search.CREATE INDEX ON docs USING gin(body_tsvector);
GiSTGeneralized Search TreeGeometric data (PostGIS), nearest-neighbor, range types.CREATE INDEX ON places USING gist(location);
BRINBlock Range Index (metadata per block)Very large, naturally ordered tables (time-series, logs). Tiny index size.CREATE INDEX ON events USING brin(created_at);
The Index Trade-off

Indexes buy read speed by making writes heavier. Every INSERT, UPDATE, or DELETE on an indexed column has to keep that index in sync. On a table with 10 indexes, one write can turn into 10 index changes. So index the columns that drive real queries, especially WHERE, JOIN, and ORDER BY. Do not index on instinct. Check the plan first.

04

Normalization vs Denormalization

Normalization: Eliminate Redundancy

Normalization removes duplication. Instead of copying a user's name into every order, store user_id in the order row and join back to the users table when you need the name. Then a name change touches one row, not ten thousand.

Normalized: user name stored once
-- users table: single source of truth for user info
users:  { id: 1, name: "Alice", email: "alice@example.com" }

-- orders table: references user by ID, not by name
orders: { id: 101, user_id: 1, amount: 49.99 }
orders: { id: 102, user_id: 1, amount: 29.99 }

-- To get "Alice's orders": JOIN users ON orders.user_id = users.id
-- If Alice changes her name → update ONE row in users. Orders are fine.

Denormalization: Trade Redundancy for Speed

Denormalization duplicates data on purpose so reads can skip joins. You might copy the user's name into the orders table and serve a dashboard from one table instead of five. That helps reads. It also means every update has more places to touch, and missed backfills leave stale copies behind.

PropertyNormalizedDenormalized
ReadsSlower (requires JOINs)Faster (data is pre-joined)
WritesFaster (update one place)Slower (update many copies)
StorageLess (no duplication)More (duplicated data)
ConsistencyStrong (single source of truth)Risk of inconsistency
When to useOLTP (user accounts, transactions)OLAP (analytics, dashboards, read-heavy)
Interview Insight

In an interview, start with the clean normalized version. Then wait for the pressure that justifies breaking it. If the dashboard now joins five tables and misses its latency target, add a summary table or materialized view and explain the trade. That answer is stronger than denormalizing everything up front.

The Query Planner — How SQL Becomes I/O

When you write SELECT * FROM orders WHERE user_id = 42, the engine does not blindly follow the text. The query planner checks table statistics, available indexes, and estimated costs, then chooses a plan.

EXPLAIN ANALYZE — see what the database actually does
-- Without index:
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42;

Seq Scan on orders  (cost=0.00..18334.00 rows=5 width=44)
  Filter: (user_id = 42)
  Rows Removed by Filter: 999995
  Planning Time: 0.08 ms
  Execution Time: 92.4 ms        ← scanned ALL 1M rows to find 5

-- Add an index:
CREATE INDEX idx_orders_user ON orders(user_id);

-- With index:
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42;

Index Scan using idx_orders_user on orders  (cost=0.42..8.44 rows=5)
  Index Cond: (user_id = 42)
  Planning Time: 0.12 ms
  Execution Time: 0.04 ms        ← B-tree traversal, read only 5 rows

-- 92.4ms → 0.04ms = 2,310× faster. One index. That's the power.
EXPLAIN is Your Best Friend

If you work with databases, you need to be comfortable reading EXPLAIN ANALYZE. Start with the obvious questions. Is there a Seq Scan on a large table? Which join type did the planner pick? Do actual rows and estimated rows line up? Big gaps there usually mean stale stats or a bad estimate somewhere upstream.

Transactions & Isolation Levels

Isolation levels decide what one transaction is allowed to observe about another. Stronger isolation blocks more anomalies, but it usually costs throughput or retry rate.

Isolation LevelDirty ReadsNon-repeatable ReadsPhantom ReadsPerformance
Read UncommittedPossiblePossiblePossibleFastest (no locks)
Read CommittedPreventedPossiblePossibleFast (PostgreSQL default)
Repeatable ReadPreventedPreventedPossibleModerate (MySQL/InnoDB default)
SerializablePreventedPreventedPreventedSlowest (full isolation)

Dirty read: Transaction A reads a value that Transaction B has not committed yet. If B rolls back, A saw something that never became real. Non-repeatable read: A reads a row twice and gets two different answers because B committed a change in between. Phantom read: A reruns a range query and sees new rows that now match the predicate.

PostgreSQL's Implementation Detail

PostgreSQL uses MVCC, so reads usually do not take read locks on rows. Each transaction gets a snapshot. Writers create new tuple versions. Readers keep seeing the version that matches their snapshot. That is why PostgreSQL handles concurrent reads well. The price is dead tuples, and dead tuples mean you need VACUUM to keep the system healthy.

Mini-Project: Schema Design & Queries

Simple: Feel What a Relational Database Does

mini_rdb.py — Python + SQLite, zero dependencies (macOS/Linux/WSL)
import sqlite3, time

db = sqlite3.connect(":memory:")  # In-memory DB — dies when script ends
db.execute("PRAGMA journal_mode=WAL")  # Enable WAL for better concurrency

# ── Schema ──
db.executescript("""
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    email TEXT UNIQUE NOT NULL
);
CREATE TABLE orders (
    id INTEGER PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    product TEXT NOT NULL,
    amount REAL NOT NULL,
    created_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_orders_user ON orders(user_id);
""")

# ── Seed data ──
users = [("Alice", "alice@example.com"), ("Bob", "bob@example.com"), ("Charlie", "charlie@example.com")]
db.executemany("INSERT INTO users (name, email) VALUES (?, ?)", users)

orders = [
    (1, "Widget", 49.99), (1, "Gadget", 29.99), (1, "Gizmo", 9.99),
    (2, "Widget", 49.99), (2, "Thingamajig", 199.00),
]
db.executemany("INSERT INTO orders (user_id, product, amount) VALUES (?, ?, ?)", orders)
db.commit()

# ── Queries ──
print("=== Total spent per user ===")
for row in db.execute("""
    SELECT u.name, COUNT(o.id) AS orders, COALESCE(SUM(o.amount), 0) AS total
    FROM users u LEFT JOIN orders o ON u.id = o.user_id
    GROUP BY u.id ORDER BY total DESC
"""):
    print(f"  {row[0]:<10} {row[1]} orders  ${row[2]:.2f}")

print("\n=== ACID test: atomic transfer ===")
try:
    db.execute("BEGIN")
    # Simulate: move an order from Alice to a nonexistent user
    db.execute("UPDATE orders SET user_id = 999 WHERE id = 1")
    # This would violate FK... but SQLite doesn't enforce FK by default!
    # Enable enforcement:
    db.execute("PRAGMA foreign_keys = ON")
    db.execute("COMMIT")
    print("  Transfer committed (FK not enforced in this demo)")
except Exception as e:
    db.execute("ROLLBACK")
    print(f"  Rolled back: {e}")

print("\n=== Index vs no-index benchmark ===")
# Insert 100K rows for a realistic test
db.executemany("INSERT INTO orders (user_id, product, amount) VALUES (?, 'Bulk', ?)",
              [((i % 3) + 1, round(i * 0.01, 2)) for i in range(100000)])
db.commit()

t0 = time.perf_counter()
db.execute("SELECT * FROM orders WHERE user_id = 2").fetchall()
with_idx = (time.perf_counter() - t0) * 1000

db.execute("DROP INDEX idx_orders_user")
t0 = time.perf_counter()
db.execute("SELECT * FROM orders WHERE user_id = 2").fetchall()
no_idx = (time.perf_counter() - t0) * 1000
print(f"  With index:    {with_idx:.1f}ms")
print(f"  Without index: {no_idx:.1f}ms")
print(f"  Speedup:       {no_idx/with_idx:.0f}×")

Production Example 1: PostgreSQL

PostgreSQL is the default relational database for a lot of modern systems, from small internal tools to large production services.

Why PostgreSQL Dominates

MVCC keeps concurrency sane: readers work from a snapshot, so they usually do not block writers.

The extension model is unusually strong: PostGIS, pg_trgm, JSONB, and pgvector all expand what you can do without leaving Postgres.

WAL is the foundation: writes hit the log before data files, which is what gives you crash recovery and streaming replication.

When to Use PostgreSQL

Use PostgreSQL unless you have a good reason not to. It is a solid fit for ACID transactions, joins, reporting queries, and most OLTP workloads. You usually leave it only when the workload wants something very different: huge write fan-in, globally distributed SQL with very low cross-region latency, or a simple key-value access pattern.

Production Example 2: Google Spanner

Spanner is Google's globally distributed relational database. It gives you ACID transactions across regions, which is exactly why it keeps showing up in database discussions. [2]

How Spanner Works

TrueTime: the clock API returns an interval [earliest, latest] with bounded uncertainty. Spanner uses that uncertainty directly in its commit protocol.

Externally consistent transactions: if transaction T1 commits before T2 starts in real time, T1's timestamp sorts before T2's. That is strict serializability with real-time order attached.

Split-based sharding: data is broken into key-range splits, each replicated across zones with Paxos. Reads can stay local; writes go through the leader for that split.

Key Lesson from Spanner

Spanner does not break CAP. It just spends a lot of engineering effort making the consistency cost tolerable. Better clocks shrink the uncertainty window, but partitions still force a choice and Spanner still lands on the CP side.

Edge Cases & Gotchas

1. N+1 Query Problem

Your ORM fetches 100 users, then runs 100 more queries to fetch each user's orders. That is 101 queries where one join would do. Fix it with eager loading, or the equivalent feature in your ORM.

2. Missing Indexes on Foreign Keys

PostgreSQL does not automatically index foreign key columns. If you keep joining on orders.user_id without an index there, the engine has no cheap access path. Add the index yourself.

3. Long-Running Transactions Block Vacuum

MVCC leaves dead row versions behind. VACUUM removes them, but it cannot remove tuples that an open transaction might still need to see. One forgotten transaction can hold that floor in place for hours and quietly bloat the table.

4. Full-Table Locks on ALTER TABLE

Some schema changes take an ACCESS EXCLUSIVE lock. On a large table, that can pause reads and writes long enough to become an incident. Use concurrent operations where PostgreSQL offers them, and plan the rest like real production work, not a casual migration.

5. Connection Exhaustion

PostgreSQL uses one process per connection. Enough idle connections can burn gigabytes of memory before query work even starts. That is why PgBouncer exists.

Production War Story

A dashboard went from 200ms to 45 seconds because someone added WHERE new_column = 'active' on a 50M-row table and never added the index. Every request turned into a full scan. One B-tree fixed it. Boring problem, expensive outage.

What If? — Interactive Exploration

Tradeoff Cards

Add more indexes

Improves: indexed reads.
Costs: slower writes, more disk, more VACUUM work.
Worth it when: the column shows up repeatedly in real WHERE, JOIN, or ORDER BY paths.

Relational (PostgreSQL) vs Document (MongoDB)

Relational: strict schema, joins, transactions, stronger invariants.
Document: flexible shape, nested records, easier denormalized modeling.
Reality: PostgreSQL with JSONB often covers the middle ground.

Read Committed vs Serializable isolation

Read Committed: usually enough for CRUD-style app traffic.
Serializable: stricter, safer, and more likely to abort and retry conflicting work.
Default: use the weaker level unless the invariant says otherwise.

Vertical scaling vs read replicas

Vertical: bigger box, simpler system, hard ceiling.
Read replicas: more read capacity, plus lag and consistency caveats.
Switch when: reads are the bottleneck and the app can tolerate stale replicas where needed.

Toy Simulation: Index vs No-Index at Scale

index_bench.py — measure the speedup as table size grows (macOS/Linux/WSL)
import sqlite3, time

for n_rows in [1_000, 10_000, 100_000, 1_000_000]:
    db = sqlite3.connect(":memory:")
    db.execute("CREATE TABLE t (id INTEGER PRIMARY KEY, val INTEGER, data TEXT)")
    db.executemany("INSERT INTO t VALUES (?,?,?)",
                   [(i, i % 100, "x" * 50) for i in range(n_rows)])
    db.commit()

    # Without index
    t0 = time.perf_counter()
    db.execute("SELECT * FROM t WHERE val = 42").fetchall()
    no_idx = (time.perf_counter() - t0) * 1000

    # With index
    db.execute("CREATE INDEX idx ON t(val)")
    t0 = time.perf_counter()
    db.execute("SELECT * FROM t WHERE val = 42").fetchall()
    with_idx = (time.perf_counter() - t0) * 1000

    speedup = no_idx / with_idx if with_idx > 0 else 0
    print(f"  {n_rows:>10,} rows | No index: {no_idx:6.1f}ms | With index: {with_idx:5.2f}ms | {speedup:5.0f}×")
    db.close()
Expected output (approximate — varies by hardware)
       1,000 rows | No index:    0.3ms | With index:  0.05ms |     6×
      10,000 rows | No index:    2.1ms | With index:  0.06ms |    35×
     100,000 rows | No index:   19.5ms | With index:  0.08ms |   244×
   1,000,000 rows | No index:  195.0ms | With index:  0.10ms |  1950×

Interview Questions & Answers

Q1: What are ACID properties and why do they matter?

Atomicity means the whole transaction commits or none of it does. Consistency means constraints still hold after commit. Isolation means concurrent work does not expose broken intermediate state. Durability means the commit survives a crash. Without ACID, a crash can leave you with half-finished transfers, broken foreign keys, or missing writes.

Q2: What is a database index and when should you create one?

An index gives the planner a faster path than scanning the whole table. Create them on columns that drive real filters, joins, and sorts. Skip the rest. Every extra index costs write time and disk. If you want to know whether one is missing, look at EXPLAIN ANALYZE and find out why the planner chose a sequential scan.

Q3: When would you choose a relational database over a NoSQL database?

Pick relational when the data is connected, the invariants matter, and joins are part of the workload. Pick a NoSQL system when the records stand mostly on their own, the schema changes constantly, or the workload is really key-value or high-ingest append traffic. A lot of teams land in the middle and use PostgreSQL with JSONB for the parts that do not fit cleanly into fixed columns.

Q4: Explain PostgreSQL's MVCC and why it matters for concurrency.

Under MVCC, each transaction reads from a snapshot. Updates create new tuple versions instead of overwriting old ones in place. That keeps readers and writers from stepping on each other as often, which is one reason PostgreSQL handles mixed workloads well. The downside is cleanup. Dead tuples build up, and if VACUUM falls behind, performance falls apart slowly and then all at once.

Q5: How would you scale a relational database that's hitting its limits?

I would scale it in layers. First, use a bigger machine and fix the obvious query problems. Then add connection pooling, replicas, and cache shielding for hot reads. After that, partition the biggest tables. Sharding comes last because it changes everything: routing, migrations, rebalancing, and cross-shard transactions.

Q6: A query that was fast yesterday is now slow. How do you diagnose it?

Start with EXPLAIN ANALYZE. Did the plan change? Did row estimates go bad? Then check for bloat, blocked sessions, dropped or unused indexes, and plain old data growth. A query can stay logically identical while the table grows from manageable to painful. If the plan still looks fine, check the host: disk, cache hit rate, and memory pressure.

Actual System Design Interview

A Prompt You Could Actually Get

Design a URL shortener with 100M new links per month and 10× more reads than writes. The database part is not just "pick PostgreSQL." You need to talk about schema shape, unique code generation, read scaling, replica lag, and what changes when the table stops fitting comfortably on one machine. [2] [3] [4]

How I'd Structure the Answer

I would start with the relational core: one urls table, one unique short_code, and the metadata that actually matters. Then I would split the write path from the read path. Writes need collision-safe code generation plus a uniqueness constraint the database can enforce. Reads are just exact-key lookups on short_code, so I would call out the B-tree index immediately and put a cache in front of it for hot links. If the system later shards, I would define the ownership key and say how routers learn shard placement during rebalance. [1] [3] [4]

Where the Interviewer Pushes

The first push is usually read pressure: the table keeps growing, but the redirect still needs to feel instant. That is where cache hit rate, replicas, and replica lag enter the answer. The next push is table size. If every query is still equality on short_code, a very large B-tree can be fine for longer than people think. But once vacuum, index maintenance, or working-set size stop fitting the host, you start talking about partitioning, archival, or sharding. Analytics is separate. Do not take row locks in the hot redirect path. [2] [3] [5]

Production Case Study Lens

Think in terms of a Bitly-style redirect service: one exact-match lookup is the hot path, replicas take some of the read load, Redis absorbs viral traffic, and a queue keeps analytics away from the redirect latency budget. The good answer is the one that separates what the database must enforce from what the surrounding systems can handle asynchronously. [2] [3] [4]

Step 2: Interviewer Follow-ups (the real test)

"How do you generate the short code? What about collisions?"

Approach 1, ID to Base62: generate a unique numeric ID and encode it. No collisions. Simple lookup path. The downside is predictability if IDs are sequential.

Approach 2, hash the URL: derive the code from the URL itself. That can deduplicate naturally, but collisions become part of the design and you need a retry or suffix strategy.

Approach 3, pre-generated code pool: mint codes in advance, store them, and claim one on insert. Operationally heavier, but easy to reason about.

I would usually pick the first approach and hide predictability with a less obvious ID scheme if needed.

"100M writes/month is ~40 writes/sec. That's easy. But 1B reads/month is ~400 reads/sec on the database. What if traffic spikes to 10K reads/sec during a viral link?"

First, cache the hot links. URL traffic is usually skewed, so Redis can peel a lot of reads away from PostgreSQL.

Then use read replicas for the misses that remain. That pushes more traffic off the primary, but it introduces lag.

For truly viral links, let the CDN cache the redirect itself. If the edge can answer, the origin does not matter.

The real question here is whether stale reads are acceptable. For a shortener, a small lag window usually is.

"The table now has 2 billion rows. Queries are slowing down. How do you handle this?"

First check the plan. If the query is still using the short_code index, the raw row count is not automatically the problem. B-trees stay shallow.

If the table is now painful operationally, partition by time, archive expired links, or shard by short_code. Those are answers to maintenance pressure and working-set size, not just row count vanity.

Since the access pattern is exact-key lookup, hash-based distribution usually fits better than range partitioning.

"A user creates a short URL and immediately shares it. Their friend clicks it but gets a 404. Why?"

This is a read-your-writes problem. The write committed on the primary, but the friend read from a replica that has not caught up yet.

Fixes: send very recent reads to the primary, require synchronous replication for that path, or write through to Redis so the redirect path sees the new code immediately.

The trade-off is straightforward: low-latency async replication gives you a lag window. Closing that window costs write latency or extra system design.

"How do you handle analytics — tracking clicks, referrers, geolocations — without slowing down the redirect?"

Do not update analytics synchronously in the redirect path. Hot links would turn one row into a write hotspot immediately.

Return the redirect first. Then enqueue a click event with the metadata you care about. A downstream consumer can batch those events into analytics storage and roll up counts later.

The redirect path wants low latency. The analytics path wants throughput. Treat them like different systems.

What the Interviewer Is Testing

The interviewer is looking for schema design, access paths, read/write trade-offs, scaling order, and consistency reasoning. A clean answer starts simple, then evolves only when the new constraint forces it.

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Relational databases store data in tables with enforced schemas and foreign-key relationships.
2. ACID is why they are trusted with money, inventory, and account state.
3. B-tree indexes change a table scan into a cheap lookup path.
4. Normalize first. Denormalize only when the workload forces it.
5. EXPLAIN ANALYZE tells you what the engine actually did, not what you hoped it did.
6. MVCC helps concurrency, but it creates cleanup work. That is why VACUUM matters.
7. Isolation levels are trade-offs, not moral categories.
8. Scale in layers before you jump to sharding.
9. PostgreSQL is the default answer for a reason.
10. Spanner shows what global SQL costs when you insist on strong consistency.

Key Intuition

Most relational database performance problems come back to three levers: indexes, cache residency, and WAL fsync. Indexes decide how much data a read has to touch. The buffer pool decides whether those reads hit RAM or disk. WAL flush decides how quickly a write becomes durable. If you keep those three in view, most slow-query stories get a lot less mysterious.

References & Further Reading

These are the sources to study if you want the machinery underneath a relational database: algebra, plans, snapshots, logging, recovery, and replication.

[1] Codd — "A Relational Model of Data for Large Shared Data Banks" (CACM 1970)

Edgar F. Codd. IBM Research. Communications of the ACM, Vol. 13, No. 6, June 1970, pp. 377-387.
https://www.seas.upenn.edu/~zives/03f/cis550/codd.pdf

• Codd defines a relation as a set of tuples over named domains. That immediately rules out any dependence on physical row order, which is why SQL needs ORDER BY when order matters.

• His treatment of nonsimple domains is the real normalization move: take repeating groups or nested records, split them into separate relations, and reconnect them with keys.

• Data independence is the part that still matters day to day. A query should keep the same meaning even if the engine swaps one file layout or index strategy for another.

• The algebra is small and concrete: restriction, projection, join. Physical operators come later. First you describe the result; then the engine decides how to get it.

• That split is the whole game in relational systems. Constraints define correctness. Physical design defines cost.

[2] Corbett et al. — "Spanner: Google's Globally-Distributed Database" (OSDI 2012)

James C. Corbett et al. Google Inc. 10th USENIX OSDI, October 2012.
https://research.google.com/archive/spanner-osdi2012.pdf

• TrueTime gives an interval, not a timestamp. Spanner treats clock uncertainty ε as a systems constant and builds commit logic around it instead of pretending clocks agree exactly.

• Commit wait is what turns that interval into external consistency. After picking commit timestamp s, Spanner waits until real time is definitely past s.

• The placement unit is a directory, basically a movable key-range shard. Split, move, and rebalance happen at that granularity, not at the level of the whole database.

• Each directory lives inside a Paxos group. Cross-directory transactions use 2PC on top of consensus-replicated participants, so prepare and commit state survives machine failure.

• The bill for global SQL is explicit here: time uncertainty, quorum writes, and cross-range coordination.

[3] PostgreSQL Documentation — Indexes, MVCC, WAL

The PostgreSQL Global Development Group.
https://www.postgresql.org/docs/current/indexes.html

• PostgreSQL’s default B-tree is page-based, so lookup cost is page traversal plus whatever heap fetches remain. That is why cache residency matters more than raw row count on hot key lookups.

• MVCC visibility comes from tuple metadata like xmin and xmax. Readers pick the version visible to their snapshot; they do not wait for writers to overwrite rows in place.

• The crash rule is WAL before data page. The heap page can hit disk later. The log record describing the change cannot.

• VACUUM is not housekeeping. It is how PostgreSQL stops dead tuples from piling up in scans, indexes, and visibility checks.

• HOT updates matter because they skip secondary-index churn when indexed columns do not change and the new tuple version still fits on the same page.

[4] Kleppmann — "Designing Data-Intensive Applications" (2017)

Martin Kleppmann. O'Reilly Media, 2017. Chapters 2, 3, 5, 6, 7 are most relevant here.
https://dataintensive.net/

• The schema chapters keep coming back to one hard tradeoff: join at read time or duplicate at write time. That is still the cleanest way to think about normalization versus denormalization.

• The storage-engine comparison between B-trees and LSM-trees is useful because it is framed in amplification terms: read, write, and space. That maps directly to production pain.

• The replication chapters give names to behaviors teams often discover the hard way: read-your-writes, monotonic reads, consistent prefix. If your replica setup cannot provide them, the app has to cope.

• The partitioning material is practical. Once data is sharded, the hard part is no longer just storage size. It is routing, rebalancing, and cross-shard work.

• The transaction chapters are worth rereading for one reason: they define isolation by which anomalies are still allowed. That is a much sharper tool than labels like "safe" or "strong."

[5] Hellerstein, Stonebraker, Hamilton — "Architecture of a Database System" (2007)

Joseph M. Hellerstein, Michael Stonebraker, James Hamilton. Foundations and Trends in Databases, Vol. 1, No. 2, 2007.
https://dsf.berkeley.edu/papers/fntdb07-architecture.pdf

• This paper breaks a DBMS into the pieces that matter under load: query processor, transaction manager, storage manager, process model. If latency is bad, one of those layers is usually where the explanation lives.

• The optimizer pipeline is spelled out cleanly: parse, rewrite, estimate cardinality, pick physical operators, build a plan. Wrong estimates are dangerous because they push the engine into the wrong join order or scan strategy.

• The storage chapters make one point very clearly: buffer pool, WAL, checkpoints, and page cleaning are coupled. You cannot reason about write latency or crash recovery by looking at any one of them in isolation.

• It also draws the line between locks and latches. Latches protect in-memory structures briefly. Locks protect logical data for the life of a transaction. Mixing those up causes confusion fast.

• It is still one of the clearest papers on what the engine is doing while your query runs.
