Key Terms

Databases

At their core, databases are specialized programs built to do two things: record data (writes) and query data (reads). They run as long-lived server processes that your application talks to over the network — typically via TCP-based protocols or HTTP APIs.

Some databases operate entirely in memory (like Redis or Memcached), and their users accept that all stored data can vanish if the process or machine dies. Most databases, however, need persistence — the guarantee that your data survives crashes. This requires writing to disk, since anything on disk remains intact through power failures and machine restarts.

In production environments where machines regularly fail, databases use dedicated disk partitions or network-attached volumes that can be recovered and re-mounted even if the original machine is permanently destroyed.

Disk

Refers to non-volatile storage — either HDD (hard-disk drive) or SSD (solid-state drive). The defining property: data written to disk survives power failures and machine crashes.

SSDs offer dramatically faster read/write speeds than HDDs (particularly for random access) but cost significantly more per gigabyte. The practical result: HDDs are used for infrequently accessed, long-retention data (archives, backups), while SSDs serve frequently accessed, latency-sensitive data (active databases, caches-on-disk).

Memory

Short for Random Access Memory (RAM). The defining property: data stored in memory is volatile — it disappears the moment the process writing it terminates or the machine loses power. What makes RAM invaluable is its speed: ~1,000× faster than SSD for random access.

Persistent Storage

Any storage medium that retains data even when the process managing it crashes or the machine powers down. Usually this means disk (HDD or SSD), but it also encompasses network-attached storage, object stores (S3, GCS), and replicated volumes in cloud environments.

01

Why Storage Matters in System Design

Every system design interview eventually comes down to this question: where does the data live? The answer determines your system's speed, cost, durability, and failure behavior. Choose wrong and you're either losing data or burning money — often both.

Storage is the bridge between computation (what the CPU does) and persistence (what survives when the power goes out). The fundamental tension is simple: fast storage is expensive and volatile (RAM), cheap storage is slow and persistent (HDD). Every system design decision in databases, caching, and data pipelines is navigating this tradeoff.

Core Intuition

Think of storage like a kitchen. Your countertop (RAM) is where you actively work — fast access, limited space, and everything gets cleared when you leave. Your refrigerator (SSD) keeps things fresh and accessible for longer. Your basement freezer (HDD) stores large quantities cheaply but takes time to get to. Your off-site warehouse (tape/cold storage) holds archives you rarely need. A well-designed kitchen uses all of them — not just one.

The Latency Gap Is Enormous

RAM is ~1,000× faster than SSD for random reads. SSD is ~50× faster than HDD. If RAM access were 1 second, SSD would be 17 minutes, and HDD would be 14 hours. This is why caching exists — moving hot data up the hierarchy saves orders of magnitude in latency. It's also why Jim Gray's "Five-Minute Rule" [2] remains one of the most important heuristics in storage engineering.

03

Memory (RAM) — Deep Dive

RAM (Random Access Memory) is where the CPU does its active work. Every running program, every variable, every data structure, every cache entry — all in RAM. The "random access" means any byte can be read in the same amount of time (~100ns), regardless of its location. This is in contrast to HDD where the read head must physically seek to the right position.

Key Properties

Volatile: RAM loses all data when power is lost. Kill the process → data gone. Reboot the machine → data gone. This is the defining characteristic that separates memory from persistent storage.

Byte-addressable: You can read or write any individual byte. No need to read a whole "block" or "page" (though CPU caches do operate in cache lines of 64 bytes).

Uniform access time: ~100ns for any address. This makes hash tables, trees, and random data structures practical. On HDD, random access is catastrophically slow.

Limited capacity: Typical servers have 64–512 GB of RAM. Large memory-optimized machines can reach multiple terabytes of RAM, but disk capacity still scales much farther and much more cheaply.

What Lives in RAM

Your application's heap and stack (variables, objects, function calls), the OS page cache (recently read disk blocks), database buffer pools (hot table pages), cache layers (Redis, Memcached — entirely in-memory), and the kernel's network buffers (TCP send/receive windows).

Why Not Just Put Everything in RAM?

Cost. DRAM is still orders of magnitude more expensive per gigabyte than HDD, and materially more expensive than SSD. Plus RAM is volatile — you'd lose everything on a power failure. The engineering challenge is figuring out which data should be in RAM (hot, frequently accessed) and which can stay on disk (cold, rarely accessed). This is exactly what the Five-Minute Rule [2] quantifies.

04

Disk — HDD vs SSD

HDD (Hard Disk Drive)

A spinning magnetic platter with a read/write head that physically moves to the correct position. Sequential reads are fast (~200 MB/s) because the head just follows the track. Random reads are slow (~5ms per seek) because the head must physically reposition. This mechanical limitation makes HDDs 50,000× slower than RAM for random reads.

Best for: Cold data, backups, data lakes, log archives — anything that's written once and read sequentially (or rarely).

SSD (Solid-State Drive)

No moving parts. Uses NAND flash memory cells that can be read electronically. Random reads are ~100µs (50× faster than HDD). Sequential reads reach 3.5+ GB/s on NVMe. However, SSDs have a write amplification problem and limited write endurance (each cell can only be written ~3,000–100,000 times depending on the type).

Best for: Database data files, frequently accessed data, write-ahead logs, OLTP workloads — anything with random access patterns.

Interview Insight — IOPS Matters More Than Throughput

When designing database storage, IOPS (I/O Operations Per Second) matters more than raw throughput for OLTP workloads. An HDD does ~150 random IOPS. An NVMe SSD does ~500,000. That's a 3,000× difference. This is why databases moved from HDD to SSD — not for the bandwidth, but for the random access performance. When you hear "the database is slow," the first question is often: "is it on SSD or HDD?"

Persistence vs Volatility

This is the core concept of storage in system design: volatile storage (RAM) is fast but loses data on failure. Persistent storage (disk) is slower but survives failures. The tension between these two defines how every database and storage system is built.

The same write can be fast-but-volatile in memory, or slower-but-durable once it is flushed to persistent storage

The fsync() Gotcha

Writing to a file doesn't mean the data is on disk. The OS buffers writes in the page cache (RAM) and flushes to disk asynchronously. If the machine crashes before the flush, your "written" data is lost. To guarantee persistence, you must call fsync() (or fdatasync()), which forces the OS to flush the buffer to physical storage. Databases like PostgreSQL call fsync on every transaction commit — that's what makes ACID durable. [3]

06

How Databases Use Storage

Databases are the most important consumers of storage in system design. Understanding how they use RAM and disk is essential. Most databases use a hybrid approach — keeping hot data in memory for speed while persisting everything to disk for durability.

The Write Path (simplified)

What happens when you INSERT a row
1. Client sends: INSERT INTO users (name) VALUES ('Alice')

2. WAL (Write-Ahead Log) — write to sequential log on disk
   → This is an APPEND to a file = sequential I/O = fast even on HDD
   → fsync() to guarantee durability
   → If machine crashes, WAL is replayed on restart

3. Buffer Pool — update the in-memory page
   → The actual table page is modified in RAM (not yet on disk)
   → This "dirty page" will be flushed to disk later (checkpoint)

4. Acknowledge to client: "Commit OK"
   → The data is durable (WAL is on disk) even though the table
      data file hasn't been updated yet

5. Background checkpoint — eventually flush dirty pages to data files
   → This is random I/O to disk (slower), but happens asynchronously

The Read Path (simplified)

What happens when you SELECT
1. Client sends: SELECT * FROM users WHERE id = 42

2. Buffer Pool check — is the page in RAM?
   → If YES: return from memory (~100ns) — "buffer hit"
   → If NO: read from disk (~100µs SSD, ~5ms HDD) — "buffer miss"

3. Disk read (on buffer miss)
   → Read the 8KB page from the data file
   → Load it into the buffer pool (evicting an old page if full)
   → Use LRU or clock-sweep eviction algorithm

4. Return the row to the client
The WAL Trick — Why Databases Are Fast

The Write-Ahead Log is the single most important storage optimization in databases. Instead of updating the table data file (random I/O, slow) on every write, the database appends to a sequential log (fast) and modifies the in-memory page. The data file is updated lazily in background checkpoints. This converts random writes → sequential writes, which is 100× faster on HDD and still 5-10× faster on SSD. WAL also enables crash recovery: replay the log to reconstruct any writes that weren't checkpointed. [4]

Mini-Project: Memory vs Disk Store

Build a Python key-value store with two backends — one in-memory (dict) and one on-disk (files) — to viscerally feel the persistence and performance difference.

kv_store.py — volatile vs persistent storage
import os, time
from http.server import HTTPServer, BaseHTTPRequestHandler

# ── Two storage backends ──
memory_store = {}                # Volatile: lives in RAM, dies with process
DISK_DIR = "/tmp/kv_disk_store"  # Persistent: survives process restarts
os.makedirs(DISK_DIR, exist_ok=True)

def mem_set(k, v): memory_store[k] = v
def mem_get(k): return memory_store.get(k)

def disk_set(k, v):
    with open(f"{DISK_DIR}/{k}", "w") as f:
        f.write(v)
        f.flush()
        os.fsync(f.fileno())  # Force write to physical disk!

def disk_get(k):
    try:
        with open(f"{DISK_DIR}/{k}") as f: return f.read()
    except FileNotFoundError: return None

class Handler(BaseHTTPRequestHandler):
    def do_POST(self):
        parts = self.path.strip("/").split("/")  # /memory/mykey or /disk/mykey
        backend, key = parts[0], parts[1]
        body = self.rfile.read(int(self.headers['Content-Length'])).decode()
        t0 = time.perf_counter_ns()
        if backend == "memory": mem_set(key, body)
        else: disk_set(key, body)
        elapsed = (time.perf_counter_ns() - t0) / 1000
        self.send_response(200)
        self.end_headers()
        self.wfile.write(f"SET {backend}/{key} in {elapsed:.1f}µs\n".encode())

    def do_GET(self):
        parts = self.path.strip("/").split("/")
        backend, key = parts[0], parts[1]
        t0 = time.perf_counter_ns()
        val = mem_get(key) if backend == "memory" else disk_get(key)
        elapsed = (time.perf_counter_ns() - t0) / 1000
        self.send_response(200)
        self.end_headers()
        result = "null" if val is None else val
        self.wfile.write(f"{result}  [{elapsed:.1f}µs]\n".encode())

    def log_message(self, *a): pass

print("[KV Store] http://localhost:8888")
print("  POST /memory/:key  — store in RAM (volatile)")
print("  POST /disk/:key    — store on disk (persistent)")
print("  GET  /memory/:key  — read from RAM")
print("  GET  /disk/:key    — read from disk\n")
HTTPServer(("", 8888), Handler).serve_forever()

What You'll Learn

Memory writes are usually much faster than disk writes that include fsync(), often by one or more orders of magnitude. But after killing and restarting the process, only the disk data survives. This is the fundamental tradeoff. Every database, cache, and storage system is a different answer to: "how much latency am I willing to pay for durability?"

Production Example 1: Redis

Redis is one of the most widely used in-memory data stores. It keeps all data in RAM for extreme speed (sub-millisecond latency), with optional disk persistence.

Storage Architecture

Primary store: RAM. All keys, values, data structures (strings, lists, sets, sorted sets, hashes) live entirely in memory. A Redis instance with 64GB RAM can hold ~64GB of data.

Optional persistence — two mechanisms:

RDB (snapshots): Periodically dumps the entire dataset to disk as a binary file. Fast to load on restart, but you lose data between snapshots (e.g., last 5 minutes).

AOF (Append-Only File): Logs every write command to disk (like a WAL). On restart, Redis replays the AOF to reconstruct the dataset. When AOF is enabled, Redis supports appendfsync always, everysec, and no; the recommended default AOF fsync policy is everysec. [5]

The Redis Tradeoff

Redis is fast because it's in-memory. But RAM is expensive and volatile. If the process dies without persistence, everything is gone. With AOF enabled and appendfsync everysec, Redis typically trades away up to about 1 second of recent writes in exchange for much better performance than fsyncing every command. For truly critical data, Redis is usually not the only store — it's a cache or fast-access layer in front of a more durable database. [5]

Production Example 2: PostgreSQL

PostgreSQL is the gold standard of persistent relational databases. It uses a sophisticated combination of RAM and disk to achieve both speed and durability.

Storage Architecture

Buffer Pool (shared_buffers): PostgreSQL keeps frequently-accessed table and index pages in a RAM cache. Typically configured to 25% of system RAM. Buffer hits avoid disk I/O entirely.

WAL (Write-Ahead Log): Every transaction is first written to the WAL (sequential disk write + fsync). This guarantees durability even if the machine crashes before the data files are updated.

Data Files: The actual table data on disk, stored as 8KB pages. Updated asynchronously by the background writer and checkpoint processes.

OS Page Cache: The OS also caches recently-read disk pages in RAM. PostgreSQL benefits from this "double caching" — a page might miss the buffer pool but still be found in the OS cache, avoiding physical disk I/O.

Key Insight — The Buffer Hit Ratio

In a read-heavy PostgreSQL system, engineers usually want a high buffer cache hit ratio so most reads come from memory instead of disk. You can inspect it with SELECT sum(blks_hit) / sum(blks_hit + blks_read) FROM pg_stat_database;, but the right target depends on workload, indexing, and whether the working set fits in memory. A low ratio is a signal to investigate RAM pressure, poor locality, or missing indexes rather than a universal "bad number." [3]

Edge Cases & Gotchas

1. The fsync() Lie

fsync() is only as trustworthy as the storage stack underneath it. Battery-backed write caches can make delayed persistence safe; dishonest drives or broken flush behavior can make it unsafe. PostgreSQL's own reliability documentation explicitly warns that durability depends on the correctness of the OS, filesystem, controller, and disk firmware underneath the database. [3]

2. SSD Write Amplification

SSDs can't overwrite individual bytes — they must erase entire blocks (128KB–512KB) before writing. A small 4KB write might trigger erasing and rewriting a full 512KB block. This "write amplification" reduces SSD lifespan and throughput under write-heavy workloads. Modern SSDs mitigate this with TRIM commands and wear-leveling algorithms.

3. OOM Killer

When Linux runs out of RAM, the Out-Of-Memory killer (OOM) selects and kills a process to free memory. If it kills your database process, you lose all in-memory state. Fix: configure vm.overcommit_memory, set appropriate memory limits, and ensure the DB process has an OOM score adjustment (oom_score_adj = -1000).

4. Page Cache Confusion

On Linux, free -h shows most memory as "used" because the OS aggressively caches disk blocks in RAM. New engineers often panic ("we're out of memory!") when in reality the "buff/cache" column is reclaimable. The "available" column is what matters.

5. Network Storage Latency

Cloud block storage (AWS EBS, GCP Persistent Disk, Azure Managed Disks) adds network hops on top of storage latency. Local NVMe is usually much lower latency; network-attached block storage is usually in the millisecond range instead of the low hundreds of microseconds. For latency-sensitive databases, instance-local NVMe can be attractive, but the tradeoff is operational: local storage disappears when the instance is terminated.

6. Cold Start / Empty Cache

After a database restart, the buffer pool is empty — every query hits disk. Performance can be dramatically worse until the cache "warms up." Some databases support buffer pool pre-warming (loading hot pages from disk on startup). PostgreSQL has pg_prewarm; MySQL has buffer pool dump/restore.

Production War Story

In 2017, GitLab suffered a major database incident that resulted in hours of downtime and roughly six hours of lost database data. The failure exposed gaps in backup coverage and restore readiness, not just in primary storage. The lesson: persistence is not only about fsync(); it is also about verified, tested backups across multiple storage tiers. A backup that has never been restored successfully is not a backup. [6]

Interview Questions & Answers

Q1: What's the difference between memory and disk? When would you use each?

Memory (RAM) is volatile, fast (~100ns random read), expensive (~$3/GB), and limited in capacity (typically 64–512 GB). Data is lost when the process or machine stops.

Disk (SSD/HDD) is persistent, slower (100µs–5ms random read), cheaper ($0.02–0.08/GB), and available in large capacities (TBs). Data survives power loss.

Use RAM for: Caches (Redis, Memcached), hot data in database buffer pools, application state that can be recomputed, session data. Use disk for: Database data files, WAL, logs, anything that must survive failures. Most real systems use both — RAM for speed, disk for durability.

Q2: How does a database achieve both speed and durability?

Through the Write-Ahead Log (WAL) pattern. On every write: (1) Append the change to a sequential log on disk (fast — sequential I/O), (2) fsync the log (guarantees durability), (3) Update the in-memory buffer pool (fast — RAM), (4) Acknowledge success to the client. The actual data files on disk are updated later by background checkpoint processes.

This achieves speed (writes go to RAM immediately) and durability (WAL on disk survives crashes). On crash recovery, the database replays the WAL to reconstruct any changes that weren't checkpointed. This pattern is used by virtually every serious database: PostgreSQL, MySQL, SQLite, MongoDB, CockroachDB.

Q3: What is fsync() and why does it matter?

fsync() is a system call that forces the OS to flush all buffered writes for a file descriptor to the physical storage device. Without it, a "write" might only update the OS page cache (RAM), not the disk. If the machine crashes, unflushed data is lost.

Databases call fsync() on WAL writes to guarantee durability. The cost: each fsync() can take 100µs–1ms (SSD) or 5–10ms (HDD), which limits throughput. This is why group commit exists — batch multiple transactions into a single fsync to amortize the cost.

Q4: SSD vs HDD — when would you still choose HDD?

HDDs still make sense for: (1) Large-scale cold storage — data lakes, backups, archives where data is rarely accessed. HDD at $0.02/GB vs SSD at $0.08/GB means 4× cost savings at petabyte scale. (2) Sequential workloads — HDDs achieve ~200MB/s sequential throughput, which is acceptable for log ingestion and batch processing. (3) Budget constraints — where the 50× random IOPS difference doesn't matter because the workload is read-once or sequential.

Choose SSD for: any workload with random access patterns (OLTP databases, key-value stores, search indexes), latency-sensitive applications, and anything where IOPS matters more than raw capacity.

Q5: A system design question says "the data must not be lost." What storage decisions does this imply?

This implies: (1) Persistent storage — data must be on disk, not just in memory. (2) Write-ahead logging — every write is logged to disk before acknowledging. (3) fsync on commit — flushes to physical medium, not just OS cache. (4) Replication — at least 2-3 copies across different machines/AZs to survive disk and machine failures. (5) Backups — periodic snapshots to a separate storage tier (S3, GCS) for disaster recovery.

The full durability stack: RAM (buffer pool) → WAL on local SSD (crash recovery) → Replicated to standby (disk failure survival) → Backed up to object storage (disaster recovery). Each layer adds latency and cost but reduces the probability of data loss.

Q6: What does the "Five-Minute Rule" say and why is it relevant?

Jim Gray's Five-Minute Rule (1987, updated in 1997, 2007, 2017) says: data that is accessed more frequently than every 5 minutes should be kept in RAM; data accessed less frequently should stay on disk. The break-even interval is computed from the cost of RAM per GB vs. the fractional cost of a disk's IOPS.

The rule has evolved: the original 5-minute interval for 1KB pages (RAM vs HDD, 1987) is now ~90 minutes for DRAM vs HDD (2017), but ~5 minutes for DRAM vs SSD. The rule matters because it gives you a quantitative framework for cache sizing: if your "hot" data fits in RAM and is accessed frequently, keep it there. If not, invest in faster disks rather than more RAM. [2]

The 10 Things You Must Know

1. RAM is fast (~100ns), volatile, expensive ($3/GB). Disk is slow (100µs–5ms), persistent, cheap ($0.02–0.08/GB).
2. Databases do two things: record (write) and query (read). Storage choice determines the speed and durability of both.
3. SSD is ~1,000× faster than HDD for random reads (IOPS), ~50× for sequential. Always use SSD for OLTP databases.
4. Persistence means surviving process/machine crashes. RAM alone is never persistent. Disk + fsync = persistent.
5. The Write-Ahead Log (WAL) pattern: append to sequential log + fsync → update memory → lazy checkpoint to data files. This is how databases achieve speed + durability.
6. fsync() forces data to physical storage. Without it, "writes" may only be in the OS page cache (still volatile!).
7. Most databases use a hybrid: RAM buffer pool for hot data + disk for durable storage. Aim for a high cache hit ratio, but judge it against the workload.
8. Redis = all-in-memory (fast, volatile by default). PostgreSQL = disk-primary with RAM cache (slower, durable).
9. Cloud storage (EBS, Persistent Disk) adds network latency. Local NVMe is faster but ephemeral.
10. Distributed storage adds replication, sharding, and specialized tradeoffs on top of these fundamentals.

Mental Model — Write Path Under Pressure

Database write path: WAL → fsync → buffer pool → async checkpoint, with pressure points
Key Intuition

The speed of writes is gated by the speed of fsync() to the WAL. Everything else (buffer pool update, data file checkpoint) happens in the background. The WAL converts random writes into sequential appends, and fsync() is the moment where "in memory" becomes "durable." This single bottleneck is why SSD vs HDD matters so much for write-heavy workloads, and why group commit exists.

References & Further Reading

[1] Latency Numbers Every Programmer Should Know

Jeff Dean, Google. Originally presented ~2009, widely updated. Compiled by Colin Scott.
https://colin-scott.github.io/personal_website/research/interactive_latency.html

• Quantifies the storage hierarchy: L1 cache (0.5ns), RAM (100ns), SSD random read (16µs in 2020), HDD seek (2ms in 2020).

• Shows how latency gaps have changed over decades — RAM has gotten faster at a different rate than disk, widening the gap.

• The source of the famous "if RAM were 1 second, disk would be X days" analogies used in every system design interview.

• Essential for napkin math — back-of-the-envelope calculations that interviewers expect you to do fluently.

[2] Gray & Putzolu — "The 5-Minute Rule for Trading Memory for Disk Accesses" (1987)

Jim Gray, Gianfranco Putzolu. ACM SIGMOD, 1987. Revised in 1997, 2007, and 2017.
https://jimgray.azurewebsites.net/5_min_rule_sigmod.pdf (1997 revision)
CACM 2019 — 30-year revision

• Provides a formula: BreakEvenInterval = (PagesPerMB_RAM / IOPS_Disk) × (PricePerDisk / PricePerMB_RAM). If data is accessed more frequently than this interval, cache it in RAM.

• In 1987, the break-even was ~5 minutes for 1KB pages (RAM vs HDD). In 2017, it's ~90 minutes for DRAM-HDD and ~5 minutes for DRAM-SSD.

• The 30-year revision shows HDD is becoming irrelevant for random-access workloads — the five-minute rule now applies to DRAM vs SSD.

• Jim Gray (Turing Award winner) predicted the storage hierarchy evolution with remarkable accuracy. His framework is still used for cache sizing decisions.

• Teaches the fundamental principle: storage engineering is economics — cost per access determines where data should live.

[3] PostgreSQL Documentation — Reliability and the Write-Ahead Log

The PostgreSQL Global Development Group.
https://www.postgresql.org/docs/current/wal-reliability.html

• Explains WAL reliability guarantees: each transaction commit writes to WAL and fsync()s before acknowledging to the client.

• Describes the fsync failure problem: if fsync fails, all dirty buffers in the OS page cache for that file become unreliable.

• Documents the full_page_writes mechanism — writing full page images on the first modification after a checkpoint to prevent partial-write corruption.

• Covers the interaction between PostgreSQL's buffer pool and the OS page cache — why both layers of caching exist and how they complement each other.

• Essential reading for understanding how a real production database achieves ACID durability using the storage hierarchy.

[4] Mohan et al. — "ARIES: A Transaction Recovery Method" (1992)

C. Mohan, Don Haderle, Bruce Lindsay, Hamid Pirahesh, Peter Schwarz. ACM TODS, Vol. 17, No. 1, March 1992.
https://cs.stanford.edu/people/chrismre/cs345/rl/aries.pdf

• The foundational paper on write-ahead logging for database crash recovery, used (in some form) by PostgreSQL, MySQL/InnoDB, SQL Server, and most modern databases.

• Describes three phases of recovery: Analysis (determine what was dirty at crash), Redo (replay all logged changes), Undo (roll back uncommitted transactions).

• Introduces physiological logging — logging physical page modifications at a logical level, balancing between full physical logging (too expensive) and pure logical logging (can't handle partial writes).

• The WAL "trick" explained in Section 6 of this document is a simplified version of the ARIES protocol.

• One of the most cited database papers. Understanding ARIES at a high level gives you deep credibility in database design interviews.

[5] Redis Documentation — Persistence

Redis Open Source Docs.
https://redis.io/docs/latest/operate/oss_and_stack/management/persistence/

• Documents Redis persistence modes: RDB snapshots, AOF, no persistence, or both together.

• Explains the three AOF fsync policies: always, everysec, and no.

• States that the suggested and default AOF fsync policy is everysec when AOF is enabled, balancing performance with durability.

[6] GitLab — Postmortem of Database Outage of January 31 (2017)

GitLab Engineering. February 2017.
https://about.gitlab.com/blog/postmortem-of-database-outage-of-january-31/

• Documents the accidental removal of data from GitLab.com's primary database and the resulting outage.

• Explains that backup and recovery procedures were incomplete or unreliable, which turned a bad production incident into a major data-loss event.

• A strong real-world reminder that durability requires not just storage, but verified restore paths and tested backups.
