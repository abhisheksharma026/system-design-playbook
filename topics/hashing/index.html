Hashing

Prerequisites & Key Terms

Hashing Function

A function that accepts an input of arbitrary size (a string, a file, a number) and produces a fixed-size output (typically an integer). A good hash function distributes outputs uniformly across its range, minimizing collisions (different inputs producing the same output). The output is deterministic — the same input always produces the same hash. Used everywhere: hash tables, checksums, caching keys, load balancer routing, and cryptography. Examples: MD5 (128-bit output, fast but broken for security), SHA-256 (256-bit, cryptographically secure), MurmurHash3 (fast, non-cryptographic, good distribution), xxHash (extremely fast, non-cryptographic).

Consistent Hashing

A hashing technique designed for distributed systems where the number of servers (buckets) changes frequently. In standard modulo hashing (hash(key) % N), changing N remaps nearly all keys. Consistent hashing maps both keys and servers onto a ring (circular hash space), and each key is assigned to the nearest server clockwise. When a server is added or removed, only ~K/N keys need to be remapped (where K is the total number of keys and N is the number of servers), rather than nearly all of them. Invented by David Karger et al. in 1997. [1] The idea shows up in systems like Cassandra, client-side Memcached sharding, cache-routing designs, and other distributed key-placement schemes.

Rendezvous Hashing

Also called Highest Random Weight (HRW) hashing. An alternative to consistent hashing that achieves the same goal (minimal key remapping on server changes) with a different mechanism. For each key, compute a score for every server using hash(key, server_id) and assign the key to the server with the highest score. When a server is removed, only its keys are redistributed — and they go to their second-highest-scoring server, which is deterministic. Simpler to implement than consistent hashing (no ring, no virtual nodes), but requires O(N) computation per key lookup (where N is the number of servers). Practical for small N (dozens of servers).

SHA (Secure Hash Algorithms)

A family of cryptographic hash functions published by NIST. SHA-1 (160-bit, deprecated — collision found in 2017 by Google/CWI). SHA-256 (256-bit, part of SHA-2 family, widely used in TLS, Bitcoin, Git). SHA-3 (based on Keccak, selected as the latest standard in 2015). For system design, SHA-256 is the standard choice when you need cryptographic properties (e.g., content addressing, integrity verification). For load balancing and caching, use faster non-cryptographic hashes like MurmurHash3 or xxHash.

01

What Is Hashing & Why It Matters

At its core, hashing converts data into a number. That number is used to determine where something goes — which bucket in a hash table, which server in a cluster, which shard in a database, which cache node holds this key.

Core Intuition

Imagine a mail sorting facility. Every letter has an address (the key), and you need to decide which delivery truck (server) carries it. The simplest approach: take the zip code, divide by the number of trucks, and use the remainder to assign a truck. That's modulo hashing. It works great — until you add or remove a truck, at which point almost every letter gets reassigned to a different truck. Consistent hashing solves this: it assigns delivery zones on a circular route, so adding a truck only shifts the letters in one zone.

Hashing appears in system design in three critical places: (1) Data partitioning — which database shard holds this row? (2) Load balancing — which server handles this request? (3) Caching — which cache node stores this key? In all three cases, the question is the same: given a key and a set of servers, which server is responsible?

Hash Functions — Properties & Types

Properties of a Good Hash Function

Deterministic: Same input always produces the same output. Uniform distribution: Outputs are spread evenly across the hash range, minimizing collisions. Avalanche effect: A small change in input (one bit flip) changes the output significantly. Fast computation: For system design (LB, caching), speed matters — non-cryptographic hashes can process GBs per second.

Hash FunctionOutput SizeTypeSpeed (approx.)Use Case
MD5128-bitCryptographic (broken)~400 MB/sChecksums, file deduplication (NOT security)
SHA-256256-bitCryptographic~200 MB/sTLS certificates, Git commits, Bitcoin, integrity verification
SHA-3 (Keccak)224–512 bitCryptographic~150 MB/sPost-quantum readiness, Ethereum
MurmurHash332 or 128-bitNon-cryptographic~3 GB/sHash tables, Bloom filters, consistent hashing
xxHash32, 64, or 128-bitNon-cryptographic~10+ GB/sHigh-throughput checksums, data processing
CRC3232-bitError-detection~5+ GB/s (hardware accelerated)Ethernet frame checks, storage checksums, file integrity
Cryptographic vs Non-Cryptographic

Cryptographic hashes (SHA-256, SHA-3) are designed to be collision-resistant and pre-image resistant — you can't reverse the hash to find the input, and it's infeasible to find two inputs with the same output. They're slower because security requires more computation. Non-cryptographic hashes (MurmurHash3, xxHash) prioritize speed and good distribution. They're fine for hash tables and load balancing — you don't need security guarantees, just even distribution. Rule of thumb: Use SHA-256 for security/integrity. Use MurmurHash3 or xxHash for everything else.

03

The Naive Approach: Modulo Hashing

The simplest way to assign keys to servers: server = hash(key) % num_servers. If you have 4 servers, key "user_42" might hash to 17, and 17 % 4 = 1, so it goes to Server 1.

Modulo hashing — works fine until a server changes
# 4 servers, 8 keys
hash("user_A") % 4 = 2  → Server 2
hash("user_B") % 4 = 0  → Server 0
hash("user_C") % 4 = 3  → Server 3
hash("user_D") % 4 = 1  → Server 1

# Now add a 5th server. EVERYTHING shifts:
hash("user_A") % 5 = 3  → Server 3  (was 2 — REMAPPED!)
hash("user_B") % 5 = 0  → Server 0  (same — lucky)
hash("user_C") % 5 = 4  → Server 4  (was 3 — REMAPPED!)
hash("user_D") % 5 = 2  → Server 2  (was 1 — REMAPPED!)

# On average, (N-1)/N = 3/4 = 75% of keys are remapped when going from 4→5 servers.
# For a cache: 75% of all cached data becomes unreachable → massive cache miss storm.
# For a database: 75% of queries go to the wrong shard → reads fail or return nothing.
Why This Is Catastrophic

In a distributed cache with modulo hashing, adding one server causes ~(N-1)/N of all keys to remap. With 100 servers, adding one remaps ~99% of keys. Every remapped key is a cache miss that hits the database. If your cache is absorbing 95% of reads, suddenly 95% × 99% ≈ 94% of all traffic floods the database simultaneously. This is the problem consistent hashing was invented to solve.

04

Consistent Hashing — The Ring

Consistent hashing (Karger et al., 1997) solves the remapping problem by mapping both keys and servers onto a circular hash space (a ring of integers from 0 to 2³²−1 or 0 to 2⁶⁴−1). [1]

How It Works

Step 1: Hash each server's identifier (hostname, IP) to a position on the ring.

Step 2: To find which server owns a given key, hash the key to a position on the ring, then walk clockwise until you encounter the first server. That server is responsible for this key.

Step 3: When a server is added, it takes a position on the ring and absorbs only the keys between it and its counter-clockwise neighbor — other servers are unaffected.

Step 4: When a server is removed, its keys are simply absorbed by the next server clockwise — again, other servers are unaffected.

Consistent hashing — what changes when a server is added
Before (3 servers on the ring):
Ring: 0 ──── A(100) ──── B(300) ──── C(600) ──── 0
Keys:
  "user_1" hashes to 50  → walks clockwise → hits A at 100
  "user_2" hashes to 200 → walks clockwise → hits B at 300
  "user_3" hashes to 400 → walks clockwise → hits C at 600
  "user_4" hashes to 700 → walks clockwise → wraps → hits A at 100

After adding server D at position 500:
Ring: 0 ──── A(100) ──── B(300) ──── D(500) ──── C(600) ──── 0
Keys:
  "user_1" → A (unchanged ✓)
  "user_2" → B (unchanged ✓)
  "user_3" → D (was C, now remapped to D)  ← only keys between B(300) and D(500)
  "user_4" → A (unchanged ✓)

Only keys in one segment of the ring moved. With N servers, adding one remaps ~K/N keys.
With 100 servers, adding one remaps ~1% of keys. Compare to modulo's ~99%!
05

Virtual Nodes — Fixing the Imbalance

With only N physical servers on the ring, the arc lengths between them can be very uneven — one server might own 50% of the ring while another owns 10%. This creates hot spots.

Virtual nodes (vnodes) solve this: instead of placing each server at one point on the ring, place it at many points (e.g., 100–200 virtual nodes per physical server). Each virtual node maps to the same physical server. With 100 vnodes per server and 10 servers, you have 1,000 points on the ring — the arcs become much more uniform.

Virtual nodes — how they improve balance
Without vnodes (3 servers, 3 ring positions):
Server A: owns 15% of ring     ← underloaded
Server B: owns 55% of ring     ← HOT SPOT!
Server C: owns 30% of ring

With 100 vnodes per server (3 servers, 300 ring positions):
Server A: owns ~33% of ring    ← balanced!
Server B: owns ~34% of ring    ← balanced!
Server C: owns ~33% of ring    ← balanced!

More vnodes = better balance, but more memory for the ring data structure.
Cassandra uses 256 vnodes per node by default.
Other large key-value systems use hashed partition spaces too, even if they do not expose a literal client-visible ring.
Why Not Just Use More Real Servers?

Virtual nodes also help with heterogeneous hardware. A powerful server can have 200 vnodes while a weaker server has 50, automatically giving the powerful server ~4× more traffic. And when a server is removed, its 100 vnodes are scattered across the ring — so its load distributes evenly among all remaining servers, not just its two ring neighbors.

Rendezvous Hashing

Rendezvous hashing (Thaler & Ravishankar, 1996) is an alternative to consistent hashing. Instead of a ring, it uses a scoring approach:

Rendezvous hashing algorithm
def pick_server_rendezvous(key, servers):
    """For a given key, compute a score for each server.
    Assign the key to the server with the highest score."""
    best_server = None
    best_score  = -1
    for server in servers:
        score = hash(f"{key}:{server}")   # Hash the (key, server) pair
        if score > best_score:
            best_score = score
            best_server = server
    return best_server

# When server X is removed:
# Only keys whose HIGHEST score was X are affected.
# Those keys now go to their SECOND-highest scoring server.
# All other keys are unchanged — same property as consistent hashing.
PropertyConsistent HashingRendezvous Hashing
MechanismRing + walk clockwiseScore all servers, pick highest
Lookup timeO(log N) with sorted ring, O(1) with jump hashO(N) — must score every server
Keys remapped on change~K/N~K/N (same!)
Implementation complexityHigher (ring, vnodes, binary search)Lower (loop over servers, compute hash)
Works well withLarge N (hundreds of servers)Small N (dozens of servers)
Used byCassandra, client-side Memcached sharding, cache routingProxy-cache routing and small-cluster placement

Mini-Project: Consistent Hash Ring

Simple: Feel What Consistent Hashing Does

consistent_hash.py — build a ring, add/remove servers, measure remapping (macOS/Linux/WSL)
import hashlib, bisect

class ConsistentHashRing:
    def __init__(self, servers, vnodes=150):
        self.ring = []        # Sorted list of (hash_value, server_name)
        self.vnodes = vnodes
        for s in servers:
            self.add_server(s)

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_server(self, server):
        for i in range(self.vnodes):
            h = self._hash(f"{server}:vnode{i}")
            bisect.insort(self.ring, (h, server))

    def remove_server(self, server):
        self.ring = [(h, s) for h, s in self.ring if s != server]

    def get_server(self, key):
        if not self.ring:
            return None
        h = self._hash(key)
        idx = bisect.bisect_right(self.ring, (h,))
        if idx == len(self.ring):
            idx = 0  # Wrap around the ring
        return self.ring[idx][1]

# ── Compare modulo vs consistent hashing ──
servers = [f"server_{i}" for i in range(5)]
keys = [f"key_{i}" for i in range(10000)]

ring = ConsistentHashRing(servers, vnodes=150)
before = {k: ring.get_server(k) for k in keys}

# Add a 6th server
ring.add_server("server_5")
after = {k: ring.get_server(k) for k in keys}

remapped = sum(1 for k in keys if before[k] != after[k])
print(f"Consistent hashing: {remapped}/{len(keys)} = {remapped/len(keys)*100:.1f}% remapped")
print(f"Modulo hashing would remap: ~{(5-1)/5*100:.0f}% = ~{int(len(keys)*4/5)} keys")
print(f"Consistent hashing expected: ~{100/6:.1f}% = ~{int(len(keys)/6)} keys")
Expected output
Consistent hashing: 1637/10000 = 16.4% remapped
Modulo hashing would remap: ~80% = ~8000 keys
Consistent hashing expected: ~16.7% = ~1667 keys

# ~16% vs ~80% — consistent hashing remaps 5× fewer keys.
# With 100 servers, it would be ~1% vs ~99%. The gap grows with scale.

Production Example 1: DynamoDB / Cassandra

Apache Cassandra uses a token ring directly, while DynamoDB uses a hashed partition-key space to route items into partitions. Both rely on hashing to keep key placement deterministic as capacity changes, but Cassandra exposes the ring model more explicitly.

How DynamoDB Uses Hash-Based Partitioning

Partition key hashing: DynamoDB hashes the partition key, then uses the hashed value to route the item into a physical partition range. Clients do not manage a ring; the service owns the partition map internally.

Automatic splitting: When a partition grows too large in size or throughput, DynamoDB splits the partition and reassigns part of that hash range to new storage. The point is the same as consistent hashing: capacity can grow without a global reshuffle of every item.

Replication: Each partition is synchronously replicated across Availability Zones. The key idea here is not clockwise replica placement on a public ring, but deterministic mapping from key hash to partition plus replicated storage behind that partition.

How Cassandra Uses Consistent Hashing

Token ring: Cassandra hashes partition keys using MurmurHash3 (by default) to a 64-bit token space. Each node owns a set of token ranges. Cassandra uses 256 vnodes per node by default to ensure even data distribution.

Replication factor: Data is replicated to the next RF-1 nodes clockwise on the ring (where RF is the configurable replication factor, typically 3).

Key Lesson

The shared systems lesson is deterministic key placement under membership change. Cassandra implements that as an explicit token ring with vnodes; DynamoDB implements it as a managed partition map over hashed key ranges. In both cases, the design goal is the same: grow capacity without forcing a full re-placement of the dataset.

Production Example 2: Google Maglev Hashing

Google's Maglev load balancer uses a custom variant of consistent hashing called Maglev hashing. [2]

Standard consistent hashing with virtual nodes can have uneven distribution — some servers may get 10-20% more traffic than others. Maglev hashing produces a lookup table where every entry maps to exactly one backend, and the table is generated so that each backend gets an exactly equal share of entries. When a backend is removed, only its entries are reassigned — minimizing disruption while guaranteeing perfectly even distribution.

Why Maglev Needed Custom Hashing

At Google's scale, even a 10% load imbalance across servers is a serious problem — it means some servers hit their capacity ceiling while others have headroom, reducing overall cluster efficiency. Maglev hashing achieves near-perfect balance (within 1% of ideal) while maintaining the minimal-disruption property of consistent hashing. The tradeoff: the lookup table must be regenerated when the backend set changes, which takes O(N × M) time (N=backends, M=table size).

Edge Cases & Gotchas

1. Hot Partition Key

Even with perfect consistent hashing, if one key is accessed far more than others (a celebrity's profile, a viral post), all traffic for that key goes to one server — creating a hot spot that no hash function can fix. Fix: split the hot key by appending a random suffix (e.g., user_celeb#0 through user_celeb#9) and scatter-gather reads across all shards.

2. Hash Collisions in Practice

With a 32-bit hash and 100K keys, the probability of at least one collision is ~69% (by the birthday paradox: 1 - e^(-n²/2m) where n=100K, m=2³²). For load balancing this is fine — collisions just mean two keys go to the same server. For deduplication or security, use 128-bit or 256-bit hashes where collisions are astronomically unlikely.

3. Non-Uniform Hash Functions

Some hash functions cluster outputs in certain ranges. If you're using a hash function that produces non-uniform output (e.g., Java's String.hashCode() has known clustering for short strings), your ring will have hot spots. Fix: use a well-tested hash function like MurmurHash3 or xxHash, or SHA-256 if you need cryptographic quality.

4. Ring Membership Propagation Delay

When a server joins or leaves the ring, all clients (LBs, application servers) must update their view of the ring. If one client updates before another, they'll disagree on key assignments, causing inconsistent reads. Fix: use a consensus service (ZooKeeper, etcd) to coordinate ring membership, and ensure all clients converge on the same ring state before serving traffic.

5. Cascading Failures on Ring

If Server A on the ring fails, all of A's keys go to the next server clockwise (Server B). If Server B was already near capacity, this extra load can cause B to fail too — and B's keys cascade to Server C, and so on. Virtual nodes mitigate this (A's load distributes across all servers, not just B), but it's still a risk. Fix: N+1 capacity planning so each server can absorb ~1/N of another server's load.

What If? — Interactive Exploration

Remapping Impact

Tradeoff Cards

More virtual nodes

Improves: Even load distribution (less variance between servers).
Costs: Larger ring data structure in memory, slightly slower lookups.
When worth it: Almost always. 100-200 vnodes per server is the standard. Below 50 vnodes, distribution can be notably uneven.

Consistent hashing vs rendezvous hashing

Consistent hashing: O(log N) lookup, needs ring + vnodes. Best for large N (hundreds of servers).
Rendezvous: O(N) lookup, trivial to implement. Best for small N (under ~50 servers).
Both: ~K/N keys remapped on server change. Choose based on cluster size.

Faster hash function

Improves: Per-request overhead. xxHash at 10+ GB/s vs SHA-256 at ~200 MB/s — 50× faster.
Costs: No cryptographic properties. Fine for routing, bad for security.
When worth it: Always for load balancing and caching. Only use SHA-256 when you need collision resistance or integrity.

Bounded-load consistent hashing

Improves: Hard cap on maximum server load (e.g., no server gets >125% of average).
Costs: Some keys may be routed to a non-nearest server on the ring (violates pure consistent hashing).
When worth it: When load variance is unacceptable. Used by Google (Vimeo's implementation is open-source).

Toy Simulation: Modulo vs Consistent vs Rendezvous

hash_compare.py — run all three, measure remapping (macOS/Linux/WSL)
import hashlib, bisect

def md5int(s):
    return int(hashlib.md5(s.encode()).hexdigest(), 16)

# ── Modulo hashing ──
def modulo_assign(keys, n):
    return {k: md5int(k) % n for k in keys}

# ── Consistent hashing (with vnodes) ──
def ch_assign(keys, servers, vnodes=150):
    ring = sorted((md5int(f"{s}:v{i}"), s) for s in servers for i in range(vnodes))
    hashes = [h for h, _ in ring]
    result = {}
    for k in keys:
        h = md5int(k)
        idx = bisect.bisect_right(hashes, h) % len(ring)
        result[k] = ring[idx][1]
    return result

# ── Rendezvous hashing ──
def rv_assign(keys, servers):
    result = {}
    for k in keys:
        result[k] = max(servers, key=lambda s: md5int(f"{k}:{s}"))
    return result

# ── Run comparison ──
N = 5
keys = [f"key_{i}" for i in range(10000)]
servers_before = [f"s{i}" for i in range(N)]
servers_after  = servers_before + ["s5"]

print(f"Adding 1 server to a {N}-server cluster (10,000 keys)\n")
for name, fn in [("Modulo", lambda k,s: modulo_assign(k, len(s))),
                  ("Consistent (150 vnodes)", lambda k,s: ch_assign(k, s)),
                  ("Rendezvous", rv_assign)]:
    before = fn(keys, servers_before)
    after  = fn(keys, servers_after)

    print(f"  {name:<30} remapped: {changed:>5} ({changed/100:.1f}%)")

print(f"\nIdeal (K/N):                    remapped: {10000//6:>5} ({100/6:.1f}%)")

Actual System Design Interview

The Kind of Prompt You Actually Hear

Design a distributed cache for user profiles. It needs to scale out, survive cache-node adds/removes, and not melt the database during rebalancing. This usually turns into a hashing discussion very quickly, but the interviewer is really probing whether you can reason about placement, churn, hot keys, and stale routing metadata. [1] [3] [4]

If I Were Answering This Out Loud

Candidate Notes

First thing I would pin down: what exactly is the routing key? If it is a profile cache, I am probably hashing user_id. If I cannot name the placement key, the rest of the answer gets fuzzy fast. [4]

Next I would choose where the ring lives: in the clients, or behind a proxy/router tier. Client-side routing saves a hop but now every app server needs an up-to-date membership view. Proxy-side routing is easier to coordinate, but it means the proxy layer becomes part of the control plane. [4] [5]

Then I would explain why modulo hashing is a trap: the first autoscaling event turns into a remapping storm. So I would switch to consistent hashing or rendezvous hashing and say out loud that I care about minimizing churn, not just placing keys “somewhere.” [1] [3]

Then I would volunteer the thing interviewers usually ask next: hot keys. A perfect hash still loses if one celebrity profile gets 1000× more traffic than the median. That is where key splitting, request coalescing, local L1 caches, or selective replication come in. [4] [5]

Finally I would talk about placement-map updates: whenever the membership changes, the routing map changes too. One version of the fleet may still think key user_123 belongs on the old node, while another version now sends it to the new node. In a cache, that creates misses, stale reads, or duplicate warming. In a primary data store, it is more dangerous because different clients can write the same key to different owners. So the real design question is not just “which hash function do we use?” but also “how do all clients converge on the same routing view quickly enough?” [4]

Where Interviewers Usually Push Back

Things They Will Not Let You Skip

Node added during peak traffic: yes, only ~K/N keys move, but those keys are cold on the new node. A good answer mentions warm-up, throttled migration, or enough spare capacity that the backing store survives the cold-start burst. [1] [5]

Heterogeneous nodes: equal key ownership is wrong if hardware is unequal. Say weighted virtual nodes or weighted rendezvous hashing and move on. [3]

Membership flapping: if health checks bounce, the ring keeps changing and the cache never settles. This is where hysteresis, draining, and conservative failure detection matter. [4]

Replication and read path: if a key exists on more than one node, say who is primary, who is backup, and what the read rule is. Otherwise “replication” is just a hand-wave. [4] [5]

Interview Questions & Answers

Q1: What is consistent hashing and why is it needed?

Consistent hashing maps keys and servers onto a circular hash space (ring). Each key is assigned to the nearest server clockwise on the ring. When a server is added or removed, only ~K/N keys are remapped (where K=total keys, N=servers). Without it, modulo hashing (hash % N) remaps ~(N-1)/N keys on any server change — devastating for caches and partitioned databases. Consistent hashing was invented by Karger et al. (MIT, 1997) for web caching and is now used by DynamoDB, Cassandra, Memcached, Akamai, and virtually every distributed system that partitions data by key.

Q2: What are virtual nodes and why do we need them?

With only N physical servers on the ring, the arc lengths between them can be very uneven — one server might own 50% of the key space. Virtual nodes solve this: each physical server is placed at 100-200 positions on the ring (each a "virtual node"). With 5 servers × 150 vnodes = 750 ring positions, the distribution becomes much more uniform. Vnodes also help with heterogeneous hardware (powerful servers get more vnodes → more traffic) and graceful failover (a removed server's load distributes across all survivors, not just one neighbor). Cassandra defaults to 256 vnodes per node.

Q3: Compare consistent hashing vs rendezvous hashing.

Both achieve the same goal: ~K/N keys remapped on server changes. Consistent hashing uses a ring with O(log N) lookup (binary search on sorted ring), needs virtual nodes for even distribution, and scales well to hundreds of servers. Rendezvous hashing scores every server per key, O(N) per lookup, trivially simple to implement (a loop), naturally even distribution (no vnodes needed), but impractical for very large N. Pick consistent hashing for large clusters (DynamoDB, Cassandra). Pick rendezvous hashing for small clusters (under ~50 servers) where simplicity matters.

Q4: What happens during a hash collision in distributed hashing?

In the context of load balancing / data partitioning, a "collision" means two different keys hash to the same position on the ring — they simply go to the same server. This is expected and harmless for routing purposes. For hash tables, collisions are handled via chaining (linked lists at each bucket) or open addressing (probing the next empty slot). For security contexts (password hashing, data integrity), collisions are dangerous — if an attacker can find two inputs with the same hash (collision attack), they can forge data. SHA-1 collisions were demonstrated in 2017 (Google/CWI SHAttered attack), which is why SHA-256 is now the minimum standard.

Q5: How would you design a distributed cache using consistent hashing?

Architecture: N cache servers (Redis/Memcached) on a consistent hash ring with 150+ vnodes each. Application servers have a client library that maintains the ring and routes GET(key) / SET(key, value) to the correct cache server. Adding a server: New server joins ring, absorbs ~K/N keys from neighbors. Those keys are cache misses on first access (cold start) — the DB handles the extra load briefly, then the new server warms up. Removing a server: Its keys shift to the next clockwise server — cache misses for those keys until repopulated. Replication: Optionally replicate each key to the next 2 servers on the ring for fault tolerance. Hot keys: Cache locally in app memory (L1) or replicate across multiple ring positions.

Q6: A distributed cache uses consistent hashing. After adding a new server, users report seeing stale data. Why?

When the new server joins, ~K/N keys are reassigned from their old server to the new one. The new server's cache is empty for those keys. If the old server is still alive and hasn't evicted those keys, you now have two servers with potentially different versions of the same data — the old server has a stale cached copy, and the new server has nothing (will fetch fresh from DB). But if the app client's ring view is slightly out of sync (propagation delay), some clients may still route those keys to the old server, getting stale data, while others route to the new server, getting fresh data. Fix: (1) Invalidate affected keys on the old server when ring membership changes. (2) Use a consensus service (ZooKeeper/etcd) to ensure all clients see the same ring state atomically. (3) TTL as a safety net — stale data expires eventually.

Q7: The interviewer says, “One node is twice as powerful as the others. What changes?”

Answer: Equal placement is no longer correct because equal ownership will underutilize the stronger node and overload the weaker ones. In a ring-based design, give the stronger node proportionally more virtual nodes. In rendezvous hashing, use weighted rendezvous so the stronger server wins a larger fraction of keys.

Thought process: First identify whether the imbalance is compute, memory, or network. Then make the placement algorithm capacity-aware. The key interview move is to say that load-balancing fairness is relative to server capacity, not raw key count.

Q8: “What if the hash function is perfect but one key is 1000× hotter than the rest?”

Answer: The problem is no longer poor distribution; it is skew. A perfect hash only balances the number of keys, not the request rate per key. Split the hot key into subkeys, replicate the hot object, add an L1 cache near the application, or introduce request coalescing so concurrent misses do not stampede the backing store.

Thought process: Say explicitly that hashing solves placement variance, not workload skew. Interviewers often ask this to see whether you confuse distribution quality with load quality.

Q9: “Clients update ring membership asynchronously. Is that acceptable?”

Answer: It depends on the workload. For an eventually consistent cache, a short period of disagreement may be acceptable if you tolerate misses and stale entries. For a partitioned primary data store, divergent views can route writes to the wrong owner and create correctness bugs.

Thought process: Classify the system first: cache, shard router, or authoritative datastore. Then explain the convergence mechanism: versioned ring metadata, leases, epoch numbers, and safe rollout windows. The real question is not “async or sync?” but “what correctness guarantees does this layer owe?”

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Hashing converts arbitrary data into fixed-size numbers. Used for hash tables, caching, load balancing, and data partitioning.
2. Modulo hashing (hash(key) % N) remaps ~(N-1)/N keys when N changes. Catastrophic for distributed caches.
3. Consistent hashing maps keys and servers to a ring. Adding/removing a server remaps only ~K/N keys.
4. Virtual nodes (100-200 per server) fix the uneven distribution problem on the ring. Cassandra defaults to 256.
5. Rendezvous hashing: score each server per key, pick highest. Same K/N remapping, O(N) lookup. Simple for small clusters.
6. Use non-cryptographic hashes (MurmurHash3, xxHash) for routing — they're 10-50× faster than SHA-256.
7. Use SHA-256 for security/integrity. SHA-1 is broken (collision demonstrated 2017). MD5 is broken (collision demonstrated 2004).
8. Hot partition keys can't be fixed by better hashing — they need key splitting or scatter-gather reads.
9. Cassandra, Memcached-style client sharding, cache-routing systems, and Maglev-style load balancers all rely on deterministic hash-based placement at production scale.
10. The birthday paradox: with a 32-bit hash, expect a collision after ~65K entries. Use 64-bit+ for large key spaces.

Key Intuition

Consistent hashing turns the "add a server, remap everything" problem into "add a server, remap 1/N of the data." This single property is what makes distributed caches, partitioned databases, and scalable load balancers possible. Without it, every scaling event would trigger a catastrophic cache miss storm. The ring is the most important data structure in distributed systems design.

References & Further Reading

Each source below is broken down by the formal property, disruption bound, or routing consequence you should take from it.

[1] Karger et al. — "Consistent Hashing and Random Trees" (STOC 1997)

David Karger, Eric Lehman, Frank Thomson Leighton, Rina Panigrahy, Matthew Levine, Daniel Lewin. MIT. ACM STOC, 1997, pp. 654-663.
https://www.cs.princeton.edu/courses/archive/fall09/cos518/papers/chash.pdf

• Defines a consistent hash function as one where the fraction of items remapped when a bucket is added/removed is at most O(1/N), proven via a monotonicity property: if a key maps to cache C in a set S, it still maps to C in any superset of S that contains C.

• Constructs the ring by hashing both objects and caches to points in [0, 1) using a universal hash family, assigning each object to the closest cache clockwise. With K objects and N caches, each cache holds at most O(K/N × log N) objects with high probability.

• Introduces random trees as an alternative to the ring — objects traverse a tree of caches, querying each level. The tree ensures O(log N) cache probes per request while bounding per-cache load to O(C/N × log N) where C is the total request volume.

• Proves a balance bound: with N caches and O(N log N) virtual replicas per cache on the ring, the maximum load per cache is within a (1 + ε) factor of the average, for any ε > 0. Without virtual replicas, the max/min ratio can be O(N).

• The adversarial model assumes an adversary controls which pages are requested but cannot see the hash function's random choices — the bounds hold against worst-case request sequences, not just average-case.

[2] Eisenbud et al. — "Maglev: A Fast and Reliable Software Network Load Balancer" (NSDI 2016)

Danielle E. Eisenbud, Cheng Yi, et al. Google Inc. 13th USENIX NSDI, 2016.
https://research.google.com/pubs/archive/44824.pdf

• Maglev hashing generates a fixed-size lookup table (default M=65537, a prime) where each entry maps to exactly one backend. The table is built by having each backend generate a permutation of [0, M-1], then backends take turns filling empty slots according to their permutation — ensuring every backend gets exactly M/N entries (perfect balance).

• On backend removal, only the removed backend's M/N entries are reassigned. The disruption is bounded: at most M/N entries change, matching consistent hashing's K/N guarantee but with zero load variance (every backend gets exactly equal slots).

• Maglev bypasses the Linux kernel network stack entirely — packets are read directly from the NIC into userspace via a custom ring buffer. A single 8-core machine achieves 12 million packets per second in benchmarks, saturating a 10Gbps link with small (100-byte) packets.

• Connection tracking uses a per-packet 5-tuple hash (src IP, dst IP, src port, dst port, protocol) to maintain affinity. If consistent hashing disagrees with the connection table (e.g., after a backend change), the connection table takes precedence to avoid breaking in-flight TCP sessions.

• Uses Direct Server Return (DSR): the backend sends response packets directly to the router, bypassing Maglev on the return path. This means Maglev only handles the smaller inbound packets, not the larger responses — critical for capacity since responses are typically 10× larger than requests.

[3] Thaler & Ravishankar — "Using Name-Based Mappings to Increase Hit Rates" (IEEE/ACM ToN 1998)

David G. Thaler, Chinya V. Ravishankar. IEEE/ACM Transactions on Networking, 6(1):1-14, 1998.
https://doi.org/10.1109/90.650137

• The assignment function server(key) = argmax_s { hash(key, s) } is proven to have the minimal disruption property: when server S is removed, only keys whose maximum-weight server was S are reassigned — they fall to their second-highest-weight server deterministically.

• Unlike consistent hashing, rendezvous hashing achieves uniform distribution without virtual nodes — each server is scored once per key, and since the hash function is random, each server wins with equal probability 1/N.

• The paper compares HRW to modular hashing and tree-based approaches for proxy cache routing, showing that HRW achieves 10-30% higher cache hit rates under server churn because fewer cache entries are invalidated during membership changes.

• The O(N) per-key computation cost is explicitly acknowledged as the tradeoff — acceptable for N in the dozens (proxy clusters) but impractical for N in the thousands. The paper suggests caching the assignment mapping to amortize this cost for frequently accessed keys.

• The hash function used must be jointly uniform over (key, server) pairs — the paper specifies that independent universal hashing suffices, and provides specific constructions using bitwise operations and primes that avoid expensive cryptographic hashes.

[4] Martin Kleppmann — "Designing Data-Intensive Applications" (2017)

Martin Kleppmann. O'Reilly Media, 2017. Chapters 6 and 7 are the most relevant here.
https://dataintensive.net/

• Distinguishes hash-based partitioning from key-range partitioning: hashing destroys locality to improve balance, while range partitioning preserves locality but risks skew. The operational point is that partition strategy encodes a tradeoff between balanced ownership and query shape.

• Treats rebalancing as a data-movement problem under load. Fixed partitioning, dynamic partitioning, and node-per-partition models all exist to control how much data must move when cluster membership changes.

• Calls out skew and hot spots as first-class failure modes: even if keys are evenly partitioned, one partition can dominate traffic. Hashing reduces average imbalance; it does not remove skewed-access pathologies.

• Emphasizes that a partitioning scheme also determines the shape of request routing metadata: clients either need a local partition map, a routing tier, or an indirection layer. In interviews, this is the hidden control-plane question behind “how do requests find the right shard?”

[5] Alex Xu — "System Design Interview" (Vol. 1 / Vol. 2)

Alex Xu. ByteByteGo, 2020 onward.
https://bytebytego.com/

• Uses consistent hashing repeatedly as an interview design primitive: caches, sharded data stores, and sticky-routing load balancers all need deterministic placement that survives node churn without global remapping.

• The practical interview pattern is: define the placement key, decide client-side versus proxy-side routing, choose replication, then explain what happens during scale-out, node failure, and hot-key skew. Those four follow-ups are where most hashing discussions become system-design discussions.

• Recommends discussing virtual nodes, metadata propagation, and rebalance windows explicitly, because the algorithm alone is not enough. The operational behavior around the ring is what interviewers usually grade.
