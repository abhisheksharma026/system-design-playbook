Key-Value Stores

A key-value store does one job well: you give it a key and it gives you a value. No joins. No SQL planner. No schema gymnastics. That narrow contract is why KV stores end up on the hot path of so many real systems.

Key Terms

Key-Value Store

A database with a very small API: GET(key) → value, PUT(key, value), DELETE(key). No tables, no joins, no SQL. The key is usually a string. The value can be raw bytes, JSON, or whatever your app wants to store. That narrow model is what makes KV stores fast enough for caches, sessions, rate limiting, feature flags, and some primary data paths.

Redis

An open-source, in-memory key-value store with more than plain strings. Redis gives you hashes, lists, sets, sorted sets, streams, and bitmaps on top of the basic KV model. It runs an event loop, keeps the common path in RAM, and usually responds in well under a millisecond. You reach for it for caching, sessions, counters, rate limiting, leaderboards, and a lot of "I just need this to be fast" problems.

etcd

A strongly consistent distributed key-value store built on Raft. A write is not committed until a majority of nodes agree on it, which is why etcd is trusted with cluster metadata, leader election, service discovery, and locks. The cost is easy to guess: quorum writes are slower and lower-throughput than a best-effort cache.

ZooKeeper

A strongly consistent hierarchical key-value store built around a tree of znodes. ZooKeeper is mostly a coordination system: leader election, locks, config, and service discovery. Older Kafka deployments leaned on it heavily for broker metadata and partition leadership. Newer systems often prefer etcd or other Raft-based tools because the API is simpler.

DynamoDB

Amazon's fully managed distributed key-value and document database. It takes a lot of the Dynamo-style operational pain off your hands: partitioning, replication, failover, and capacity management. Reads can be eventually consistent or strongly consistent depending on the workload. You pay for that convenience, but plenty of teams decide it is worth it.

01

What Is a Key-Value Store?

A key-value store is a database stripped down to one idea: map a key to a value. If you understand a Python dict or a JavaScript object, you already understand the data model.

The entire KV store API
PUT("user:42:session", '{"token":"abc","expires":1711900000}')
GET("user:42:session")  →  '{"token":"abc","expires":1711900000}'
DELETE("user:42:session")

That's it. No schema. No types. No joins. No SQL.
The key is a string. The value is opaque bytes — the DB doesn't care what's inside.
Core Intuition

A relational database is a library. A key-value store is a coat check. The coat check is much faster, but it can answer only one question: "what is behind ticket 42?" That trade-off is the whole topic.

That narrow API gives the implementation room to optimize hard. No join planner. No complex query parser. No relational constraints to check on every lookup. Just find the key and return the bytes.

The sorting question is simple: does the data need to survive a crash?

Persistence Options

RDB: periodic snapshots of the whole dataset. Restarts are fast, but you lose whatever happened after the last snapshot.

AOF: an append-only log of writes. Better durability, slower write path, and a replay cost on restart. The appendfsync setting is the real contract.

RDB + AOF: the usual production answer when you want both quicker restarts and a smaller loss window.

Redis is Single-Threaded (Mostly)

Redis executes commands in a single-threaded event loop. That keeps the common path simple and avoids lock contention. It also means one bad command, like KEYS * on a huge dataset, can stall everyone else. That trade is worth understanding because it explains both Redis's speed and its failure modes.

04

etcd / ZooKeeper — Strongly Consistent KV

When you need every read to see the latest committed write, even across machines, you are no longer in cache territory. You are in consensus territory.

etcd (Raft-based)

etcd stores data in a B-tree backed by BoltDB. Writes go through Raft, so the leader proposes an entry and a majority has to persist it before the write counts as committed.

Kubernetes depends on etcd for cluster state. If etcd is unhealthy, the control plane starts falling apart quickly because that metadata store is the source of truth.

ZooKeeper (ZAB-based)

ZooKeeper organizes data as a tree of znodes, closer to a filesystem than a flat KV map. Its ephemeral and sequential nodes are the parts people care about most because they make service discovery, locks, and leader election practical.

etcd vs ZooKeeper — When to Pick Which

etcd: simpler API, easier fit for modern control planes, especially around Kubernetes. ZooKeeper: older, battle-tested, and still useful when the tree model or ecosystem fit matters. Both: coordination systems first, not general-purpose high-throughput data stores.

05

DynamoDB — Managed, Distributed KV

DynamoDB is Amazon's managed KV and document store. [1] It handles partitioning, replication, failover, and most of the day-2 operational work that teams otherwise end up owning themselves.

Data Model

Partition key: the field that decides where the item lives. If you pick a bad one, you create hot partitions. Sort key: optional, but useful when you need ordered access within one partition, such as all orders for one user sorted by time.

Consistency Options

Eventually consistent reads: cheaper and usually good enough. Strongly consistent reads: fresher, more expensive, and worth using only when the stale-read window actually matters.

When to Use DynamoDB

Great for: serverless applications, user profiles, sessions, game state, and other workloads with a clean partition key and mostly key-based access. Not great for: ad-hoc analytics, awkward access patterns, or workloads that create hot partitions faster than DynamoDB can absorb them.

Mini-Project: KV Store from Scratch

Simple: Feel What a KV Store Does

kv_server.py — HTTP KV store in 30 lines (macOS/Linux/WSL, Python 3.7+)
from http.server import HTTPServer, BaseHTTPRequestHandler
import json, time

store = {}  # The entire "database" — a Python dict in RAM

class KV(BaseHTTPRequestHandler):
    def do_GET(self):
        key = self.path.lstrip("/")
        if key in store:
            val, exp = store[key]
            if exp and time.time() > exp:
                del store[key]
                self._respond(404, {"error": "expired"})
            else:
                self._respond(200, {"key": key, "value": val})
        else:
            self._respond(404, {"error": "not found"})

    def do_PUT(self):
        key = self.path.lstrip("/")
        body = json.loads(self.rfile.read(int(self.headers["Content-Length"])))
        ttl = body.get("ttl")
        exp = time.time() + ttl if ttl else None
        store[key] = (body["value"], exp)
        self._respond(201, {"key": key, "stored": True, "ttl": ttl})

    def do_DELETE(self):
        key = self.path.lstrip("/")
        deleted = store.pop(key, None) is not None
        self._respond(200 if deleted else 404, {"deleted": deleted})

    def _respond(self, code, data):
        self.send_response(code); self.send_header("Content-Type","application/json")
        self.end_headers(); self.wfile.write(json.dumps(data).encode())
    def log_message(self, *a): pass

print("[KV Store] http://localhost:8080/{key}")
HTTPServer(("", 8080), KV).serve_forever()
Test it with curl
# PUT a key with a 10-second TTL
$ curl -X PUT localhost:8080/greeting -d '{"value":"hello","ttl":10}'
{"key":"greeting","stored":true,"ttl":10}

# GET it
$ curl localhost:8080/greeting
{"key":"greeting","value":"hello"}

# Wait 10 seconds, try again
$ curl localhost:8080/greeting
{"error":"expired"}   ← TTL-based expiry, just like Redis SETEX!

# DELETE a key
$ curl -X DELETE localhost:8080/greeting
{"deleted":true}

Production Example 1: Amazon Dynamo

The original Amazon Dynamo paper still matters because so many later KV systems borrowed from it. [1] DynamoDB, Cassandra, Riak, and Voldemort all carry some of that DNA.

Key Design Decisions

Always writable: Dynamo leans toward availability. Even during partitions, it keeps accepting writes and deals with conflicts later through vector clocks.

Sloppy quorum + hinted handoff: if a preferred replica is down, the write goes to another healthy node and gets replayed later. That keeps the system moving, but it also means repair work never really goes away.

Merkle trees for anti-entropy: replicas compare tree hashes instead of scanning every key, which makes background repair practical at scale.

Production Example 2: Discord's Use of Redis

Discord uses Redis where low latency matters more than fancy query power.

How Discord Uses Redis

Presence: online status lives behind keys with TTLs. When heartbeats stop, the keys expire and the user looks offline without extra cleanup logic.

Rate limiting: API calls use Redis counters and expirations, which is a natural fit for fast shared state.

Message caching: the permanent history can live somewhere heavier, but the hot path only needs the recent messages now.

Edge Cases & Gotchas

1. Redis Memory Exhaustion

Redis stores everything in RAM. If your dataset exceeds available memory, Redis either rejects new writes (maxmemory-policy noeviction) or starts evicting keys based on LRU/LFU (allkeys-lru). If you don't set a maxmemory limit, Redis will use all available RAM and the OS will kill it (OOM). Always set maxmemory in production.

2. The KEYS * Command in Production

KEYS * scans every key in the database — O(N) and single-threaded. On a Redis instance with 10M keys, this blocks all other clients for seconds. Never use KEYS in production. Use SCAN instead — it iterates incrementally without blocking.

3. Hot Key on a Redis Cluster

In Redis Cluster, keys are distributed across slots (16384 slots). If one key gets extreme traffic (a viral post, a global config key), the single node owning that slot becomes the bottleneck. Fix: replicate the hot key to multiple nodes using read replicas, or use client-side caching for the hottest keys.

4. etcd Data Size Limits

etcd is designed for small metadata, not large datasets. The default maximum database size is 2GB. Each value is limited to 1.5MB. If you try to use etcd as a general-purpose KV store, you'll hit these limits quickly. etcd is for coordination metadata (K8s pod specs, config, leader locks) — use Redis or DynamoDB for application data.

5. DynamoDB Hot Partition

DynamoDB distributes data by hashing the partition key. If your partition key has low cardinality (e.g., status: "active" for 90% of items), most data lands on one partition → throttling. Fix: choose a high-cardinality partition key (user_id, device_id), or add a random suffix for write-sharding.

Production War Story

A startup put all user sessions in Redis and forgot to set maxmemory. As traffic grew, Redis consumed the host's RAM until Linux killed it. AOF was disabled, so every session vanished and everyone got logged out. The fix was boring but necessary: set a real memory limit, choose an eviction policy, and turn on persistence.

What If? — Interactive Exploration

Tradeoff Cards

Redis vs Memcached

Redis: Rich data structures, persistence, pub/sub, Lua scripting, cluster mode. More features, slightly more memory overhead per key.
Memcached: Multi-threaded (better per-instance throughput for simple GET/SET), simpler, less memory per key. No persistence, no data structures beyond strings.
Rule: Redis unless you need only simple caching and want maximum GET/SET throughput.

In-memory KV vs persistent KV

In-memory (Redis): Sub-ms latency, limited by RAM, data loss risk on crash (unless AOF).
Persistent (DynamoDB, Cassandra): 1-10ms latency, unlimited storage, durable by default.
Decision: "Does this data need to survive a server crash?" Yes → persistent. No → in-memory. Many systems use both: Redis as cache in front of DynamoDB as source of truth.

Strong consistency (etcd) vs eventual consistency (Redis Cluster)

Strong: Every read sees the latest write. Required for leader election, locks, K8s state. Costs: higher latency (Raft round-trips), lower throughput.
Eventual: Reads may be stale briefly. Acceptable for caching, sessions, counters. Benefits: higher throughput, lower latency, better availability during partitions.

KV store vs relational DB for user profiles

KV (DynamoDB/Redis): Fast single-key lookups. No joins. Schema-free. Horizontal scaling is trivial.
Relational (PostgreSQL): JOINs, complex queries, ACID transactions, strong consistency, but harder to shard.
Decision: If you only ever access profiles by user_id → KV is optimal. If you need "find all users in Berlin who signed up last week" → relational.

Interview Questions & Answers

Q1: When would you use a KV store instead of a relational database?

Use a KV store when access is mostly by key, speed matters a lot, and the data model does not need joins or rich query semantics. Stick with a relational database when the data is connected, the queries are richer, or the invariants span multiple entities. In practice, most systems use both: PostgreSQL for source of truth and Redis for the hot path.

Q2: How would you implement rate limiting with Redis?

Fixed window: use a key like ratelimit:{user_id}:{current_minute}, then INCR and EXPIRE. Sliding window log: use a Sorted Set, trim old timestamps, then count what remains. It is more accurate but costs more memory. Token bucket: keep a token counter and refill it at a steady rate, usually with a Lua script to keep the operation atomic.

Q3: What's the difference between etcd and Redis? When would you use each?

Redis: built for speed, great for caches, sessions, counters, and other hot-path data. etcd: built for correctness, better for leader election, cluster metadata, locks, and discovery. Redis is where you put fast-changing data you can tolerate losing briefly. etcd is where you put small pieces of metadata that have to be right.

Q4: Redis is single-threaded. How can it be so fast?

(1) All data in RAM — no disk I/O for reads. RAM access is ~100ns vs SSD at ~100μs (1000× faster). (2) Simple data structures — hash table lookup is O(1). No query parsing, no execution planning. (3) Non-blocking I/O — epoll/kqueue event loop handles thousands of connections without thread context switching. (4) No locks — single-threaded means no mutex contention, no cache-line bouncing. (5) Efficient protocol — RESP (Redis Serialization Protocol) is simple and fast to parse. The single-threaded model means one slow command (e.g., KEYS *) blocks everything — which is why KEYS is banned in production. But for typical GET/SET workloads, one CPU core can serve ~100K ops/sec.

Q5: How does DynamoDB handle hot partitions, and what can you do about it?

DynamoDB distributes items across partitions by hashing the partition key. If one partition key is accessed far more than others (e.g., a trending product page where partition_key = product_id), that partition's throughput capacity is exhausted while other partitions sit idle. DynamoDB will throttle requests to the hot partition (ProvisionedThroughputExceededException). Adaptive capacity (enabled by default since 2018) redistributes some unused capacity from cold partitions, but it doesn't eliminate the problem for extreme hot spots.

Fixes: (1) Write sharding: append a random suffix to the partition key (e.g., product_123#shard_7) and scatter-gather on reads. (2) Cache hot keys in Redis/DAX — absorb the read traffic before it hits DynamoDB. (3) Redesign the partition key — use a higher-cardinality key. Reference: DDIA Ch. 6, "Skewed Workloads and Relieving Hot Spots."

Actual System Design Interview — "Design a Rate Limiter"

This interview question is really about shared counters, atomic operations, and what happens when the fast path breaks.

The Prompt

"Design a rate limiter that enforces 100 requests per minute per user across a distributed system with multiple API servers. It should handle 50K RPS total and not become a bottleneck."

Distributed rate limiter: API servers share one Redis-backed counter per user per time window, so enforcement remains global instead of fragmenting across instances.

Interviewer Follow-ups

"Why Redis and not a local in-memory counter on each API server?"

With N API servers and a local counter on each, the user's traffic gets split across machines. That means the "100 requests per minute" limit quietly turns into something much larger. The count has to be shared if you want the answer to mean anything. Redis is the usual choice because INCR is atomic, EXPIRE is built in, and the latency stays low.

You could use a consensus system instead, but that is usually too expensive for a hot path like this. Rate limiting rarely needs perfect global truth. It needs a fast shared approximation that behaves well enough.

"What if Redis goes down? Do you block all requests or let them through?"

Fail open keeps the API up and temporarily gives up accurate rate limiting. Fail closed protects the backend but can turn a Redis outage into an application outage. Most public APIs choose fail open unless unchecked traffic is itself dangerous.

Mitigation: run Redis with replicas and automatic failover, then decide what the app should do during the short failover gap.

"Fixed window has a burst problem at window boundaries. How do you handle it?"

With a fixed window, someone can burst at the end of one minute and the start of the next. If that matters, switch to a sliding window log in a Sorted Set or a sliding window counter that weights the previous bucket. The former is more accurate. The latter is cheaper.

What They Are Looking For

The interviewer is checking whether you understand shared state, KV store choice, failure behavior, algorithm accuracy, and the operational details that turn a toy rate limiter into a production one.

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. KV stores do one thing well: map a key to a value.
2. Redis is the default hot-path store for caches, sessions, counters, and short-lived shared state.
3. etcd and ZooKeeper are coordination systems, not general-purpose app databases.
4. In DynamoDB, partition-key choice usually matters more than any other schema decision.
5. Redis is single-threaded on the command path, which is why it is both fast and easy to stall with a bad command.
6. RDB and AOF are different durability stories. Know what loss window each one implies.
7. If you do not set Redis memory limits and eviction behavior, the OS will eventually make that decision for you.
8. Rate limiting is a classic KV problem because it needs fast shared counters with expiration.
9. Dynamo is still worth reading because it shows what KV systems look like when availability wins the argument.
10. Most systems use BOTH: PostgreSQL (source of truth, ACID) + Redis (speed layer, caching). KV stores complement relational DBs, not replace them.

References & Further Reading

[1] DeCandia et al. — "Dynamo: Amazon's Highly Available Key-value Store" (SOSP 2007)

Giuseppe DeCandia, Deniz Hastorun, et al. Amazon. 21st ACM SOSP, October 2007.
https://www.allthingsdistributed.com/files/amazon-dynamo-sosp2007.pdf

• The paper's API is deliberately tiny: get(key) can return multiple versions plus a context, and put(key, context, value) writes a new version. That context is the vector clock; it is the metadata Dynamo uses to decide whether one write supersedes another or whether the versions are concurrent.

• Vector clocks are the core write-conflict mechanism. If clock A dominates clock B, B is obsolete. If neither dominates, Dynamo returns both versions to the caller. The shopping-cart example merges by set union, which shows the actual contract: the storage layer preserves causality, but conflict resolution can be pushed to application code.

• Sloppy quorum and hinted handoff are the availability path. Writes target the first N reachable nodes in the preference list, not strictly the ideal replicas. A temporary substitute node stores the write along with a hint about the intended owner, then replays it later. That is how Dynamo keeps accepting writes during node or rack failure without waiting for topology to heal first.

• Read repair and Merkle-tree anti-entropy are separate repair loops. Read repair fixes divergence only for keys that are being read. Merkle trees compare replica subranges by hash so background repair can locate a mismatching branch and ship only that slice instead of re-copying the entire partition.

• The engineering target is the tail: the paper explicitly optimizes for the 99.9th percentile, not the average. That matters because every mechanism in the design, from leaderless writes to replica selection, is there to keep a single slow node from dominating customer-visible latency.

[2] Redis Documentation — Data Types, Persistence, Cluster

Redis Ltd.
https://redis.io/docs/latest/

• The docs make Redis's execution model explicit: network I/O may be threaded, but command execution is still serialized through the main event loop. That is why INCR, HSET, and small Lua scripts behave atomically without application-side locking, and also why one long-running command can stall everyone behind it.

• The persistence pages separate RDB snapshots from AOF replay. RDB is a point-in-time dump produced from a fork()ed child with copy-on-write memory semantics. AOF records each write command in order, with durability controlled by appendfsync always|everysec|no. Those are different failure envelopes, not just different file formats.

• The cluster docs spell out the actual shard map: 16384 hash slots, assigned by CRC16(key) mod 16384. Multi-key atomicity only exists when the keys land on the same node, which is why hash tags like {user:42}:profile matter. Replication is async, so a primary can ACK a write that is still absent from its replica.

• The data-structure docs are worth reading as implementation notes. A Sorted Set combines a hash table for member lookup with a skip list for ordered traversal, which is why ZADD, ZRANK, and score-range queries stay efficient on the same object.

Martin Kleppmann. O'Reilly Media, 2017. Chapters 5, 6, 8 most relevant.
https://dataintensive.net/

• Chapter 5 formalizes the leaderless replication model behind Dynamo-style systems. The useful detail is not just R + W > N, but the failure cases around it: overlapping quorums can still return stale data if replicas miss writes, clocks disagree, or concurrent versions need reconciliation.

• Chapter 6 is the cleanest explanation of partitioning trade-offs. Hash partitioning gives load spread but destroys key locality. Range partitioning preserves ordered scans and prefix locality but turns monotonically increasing keys into hot tablets. That one chapter explains most partition-key mistakes people make in DynamoDB, Cassandra, and Bigtable-like systems.

• Chapter 8 is where the book gets precise about locks and leases. A lock in etcd or ZooKeeper is only safe if downstream systems reject stale holders, which is where fencing tokens come in. Without that extra monotonic token, a paused client can resume after its lease expired and still damage shared state.

• The consistency chapters also cleanly separate linearizability, serializability, and eventual consistency. For KV stores that distinction matters because many systems give you per-key ordering, but not a single serial history across keys, partitions, or replicas.

[4] Ongaro & Ousterhout — "In Search of an Understandable Consensus Algorithm (Raft)" (2014)

Diego Ongaro, John Ousterhout. USENIX ATC, 2014.
https://raft.github.io/raft.pdf

• Raft defines the commit rule that etcd relies on: a leader appends a log entry, replicates it to followers, and only marks it committed once a majority has stored it. An ACK before that point is not durable; an ACK after that point survives follower loss.

• The safety section is the important part for KV systems. Election restriction prevents a node with an older log from becoming leader, and log matching guarantees that if two entries share an index and term, their entire prefix is identical. That is why a read-after-write system can trust the replicated log as the source of truth.

• Terms, heartbeats, and AppendEntries are not background protocol trivia; they are the exact mechanism that converts a cluster of disks into one ordered write stream. This is the paper to read if you want to understand why etcd is good at tiny metadata writes and bad at cache-like request rates.
