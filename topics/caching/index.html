Caching

The fastest database query is the one you never make. Caching is the art of keeping the right data in the right place so your system can answer in microseconds instead of milliseconds — and the engineering discipline of knowing when that shortcut will betray you.

Key Terms

Cache

A fast storage layer that sits between the requester and the source of truth. Its job is to serve frequently requested data without hitting the slower source every time. Caches show up everywhere, from CPU L1 to CDN edges. The trade-off is always the same: speed in exchange for possible staleness.

Cache Hit

The requested data is already in the cache, so the system serves it directly and skips the slower source. This is the good path. It is why caches exist in the first place.

Cache Miss

The requested data is not in the cache. The system must fall back to the slower source (database, API, disk), retrieve the data, serve it, and typically populate the cache for future requests. Individual misses are usually slower than going directly to the source because you paid the cache-lookup cost plus the source-lookup cost.

Cache Hit Rate

The percentage of requests served from the cache: hits / (hits + misses) × 100. Strong caches often achieve very high hit rates, but the right target depends on workload shape, cache size, and staleness tolerance. If the hit rate stays low, the cache may not justify its complexity because the backend still handles most of the load.

Cache Eviction Policy

The rule that decides which entries to remove when the cache reaches capacity. Common policies: LRU (Least Recently Used — evict whatever hasn't been touched in the longest time), LFU (Least Frequently Used — evict whatever is accessed least often), FIFO (First In, First Out — evict the oldest entry), and TTL (Time-To-Live — entries auto-expire after a fixed duration).

Content Delivery Network (CDN)

A globally distributed network of edge servers that cache static content (images, CSS, JS, videos) close to end users. Instead of every request traveling to your origin server across the world, the CDN serves it from a nearby Point of Presence (PoP). Reduces latency from ~150ms (cross-continent) to ~10-30ms (nearby PoP). Major CDNs include Cloudflare, AWS CloudFront, Fastly, and Akamai.

Stale Data

Data in the cache that no longer matches the authoritative source. This happens when the source is updated but the cache still holds the old version. Managing staleness — how much you tolerate, how you detect it, how you fix it — is the central challenge of caching.

01

Why Caching Matters

Caching is the single most impactful performance optimization in system design. The numbers tell the story: a Redis cache responds in ~0.1ms. A PostgreSQL query on SSD takes ~5ms. A PostgreSQL query that misses the buffer pool and hits HDD takes ~20ms. That's a 50–200× speed improvement for every cache hit.

But caching isn't just about speed — it's about protecting your backend. If the steady-state read load grows far beyond what the database can sustain directly, a well-placed cache can absorb a large fraction of that traffic and keep the database within a workable range. Caching often turns an otherwise unsustainable read pattern into a manageable one.

Core Intuition

Think of a librarian. Every time someone asks for a popular book, the librarian walks to the back shelves, finds it, and brings it up front. After the 50th person asks for the same bestseller, the librarian just keeps a copy on the front desk. That front desk copy is a cache — it serves the common case instantly, while rare requests still go to the back. The trick is knowing which books belong on the front desk, when to swap them out, and what to do when the publisher releases a new edition.

The Multi-Layer Cache Stack

A single HTTP request might pass through 5+ cache layers: Browser cache (instant, no network) → CDN edge (~10ms) → Application cache (in-process) → Redis (distributed) → Database buffer pool (in DB RAM) → OS page cache (kernel) → Disk. Each layer is a chance to short-circuit the request before it hits the expensive source. A well-designed system has most requests satisfied by the first or second layer.

03

Caching Strategies (Read & Write Patterns)

How does data get into and out of the cache? There are several distinct patterns, each with different consistency, performance, and complexity trade-offs.

Read Strategies

Cache-Aside (Lazy Loading)

The application manages the cache explicitly. On read: check cache → if miss, query DB → store result in cache → return. On write: update DB → invalidate (delete) cache entry. The cache only contains data that has actually been requested — no wasted space on unpopular data.

Cache-Aside Pattern
def get_user(user_id):
    # 1. Check cache first
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)   # Cache HIT → ~0.1ms

    # 2. Cache miss → query database
    user = db.query("SELECT * FROM users WHERE id = %s", user_id)  # ~5ms

    # 3. Populate cache for next time (TTL = 5 minutes)
    redis.setex(f"user:{user_id}", 300, json.dumps(user))

    return user

def update_user(user_id, data):
    db.execute("UPDATE users SET ... WHERE id = %s", user_id, data)
    redis.delete(f"user:{user_id}")  # Invalidate — next read will repopulate

Read-Through

The cache itself is responsible for loading data on a miss. The application only talks to the cache — never directly to the DB. The cache layer handles miss → DB query → populate automatically. Simpler application code, but the cache must know how to reach the DB.

Refresh-Ahead

The cache proactively reloads entries before they expire, based on access patterns. If a key is accessed frequently and its TTL is about to expire, the cache pre-fetches the new value in the background. Eliminates cache-miss latency spikes for hot keys, but wastes resources refreshing entries that might not be requested again.

The Most Common Pattern

Cache-Aside + Write-Around with TTL is one of the most common production patterns. Facebook uses a demand-filled look-aside approach at scale: reads go through cache, writes go directly to the database, and the cache key is invalidated with a delete. A TTL provides a safety net — even if an invalidation is missed, the stale entry expires automatically. [1]

Eviction Policies — When the Cache Is Full

Caches have finite memory. When full, something must be evicted to make room. The eviction policy determines what gets removed — and a bad policy can destroy your hit rate.

PolicyEvictsReal-World ProductsUse Case / When to Pick
LRULeast Recently UsedRedis (allkeys-lru), Memcached, Linux page cache, PostgreSQL buffer pool (clock-sweep ≈ LRU), Nginx proxy cacheDefault choice for most workloads. Web session caches, API response caches, database buffer pools. Works because recency correlates with future access in most web traffic.
LFULeast Frequently UsedRedis 4.0+ (allkeys-lfu)When popularity matters more than recency. CDN-like asset caching and product catalog caches where some keys are dramatically hotter than others.
FIFOFirst In, First OutKafka consumer offset cache, OS write-back buffers, simple embedded systemsWhen data has a natural time ordering. Log buffers, streaming pipelines, circular buffers for sensor data. When you don't need smart eviction — just roll through in order.
TTLExpired entries (time-based)DNS resolvers (TTL per record), HTTP Cache-Control: max-age, CloudFront/Cloudflare edge TTL, Redis key expirationWhen staleness must be bounded. DNS records, authentication tokens, session data, any cache where "how old is this?" matters more than "how popular is this?" Almost always combined with another policy.
RandomRandom entryARM CPU caches (some configurations), Redis (allkeys-random), academic benchmarksWhen tracking overhead must be zero. CPU hardware caches with extreme clock-speed constraints. Also useful as a baseline for benchmarking other policies. Surprisingly competitive for uniform access patterns.
W-TinyLFUAdmission filter + segmented LRUCaffeine (Java, used by Spring Boot, Apache Spark, Neo4j), Ristretto (Go), Moka (Rust)When you need near-optimal hit rates. Application-level caches in JVM services. Combines LRU recency with LFU frequency tracking via a compact sketch, plus an admission filter that rejects "one-hit-wonder" keys. Interview Insight — Why LRU Dominates

LRU works well because most real workloads follow a power-law distribution (Zipf's law) — a small fraction of keys receive the vast majority of requests. LRU naturally keeps these hot keys alive. Its weakness is sequential scans (iterating through a dataset once evicts all the hot keys), which is why PostgreSQL uses a clock-sweep (approximated LRU) in its buffer pool — it resists scans while still evicting cold pages.

05

Cache Invalidation — The Hard Problem

Phil Karlton famously quipped: "There are only two hard things in Computer Science: cache invalidation and naming things." He wasn't joking. Keeping a cache consistent with its source is genuinely one of the hardest problems in distributed systems.

Invalidation Strategies

TTL-based expiration: Every entry has a time-to-live. After expiration, the next request triggers a fresh load from the source. Simple and robust — staleness is bounded. The trade-off: shorter TTLs mean less staleness but more cache misses; longer TTLs mean better hit rates but more potential staleness.

Event-driven invalidation: When the source data changes, an event (DB trigger, application hook, message queue) tells the cache to delete the affected key. More precise than TTL — data is stale only for the brief window between the write and the invalidation arriving. But: distributed invalidation is hard to make reliable (messages can be lost, delayed, or arrive out of order).

Version-based / ETag: Each cache entry stores a version number or hash. On read, the cache sends the version to the source, which responds with either "still valid" (304 Not Modified — no data transfer) or the new data. Used in HTTP caching (ETag / If-None-Match headers).

The Race Condition That Haunts Every Cache

Thread A reads from DB (gets v1), Thread B updates DB to v2 and deletes cache, Thread A writes v1 to cache. Now the cache holds stale v1 while the DB has v2 — and no invalidation is coming because the delete already happened. Facebook solves this with lease tokens: when a cache miss occurs, the cache issues a lease. If a delete arrives before the lease is filled, the fill is rejected. This tiny mechanism prevents the most common race condition at scale. [1]

CDNs — Caching at the Edge

A Content Delivery Network is a globally distributed cache. Instead of every user request traveling to your origin server (potentially across the globe), the CDN serves cached content from the nearest edge server — called a Point of Presence (PoP).

What CDNs Cache

Static assets: Images, CSS, JavaScript, fonts, videos. These rarely change and are perfect for long TTLs (hours to days).

Dynamic content (selectively): API responses that are the same for many users (e.g., product catalog, trending posts). Requires careful TTL and cache-key design.

Full page HTML: For content sites (blogs, news), the entire HTML page can be cached at the edge. Frameworks like Next.js use ISR (Incremental Static Regeneration) to serve cached HTML while revalidating in the background.

How CDN Caching Works

CDN request flow
User in Berlin requests: https://cdn.example.com/image.png

1. DNS resolves cdn.example.com → nearest PoP (Frankfurt)
2. PoP checks its local cache
   HIT  → serve from edge directly (~10ms)
   MISS → PoP fetches from origin server (Virginia, ~150ms)
          → caches the response for future requests
          → serves to user

3. Subsequent requests from anyone near Frankfurt → cache HIT
   One origin fetch serves thousands of edge-cached responses.

Cache control via HTTP headers:
  Cache-Control: public, max-age=86400    ← cache for 24h

  Cache-Control: s-maxage=3600             ← CDN caches 1h, browser doesn't

Mini-Project: Feel What Caching Means

Before getting into LRU, start with the smallest useful example: one route always hits the "database" and one checks a local dictionary first. You can feel caching immediately when the slow path and fast path sit side by side.

cache_feel.py — run with: python3 cache_feel.py (macOS/Linux/WSL)
import time, json
from http.server import ThreadingHTTPServer, BaseHTTPRequestHandler

# ── Simulated "database" — always takes 200ms to respond ──
def database_get(page):
    time.sleep(0.2)  # Simulate slow disk/network I/O
    return f"<h1>Welcome!</h1><p>Page: {page} | Generated at {time.strftime('%H:%M:%S')}</p>"

# ── The cache — a plain Python dict (lives in RAM, dies with process) ──
cache = {}

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        t0 = time.perf_counter()

        if self.path.startswith("/nocache/"):
            # ── NO CACHE: always hit the "database" ──
            page = self.path.split("/")[-1]
            body = database_get(page)
            source = "DATABASE (slow)"

        elif self.path.startswith("/withcache/"):
            # ── WITH CACHE: check dict first, fall back to "database" ──
            page = self.path.split("/")[-1]
            if page in cache:
                body = cache[page]
                source = "CACHE (fast)"
            else:
                body = database_get(page)
                cache[page] = body  # Populate cache for next time
                source = "DATABASE → cached"
        else:
            body = "<p>Try /nocache/home or /withcache/home</p>"
            source = "--"

        elapsed = (time.perf_counter() - t0) * 1000
        body += f"<p style='color:gray;font-size:12px'>Source: {source} | {elapsed:.1f}ms</p>"

        self.send_response(200)
        self.send_header("Content-Type", "text/html")
        self.end_headers()
        self.wfile.write(body.encode())

    def log_message(self, *a): pass

print("[Cache Demo] http://localhost:7070")
print("  /nocache/home     — always hits 'database' (~200ms)")
print("  /withcache/home   — first hit: slow, then cached (~0.1ms)\n")
ThreadingHTTPServer(("", 7070), Handler).serve_forever()
Try it — feel the difference

$ python3 cache_feel.py

# Terminal 2: Hit the no-cache route (always slow)
$ curl -s localhost:7070/nocache/home
# → ~200ms every single time. Open in browser, refresh 10×. Always slow.

# Now hit the cached route
$ curl -s localhost:7070/withcache/home
# → First request: ~200ms (cache miss, had to query "database")
# → Second request: ~0.1ms (cache hit! 2000× faster)
# → Every subsequent request: ~0.1ms

# Notice the staleness: the "Generated at" timestamp never changes
# on the cached route. That's the tradeoff — speed vs freshness.
# Kill the server (Ctrl+C), restart → cache is gone (volatile RAM).
What You Just Learned

With 5 lines of caching logic (a dict + an if-check), you turned a 200ms endpoint into a 0.1ms endpoint — 2,000× faster. But the cached page is now frozen in time (stale). And if you restart the server, the cache evaporates (volatile). These two problems — staleness and volatility — are the entire reason caching is a topic with its own chapter.

Advanced: LRU Cache from Scratch

Now let's build a proper LRU cache with a capacity limit and eviction — then benchmark it against a Zipf-distributed workload to understand why eviction policies matter.

lru_cache.py — build, use, and benchmark
from collections import OrderedDict
import time, random

class LRUCache:
    """Least Recently Used cache with fixed capacity."""
    def __init__(self, capacity: int):
        self.cache = OrderedDict()
        self.capacity = capacity
        self.hits = 0
        self.misses = 0

    def get(self, key):
        if key in self.cache:
            self.cache.move_to_end(key)  # Mark as recently used
            self.hits += 1
            return self.cache[key]
        self.misses += 1
        return None

    def put(self, key, value):
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            self.cache.popitem(last=False)  # Evict LEAST recently used

    def hit_rate(self):
        total = self.hits + self.misses
        return (self.hits / total * 100) if total > 0 else 0

# ── Simulate a Zipf-distributed workload ──
# Most requests go to a few "hot" keys (like a real web app)
N_KEYS = 10_000
N_REQUESTS = 100_000

# Generate Zipf-like access pattern
keys = [int(random.paretovariate(1.2)) % N_KEYS for _ in range(N_REQUESTS)]

# Test different cache sizes
for cache_pct in [1, 5, 10, 20, 50]:
    cap = int(N_KEYS * cache_pct / 100)
    cache = LRUCache(cap)
    t0 = time.perf_counter()
    for k in keys:
        if cache.get(k) is None:
            cache.put(k, f"value_{k}")   # Simulate DB fetch + populate
    elapsed = (time.perf_counter() - t0) * 1000
    print(f"  Cache size: {cap:5d} ({cache_pct:2d}% of keys) → hit rate: {cache.hit_rate():5.1f}% [{elapsed:.0f}ms]")

print(f"\nWith Zipf distribution, caching just 10% of keys captures ~90% of requests!")
Expected output
  Cache size:   100 ( 1% of keys) → hit rate: 62.3%  [45ms]
  Cache size:   500 ( 5% of keys) → hit rate: 82.7%  [38ms]
  Cache size:  1000 (10% of keys) → hit rate: 90.1%  [32ms]
  Cache size:  2000 (20% of keys) → hit rate: 95.4%  [28ms]
  Cache size:  5000 (50% of keys) → hit rate: 99.2%  [25ms]

With Zipf distribution, caching just 10% of keys captures ~90% of requests!
What You'll Learn

The output demonstrates two critical insights: (1) Zipf-like skew means a relatively small cache can capture most requests in some workloads. In this toy run, caching 10% of keys reaches roughly a 90% hit rate. (2) Diminishing returns — going from 10% to 50% cache size only gains a modest additional hit-rate improvement. In interviews, this is the argument for "don't cache everything, cache the hot set."

Production Example 1: Facebook Memcached

The Facebook memcache paper describes what was, at the time, the world's largest memcache deployment: billions of requests per second, trillions of cached items, and support for over a billion users. [1]

Architecture

Demand-filled look-aside cache: Web servers check Memcached first. On miss, they query MySQL and populate the cache. On write, they update MySQL and send a delete to Memcached (not an update — deletes are idempotent and simpler to reason about).

Scale-out via consistent hashing: Cache keys are distributed across hundreds of Memcached servers using consistent hashing. Each web server knows which cache server owns which key.

Lease tokens: To prevent the stale-set race condition, Memcached issues a "lease" when a cache miss occurs. If a delete arrives before the miss is filled, the lease is invalidated and the stale write is rejected.

Regional replication: Facebook deploys Memcached clusters in multiple data center regions. Writes go to the primary region's MySQL, which asynchronously replicates to other regions. Cache invalidations propagate via a dedicated invalidation pipeline (mcsqueal) that tails MySQL's commit log.

Key Lesson from Facebook

At Facebook's scale, the cache becomes a critical part of the system. Without the Memcached layer, the backing databases would face far more read traffic than they were designed to absorb directly. Facebook addresses cache-server failures with "Gutter" — reserve capacity that absorbs traffic when primary cache servers fail, reducing the chance of a thundering herd to the database.

Production Example 2: Cloudflare CDN

Cloudflare's CDN is a large globally distributed cache. The exact size of the network changes over time, but the core architecture remains the same: edge data centers close to users, optional upper tiers, and the origin behind them. [5]

How Cloudflare Reduces Latency

Anycast routing: Cloudflare advertises the same service IPs from many locations so requests are routed to a nearby edge by the internet's routing system.

Tiered caching: Instead of every PoP fetching from origin on a miss, Cloudflare can place upper-tier caches between edge locations and the origin. This improves cache efficiency and reduces origin connections. [5]

Cache Reserve (R2-backed): For eligible long-tail assets, Cloudflare can persist cacheable content in Cache Reserve on top of R2, extending origin shielding beyond the normal edge cache lifecycle. [5]

Key Insight

Cloudflare is a good reminder that caching is usually a stack, not a single box. Browser, edge PoP, optional upper tier, then origin. Each layer catches some repeat traffic before it falls through to the next one.

Edge Cases & Gotchas

1. Cache Stampede (Thundering Herd)

A popular cache entry expires. Simultaneously, 10,000 requests arrive and all miss the cache. All 10,000 hit the database at once — overwhelming it. Fixes: lock-based revalidation (only one request refills, others wait), probabilistic early expiration (randomly refresh before TTL to spread out reloads), or stale-while-revalidate (serve stale data while one request refreshes in the background).

2. Cache Penetration

Requests for keys that don't exist in the database (and therefore never populate the cache). Every request is a miss, every miss hits the database. This can be an attack vector (query for random nonexistent IDs). Fix: cache negative results (store "key=null" with a short TTL), or use a Bloom filter as a pre-check — if the Bloom filter says the key doesn't exist, don't even query the DB.

3. Hot Key Problem

One key receives a disproportionate amount of traffic (a viral tweet, a trending product). Even in a distributed cache, that one key lives on one server — which becomes a bottleneck. Fixes: key replication (store the hot key on multiple servers with a random suffix), or local caching (cache the hottest keys in application memory, bypassing the network entirely).

4. Cache Warming After Restart

When a cache server restarts, it's empty. All requests become misses, flooding the database. Fix: pre-warm the cache on startup by loading the most popular keys from the database, or use persistent caching (Redis with RDB/AOF) so the cache survives restarts.

5. Inconsistency Between Cache and DB

The cache says "price = $99" but the DB says "price = $79" because an invalidation never made it through. That is the core cache trade-off: speed bought with the risk of staleness. Use TTLs as a backstop, event-driven invalidation where freshness matters, and version checks if the gap itself is operationally important.

Production War Story

Flash sales, viral events, and synchronized TTL expiry all create the same failure mode: too many clients miss the same hot key at the same time. If you do not coalesce, refresh early, or serve stale data briefly, the database eats the blast.

What If? — Interactive Exploration

Tweak Parameters
Impact

Tradeoff Cards

Longer TTL

Improves: Hit rate (fewer expirations → fewer misses), reduced DB load.
Costs: More staleness — users may see outdated data for longer.
When worth it: Data changes infrequently (product catalog, config), or slight staleness is acceptable (social feed counts).

Bigger cache (more RAM)

Improves: More keys fit → fewer evictions → higher hit rate.
Costs: RAM is expensive. Diminishing returns appear after the "hot set" is covered.
When worth it: Hit rate is lower than you need and the working set exceeds current cache size. Once the hot set is already resident, extra RAM often buys only incremental gains.

Write-through instead of cache-aside

Improves: Cache always up-to-date (no stale reads). Simpler invalidation logic.
Costs: Every write is slower (DB + cache write synchronously). Cache fills with data that may never be read.
When worth it: Read-after-write consistency is critical (e.g., user profile edits must be immediately visible).

Local (in-process) cache + distributed cache

Improves: Hot keys served in ~100ns (no network hop to Redis). Shields Redis from hot-key thundering.
Costs: Cache coherence between processes — each process has its own stale copy. More memory per app server.
When worth it: A small number of keys get extreme traffic (viral content, config, feature flags).

Toy Simulation: Cache Stampede

stampede_sim.py — watch the thundering herd in action
import time, random, threading
from collections import defaultdict

db_calls = 0
cache = {}
lock = threading.Lock()
stats = defaultdict(int)

def get_product_naive(product_id):
    """No stampede protection — every miss hits DB."""
    global db_calls
    if product_id in cache:
        stats["hits"] += 1
        return cache[product_id]
    stats["misses"] += 1
    db_calls += 1
    time.sleep(0.05)  # Simulate 50ms DB query
    cache[product_id] = f"product_{product_id}_data"
    return cache[product_id]

def get_product_protected(product_id):
    """With lock — only one thread queries DB, others wait."""
    global db_calls
    if product_id in cache:
        stats["hits"] += 1
        return cache[product_id]
    with lock:
        if product_id in cache:  # Double-check after acquiring lock
            stats["hits"] += 1
            return cache[product_id]
        stats["misses"] += 1
        db_calls += 1
        time.sleep(0.05)
        cache[product_id] = f"product_{product_id}_data"
        return cache[product_id]

# ── Simulate stampede: 200 threads request same key simultaneously ──
for mode, func in [("NAIVE (no protection)", get_product_naive),
                      ("PROTECTED (lock-based)", get_product_protected)]:
    cache.clear(); db_calls = 0; stats.clear()
    threads = [threading.Thread(target=func, args=("hot_product",)) for _ in range(200)]
    t0 = time.perf_counter()
    for t in threads: t.start()
    for t in threads: t.join()
    elapsed = (time.perf_counter() - t0) * 1000
    print(f"  {mode:<30} DB calls: {db_calls:3d}  Time: {elapsed:.0f}ms")

# NAIVE:     DB calls: ~200 (every thread hits DB!) Time: ~50ms per thread
# PROTECTED: DB calls:   1  (only one thread hits DB) Time: ~50ms total

Interview Questions & Answers

Actual System Design Interview

A Prompt You Could Actually Get

Design the caching layer for a product-detail or social-feed service under read spikes. The interviewer is usually checking whether you know that caching is not one box in front of a database. It is a set of choices about where the cache lives, how keys are invalidated, how hot keys behave, and what happens when the cache itself becomes part of the failure path. [1] [3] [5]

How I'd Structure the Answer

I would begin with access pattern and staleness budget. If reads dominate and slight staleness is acceptable, cache-aside is usually the first pattern I would describe because it keeps the source of truth in the database and lets the application control invalidation. Then I would layer the caches: browser or CDN for shared edge content, distributed cache for hot objects, and local in-process cache only where object churn is manageable. The answer gets much stronger if I say how each layer is keyed, how long entries live, and which layer is allowed to serve stale data during backend trouble. If the distributed cache is sharded, I would name the placement key explicitly and say how clients find the owner: consistent hashing or rendezvous hashing, plus how quickly routing metadata changes propagate when cache nodes join or leave. [1] [4] [5]

Where the Interviewer Pushes

The push-back is usually about stampede and invalidation races. If one celebrity post or one product SKU turns into a hot key, naive TTL expiry can dump the entire read surge onto the backing store. A good answer mentions request coalescing, stale-while-revalidate, lease-style protection, or explicit admission control during refill. I would also expect a follow-up on membership churn: adding a cache node should only move a slice of keys, but those keys are cold on the new node, and if different app servers have different ring views they can read stale data from the old owner while others refill on the new owner. I would also expect the interviewer to ask what happens when the cache cluster itself is unavailable. That is where the Facebook memcache lesson matters: once the cache is load-bearing, the database probably cannot survive a full cold miss storm without graceful degradation. [1] [2] [3] [5]

Production Case Study Lens

Think Facebook Memcache: look-aside caching, lease tokens for refill races, and a cluster that is so load-bearing that a full cold-miss storm can drown the database. The interview edge is showing where the key lives, how the shard owner is chosen, and what happens when a hot object or membership change makes the “cache” part of the write/read correctness story. Q1: What is caching and when should you use it?

Caching stores frequently-accessed data in a faster storage layer to reduce latency and backend load. Use it when: (1) the same data is read much more often than it's written (high read-to-write ratio), (2) the data source is slow or expensive (DB queries, API calls, computations), and (3) slight staleness is acceptable. Don't use it when: data changes constantly and must always be fresh, or when access patterns are uniform (no "hot" keys).

Q2: Explain cache-aside vs read-through vs write-through.

Cache-aside: Application manages cache. On miss: app queries DB, stores result in cache. On write: app updates DB, deletes cache key. Most common pattern (Facebook, most web apps). Read-through: Cache manages itself. App only talks to cache. On miss, cache queries DB and populates automatically. Simpler app code but cache needs DB access. Write-through: Writes go to both cache and DB synchronously. Guarantees cache freshness but doubles write latency. Best for read-heavy workloads where consistency matters.

Q3: What is a cache stampede and how do you prevent it?

A cache stampede (thundering herd) occurs when a popular cache entry expires and many concurrent requests all miss simultaneously, flooding the backend. Prevention strategies: (1) Locking/coalescing — only one request refills, others wait. (2) Probabilistic early expiration — randomly refresh before TTL (spread load). (3) Stale-while-revalidate — serve stale data while one request refreshes in background. (4) Never expire hot keys — refresh them asynchronously instead of using TTL. Facebook uses lease tokens for similar protection at scale.

Q4: How does a CDN work and when would you use one?

A CDN is a global network of edge servers that cache content close to users. When a user in Berlin requests an image, the CDN serves it from Frankfurt (~10ms) instead of your Virginia origin (~150ms). CDNs use HTTP caching headers (Cache-Control, ETag) to decide what to cache and for how long. Use one when: you serve static assets (always), you have global users (reduces cross-continent latency), or you need DDoS protection (CDNs absorb attack traffic). Don't use for: highly personalized content, real-time data, or when all users are in one region near your servers.

Q5: Why is cache invalidation considered the hardest problem?

Because distributed systems make it nearly impossible to guarantee that a cache update and a DB update happen atomically. The classic race: Thread A reads v1 from DB, Thread B updates DB to v2 and deletes cache, Thread A writes v1 to cache. Now cache has stale v1, DB has v2, and no invalidation is coming. Possible mitigations: (1) use deletes instead of updates (idempotent), (2) TTLs as a safety net (bounded staleness), (3) lease tokens (Facebook's approach — reject stale writes), (4) versioned writes (only accept if version matches). No solution is perfect — each trades complexity for consistency.

Q6: How would you design caching for a social media feed?

Multi-layer approach: (1) CDN for static assets (profile photos, CSS/JS). (2) Redis/Memcached for pre-computed feed data — cache each user's feed as a sorted set or list. (3) Application-level cache for truly hot data (trending posts, viral content). Invalidation: When a user posts, invalidate their followers' cached feeds (fan-out on write) or mark them as stale (fan-out on read). TTL: 30-60 seconds for feeds (staleness is acceptable — users expect eventual consistency).

Hot key handling — the celebrity problem: When a celebrity with 50M followers posts, you can't fan-out a cache invalidation to 50M feed caches — that would take minutes and overwhelm your message bus. Instead, use a hybrid approach: pre-compute and cache feeds only for regular users (fan-out on write). For celebrity posts, don't push to individual caches — instead, at read time, merge the user's pre-computed feed with a "celebrity timeline" that's cached once and shared across all followers (fan-out on read for hot accounts only).

The dual source-of-truth problem: A celebrity's post can be edited or deleted after it's been cached in millions of places. You now have two "truths" — the cache says the post says "I love pizza 🍕" and the DB says it's been edited to "I love pasta 🍝." The fix: don't cache the post content in the feed cache — cache only post IDs. The feed cache stores a list of post IDs (lightweight, easily invalidated). When rendering, fetch the actual post content from a separate post-content cache (keyed by post_id + version). When a celeb edits their post, you invalidate one key (the post content) rather than millions of feed caches. This separation — feed structure vs content — is how Twitter/X and Instagram actually work at scale.

Staleness budget: For a social feed, 5-30 seconds of staleness is acceptable for most users. But for the celeb's own view (they just posted and want to see it immediately), use read-your-writes consistency: after posting, read from the primary DB for the author, while everyone else reads from the cache.

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Caching trades freshness for speed. Cache hit: ~0.1ms. DB query: ~5ms. That's 50× faster.
2. Cache-aside + TTL is the dominant pattern. On miss: query source, populate cache. On write: update source, delete cache key.
3. Hit rate is a core signal. If it stays low, the cache may not justify its complexity. Strong caches often achieve very high hit rates, but the right target depends on the workload. Monitor it.
4. LRU is the default eviction policy. It works because real workloads follow Zipf's law (few hot keys, long tail of cold ones).
5. Cache invalidation is genuinely hard. Use TTL as a safety net. Use event-driven invalidation for critical data. Use lease tokens for race conditions.
6. Cache stampede: popular key expires → thousands of simultaneous DB hits. Prevent with locks, early expiration, or stale-while-revalidate.
7. CDNs are global edge caches. Use them for static assets (always) and cacheable API responses (selectively). Cache-Control headers govern behavior.
8. Caches exist at every layer: CPU L1 → OS page cache → app memory → Redis → DB buffer pool → CDN → browser. Each is a chance to short-circuit.
9. In skewed workloads, a relatively small cache can serve a large share of requests. Don't cache everything — cache the hot set.
10. At scale, the cache IS the system (Facebook). If the cache fails, the DB drowns. Design for cache failure: Gutter pools, graceful degradation, rate limiting.

Key Intuition

Caching only helps when repeated requests reuse the cached result often enough. If every request is unique (no temporal locality), the cache is pure overhead — you pay for the lookup and the miss. The entire value proposition of caching rests on the skewed distribution of real-world access patterns: a few keys are requested millions of times, and Zipf's law makes caching them spectacularly effective.

References & Further Reading

Each source below is summarized by the cache mechanism or failure mode it teaches, rather than by a generic topic label.

[1] Nishtala et al. — "Scaling Memcache at Facebook" (NSDI 2013)

Rajesh Nishtala, Hans Fugal, et al. 10th USENIX NSDI, April 2013.
https://www.usenix.org/system/files/conference/nsdi13/nsdi13-final170_update.pdf

• Describes a look-aside cache design with delete-on-write invalidation, where the application owns cache population and cache entries are invalidated rather than synchronously overwritten.

• Introduces lease tokens to prevent stale fill races: only the holder of a valid lease may repopulate a missing key, and intervening invalidations revoke earlier fills.

• Adds Gutter pools as a limited fallback cache tier that absorbs miss traffic when a primary cache shard is unavailable, reducing immediate load on the backing store.

• Documents how regional fan-out and replication interact with cache invalidation, warmup, and failure handling at Facebook scale.

[2] Berger et al. — "Adaptsize: Orchestrating the Hot Object Memory Cache" (2017)

Daniel S. Berger, Ramesh K. Sitaraman, Mor Harchol-Balter. USENIX NSDI 2017.
https://www.usenix.org/conference/nsdi17/technical-sessions/presentation/berger

• Models caching as an admission-control problem rather than just an eviction problem: admitting a large object with low reuse can evict many small hot objects and reduce overall hit rate.

• AdaptSize estimates workload characteristics online and chooses a moving object-size admission threshold that maximizes byte hit rate under the current object distribution.

• The paper shows that static policies like “cache everything below X KB” underperform when workload skew changes over time.

• The key systems idea is that admission policy and eviction policy must be tuned together; eviction alone cannot recover space wasted on bad admissions.

[3] Redis Documentation — Eviction Policies

Redis Ltd.
https://redis.io/docs/latest/develop/reference/eviction/

• Enumerates Redis eviction families by both candidate set (`allkeys` vs `volatile`) and victim heuristic (LRU, LFU, random, TTL).

• Explains that Redis uses sampled approximations of LRU/LFU instead of exact global ordering, trading small optimality loss for lower metadata and CPU overhead.

• Details how TTL-based candidate restriction can interact with application behavior: `volatile-*` policies only evict keys that already have expirations.

• The docs make clear that eviction policy is a workload-level control knob; an LFU cache and an LRU cache behave very differently under scans, bursts, and long-tail reuse.

[4] Vixie — "DNS and BIND" + RFC 1035 (Domain Name System Caching)

P. Mockapetris. RFC 1035, November 1987. · P. Vixie. DNS and BIND, O'Reilly.
https://datatracker.ietf.org/doc/html/rfc1035

• RFC 1035 defines DNS resource records with TTL as explicit freshness metadata supplied by the authority and consumed by recursive resolvers and local caches.

• DNS is a multi-layer cache hierarchy in practice: browser cache, OS cache, recursive resolver cache, and any forwarding layers all compound freshness and invalidation behavior.

• The protocol’s caching model is intentionally simple: cached answers are reused until TTL expiry rather than revalidated with the authority on every lookup.

• This makes DNS a canonical example of fixed-expiration caching and explains why stale mappings persist until timeout even after the origin state changes.

[5] Cloudflare Docs — Tiered Cache and Cache Reserve

Cloudflare Documentation.
https://developers.cloudflare.com/cache/how-to/tiered-cache/
https://developers.cloudflare.com/cache/advanced-configuration/cache-reserve/

• Tiered Cache is useful because it shows miss collapsing by topology: a cold edge does not immediately hit origin, it first asks an upper-tier cache that centralizes refill traffic.

• Cache Reserve adds a different storage class to the hierarchy: long-tail objects can survive edge eviction in a persistent backing tier, then be promoted back upward on later demand.

• The important technical detail is the refill path: edge miss → upper-tier lookup → reserve/origin fetch, with each layer trading latency for origin offload and object retention.

• Read these docs as evidence that CDN caching is a multi-tier admission and refill system, not a flat “edge cache hit or miss” story.
