Replication & Sharding

Replication gives you more copies. Sharding gives you more owners. Most distributed databases need both, and most painful trade-offs show up exactly where those two ideas meet.

Key Terms

Replication

Replication means keeping multiple copies of the same data on different machines. You do it for fault tolerance, read scaling, and sometimes regional latency. The hard part is familiar: every extra copy makes failures safer, but it also makes writes more coordinated. That is where sync vs async replication, quorum rules, and replica lag all come from.

Sharding

Sharding means splitting one logical dataset across many machines so each machine owns only a subset of the rows. This is how storage footprint and write throughput scale past one box. The whole design hangs on the shard key: it decides where a row lives, which queries stay local, and whether one shard becomes the bottleneck while the rest sit mostly idle.

Hot Spot

A hot spot is one shard getting far more traffic than the others. That usually happens because the shard key was too low-cardinality, too time-skewed, or tied to a celebrity-style workload. Adding more shards does not fix the hot one unless the key space itself changes. That is why hot-spot reasoning is one of the first things interviewers push on in sharding questions.

Kleppmann’s framing is useful because it turns topology into coordination cost. Single-leader is easy to reason about but caps write scale. More write freedom almost always means more correctness work. [2]

Synchronous vs Asynchronous Replication

The fundamental write-path decision
SYNCHRONOUS replication:
Client → Leader: "INSERT INTO orders ..."
Leader → Follower(s): "replicate this write"
Follower(s) → Leader: "ACK — write is durable on my disk"
Leader → Client: "COMMIT OK"

Guarantee: if the leader dies AFTER returning "COMMIT OK",
the follower has the data. ZERO data loss.
Cost: every write waits for network round-trip to follower(s).
If follower is slow or unreachable → write hangs.

ASYNCHRONOUS replication:
Client → Leader: "INSERT INTO orders ..."
Leader: writes to local WAL, immediately returns "COMMIT OK"
Leader → Follower(s): "here's the WAL stream" (in background)

Guarantee: write is durable on the LEADER only.
If leader dies BEFORE the follower catches up → data is LOST.
Benefit: writes are fast (no waiting for followers).
This is the PostgreSQL default: streaming replication is async.

SEMI-SYNCHRONOUS (compromise):
Write waits for at least ONE follower to ACK (not all).
If that follower dies, another is promoted to "sync standby."
PostgreSQL: synchronous_commit = on (per-transaction configurable).
MySQL: semi-sync replication plugin (rpl_semi_sync_master_enabled).
The Replication Lag Problem

With async replication, followers are always slightly behind the leader — the replication lag (typically 10-500ms, but can spike to seconds under load). This creates three classic anomalies (DDIA Ch. 5): (1) Read-your-writes violation: you write on the leader, then read from a follower that hasn't caught up — your write appears lost. (2) Monotonic reads violation: two consecutive reads hit different followers at different lag states — you see a newer value then an older one. (3) Causal ordering violation: you see a reply to a comment before the comment itself. Fixes: route reads to the leader for recently-written data, pin a user to one replica per session, or use causal consistency protocols.

04

Consensus & the Raft Protocol

For strongly consistent replication, you need a consensus protocol — an algorithm where a majority of nodes must agree on every write before it's committed. [1]

Raft in Three Sentences

Leader election: Nodes start as followers. If a follower doesn't hear from a leader within a random timeout (150-300ms), it becomes a candidate and requests votes from peers. A candidate that receives votes from a majority becomes the leader for that term.

Log replication: The leader receives all writes, appends them to its log, and replicates log entries to followers. A log entry is committed only when a majority of nodes have persisted it. Committed entries are applied to the state machine in order.

Safety: Raft guarantees that if a log entry is committed, it will be present in the logs of all future leaders. A candidate can only win an election if its log is at least as up-to-date as a majority of nodes — ensuring committed entries are never lost.

Where Raft Is Used

etcd (Kubernetes control plane) uses Raft for all cluster state. CockroachDB uses Raft per range (each data range is a separate Raft group). TiKV (TiDB's storage engine) uses Raft. Consul uses Raft for service discovery. ClickHouse Keeper replaced ZooKeeper with a Raft-based implementation. The Raft paper was awarded Best Paper at USENIX ATC 2014 specifically because it made consensus understandable enough to implement correctly — a direct response to the decades of incorrect Paxos implementations.

The Shard Key is the Most Important Decision

Once data is sharded, the shard key determines everything: which queries are efficient (single-shard lookup = fast, cross-shard scatter-gather = slow), how evenly load distributes, and how painful resharding will be. A good shard key has: high cardinality (millions of distinct values), even distribution (no one value dominates), and alignment with your query patterns (most queries include the shard key in the WHERE clause). Common choices: user_id (social apps), tenant_id (SaaS), workspace_id (Slack), order_id (e-commerce). Bad choices: country_code (too few values), created_at (today's shard gets all writes), status (most rows are "active"). Reference: DDIA Ch. 6. [2]

06

Rebalancing — The Hard Part

When you need to add or remove shards, data must move between machines. This is rebalancing — and getting it wrong causes downtime, data loss, or performance degradation.

Rebalancing Approaches

ApproachHow It WorksDisruptionUsed By
Fixed partition countCreate many more partitions than nodes (e.g., 1000 partitions on 10 nodes = 100 per node). To add a node, move some partitions to it.Low. Only moved partitions are briefly unavailable.Cassandra (256 vnodes default), Elasticsearch (5 primary shards default)
Dynamic splittingStart with few partitions. When a partition exceeds a size threshold, split it in half. When a node is overloaded, migrate partitions to a new node.Medium. Split requires a brief write pause to the affected partition.HBase, Bigtable, CockroachDB (64MB default range size)
Consistent hashingNodes own positions on a hash ring. Adding a node absorbs only ~K/N keys from neighbors.Low. Only the transferring key range is affected.DynamoDB, Cassandra (token ring)
Rebalancing War Story

A fintech company sharded their PostgreSQL by hash(user_id) % 8. When they needed to grow to 16 shards, modulo-based sharding remapped ~50% of all rows. The migration took 3 days, required double the storage (old + new layout simultaneously), and caused intermittent read failures when queries hit rows mid-migration. If they had used consistent hashing from the start, adding 8 shards would have moved only ~50% of data (K/N per new shard), and each migration chunk would have been small and atomic. Lesson: choose your partitioning strategy before you need to rebalance.

Production: Vitess (YouTube / Slack)

Vitess is an open-source sharding middleware for MySQL, originally developed at YouTube (Google) to scale their MySQL infrastructure. Slack migrated to Vitess to shard their MySQL databases by workspace_id.

How Vitess Works

VSchema: A declarative sharding configuration that maps logical tables to shards. The application sends queries to a Vitess proxy (VTGate), which parses the SQL, inspects the VSchema, and routes the query to the correct shard(s). If the query includes the shard key in the WHERE clause → single-shard (fast). If not → scatter-gather across all shards (slow).

Online resharding: Vitess can split or merge shards without downtime. It creates new shards, starts replicating data to them, then atomically switches traffic — the application never sees a gap. This is the operational superpower: resharding is a zero-downtime operation, not a multi-day migration.

Production: CockroachDB Range Replication

CockroachDB combines sharding and replication in a single system. Data is automatically divided into ranges (contiguous key spans, default 512MB). Each range is replicated via Raft across 3+ nodes.

Key Design Decisions

Automatic range splitting: When a range exceeds the size threshold, CockroachDB splits it into two ranges — no operator intervention. The split is transparent to the application.

Raft per range: Each range is an independent Raft group. A cluster with 1000 ranges has 1000 concurrent Raft groups. This provides fine-grained fault isolation — a leader failure for one range doesn't affect others.

Leaseholder: One replica per range holds the lease — it serves all reads and proposes all writes. The leaseholder is typically the Raft leader. Reads from the leaseholder are strongly consistent without a Raft round-trip (the leaseholder knows it has the latest data because it proposed all writes).

Edge Cases & Gotchas

1. Split-Brain After Network Partition

If a single-leader system's network partitions, the old leader (on one side) and a newly elected leader (on the other side) both accept writes. When the partition heals, conflicting writes must be resolved. Fix: fencing tokens (DDIA Ch. 8) — each leader gets a monotonically increasing token; storage rejects writes from stale tokens. Or use Raft, which prevents split-brain by requiring a majority vote.

2. Cross-Shard Transactions

A user transfer (debit shard A, credit shard B) requires atomicity across two shards. Without a coordination protocol, one shard can commit while the other aborts — money vanishes or appears from nowhere. Fix: two-phase commit (2PC) with a coordinator, or Saga pattern (compensating transactions on failure). 2PC blocks on coordinator failure; Sagas are eventually consistent but non-blocking.

3. Replication Lag Amplified by Sharding

With 100 shards, each with async replicas, the probability of at least one shard having high replication lag at any moment is much higher than for a single database. A user's request might scatter-gather across 10 shards, and if one shard's replica is lagging, the entire response reflects stale data from that shard. Fix: read from leaders for consistency-sensitive queries, or implement per-shard lag tracking and route away from lagging replicas.

4. Resharding a Live System Without Vitess

If you sharded manually (hash % N in application code), resharding means: (1) create new shards, (2) double-write to old and new shards, (3) backfill historical data, (4) verify consistency, (5) switch reads, (6) stop double-writes, (7) decommission old shards. Each step has failure modes. This can take weeks. This is why managed sharding layers (Vitess, Citus, CockroachDB) exist — they automate this nightmare.

5. JOIN Across Shards

If users are on shard A and orders are on shard B, a SELECT * FROM users JOIN orders ON users.id = orders.user_id must fetch data from both shards and join in the application layer. This is orders of magnitude slower than a local JOIN. Fix: co-locate related data on the same shard (shard both users and orders by user_id), or denormalize (embed user name in the orders table).

What If? — Interactive Exploration

Tradeoff Cards

Sync vs async replication

Sync: Zero data loss on failover. But write latency increases by 1 RTT to the replica (~1-5ms in-region, 50-200ms cross-region). One slow replica stalls all writes.
Async: Fast writes (no waiting). But if the leader crashes before replication, committed writes are lost. Replication lag creates consistency anomalies.
Production default: Semi-sync (PostgreSQL synchronous_standby_names, MySQL semi-sync plugin). Waits for ONE replica, not all.

Hash sharding vs range sharding

Hash: Even distribution, no hot spots. But range queries (ORDER BY created_at LIMIT 100) scatter across all shards — N-way merge required.
Range: Efficient range scans and sorted queries. But sequential inserts (auto-increment IDs, timestamps) create a hot shard at the end of the range.
Hybrid (CockroachDB): Hash the first part of the key for distribution, use a sorted suffix for range queries within a shard.

More replicas vs more shards

More replicas (same data): Scales reads. Does NOT scale writes. Every write still goes to every replica.
More shards (split data): Scales writes AND reads. But introduces cross-shard query complexity and operational burden.
Decision: Read-bound? → Add replicas. Write-bound? → Add shards. Most systems hit read bottlenecks first.

Application-level vs middleware sharding

Application-level: Routing logic in your code (hash(key) % N). Simple, no new infrastructure. But resharding requires code changes and data migration.
Middleware (Vitess, ProxySQL): Transparent routing. Application sees a single DB. Online resharding without code changes. But another system to operate.
Auto-sharding DB (CockroachDB, Spanner): The database handles everything. Simplest operationally, but more expensive and less control.

Interview Questions & Answers

Q1: Explain the difference between replication and sharding.

Replication copies the entire dataset to multiple nodes. It provides fault tolerance (data survives node failure) and read scaling (spread reads across replicas), but does NOT scale writes — every write must be applied to every replica. Sharding splits the dataset into disjoint subsets across nodes. Each shard holds a portion of the data. It scales both writes (each write goes to exactly one shard) and storage (total capacity = sum of all shards). In production, systems use BOTH: each shard is replicated. Instagram's PostgreSQL: sharded by user_id into multiple databases, each with 2 replicas. The shard key determines write routing; replicas handle read scaling within each shard.

Q2: How do you choose a shard key? What makes a bad choice?

A good shard key has: (1) High cardinality — millions of distinct values (user_id, order_id). (2) Even distribution — no single value dominates traffic. (3) Query alignment — most queries include the shard key in WHERE, enabling single-shard lookups. Bad choices: country_code (low cardinality — US shard is 10× larger), created_at (today's shard gets 100% of writes), status (90% of rows are "active" → one hot shard), boolean fields. Worst case: a key that creates hot spots under viral load. If a celebrity user_id suddenly gets 100× traffic, that user's shard becomes the bottleneck. Fix: compound shard key (user_id + random_suffix) to scatter even within a single user's data.

Q3: What happens when you need to add shards to a running system?

With modulo-based sharding (hash % N), changing N remaps ~(N-1)/N of all keys — catastrophic. You'd need to double-write during migration, backfill historical data, and do a coordinated cutover. With consistent hashing, adding a shard remaps only ~K/N keys — much less disruptive. With managed middleware (Vitess), resharding is online: new shards are created, data is streamed to them while the old shards continue serving, and traffic is atomically cut over — zero downtime. With auto-sharding databases (CockroachDB, Spanner), you simply add a node and the system rebalances automatically. The lesson: plan your sharding strategy for growth from day one. Retrofitting is painful.

Q4: You have a single-leader PostgreSQL with async replication. A user writes a comment, refreshes the page, and doesn't see their comment. Why?

The write went to the leader. The page refresh read from a follower that hasn't received the replication update yet — the follower is behind by the replication lag (maybe 200ms). This is the read-your-writes consistency violation (DDIA Ch. 5). Fixes: (1) Read from leader for own writes — if the user just wrote, route their subsequent reads to the leader for the next N seconds. (2) Sticky sessions — pin the user to one replica per session (ensures monotonic reads). (3) Semi-sync replication — wait for at least one replica to ACK before returning commit (adds ~1-5ms latency). (4) Client-side optimistic rendering — show the comment locally immediately, before the server confirms persistence.

Q5: How does Raft prevent split-brain?

Raft requires a majority vote (quorum) for both leader election and log commitment. In a 5-node cluster, any action requires 3 votes. If the network partitions into groups of [2] and [3], only the group of 3 can elect a leader — the group of 2 cannot reach a majority. The old leader in the minority partition will stop accepting writes (its AppendEntries RPCs to 2 followers succeed, but that's not a majority). When the partition heals, the minority group's leader steps down (its term is lower), and nodes catch up to the majority leader's log. Key invariant: at most one leader per term, and a leader must have a log at least as up-to-date as any majority — guaranteeing committed entries are never lost. [1]

Interviewer Follow-ups

"Why shard by conversation_id and not by user_id?"

A message belongs to a conversation, not a single user. If you shard by user_id, sending a message in a group chat requires writing to N shards (one per participant) — an N-way cross-shard transaction. Sharding by conversation_id means all messages in a conversation are co-located on one shard. Writing a message is a single-shard append. Reading conversation history is a single-shard range scan (ordered by timestamp within the conversation). The trade-off: fetching a user's list of conversations requires a scatter-gather (each conversation might be on a different shard). Fix: maintain a secondary index — a per-user "inbox" table (sharded by user_id) that maps user_id → list of conversation_ids. This is the multi-table, multi-shard pattern used by Meta's messaging infrastructure.

"How do you handle the 500ms delivery SLA while replicating across datacenters?"

Use async replication to the remote DC. The write path: (1) message arrives at the local DC, (2) is written to the shard leader's WAL and committed locally (~5-10ms), (3) the sender receives "delivered" ACK, (4) the message is pushed to the recipient via WebSocket (if online) or queued for push notification. Cross-DC replication happens in the background (~50-200ms). The 500ms budget is for end-to-end delivery, not replication. If the local DC dies before replication, some very recent messages may be lost — this is the trade-off for speed. For strict durability, you'd need sync replication to at least one remote DC (adding ~50-100ms to every write), which would leave tight margin for the 500ms SLA. In practice, major messaging services accept the tiny risk of async data loss during catastrophic DC failure — the last few seconds of messages may be lost, but the system stays fast for billions of messages daily.

"A celebrity with 10 million followers sends a message to a public channel. How does your design handle this?"

This is the fan-out on read vs fan-out on write problem (DDIA Ch. 11). Fan-out on write: Write the message to each follower's inbox — 10M writes. Even at 100K writes/sec, this takes 100 seconds. Unacceptable. Fan-out on read: Store the message once on the channel's shard. When a follower opens the app, they query the channel's shard for new messages. 10M reads will hit the shard, but these can be served from replicas and cached aggressively (the message is immutable). Hybrid approach (what Meta and Twitter actually do): For users with <10K followers, fan-out on write (push to each follower's feed). For celebrities with millions of followers, fan-out on read (pull from the celebrity's timeline when each follower loads their feed). The threshold is tuned based on SLA and infrastructure capacity.

What the Interviewer Is Testing

(1) Shard key reasoning — why conversation_id over user_id (co-location of conversation messages). (2) Replication trade-offs — async for speed, accepting tiny data loss window during DC failure. (3) Fan-out modeling — understanding that celebrity broadcasts are fundamentally different from 1:1 messages. (4) Numbers — 100B messages/day = ~1.15M messages/sec. With 1000 shards, each handles ~1150 writes/sec — easily within a single DB instance's capacity. (5) Multi-index pattern — message table sharded by conv_id + inbox table sharded by user_id.

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Replication copies data for fault tolerance and read scaling. Sharding splits data for write scaling and storage scaling. Production uses BOTH.
2. Three replication topologies: single-leader (simple, limited writes), multi-leader (higher writes, conflict resolution), leaderless (highest availability, most complex).
3. Sync replication: zero data loss, slower writes. Async: fast writes, risk of data loss. Semi-sync: one replica ACK, best compromise.
4. Raft: leader election + log replication + safety. Majority vote prevents split-brain. Used by etcd, CockroachDB, TiKV.
5. Shard key choice is the #1 design decision. Needs: high cardinality, even distribution, query alignment. Bad: country, status, timestamps.
6. Hash sharding: even distribution, no range queries. Range sharding: efficient range scans, risk of hot spots from sequential writes.
7. Rebalancing is hard. Fixed partition count (Cassandra), dynamic splitting (CockroachDB), or consistent hashing (DynamoDB) minimize disruption.
8. Cross-shard JOINs are expensive. Co-locate related data on the same shard (shard users + orders both by user_id).
9. Replication lag creates read-your-writes, monotonic reads, and causal ordering violations. Fixes: read from leader, sticky sessions, causal tokens.
10. In interviews: always state the shard key first, then explain what queries become fast (single-shard) vs slow (scatter-gather).

Key Intuition

Replication is about copies. Sharding is about division. Replication makes your data survive failures and speeds up reads — but every write still hits every copy. Sharding makes your data fit in more machines and speeds up writes — but queries that cross shard boundaries get expensive. The art of distributed database design is choosing the shard key that keeps most queries within a single shard, while replicating each shard for fault tolerance. Get the shard key right, and the system scales. Get it wrong, and no amount of hardware can save you.

References & Further Reading

[1] Ongaro & Ousterhout — "In Search of an Understandable Consensus Algorithm" (USENIX ATC 2014)

Diego Ongaro, John Ousterhout. Stanford University. USENIX ATC 2014, pp. 305-319. Best Paper Award.
https://raft.github.io/raft.pdf

• Decomposes consensus into three sub-problems that can be reasoned about independently: leader election (randomized election timeouts, majority vote per term), log replication (leader appends entries, followers replicate in order), and safety (a committed entry is present in all future leaders' logs, proven via the Election Restriction: a candidate's log must be at least as up-to-date as a majority).

• The strong leader property distinguishes Raft from Paxos: log entries only flow from leader to followers (never the reverse). This eliminates the complexity of merging logs from multiple proposers that makes multi-decree Paxos implementations notoriously error-prone.

• Cluster membership changes use a joint consensus approach: the old and new configurations overlap, requiring a majority from BOTH configurations during the transition. This avoids the window where two independent majorities could elect different leaders.

• Formal user study with 43 students at Stanford and UC Berkeley: after learning both Raft and Paxos, 33 of 43 students answered Raft questions more correctly than Paxos questions. The paper's thesis is that understandability is a first-class design goal — a protocol that's easier to understand produces fewer implementation bugs.

• Performance is equivalent to multi-Paxos: leader election completes in one round-trip (median 12ms in the paper's experiments), and log replication requires one round-trip per entry (leader → majority ACK). The bottleneck is disk fsync at each node, not the consensus protocol itself.

[2] Kleppmann — "Designing Data-Intensive Applications" (2017), Ch. 5-7

Martin Kleppmann. O'Reilly Media, 2017.
https://dataintensive.net/

• Chapter 5 defines five consistency guarantees as a hierarchy: read-your-writes (you always see your own writes), monotonic reads (you never see time go backward), consistent prefix reads (you see causally ordered writes in order), culminating in linearizability (all operations appear to happen in a single total order). Each guarantee requires increasing coordination cost.

• Chapter 6 on partitioning introduces the concept of secondary indexes on partitioned data: a document-partitioned (local) index is fast to write but requires scatter-gather on read; a term-partitioned (global) index is efficient to read but requires cross-partition writes. This trade-off appears in every sharded system.

• Chapter 6 distinguishes three rebalancing strategies: fixed number of partitions (create N >> nodes, redistribute whole partitions), dynamic partitioning (split when too large, merge when too small), and proportional to nodes (fixed partitions per node, split on node addition). Each has different operational characteristics.

• Chapter 7 analyzes write skew as a phenomenon unique to sharded/replicated systems: two transactions read the same data, make decisions based on it, then write to different rows — the combined result violates an invariant that neither transaction individually violated. Only Serializable isolation prevents this.

[3] Vitess Documentation — VSchema, Routing, and Resharding

Vitess project.
https://vitess.io/docs/

• The key technical idea is the VSchema: Vitess separates the application's logical schema from the physical shard layout, so routing decisions like "which keyspace and shard owns this row?" are computed from metadata instead of being hard-coded into clients.

• The sharding docs make the routing contract explicit. Queries that include the sharding key can be routed to one shard. Queries that do not include it can become scatter-gather fan-out operations across many shards, which is why shard-key choice dominates performance.

• The resharding material is the operational lesson. Vitess uses filtered replication to copy a moving subset of rows into the new shard layout while the old layout still serves traffic, then cuts reads and writes over once lag is under control.

• Read the Vitess docs for the mechanics that real teams eventually face: online resharding, tablet role promotion, primary failover, and query routing through a control plane that has to stay correct while shards are moving.
