01Key Terms

Polling
A client-driven pattern where the browser (or any consumer) sends repeated requests to the server at a fixed interval to check for new data. Think of refreshing your email inbox every 30 seconds — simple to implement, but you burn bandwidth and CPU cycles even when nothing has changed. Used by early AJAX dashboards, Kubernetes controller loops, and Prometheus metric scrapers.
Long Polling
An evolution of short polling where the client sends a request but the server holds the connection open until it has new data to send (or a timeout fires). The moment the client gets a response, it immediately opens a new long-poll request. Facebook's original chat in 2008 and early Slack prototypes used long polling before migrating to WebSockets. Reduces empty responses but ties up a server thread (or connection slot) per waiting client.
Streaming (Server-Sent Events / SSE)
A server-push mechanism over a single long-lived HTTP connection. The server sends events as they occur using the text/event-stream MIME type, and the browser's built-in EventSource API handles reconnection, last-event-ID tracking, and parsing automatically. Unidirectional (server → client only). Ideal for live feeds: stock tickers, CI/CD build logs, ChatGPT's token-by-token output. Standardized by the WHATWG as part of the HTML Living Standard.
WebSocket
A full-duplex, bidirectional communication protocol (RFC 6455) that starts with an HTTP Upgrade handshake and then switches to a lightweight framed binary protocol over a single TCP connection. Either side can send messages at any time without waiting for a request. Per-frame overhead is just 2–14 bytes vs. hundreds of bytes for HTTP headers. Used by Discord (voice + text), multiplayer games, collaborative editors (Figma), and financial trading platforms.
Socket
A general-purpose OS-level abstraction representing one endpoint of a two-way network communication link, identified by an IP address + port number. Every TCP connection is backed by a pair of sockets. The term "WebSocket" borrows the name but operates at a higher layer (application-level framing on top of TCP). Don't confuse raw TCP sockets with the WebSocket protocol — they're different abstraction levels.
HTTP Upgrade
The mechanism in HTTP/1.1 that lets a client request switching to a different protocol on the same TCP connection. The client sends Connection: Upgrade and Upgrade: websocket headers; if the server agrees, it responds with 101 Switching Protocols. After this handshake, HTTP is no longer spoken on that connection — it's purely WebSocket frames.
Heartbeat / Ping-Pong
A keep-alive mechanism where one side of a persistent connection periodically sends a small control message (ping) and expects a reply (pong) to confirm the other side is still alive. WebSocket defines native ping/pong control frames. Without heartbeats, connections can silently die behind NATs, load balancers, or mobile network switches — and neither side knows until the next data send fails.

02The Core Problem

HTTP was designed as a request-response protocol: the client asks, the server answers, and the connection is done. This works perfectly for loading web pages — but it breaks down the moment your application needs real-time updates.

Consider a chat application. When Alice sends a message, Bob's browser needs to know about it. But HTTP gives the server no way to reach out to Bob — the server can only respond to requests Bob has already made. This is the fundamental tension: HTTP is client-initiated, but real-time data is server-initiated.

Every technique in this topic is a different answer to the same question: How does the server tell the client that something has changed?

The spectrum: Short polling is the simplest but most wasteful. Long polling is more efficient but awkward. SSE is elegant for one-way push. WebSockets give you a full bidirectional pipe at the cost of operational complexity. The right choice depends on your data flow direction, update frequency, and infrastructure constraints.

03Short Polling

The simplest possible approach: set a timer, fire a request, process the response, repeat.

// Client polls every 3 seconds
setInterval(async () => {
  const res = await fetch('/api/messages?after=' + lastSeenId);
  const newMsgs = await res.json();
  if (newMsgs.length > 0) {
    render(newMsgs);
    lastSeenId = newMsgs[newMsgs.length - 1].id;
  }
}, 3000);

How It Works

1. Client sends a GET request to the server. 2. Server immediately returns whatever data is currently available (possibly an empty array). 3. Client waits for the interval timer to fire. 4. Goto step 1.

When It's Actually Fine

Low-frequency dashboards: A Grafana panel refreshing every 30s doesn't need a persistent connection — polling is simpler and the overhead is negligible.

Batch-oriented workflows: Checking if a CI build has finished, or if a file-processing job is complete. You only need to check once per minute.

Behind firewalls/proxies that kill long connections: Some enterprise networks terminate idle connections after 30–60 seconds. Short polling works regardless of infrastructure constraints.

The Cost

If you have 10,000 users polling every 3 seconds, that's ~3,333 requests/second hitting your server — most returning empty responses. Each request carries ~500 bytes of HTTP headers. That's 1.6 MB/s of pure overhead with zero useful data. At scale, the wasted compute (TCP handshake, TLS negotiation, request parsing, DB query for "nothing new") dominates your server's resource budget.

04Long Polling (Hanging GET)

Long polling flips the cost model: instead of the client asking repeatedly and mostly getting nothing, the server holds the request open until it actually has something to say.

// Client — immediately reconnects after each response
async function longPoll() {
  try {
    const res = await fetch('/api/messages/subscribe?timeout=30');
    const data = await res.json();
    if (data.messages) render(data.messages);
  } catch (e) {
    await sleep(1000); // back-off on error
  }
  longPoll(); // immediately re-open
}
longPoll();

Server-Side Mechanics

The server receives the request and, if no new data is available, parks the request in a waiting queue rather than sending an immediate response. When an event occurs (e.g., a new chat message is written to the DB), the server iterates through all parked requests that are subscribed to that event, writes the response, and closes each connection. The client, upon receiving the response, immediately opens a new long-poll request.

The Timeout Problem

If nothing happens for a long time, the connection will eventually time out — either from the server's configured timeout, a reverse proxy (nginx default: 60s), a load balancer (AWS ALB: 60s default, max 4000s), or a corporate firewall. The typical pattern: server responds with an empty body after 30 seconds to preempt upstream timeouts, and the client immediately reconnects.

Thread exhaustion: In thread-per-request servers (classic Tomcat, Apache + mod_php), each parked long-poll request occupies a thread that does nothing but wait. With 10,000 concurrent users, you need 10,000 threads just idling. This is why long polling pushed adoption of event-loop architectures (Node.js) and async I/O (Netty, Go goroutines, Python asyncio).

Message ordering gap: Between the moment a long-poll response is sent and the moment the client's next request arrives, events can occur that the client misses. The fix: include a cursor/sequence ID so the server can replay anything the client hasn't seen.

05Server-Sent Events (SSE)

SSE is a W3C standard that gives the browser a built-in, production-ready streaming client via the EventSource API. The server holds a single HTTP connection open and writes events as they happen. No custom client code for reconnection — the browser handles it.

The Wire Format

HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

data: {"user":"alice","msg":"hello"}

event: typing
data: {"user":"bob"}

id: 42
data: {"user":"bob","msg":"hi back"}

: this is a comment (used as keepalive)

Each event is a set of field: value lines terminated by a blank line. The data: field carries the payload. The id: field sets a cursor — if the connection drops, the browser automatically reconnects and sends Last-Event-ID: 42 in the request headers, enabling the server to resume from where it left off. The event: field lets you dispatch to different listeners on the client side.

Client Code

const source = new EventSource('/api/stream');

source.onmessage = (e) => {
  const msg = JSON.parse(e.data);
  render(msg);
};

source.addEventListener('typing', (e) => {
  showTypingIndicator(JSON.parse(e.data));
});

source.onerror = (e) => {
  // Browser auto-reconnects; you can show a "reconnecting..." banner
  console.log('SSE connection lost, reconnecting...');
};

Limits and Gotchas

Unidirectional only: Data flows server → client. If the client needs to send data (e.g., a chat message), it still uses a regular POST/PUT request. This is fine for dashboards, live feeds, and notification streams, but not for truly interactive protocols.

HTTP/1.1 connection limit: Browsers allow only ~6 concurrent connections per domain per browser. If you open 6 tabs each with an SSE connection to the same domain, no more HTTP requests can be made to that domain from that browser. HTTP/2 solves this (streams are multiplexed over one TCP connection, default limit: 100 streams).

No binary data: SSE is text-only (UTF-8). You'd need to Base64-encode binary payloads, adding ~33% overhead.

Real-World Usage

ChatGPT's streaming response (tokens appearing one by one) uses SSE. The OpenAI API with stream: true returns text/event-stream with each token as a separate data: line. This is a perfect SSE use case: unidirectional (server → client), text-based, and the client only needs to render what arrives.

06WebSockets (RFC 6455)

WebSockets provide full-duplex, bidirectional communication over a single TCP connection. After an initial HTTP handshake, the protocol switches entirely — no more HTTP headers, no more request/response model. Either side can send a message at any time.

The Upgrade Handshake

// Client → Server
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

// Server → Client
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

The Sec-WebSocket-Key is a random Base64 nonce. The server concatenates it with a magic GUID, SHA-1 hashes it, and Base64-encodes the result as Sec-WebSocket-Accept. This proves the server understands WebSocket — it's not a security mechanism, it's a protocol identification check to prevent accidental upgrades.

Frame Structure

After the handshake, data travels in lightweight frames. Each frame has a 2-byte minimum header: 1 bit for FIN (is this the last fragment?), 4 bits for opcode (text=0x1, binary=0x2, close=0x8, ping=0x9, pong=0xA), 1 bit for MASK, and 7+ bits for payload length. Client-to-server frames are XOR-masked (4-byte mask key) to prevent cache poisoning attacks on intermediary proxies. The total per-message overhead is 2–14 bytes — compare that to ~500+ bytes of HTTP headers per request.

Client Code

const ws = new WebSocket('wss://server.example.com/chat');

ws.onopen = () => {
  ws.send(JSON.stringify({ type: 'join', room: 'general' }));
};

ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  if (msg.type === 'chat') render(msg);
  if (msg.type === 'presence') updateUserList(msg);
};

ws.onclose = (event) => {
  console.log(`Closed: code=${event.code} reason=${event.reason}`);
  // Reconnect with exponential backoff
  setTimeout(connect, backoff());
};

Operational Complexity

Load balancer stickiness: A WebSocket connection is long-lived and stateful. If your load balancer reassigns the TCP connection to a different backend server, the connection breaks. You need layer-4 (TCP) load balancing with sticky sessions, or layer-7 balancers that understand WebSocket (nginx, HAProxy, AWS ALB all support this).

Horizontal scaling: If user A is connected to Server 1 and user B to Server 2, how does A's message reach B? You need a pub/sub backplane (Redis Pub/Sub, NATS, Kafka) to fan out messages across servers. This is the #1 operational challenge of WebSocket at scale.

Connection state management: Each server must track which users are on which connections. If a server crashes, all its connections die. Clients must implement reconnection with exponential backoff. The server must reconcile reconnecting clients with any messages they missed (message queue + cursor/offset).

Proxy/firewall traversal: Some corporate proxies don't pass through WebSocket Upgrade requests. The workaround: always use wss:// (WebSocket over TLS on port 443) — TLS traffic is opaque to proxies, so the Upgrade succeeds inside the encrypted tunnel.

11Mini-Project: Chat Room (Poll vs. Stream)

Build a tiny chat system that supports both modes to feel the difference firsthand. One server, two client modes.

Simple Mini-Project: Polling vs SSE Side-by-Side

Goal: Run a server with an in-memory message store. One client polls every 2s. Another client uses SSE. Send messages via curl POST. Observe the latency difference.

# Save as chat_server.py
import json, time, threading
from http.server import HTTPServer, BaseHTTPRequestHandler

messages = []
lock = threading.Lock()
sse_clients = []

class ChatHandler(BaseHTTPRequestHandler):
    def do_POST(self):
        length = int(self.headers.get('Content-Length', 0))
        body = json.loads(self.rfile.read(length)) if length else {}
        msg = {'id': len(messages), 'user': body.get('user','anon'),
               'text': body.get('text',''), 'ts': time.time()}
        with lock:
            messages.append(msg)
            # Notify SSE clients
            dead = []
            for client_wfile in sse_clients:
                try:
                    client_wfile.write(
                        f"id: {msg['id']}\ndata: {json.dumps(msg)}\n\n".encode())
                    client_wfile.flush()
                except: dead.append(client_wfile)
            for d in dead: sse_clients.remove(d)
        self.send_response(201)
        self.end_headers()
        self.wfile.write(json.dumps(msg).encode())

    def do_GET(self):
        if self.path.startswith('/poll'):
            # Return messages after ?after=ID
            after = int(self.path.split('after=')[-1]) if 'after=' in self.path else -1
            with lock:
                new = [m for m in messages if m['id'] > after]
            self.send_response(200)
            self.send_header('Content-Type','application/json')
            self.end_headers()
            self.wfile.write(json.dumps(new).encode())

        elif self.path.startswith('/stream'):
            self.send_response(200)
            self.send_header('Content-Type','text/event-stream')
            self.send_header('Cache-Control','no-cache')
            self.end_headers()
            # Send existing messages
            with lock:
                for m in messages:
                    self.wfile.write(
                        f"id: {m['id']}\ndata: {json.dumps(m)}\n\n".encode())
                self.wfile.flush()
                sse_clients.append(self.wfile)
            # Keep connection open
            try:
                while True: time.sleep(1)
            except: pass

    def log_message(self, *a): pass

print("Chat server on :8080  |  POST /send  |  GET /poll?after=ID  |  GET /stream")
HTTPServer(('', 8080), ChatHandler).serve_forever()
# Terminal 1: Start server
$ python3 chat_server.py

# Terminal 2: Polling client (checks every 2s)
$ LAST=-1; while true; do
  RESP=$(curl -s "http://localhost:8080/poll?after=$LAST")
  echo "[POLL $(date +%H:%M:%S)] $RESP"
  # Update cursor to latest message ID
  NEW_LAST=$(echo "$RESP" | python3 -c "
import sys,json
msgs=json.load(sys.stdin)
print(msgs[-1]['id'] if msgs else $LAST)" 2>/dev/null)
  LAST=${NEW_LAST:-$LAST}
  sleep 2
done

# Terminal 3: SSE client (instant delivery)
$ curl -N http://localhost:8080/stream

# Terminal 4: Send messages
$ curl -X POST http://localhost:8080/send \
  -H "Content-Type: application/json" \
  -d '{"user":"alice","text":"hello!"}'

What to observe: The SSE client (Terminal 3) shows the message instantly. The polling client (Terminal 2) shows it on the next poll cycle — up to 2 seconds later. Send several messages rapidly and note how the polling client batches them while the SSE client shows them one by one in real time.

12Production Examples

Slack: The Long Polling → WebSocket Migration

Slack's original real-time messaging system (RTM API) used WebSockets, but their public API initially exposed long polling for bot integrations. The challenge: at peak, Slack was maintaining millions of concurrent WebSocket connections across their fleet. Each WebSocket server tracked which channels each user was in, creating a massive in-memory state map.

Scaling pattern: Each WS server subscribes to a Redis Pub/Sub channel per Slack workspace. When a user sends a message, the API server writes it to the database, then publishes to the workspace's Redis channel. Every WS server subscribed to that workspace receives the message and fans it out to all connected users in the relevant channel. The Redis Pub/Sub layer is the glue that lets WS servers scale horizontally.

Lesson: The hardest part of WebSocket at scale isn't the protocol — it's the state management and cross-server message routing. Slack eventually built a custom message bus to replace Redis Pub/Sub at their scale.

OpenAI / ChatGPT: SSE for Token Streaming

When you see ChatGPT "typing" a response word by word, that's SSE. The API endpoint returns Content-Type: text/event-stream and emits each token (or small batch of tokens) as a separate SSE event as the model generates them. The final event is data: [DONE].

Why SSE, not WebSockets? The data flow is strictly unidirectional during generation (server → client). The client's only action is to send the initial prompt (a regular POST), then receive the stream. WebSockets would add connection management complexity (heartbeats, reconnection, protocol state) for zero benefit. SSE's built-in EventSource auto-reconnection and Last-Event-ID header are bonuses that come for free.

Infrastructure advantage: SSE works through every HTTP proxy and CDN without special configuration. WebSocket Upgrade requests can be blocked by enterprise proxies, WAFs, and certain CDNs. For a public API serving millions of developers across every network environment, this matters enormously.

Figma: WebSocket for Collaborative Editing

Figma's multiplayer design tool uses WebSockets for real-time cursor tracking, layer updates, and conflict-free collaborative editing. Multiple users editing the same design file need sub-100ms update propagation — SSE can't handle the bidirectional nature (each user is both producing and consuming changes), and polling would add unacceptable latency.

CRDT integration: Changes are encoded as CRDT (Conflict-free Replicated Data Type) operations and broadcast over WebSocket. Each client applies operations locally for instant feedback, then the server resolves conflicts and broadcasts the canonical state. The WebSocket channel carries both the outbound edits and the inbound conflict resolutions.

13Edge Cases & War Stories

The Thundering Herd on Reconnect: Your WebSocket server crashes. 50,000 clients all try to reconnect at the exact same instant. The replacement server gets hammered with 50K concurrent TCP handshakes + TLS negotiations + WebSocket upgrades. Fix: clients must use jittered exponential backoff — start at 1s, double each attempt, add random jitter of ±30%. Without jitter, wave after wave of synchronized retries keeps the server overloaded.

Mobile Network Switching: A user walks from WiFi to cellular. The TCP connection silently dies — neither the phone's TCP stack nor the server is notified immediately (TCP keepalive defaults to 2 hours!). The user sees stale data until the client's heartbeat timer fires and detects the dead connection. Discord's mobile app sends application-level heartbeats every 41.25 seconds; if two consecutive heartbeats get no response, it triggers reconnection.

The Long-Poll Gap: Client A's long-poll response returns at time T. Between T and T+δ (when A's next long-poll request arrives), a message is sent. A misses it entirely because no request was open. Fix: every long-poll request includes the last seen sequence number. The server replays anything > that sequence, whether it happened during the gap or not. This is essentially reinventing message queues at the application layer — one reason to prefer SSE or WebSocket when possible.

SSE and HTTP/1.1 Connection Exhaustion: A user opens 7 tabs of your dashboard, each with an SSE connection to the same domain. Tab #7's SSE fails to connect because browsers enforce a 6-connections-per-domain limit for HTTP/1.1. All other HTTP requests to your domain from that browser are also blocked. The fix: serve your API over HTTP/2 (multiplexes all streams over a single TCP connection, default 100 stream limit), or use a different subdomain for the SSE endpoint.

Load Balancer Idle Timeout: AWS ALB's default idle timeout is 60 seconds. If your WebSocket connection has no traffic for 60s (e.g., a quiet chat room at 3 AM), ALB silently drops the connection. Neither client nor server gets a FIN — the connection just vanishes. Fix: implement server-side ping frames every 30s to keep the connection alive through the LB. Configure ALB idle timeout to 300s+ for WebSocket target groups.

14Interview Q&A

Q1: You're designing a live sports score update system. Would you use polling, SSE, or WebSockets? Why?

SSE is the ideal choice here. The data flow is purely unidirectional (server → millions of viewers). Viewers don't send data back — they just consume score updates. SSE gives you: automatic reconnection with Last-Event-ID, works through every CDN and proxy, and each connection is a plain HTTP response (easy to cache at the edge with short TTLs). WebSocket would work but adds unnecessary bidirectional complexity and doesn't cache through CDNs. Polling would work for low-frequency updates (scores change every few minutes) but SSE is cleaner.

Deep follow-up — scale: With millions of concurrent viewers, you don't terminate SSE connections at the origin. You use a CDN or edge proxy layer (e.g., Fastly, CloudFlare Workers) that maintains one upstream SSE connection per PoP and fans out to thousands of downstream viewers. This is HTTP multicast — one origin connection serves an entire geographic region.

Q2: How do WebSocket connections survive rolling deployments?

They don't — and that's by design. During a rolling deploy, you drain connections from old instances. The process: (1) stop accepting new WS connections on the old instance, (2) send a custom "please reconnect" control message to all connected clients, (3) wait a grace period (e.g., 30s) for clients to reconnect to new instances, (4) forcibly close any remaining connections. Clients must handle this gracefully with reconnection + message replay from their last cursor position.

Follow-up nuance: Some systems (e.g., Kubernetes with preStop hooks + graceful shutdown periods) coordinate this with the load balancer. The pod enters "draining" state: the LB stops routing new connections to it, existing connections get a close frame, and after the grace period, the pod terminates. The key invariant: clients should never trust that a connection is permanent. Reconnection logic isn't a fallback — it's the primary path.

Q3: When would you choose long polling over WebSockets for a chat application?

Rarely, but here are legitimate reasons: (1) Enterprise firewalls that strip WebSocket Upgrade headers — common in healthcare and financial institutions. Long polling uses standard HTTP requests and works everywhere. (2) Serverless backends (AWS Lambda, Cloud Functions) where you can't hold persistent connections — each Lambda invocation is stateless and short-lived. Long polling fits naturally: each poll is an independent invocation. (3) Very low traffic where the operational overhead of a WebSocket server (heartbeats, connection state, pub/sub backplane) isn't justified for 50 concurrent users.

The gotcha: If an interviewer suggests "just use long polling" for anything at scale, push back on thread/connection exhaustion. You need an async runtime (Node.js, Go, Netty) to make long polling viable beyond a few thousand concurrent users.

Q4: A WebSocket server is handling 100K connections. Users report messages being delivered to the wrong chat rooms. What's the most likely cause?

The most likely cause is a connection-to-room mapping bug after reconnection. When a client reconnects (e.g., after a network blip), it gets a new connection ID but the server may not have cleaned up the old mapping. Scenario: User A was in Room X. Connection drops. Server still has old connection → Room X mapping. User A reconnects and joins Room Y. If the cleanup of the old mapping races with the new subscription, the user briefly exists in both rooms — messages from Room X leak to A's new connection, or A's messages are broadcast to Room X instead of Room Y.

Fix: Each reconnection must (1) atomically remove all old subscriptions for that user, then (2) re-subscribe based on the client's declared room. Use a user_id → set(room_ids) mapping, not connection_id → room_id, so the identity survives across connection cycles. Guard the update with a lock or use a serialized update queue per user.

Q5: How does SSE handle network failures differently from WebSocket?

SSE has built-in automatic reconnection in the browser's EventSource API — the browser will retry after a configurable delay (default: ~3 seconds, server can override with retry: field). On reconnection, it sends the Last-Event-ID header so the server can resume from where the client left off. This is all free — zero client code needed.

WebSocket reconnection is entirely your responsibility. The browser's WebSocket API has no auto-reconnect, no cursor tracking, no retry logic. You must implement: (1) detecting the close event, (2) exponential backoff with jitter, (3) tracking the last received message sequence, (4) requesting replay of missed messages on reconnect, (5) handling the case where the replay window has expired. Libraries like Socket.io abstract this, but it's hundreds of lines of code you get for free with SSE.

Interview takeaway: If you don't need bidirectional communication, SSE's built-in resilience is a massive operational advantage over WebSocket.

Q6: You need to push 50,000 events/second to connected clients. Discuss the tradeoffs between SSE and WebSocket for this workload.

At 50K events/sec, the key differentiator is per-message overhead. Each SSE event carries ~50 bytes of framing (id: ...\ndata: ...\n\n) plus the HTTP chunked transfer encoding overhead. A WebSocket frame carries 2–14 bytes. At 50K events/sec, that's ~2.5 MB/s of framing overhead for SSE vs ~700 KB/s for WebSocket. Over millions of connections, this adds up to significant bandwidth savings.

However, SSE over HTTP/2 multiplexes streams on a single TCP connection — so you avoid per-event TCP overhead. And SSE's text-only constraint means you can't send binary data efficiently; if your events are small JSON objects, this doesn't matter, but if they include compressed data or binary payloads, WebSocket's native binary frame support wins.

The nuance: At this volume, you're almost certainly fan-out broadcasting (one source → many subscribers), not peer-to-peer messaging. SSE + HTTP/2 + edge CDN caching can handle this with less infrastructure than WebSocket, because you don't need a pub/sub backplane — the CDN acts as your fan-out layer.

15Actual System Design Interview

Scenario: Design the real-time notification system for a social media platform (like Instagram). Users should see notifications (likes, comments, follows, DMs) within 1–2 seconds of the event occurring. The Prompt

"Design a notification delivery path that keeps user-facing latency under two seconds, works across web and mobile, and survives gateway restarts without dropping unread events."

Round 1: Requirements & Protocol Choice

Interviewer: "Walk me through your high-level approach."

The notification flow is predominantly unidirectional — events originate from other users' actions and need to reach the recipient's device. The recipient occasionally marks notifications as read (a simple POST), but the core data flow is server → client.

For the persistent connection layer, SSE over HTTP/2 is my first choice. Justification: (1) unidirectional push fits SSE perfectly, (2) auto-reconnect with Last-Event-ID eliminates an entire class of bugs, (3) works through every proxy/CDN without special config, (4) HTTP/2 multiplexing means multiple SSE streams per TCP connection. WebSocket would also work but adds bidirectional complexity we don't need.

For mobile clients on flaky connections, we'd add a push notification fallback (APNs/FCM) — if the SSE connection has been dead for >30s and there's a pending notification, fire a mobile push to wake the app.

Round 2: Architecture Deep-Dive

Interviewer: "50M concurrent connections is a lot. How do you handle the fan-out from an event like a celebrity posting a photo that triggers notifications for 10M followers?"

Two-tier fan-out:

For normal users (< 10K followers): when Alice likes Bob's post, the notification service writes to a per-user notification queue (e.g., a Kafka topic partitioned by user_id, or a Redis Sorted Set per user). Bob's SSE connection server reads from Bob's queue and pushes the event.

For celebrities (> 100K followers): we don't write 10M individual notification records at write time. Instead, we write a single "celebrity event" record. When each follower's SSE connection checks for new notifications, the gateway merges the follower's personal notification queue with a "celebrity timeline" — this is the fan-out-on-read pattern from DDIA (Kleppmann, Chapter 11: "pulling rather than pushing for accounts with many followers"). This avoids a 10M write amplification storm.

Connection layer architecture: A fleet of stateless SSE gateway servers. Each gateway holds open SSE connections and subscribes to a message bus (NATS or Redis Streams) for the user IDs it serves. A consistent hash ring maps user_id → gateway partition, so the upstream notification writer knows which gateway to notify. On gateway crash, clients reconnect with Last-Event-ID and get routed to a new gateway that replays from the per-user queue.

Round 3: Edge Cases

Interviewer: "What happens when a user opens the app on two devices simultaneously?"

Both devices maintain independent SSE connections, possibly to different gateway servers. Both must receive the same notifications. The per-user notification queue is the source of truth — both gateways subscribe to it. When Alice receives a notification, both her phone and tablet show it. When she marks it as read on one device, that action is written to the queue as a "read receipt" event, which propagates to the other device's SSE stream, updating the UI to show the notification as read.

Deduplication: Each notification has a unique ID. Clients maintain a local set of seen IDs and skip duplicates. This is essential because after a reconnection with replay, some already-displayed notifications will be re-sent.

Interviewer: "How do you handle the transition from SSE to push notifications on mobile when the app backgrounds?"

When the mobile client detects it's entering the background (iOS applicationDidEnterBackground, Android onPause), it closes the SSE connection and registers with a "device presence service" that the SSE connection is gone. New notifications for that user are now routed to a push notification pipeline (APNs/FCM). When the app returns to foreground, it re-opens the SSE stream with the last seen ID, replays anything missed during background, and deregisters from the push path. The presence service acts as a router: if SSE is alive, use it; if not, use mobile push.

18References & Further Reading

Use these sources to learn the protocol mechanics, connection lifecycle rules, and failure modes behind real-time delivery. The useful part is not the headline that "WebSockets are realtime," but the specific wire behavior, reconnect contract, and infrastructure breakpoints each source explains.

[1] RFC 6455 — The WebSocket Protocol (2011)

I. Fette, A. Melnikov. IETF, December 2011.
https://datatracker.ietf.org/doc/html/rfc6455

• Defines the WebSocket handshake: HTTP GET with Upgrade: websocket + Sec-WebSocket-Key nonce → server responds 101 Switching Protocols with SHA-1-hashed accept header proving protocol comprehension.

• Frame format: 2-byte minimum header with FIN bit, 4-bit opcode (text/binary/close/ping/pong), 1-bit mask flag, 7+ bit payload length. Client-to-server frames must be XOR-masked with a 4-byte random key to prevent cache poisoning on intermediary proxies.

• Defines six frame types (text, binary, close, ping, pong, continuation) and reserves ten for future extensions. The close handshake is bidirectional — either side can initiate, the other must respond with a close frame before the TCP connection is torn down.

• Connection can be extended via Sec-WebSocket-Extensions header (e.g., RFC 7692 permessage-deflate for compression) and subprotocols via Sec-WebSocket-Protocol.

• Designed explicitly for compatibility with HTTP infrastructure: uses port 80/443, survives through HTTP proxies when encrypted (wss://), and the upgrade mechanism reuses the initial HTTP connection's TCP socket.

[2] RFC 6202 — Known Issues and Best Practices for Long Polling and Streaming in Bidirectional HTTP (2011)

S. Loreto, P. Saint-Andre, S. Salsano, G. Wilkins. IETF, April 2011.
https://datatracker.ietf.org/doc/html/rfc6202

• Separates two very different HTTP realtime patterns: long polling, where one request blocks until data is ready and then closes, and streaming, where one response stays open and carries multiple events. That distinction matters because timeout handling, retry logic, and intermediary behavior differ.

• Explains the resource failure mode that made async runtimes necessary: in a synchronous server, every parked long-poll request consumes a thread or process slot, so concurrency is capped by idle waiters rather than useful work.

• Calls out intermediary breakpoints such as shared reverse-proxy connection pools and buffering behavior. In practice that is why a "working" long-poll prototype can fall apart once nginx, Apache, or a load balancer is inserted in front of it.

• Recommends sequence numbers or cursors on long-poll responses so clients can replay what arrived during the reconnect gap. That is the core correctness fix, not an implementation detail.

[3] HTML Living Standard — Server-Sent Events (WHATWG)

WHATWG. Ongoing.
https://html.spec.whatwg.org/multipage/server-sent-events.html

• Defines the on-wire SSE contract: a response with text/event-stream stays open, and the payload is framed as blocks of field: value lines terminated by a blank line. The important fields are data, event, id, and retry.

• The critical delivery mechanism is the id field. Once the server emits it, the browser stores that cursor and sends it back as Last-Event-ID on reconnect, which is what makes replay and gap recovery possible.

• The API exposes a narrow state machine through readyState and automatic reconnect behavior. That is why SSE is operationally attractive for one-way streams: the browser already owns reconnect timing and stream resumption semantics.

• The real infrastructure constraint is transport-level concurrency. Under HTTP/1.1, SSE consumes one of the browser's per-origin connection slots; under HTTP/2, the stream becomes one logical channel on a multiplexed connection, which changes whether multi-tab usage is viable.

[4] Designing Data-Intensive Applications — Chapter 11: Stream Processing (Kleppmann, 2017)

Martin Kleppmann. O'Reilly, 2017.
https://dataintensive.net

• Gives the right abstraction boundary for this topic: a live connection is only the delivery channel. The durable part of the system is still an event log or per-user queue that lets gateways replay after disconnects and failover.

• Breaks down fan-out-on-write versus fan-out-on-read, which is exactly the trade-off in large notification systems. Small accounts can afford eager writes to each follower queue; celebrity-scale accounts usually need a read-time merge path.

• Uses log-based brokers to show why reconnect safety depends on ordered offsets. If the client or gateway can name "the last event I applied," then replay becomes a bounded catch-up problem instead of guesswork.

• Frames delivery semantics honestly: most realtime systems are at least once after reconnect or retry, so duplicate suppression belongs in the client or consumer contract.

[5] RFC 8441 — Bootstrapping WebSockets with HTTP/2 (2018)

M. Thomson, C. Benfield. IETF, September 2018.
https://datatracker.ietf.org/doc/html/rfc8441

• Explains the protocol mismatch directly: HTTP/1.1 WebSockets rely on Upgrade, but HTTP/2 has no equivalent hop-by-hop upgrade path because everything is already framed as multiplexed streams.

• Introduces the exact replacement mechanism: the server advertises SETTINGS_ENABLE_CONNECT_PROTOCOL, and the client then uses extended CONNECT to open a WebSocket stream inside the H2 session.

• This RFC is useful because it separates transport efficiency from application behavior. H2 reduces connection sprawl, but it does not remove the need for heartbeats, reconnect, replay, draining, or cross-gateway routing.
