Key Terms

Latency

The duration between kicking off an operation and seeing its result. In web systems, this is typically the round-trip time from sending a request to receiving the full response — measured in milliseconds or seconds. These are the critical orders of magnitude to internalize:

OperationLatencyHuman Scale
Reading 1 MB from RAM250 µs (0.25 ms)A camera shutter click
Reading 1 MB from SSD1,000 µs (1 ms)A finger snap
Transfer 1 MB over 1 Gbps network10,000 µs (10 ms)A deep breath
Reading 1 MB from HDD20,000 µs (20 ms)An eye blink
Inter-continental round trip150,000 µs (150 ms)Berlin → Virginia → Berlin

Throughput

How many operations a system can process within a given time window. For a web server, this is typically expressed as requests per second (RPS) or queries per second (QPS). For a network link, it's bits or bytes per second. A system's throughput ceiling is determined by its most constrained resource — CPU, memory, disk I/O, or network bandwidth.

01

Why These Two Numbers Rule Everything

Every system design decision you'll ever make — choosing a database, designing an API, sizing a server cluster, picking a cache strategy — ultimately comes down to two questions: "how fast?" (latency) and "how much?" (throughput). They are the metrics by which all systems are judged.

Core Intuition — The Highway Analogy

Latency = how long it takes one car to drive from point A to point B. Throughput = how many cars can arrive at point B per hour. A country road has low latency (short distance) but low throughput (one lane). A highway has higher latency (longer on-ramp) but massive throughput (8 lanes). A system designer's job is to optimize both — build highways with fast speed limits.

Critically, they are not the same thing, and optimizing one can sometimes hurt the other. A server that batches requests to increase throughput will add latency (waiting for the batch to fill). A server that processes each request immediately has great latency but may have lower throughput (no batch efficiency). Understanding when to optimize for which — and the math that connects them — is the mark of a senior engineer.

02

Latency — Deep Dive

Latency is the time elapsed between initiating an operation and getting the result. In a client-server context: the time from when the client sends a request to when it receives the complete response. Also called response time or round-trip time (RTT).

Anatomy of a Request's Latency

WHERE DOES THE TIME GO? │ ├── Network latency (client → server) ~1-150 ms │ Speed-of-light delay + routing hops + TCP overhead │ Same datacenter: ~0.5ms. Same continent: ~30ms. Cross-ocean: ~150ms. │ ├── Queue wait time ~0-500+ ms │ If the server is busy, the request waits in a queue │ THIS is where latency explodes under load │ ├── Processing time ~1-1000 ms │ CPU computation, DB queries, external API calls │ A simple key lookup: ~1ms. A complex SQL join: ~100ms+. │ ├── I/O wait time ~0-20 ms │ Waiting for storage, cache lookup, or a downstream network call │ In-memory cache: microseconds. SSD: ~0.1-1ms. HDD: ~5-10ms. Remote DB: ~5-50ms. │ └── Network latency (server → client) ~1-150 ms Response travels back + serialization + TLS overhead TOTAL = Network (2×) + Queue + Processing + I/O Typical API call within datacenter: 5-50 ms Typical API call cross-continent: 200-500 ms

What Causes High Latency?

Physical distance: Light travels ~200,000 km/s in fiber. NYC → London = 5,500 km ≈ ~28ms one-way minimum. You can't beat physics — this is why CDNs exist.

Queueing: The #1 cause of latency spikes in production. When a server is at 80%+ utilization, queue depth grows non-linearly. At 95% utilization, a request might wait 20× its processing time.

Unnecessary I/O: Reading from disk when data could be cached in RAM. Making a synchronous external API call that could be async. N+1 database query patterns.

Serialization/deserialization: Converting objects to JSON (or protobuf). Large payloads take time to serialize, transmit, and deserialize.

04

Throughput — Deep Dive

Throughput is the number of operations a system can handle per unit time. It's measured in different units depending on context:

ContextUnitTypical Values
Web serverRPS (requests per second)1,000–100,000 RPS
DatabaseQPS (queries per second)1,000–50,000 QPS
Database (transactions)TPS (transactions per second)100–10,000 TPS
NetworkGbps (gigabits per second)1–100 Gbps
DiskMB/s or IOPSSSD: 3,500 MB/s / 500K IOPS
Message queuemsgs/sKafka: workload-dependent; high hundreds of thousands to millions/s at cluster scale

What Limits Throughput?

CPU: Computation-bound tasks (encryption, compression, serialization). More cores or faster CPUs increase throughput.

Memory: If the working set exceeds RAM, thrashing (swapping to disk) destroys throughput. Each swap adds milliseconds.

Disk I/O: HDD at 150 IOPS is the classic bottleneck. Upgrading to SSD (500K IOPS) is often the single biggest throughput win.

Network bandwidth: A 1 Gbps link can carry ~125 MB/s. If each response is 100KB, max throughput = ~1,250 RPS from bandwidth alone.

External dependencies: If your server calls a downstream API with 50ms latency and 100 connections, max throughput = 100 / 0.05 = 2,000 RPS (Little's Law).

05

Little's Law — The Missing Link

Latency and throughput are not independent — they're connected by Little's Law, one of the most powerful equations in system design. [2]

Little's Law

L = λ × W

L = average number of requests in the system (concurrency)
λ = average throughput (requests/second)
W = average latency per request (seconds)

Rearranged: Throughput = Concurrency / Latency

Practical Examples

Capacity Planning with Little's Law
Example 1: How many connections do I need?
  Target throughput:  10,000 RPS
  Average latency:    50 ms (0.05 s)
  Concurrency needed: 10,000 × 0.05 = 500 concurrent connections
  → You need a connection pool of at least 500.
  → If you only have 200 connections, max throughput = 200/0.05 = 4,000 RPS.

Example 2: What happens when latency doubles?
  Same 500 connections, but latency spikes to 100 ms
  New throughput: 500 / 0.1 = 5,000 RPS (halved!)
  → Doubling latency HALVES throughput at fixed concurrency.
  → This is why latency optimization is a throughput optimization.

Example 3: How many servers do I need?
  Each server handles: 1,000 RPS
  Target total:        50,000 RPS
  Servers needed:      50,000 / 1,000 = 50 servers
  → But add 20% headroom for spikes: 60 servers.
  → If latency per request drops from 10ms to 5ms,
     each server handles 2,000 RPS → only 25 servers needed. Half the cost!
Why Little's Law Is So Powerful

It works for any stable system, regardless of distribution, ordering, or architecture. It applies to database connection pools, thread pools, message queues, checkout lines, hospital ERs — anything with arrivals, processing, and departures. In system design interviews, it lets you do instant capacity math: "if each request takes X ms and we need Y RPS, we need Z concurrent workers."

06

Latency vs Throughput — The Tradeoff

Optimizing for latency and throughput often pulls in opposite directions. Understanding when to prioritize which is a key design skill.

TechniqueEffect on LatencyEffect on ThroughputExample
BatchingIncreases (wait to fill batch)Increases dramaticallyKafka producer batches messages; DB batch inserts
CachingDecreases (RAM vs disk)Increases (fewer I/Os)Redis in front of Postgres; CDN for static assets
CompressionMixed (CPU time vs transfer)Increases (smaller payloads)gzip HTTP responses; Snappy in Kafka
Connection poolingDecreases (skip handshake)Increases (reuse connections)Database connection pools; HTTP keep-alive
Async / non-blockingUnchanged per requestIncreases (no thread blocking)Node.js event loop; Python asyncio
Replication (read replicas)Decreases (local reads)Increases (more read capacity)PostgreSQL read replicas; Redis cluster
ShardingDecreases (smaller dataset)Increases (parallel processing)Cassandra partitions; MongoDB shards
Geo-distributionDecreases (closer to user)Depends (cross-region sync cost)Cloudflare edge; AWS multi-region
The Golden Rule

For user-facing systems, optimize for latency first. Users perceive latency directly — anything above 100ms feels sluggish, above 1s feels broken. For data pipelines and batch systems, optimize for throughput first. Nobody cares if a single Kafka message takes 50ms if you're processing 2 million per second.

Napkin Math — Back-of-Envelope Calculations

System design interviews expect you to do rough calculations quickly. Here's how to think about it. [3]

Napkin Math Exercises
Q: How long to read a 1GB file from SSD sequentially?
  SSD sequential throughput: ~3.5 GB/s (NVMe)
  Time: 1 GB / 3.5 GB/s = ~0.3 seconds

Q: How long to send 1GB over a 1 Gbps network?
  1 Gbps = 125 MB/s (bits vs bytes!)
  Time: 1,000 MB / 125 MB/s = ~8 seconds

Q: How many requests/day if we handle 1,000 RPS?
  1,000 × 86,400 (seconds/day) = ~86.4 million requests/day
  Handy shortcut: 1,000 RPS ≈ ~100M requests/day

Q: How much RAM to store 10M user sessions of 1KB each?
  10,000,000 × 1 KB = 10 GB
  → Fits in a single Redis instance. No problem.

Q: How many servers if each handles 500 RPS and we need 50K RPS?
  50,000 / 500 = 100 servers
  + 20% headroom = 120 servers

Q: If a page has 50 resource requests and each takes 50ms, total?
  Sequential: 50 × 50ms = 2,500 ms (unacceptable!)
  With HTTP/2 multiplexing (6 parallel): ~50 × 50ms / 6 ≈ ~420ms
  With CDN (80% cached, 10ms local): 40×10ms + 10×50ms = ~900ms → ~150ms
Shortcut: Powers of 2 and Time Constants

2^10 = 1,024 ≈ 1 thousand. 2^20 ≈ 1 million. 2^30 ≈ 1 billion. 86,400 seconds/day ≈ 10^5. 2.6M seconds/month ≈ 2.5 × 10^6. 1 Gbps = 125 MB/s. Memorize these and napkin math becomes second nature.

Percentiles — Why Averages Lie

Never report average latency in a system design discussion. It hides the pain. Use percentiles instead.

PercentileMeaningWhy It Matters
p50 (median)50% of requests are faster than thisThe "typical" experience
p9595% of requests are faster; 5% are slowerThe experience for 1 in 20 users
p9999% are faster; 1% are slowerThe experience for 1 in 100 users
p99.999.9% are faster; 0.1% are slowerThe "tail" — often 10-100× the median
Why averages lie — a more realistic example
Imagine 1,000 requests:
  990 requests complete in 5 ms
  9 requests take 120 ms (cache miss / extra hop)
  1 request takes 5,000 ms (timeout / cold path / bad retry)

  Average: (990×5 + 9×120 + 5000) / 1000 = 11.03 ms  ← looks great!
  p50:    5 ms                                 ← typical request
  p99:    120 ms                               ← 1 in 100 users notices
  p99.9:  5,000 ms                             ← tail event is catastrophic

The average says "11ms, looks great!" but the tail says your slowest
users are still waiting seconds. That's why p95/p99/p99.9 matter.
The Tail-at-Scale Problem

Google's seminal paper [4] showed that in fan-out architectures (where one user request touches 100+ backend servers), even small tail latencies compound. If each server has a 1% chance of being slow, and you query 100 servers, you have a 63% chance that at least one is slow — and your response is as slow as the slowest server. This is why Google obsesses over p99.9 and uses techniques like hedged requests (send the same request to two servers, take the faster response).

Mini-Project: Latency & Throughput Profiler

Build a Python server that measures its own latency distribution and throughput in real time, and reports p50/p95/p99 percentiles.

latency_profiler.py — self-instrumenting HTTP server
import time, json, random, statistics
from http.server import ThreadingHTTPServer, BaseHTTPRequestHandler
from collections import deque

latencies = deque(maxlen=10000)  # rolling window of last 10K requests
request_times = deque(maxlen=10000)

def percentile(data, p):
    if not data: return 0
    s = sorted(data)
    k = int(len(s) * p / 100)
    return s[min(k, len(s)-1)]

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        t0 = time.perf_counter()
        request_times.append(t0)

        if self.path == "/stats":
            # Calculate throughput: requests in last 1 second
            now = time.perf_counter()
            rps = sum(1 for t in request_times if now - t < 1.0)
            lats = list(latencies)
            stats = {
                "total_requests": len(lats),
                "throughput_rps": rps,
                "latency_ms": {
                    "p50":  round(percentile(lats, 50), 3),
                    "p95":  round(percentile(lats, 95), 3),
                    "p99":  round(percentile(lats, 99), 3),
                    "p999": round(percentile(lats, 99.9), 3),
                    "max":  round(max(lats), 3) if lats else 0,
                }
            }
            body = json.dumps(stats, indent=2)
        else:
            # Simulate variable work — 95% fast, 5% slow (cache miss)
            if random.random() < 0.05:
                time.sleep(random.uniform(0.05, 0.2))  # 50-200ms "disk read"
            else:
                time.sleep(random.uniform(0.001, 0.005))  # 1-5ms "cache hit"
            body = '{"status":"ok"}'

        elapsed_ms = (time.perf_counter() - t0) * 1000
        latencies.append(elapsed_ms)

        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.end_headers()
        self.wfile.write(body.encode())

    def log_message(self, *a): pass

print("[Profiler] http://localhost:9090")
print("  GET /         — simulated endpoint (95% fast, 5% slow)")
print("  GET /stats    — live p50/p95/p99 latency + throughput\n")
ThreadingHTTPServer(("", 9090), Handler).serve_forever()
Run and experiment
# Terminal 1: Start the server
$ python3 latency_profiler.py

# Terminal 2: Generate load (100 requests)
$ for i in $(seq 1 100); do curl -s localhost:9090/ > /dev/null & done; wait

# Terminal 2: Check the stats
$ curl -s localhost:9090/stats | python3 -m json.tool
{
  "total_requests": 100,
  "throughput_rps": 42,
  "latency_ms": {
    "p50":  3.1,       ← most requests are fast (cache hit)
    "p95":  112.4,     ← 5% are slow (simulated cache miss)
    "p99":  187.2,     ← the long tail
    "p999": 198.1,
    "max":  199.8
  }
}
# Notice: p50 is ~3ms but p99 is ~187ms — the average would hide this!

Production Example 1: Google's Tail Latency

A single Google Search query fans out to hundreds of backend servers in parallel. The response time = the slowest server. [4]

Why tail latency matters at Google scale
A Google search query touches ~100 index servers in parallel.
Each server has p99 latency = 10ms.
Probability ALL 100 are fast: 0.99^100 = 36.6%
Probability at LEAST ONE is slow: 1 - 0.366 = 63.4%

So 63% of user searches are bottlenecked by tail latency!

Google's mitigations:
1. Hedged requests:   Send request to 2 replicas, take faster response.
                       Cost: 2× backend load. Benefit: p99 drops dramatically.
2. Tied requests:      Send to 2 servers but let them coordinate —
                       when one starts, it tells the other to cancel.
3. Micro-partitioning: Split data into many small partitions so work
                       can be rebalanced quickly away from slow machines.
4. Background warm-up: Pre-warm caches so cold misses don't cause spikes.

Production Example 2: Kafka Throughput

Apache Kafka is optimized for throughput over latency — a deliberate design choice for data pipelines. Kafka's own design docs emphasize batching, sequential disk access, compression, and zero-copy transfers as the main efficiency levers. [5]

How Kafka is designed for high throughput
Technique 1: Batching
  Producer groups records into batches before sending them
  linger.ms lets the producer wait briefly for a fuller batch
  → Fewer requests, larger packets, better network and broker efficiency

Technique 2: Sequential I/O
  Kafka appends to partition logs sequentially instead of doing random updates
  → Sequential access is much friendlier to disks and page cache than random I/O

Technique 3: Zero-copy (sendfile)
  Brokers can use sendfile-style transfer paths when serving data to consumers
  → Fewer copies and fewer context switches on the hot path

Technique 4: Partitioning
  A topic is split into partitions, each an independent append-only log
  → More partitions create more parallelism, subject to broker, disk, and consumer limits

Result:
  High sustained throughput for log-style workloads
  Exact rates depend on record size, replication factor, acks, compression,
  hardware, and whether the bottleneck is disk, CPU, or network.
The Kafka Tradeoff

Kafka explicitly trades some per-record latency for better throughput. linger.ms lets the producer wait briefly for more records before sending a batch. Kafka's official producer docs note that the default is 5ms in Kafka 4.0, and that larger batches often improve efficiency and can even keep end-to-end latency similar or lower under load. The tradeoff is still real: aggressive batching helps throughput, but it can add queueing delay for individual records. [5]

Edge Cases & Gotchas

1. Coordinated Omission

Many benchmarking approaches (including naive use of curl in a loop) have a fatal flaw: if request 1 takes 5 seconds, request 2 doesn't start until request 1 finishes. This hides the impact of slow requests on subsequent ones. Real users don't wait — they arrive continuously. wrk is useful for generating load and seeing throughput/latency distributions, but wrk2 was created specifically to maintain a constant request rate and expose coordinated omission more accurately. Gil Tene's "How NOT to Measure Latency" talk is essential viewing.

2. Goodhart's Law in Latency

"When a measure becomes a target, it ceases to be a good measure." If you optimize purely for p50 latency, you might make the tail worse (e.g., deferring garbage collection → occasional 500ms GC pauses). Always monitor p50, p95, p99, AND p99.9 together.

3. Bandwidth ≠ Throughput

A 10 Gbps network link doesn't mean 10 Gbps of useful throughput. Protocol overhead (TCP/IP headers), small packet sizes, congestion, and application-level framing all reduce effective throughput. A single TCP stream over a 100ms RTT link might only achieve ~300 Mbps due to the congestion window. You need multiple parallel streams to saturate the link.

4. The "Thundering Herd" Throughput Cliff

When a cache expires and 10,000 requests simultaneously hit the backend, throughput drops to zero while the backend is overwhelmed. Fix: cache stampede protection (only one request refills the cache, others wait), probabilistic early expiration, or stale-while-revalidate.

5. Latency Spikes from GC Pauses

JVM garbage collection (Java, Scala, Kotlin) can cause stop-the-world pauses of 50-500ms. Go's GC is better (~1ms), but still non-zero. This creates p99 latency spikes that don't show up in p50. Modern approach: use low-latency GC (ZGC, Shenandoah) or GC-free languages (Rust, C++) for latency-critical paths.

Production War Story

Large internet products have repeatedly found the same pattern: added latency hurts engagement, conversion, and perceived quality. The exact impact varies by product and user behavior, but the systems lesson is stable: speed is a feature, and latency compounds into business pain surprisingly fast.

Interview Questions & Answers

Q1: What's the difference between latency and throughput?

Latency = how long one operation takes (time per request). Throughput = how many operations per unit time (requests per second). They're connected by Little's Law: Throughput = Concurrency / Latency. A system can have low latency but low throughput (single-threaded), or high latency but high throughput (batched processing). The goal is usually to minimize latency while maximizing throughput, but these sometimes conflict.

Q2: You need to support 100K RPS with 10ms average latency. How many servers?

Use Little's Law: Concurrency per server = Throughput × Latency. If each server has 100 connections: Throughput per server = 100 / 0.01 = 10,000 RPS. Servers needed = 100,000 / 10,000 = 10 servers. Add 20-30% headroom for spikes: 12-13 servers. This assumes the bottleneck is concurrency, not CPU or memory — in practice, benchmark a single server first to find its actual capacity.

Q3: Why should you use p99 latency instead of average?

Averages hide tail latency. A service can look healthy on average while still producing painful outliers for real users. In fan-out architectures, tail latencies compound — if a request touches 100 services, the slowest dependency often dominates the end-to-end response. That's why serious systems track p50, p95, p99, and sometimes p99.9 together instead of relying on a single average.

Q4: How does caching reduce latency? Give specific numbers.

Caching moves data up the storage hierarchy. A disk-backed read might take milliseconds. An SSD-backed read is often sub-millisecond to low-millisecond. A Redis lookup is typically sub-millisecond, and an in-process memory cache is usually measured in microseconds. That gap is why caching is so powerful: it avoids slow storage and network round trips for hot data. The important metrics are cache hit ratio, TTL, eviction policy, and cache invalidation strategy.

Q5: A service has 50ms latency at low load but 500ms under high load. Why?

Queueing theory. At low load, requests are processed immediately. As utilization increases, requests start waiting in queues. The queueing delay grows non-linearly — at 50% utilization it's roughly equal to processing time (doubling latency), at 90% it's 9× processing time, and at 99% it's 99×. This follows the M/M/1 queue model: wait time = processing_time × (utilization / (1 - utilization)). Fixes: add more capacity (horizontal scaling), reduce processing time (optimization), or shed load (rate limiting, circuit breakers).

Q6: State Little's Law and use it to solve a capacity problem.

L = λ × W, where L = concurrency, λ = throughput, W = latency. Example: API gateway needs 50,000 RPS, each request takes 20ms (0.02s). Concurrency = 50,000 × 0.02 = 1,000. You need 1,000 concurrent connections. If your connection pool is sized at 500, max throughput = 500 / 0.02 = 25,000 RPS — you'll hit a ceiling. Solution: either increase the pool to 1,000, or reduce latency to 10ms (then 500 connections support 50,000 RPS). Little's Law shows that latency reduction is capacity creation.

Q7: Name 5 techniques to reduce latency in a web application.

(1) Caching — put hot data in Redis or CDN to avoid disk/network round trips. (2) Geo-distribution — serve users from nearby data centers/edge locations to reduce network RTT. (3) Connection pooling — reuse TCP/TLS connections to avoid handshake latency. (4) Async/parallel I/O — issue multiple downstream requests in parallel instead of sequentially. (5) Data locality — co-locate compute near data (same AZ, same rack) to minimize network hops. Bonus: (6) Compression — reduce payload size to reduce transfer time. (7) Protocol optimization — HTTP/2 multiplexing, HTTP/3 QUIC 0-RTT.

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Latency = time per operation. Throughput = operations per time. They're connected by Little's Law: L = λ × W.
2. Memorize: RAM ~250µs/MB, SSD ~1ms/MB, Network ~10ms/MB, HDD ~20ms/MB, cross-continent ~150ms round trip.
3. Use percentiles (p50, p95, p99), never averages. Averages hide tail latency that hurts real users.
4. Little's Law: Throughput = Concurrency / Latency. Halving latency doubles throughput at fixed concurrency.
5. Queueing is the #1 cause of latency spikes. At 90% utilization, wait time ≈ 9× processing time.
6. Caching moves data up the hierarchy: disk is milliseconds, SSD is often sub-ms to low-ms, RAM/Redis is sub-ms, local memory is microseconds.
7. Batching trades latency for throughput. Fan-out can reduce wall-clock time, but it increases backend load and amplifies tail-latency risk.
8. For user-facing: optimize latency. For data pipelines: optimize throughput. Know which you're designing.
9. Tail latency compounds in fan-out: p99 of 100 services is your real latency. Use hedged requests.
10. Napkin math is a superpower: 1,000 RPS ≈ 86M requests/day. 1 Gbps ≈ 125 MB/s. Know the shortcuts.

Mental Model — Queueing Under Saturation

Latency vs utilization: the "hockey stick" — flat until ~70%, then exponential

Diagram Interpretation

Top: the queueing model. Requests arrive at 900/s. The server processes at 1000/s capacity (1ms each). At 90% utilization, the system looks fine on paper — headroom exists. But the queue tells a different story.

The curve: This is the M/M/1 queueing model. At low utilization (10-50%), latency is barely above the processing time (~1ms). At 70%, it's ~3ms. At 90%, it's ~10ms (10× processing time). At 95%, it's ~20ms. At 99%, it's ~100ms. The curve is exponential — latency doesn't degrade linearly, it explodes.

The danger zone (>80%): This is where most production systems break. Traffic spikes push utilization from 70% to 90%, and latency jumps 5× in seconds. Users start seeing timeouts, retries add more load, and the system enters a death spiral. This is why capacity planning targets ~60-70% utilization — the remaining headroom absorbs spikes without entering the hockey stick.

The fix: Either add capacity (more servers → lower utilization per server) or reduce processing time (optimize queries, add caching → shift the curve left). Both work, but caching is usually cheaper and faster to implement.

Key Intuition

Latency is flat until utilization crosses ~70%, then it explodes exponentially. This is the single most important graph in system design. When someone says "the system is slow under load," this curve is why. Never run a production system above 70-80% average utilization — the remaining headroom isn't waste, it's your insurance against the hockey stick.

References & Further Reading

[1] Jeff Dean / Peter Norvig — "Latency Numbers Every Programmer Should Know"

Jeff Dean, Google. Originally presented around 2009 and widely circulated in systems courses and talks.
Interactive version (Colin Scott) · Modern update: sirupsen/napkin-math

• The canonical reference for memory, disk, and network latencies — L1 cache (0.5ns) through inter-continental RTT (150ms).

• The "humanized" version multiplies all times by 1 billion: RAM access = 1.5 minutes, SSD read = 1.7 days, HDD seek = 16.5 weeks.

• sirupsen/napkin-math is a practical, modern companion reference with broader benchmarking examples and cost intuition.

• These numbers are the foundation of napkin math — the back-of-envelope calculations expected in every system design interview.

[2] Little, J.D.C. — "A Proof for the Queuing Formula: L = λW" (1961)

John D.C. Little. Operations Research, Vol. 9, No. 3, 1961, pp. 383-387. DOI: 10.1287/opre.9.3.383.
https://doi.org/10.1287/opre.9.3.383

• Proves that for any stable system: average items in system (L) = arrival rate (λ) × average time per item (W). Remarkably, this holds regardless of arrival distribution, service order, or number of servers.

• Applied to system design: Concurrency = Throughput × Latency. This single equation drives capacity planning for connection pools, thread pools, and server fleets.

• Rearranged: Throughput = Concurrency / Latency. Reducing latency by half doubles throughput at fixed concurrency — latency optimization IS capacity creation.

• Used daily by engineers at every scale — from sizing a database connection pool to planning a fleet of 10,000 servers.

[3] sirupsen/napkin-math — "Napkin Math: Estimate System Performance"

Simon Eskildsen (former Principal Engineer, Shopify).
https://github.com/sirupsen/napkin-math

• A practical, modern collection of latency, throughput, and cost benchmarks for back-of-the-envelope capacity reasoning.

• Numbers are benchmarked using Criterion.rs on real hardware, not estimates. Each number has a referenced source.

• Useful as a sanity-checking companion when you need rough orders of magnitude for storage, network, and cloud primitives.

• Essential reference for system design interviews where you need to estimate "how fast" and "how much" for real infrastructure.

[5] Apache Kafka Documentation — Design and Producer Configuration

Apache Kafka Project. Official documentation for Kafka's design tradeoffs and producer batching behavior.
Design efficiency · Producer configs

• Kafka's design docs emphasize batching, sequential I/O, compression, and reduced byte-copying as the main reasons log-style workloads can achieve very high throughput.

• The producer docs explain how batch.size and linger.ms influence batching behavior and throughput.

• The official docs note that linger.ms defaults to 5 in Kafka 4.0, and that larger batches often improve efficiency enough to keep observed latency similar or lower under load.

[4] Dean & Barroso — "The Tail at Scale" (2013)

Jeffrey Dean, Luiz André Barroso. Communications of the ACM, Vol. 56, No. 2, February 2013, pp. 74-80.
https://research.google/pubs/the-tail-at-scale/

• Demonstrates that in fan-out architectures, tail latencies (p99, p99.9) dominate user experience — if one request touches 100 servers, there's a 63% chance at least one is slow.

• Introduces hedged requests: send the same request to two replicas, use whichever responds first. Reduces tail latency at the cost of 2× backend load.

• Introduces tied requests: cross-server cancellation to avoid wasting resources on the slower replica.

• Argues that reducing variability is more important than reducing average latency — a system with p50=5ms and p99=5ms beats one with p50=2ms and p99=500ms.

• One of the most important systems papers of the 2010s. Required reading for anyone building large-scale distributed services.
