Load Balancers

A single server has a ceiling. Load balancers exist to make many servers look like one — distributing requests so no individual machine drowns while others sit idle. Every large-scale system has at least one, and most have several at different layers.

Prerequisites & Key Terms

Load Balancer

A device or software process that distributes incoming network traffic across a pool of backend servers. It's a specialized form of reverse proxy whose primary job is ensuring no single server is overwhelmed while others are underutilized. Load balancers appear at multiple layers of a system — between users and web servers, between web servers and application servers, between application servers and databases. Real-world examples: Nginx, HAProxy, AWS ALB/NLB, Google Cloud Load Balancer, F5 BIG-IP, Envoy.

Server-Selection Strategy

The algorithm a load balancer uses to decide which backend receives the next request. The choice of strategy directly impacts load distribution, latency, and whether "sticky" behavior (same user → same server) is maintained. Common strategies include round-robin, weighted round-robin, least connections, IP hash, random, and least response time. Each has trade-offs — simpler algorithms are faster but less adaptive; smarter algorithms add latency per decision.

Hot Spot

When traffic concentrates on a subset of servers in the pool, overloading them while others remain idle. This can happen due to a poor selection algorithm (e.g., hash collisions sending many users to the same server), a skewed workload (certain user IDs generate far more traffic), or heterogeneous server capacity (all servers weighted equally even though some are weaker). Hot spots defeat the purpose of load balancing and are a primary failure mode to watch for.

Health Check

A periodic probe the load balancer sends to each backend to verify it's alive and responding correctly. Backends that fail health checks are temporarily removed from the pool so traffic isn't sent to dead or degraded servers. Health checks can be shallow (TCP connect — is the port open?) or deep (HTTP GET /health — does the app return 200 with valid data?). Deep checks catch more failures but cost more resources.

01

Why Load Balancers Exist

A single server has hard limits — CPU cores, RAM, network bandwidth, connection slots. Once traffic exceeds those limits, the only options are vertical scaling (bigger server — expensive, has a ceiling) or horizontal scaling (more servers — cheaper, nearly unlimited). Load balancers enable horizontal scaling by making a fleet of servers act as one.

Core Intuition

Imagine a grocery store with one checkout lane. During rush hour, the line wraps around the store. The fix isn't a faster cashier (vertical scaling) — it's opening more lanes (horizontal scaling). But then you need someone at the front directing customers to available lanes — that's the load balancer. A good one sends customers to the shortest line (least connections). A basic one just alternates lanes (round-robin). A bad one sends everyone to lane 3 (hot spot).

Load balancers provide three core benefits: (1) Throughput — distribute requests across N servers, roughly multiplying capacity by N. (2) Availability — if one server dies, the LB routes around it. (3) Flexibility — add/remove servers without users noticing (zero-downtime deployments, auto-scaling).

L4 vs L7 — The Two Types

Load balancers operate at two distinct network layers, and the choice has major implications for what decisions they can make.

PropertyL4 (Transport Layer)L7 (Application Layer)
Operates onTCP/UDP packets (IP + port)HTTP requests (URL, headers, cookies, body)
SeesSource IP, dest IP, source port, dest port, protocolEverything L4 sees + HTTP method, URL path, Host header, cookies, query params
Routing decisionsBased on IP/port only (can't inspect content)Based on URL path, hostname, headers, cookies — rich routing rules
SSL/TLSPasses encrypted traffic through (TCP passthrough) OR terminates TLSAlways terminates TLS (must read HTTP)
PerformanceFaster — less parsing, kernel-level forwarding possibleSlower — must parse HTTP, buffer headers, potentially inspect body
Sticky sessionsIP hash only (coarse)Cookie-based (precise, per-user)
Use caseHigh-throughput TCP (databases, game servers, gRPC, any non-HTTP)HTTP APIs, web apps, microservices, path-based routing
ProductsAWS NLB, HAProxy (TCP mode), Google Maglev, Linux IPVSAWS ALB, Nginx, HAProxy (HTTP mode), Envoy, Traefik
When to Pick Which

Default to L7 for HTTP/HTTPS traffic — you get path-based routing, cookie stickiness, header inspection, and HTTP-level health checks. Use L4 when you need raw TCP/UDP forwarding (databases, message queues, game servers), or when you need maximum throughput with minimal latency and don't need content-based routing. Many architectures use both: an L4 NLB in front for raw packet distribution, with L7 ALBs behind for HTTP routing.

Interview Insight — Power of Two Choices

The "Power of Two Random Choices" algorithm is a strong interview talking point because it keeps most of the benefit of global load awareness without scanning every backend. In the classic balls-into-bins analysis, sampling two candidates dramatically improves over pure random assignment, and Envoy's least request policy uses a two-choice strategy by default within that policy. Health Checks

A load balancer is only as good as its ability to detect dead or degraded backends. Health checks are the mechanism for this.

TypeHow It WorksDetectsMisses
TCP ConnectTry to open a TCP connection to the backend portProcess crashed, port not listening, machine downApp is up but returning errors (500s), slow but alive
HTTP GETSend GET /health, expect 200 OKApp crashes, misconfigurations, dependency failures (if /health checks deps)App returns 200 but is actually degraded for real traffic
Deep Health/health checks DB connectivity, cache availability, disk spaceDependency failures, resource exhaustionExpensive — can itself become a bottleneck if checks are heavy
Typical health check configuration (Nginx-style)
# Check every 10 seconds. If 3 consecutive failures → mark unhealthy.
# If 2 consecutive successes → mark healthy again.
health_check interval=10s fails=3 passes=2;

# Detection time for a crash:
#   Worst case: interval × fails = 10s × 3 = 30 seconds
#   During those 30s, ~1/N of all requests go to the dead server → errors
#   Fix: shorter intervals (5s) or fewer fails (2) — but more false positives

# Recovery time:
#   interval × passes = 10s × 2 = 20 seconds before re-entering pool
#   This prevents a flapping server from cycling in and out

DNS Load Balancing

The simplest form of load balancing: return different IP addresses from DNS queries. Large services like Google often return multiple frontend IPs for a single domain, and repeated dig queries may produce different answers over time.

See DNS load balancing in action (macOS/Linux/WSL — dig is pre-installed on most systems)
# Query google.com twice — often returns different IPs
$ dig +short google.com
142.250.185.78

$ dig +short google.com
142.250.185.110     ← different IP! DNS rotated it

# dig is installed by default on macOS and most Linux distros.
# On Ubuntu if missing: sudo apt install dnsutils

# See ALL IPs returned (Google often returns multiple A records):
$ dig google.com A +short
142.250.185.78
142.250.185.110
142.250.185.46     ← 3 IPs. Client picks one (usually the first).

# Each IP points to a different data center or server cluster.
# This IS load balancing — at the DNS layer.
DNS LB Limitations

No health awareness: DNS doesn't know if a server is healthy — it returns IPs from a static list. If a server dies, DNS keeps returning its IP until someone manually removes it or the TTL-based record propagates. Client caching: Browsers and OSes cache DNS responses. Even after you remove a dead IP, clients may use the stale entry for minutes or hours. No fine-grained control: DNS can't do least-connections, weighted routing, or path-based decisions. Use DNS LB for geo-routing between regions, but use a real LB (Nginx, ALB) within each region for fine-grained distribution.

Mini-Project: Build a Load Balancer

Simple: Feel What Load Balancing Means

simple_lb.py — the smallest load balancer (macOS/Linux/WSL, Python 3.7+)
import http.server, urllib.request, itertools

# Round-robin pool of backends
BACKENDS = ["http://127.0.0.1:8001", "http://127.0.0.1:8002"]
pool = itertools.cycle(BACKENDS)

class LB(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        backend = next(pool)
        try:
            resp = urllib.request.urlopen(f"{backend}{self.path}", timeout=3)
            self.send_response(200)
            self.send_header("X-Backend", backend)
            self.end_headers()
            self.wfile.write(resp.read())
        except:
            self.send_response(502); self.end_headers()
            self.wfile.write(f"Backend {backend} down".encode())
    def log_message(self, *a): pass

print("[LB] localhost:7070 → round-robin across", BACKENDS)
http.server.HTTPServer(("", 7070), LB).serve_forever()

Advanced: Least-Connections + Health Checks

smart_lb.py — least-connections with health checks (macOS/Linux/WSL, Python 3.7+)
import http.server, urllib.request, threading, time

BACKENDS = ["http://127.0.0.1:8001", "http://127.0.0.1:8002", "http://127.0.0.1:8003"]
conns = {b: 0 for b in BACKENDS}    # Active connections per backend
healthy = {b: True for b in BACKENDS}  # Health status

def health_checker():
    """Check each backend every 5 seconds."""
    while True:
        for b in BACKENDS:
            try:
                urllib.request.urlopen(b, timeout=2)
                if not healthy[b]: print(f"  [HC] {b} recovered ✓")
                healthy[b] = True
            except:
                if healthy[b]: print(f"  [HC] {b} FAILED ✗")
                healthy[b] = False
        time.sleep(5)

class SmartLB(http.server.BaseHTTPRequestHandler):
    def do_GET(self):
        # Pick the healthy backend with the fewest active connections
        candidates = [b for b in BACKENDS if healthy[b]]
        if not candidates:
            self.send_response(503); self.end_headers()
            self.wfile.write(b"No healthy backends"); return

        backend = min(candidates, key=lambda b: conns[b])
        conns[backend] += 1
        try:
            resp = urllib.request.urlopen(f"{backend}{self.path}", timeout=5)
            self.send_response(200)
            self.send_header("X-Backend", backend)
            self.send_header("X-Connections", str(conns))
            self.end_headers()
            self.wfile.write(resp.read())
        except:
            self.send_response(502); self.end_headers()
            self.wfile.write(f"Backend error: {backend}".encode())
        finally:
            conns[backend] -= 1
    def log_message(self, *a): pass

threading.Thread(target=health_checker, daemon=True).start()
print(f"[Smart LB] localhost:7070 → least-connections with health checks")
http.server.HTTPServer(("", 7070), SmartLB).serve_forever()
Experiments to Try

1. Start 3 backends, run the smart LB, send requests — watch the X-Connections header show even distribution.
2. Kill one backend — the health checker detects it within 5s and stops routing to it.
3. Restart the backend — the health checker recovers it and starts routing again.
4. Add a time.sleep(0.5) to one backend to simulate slowness — the least-connections algorithm naturally sends fewer requests to the slow one because its connection count stays higher.

Production Example 1: Google Maglev

Maglev is Google's software network load balancer — a distributed L4 system running on commodity Linux servers that has handled Google's incoming traffic since 2008. [1]

Key Design Decisions

Software, not hardware: Traditional hardware LBs (F5, Citrix) are expensive, hard to scale, and provide only 1+1 (active-passive) redundancy. Maglev runs on the same commodity servers as Google's services, enabling N+1 redundancy — all machines are active, and capacity scales by adding more machines.

ECMP distribution: Network routers use Equal-Cost Multipath routing to distribute packets evenly across all Maglev machines. No single Maglev is a bottleneck or single point of failure.

Consistent hashing: Maglev uses a custom consistent hashing algorithm (Maglev hashing) that provides better load distribution than standard consistent hashing while minimizing disruption when machines are added or removed.

Line-rate performance: A single Maglev machine can saturate a 10Gbps link with small packets. This is achieved by bypassing the Linux kernel network stack and using kernel bypass techniques for packet processing.

Key Lesson from Maglev

Maglev demonstrates that software load balancers on commodity hardware can compete with and often outperform dedicated hardware LBs while being easier to scale operationally. It is a strong case study for why cloud platforms favored software-defined load balancing instead of proprietary appliances. If an interviewer asks about hardware vs software LBs, Maglev is one of the best examples to cite. [1]

Edge Cases & Gotchas

1. The Sticky Session Trap

IP-hash or cookie-based stickiness sends the same user to the same backend — useful for in-memory sessions. But if that backend dies, the user loses their session. And during scale-in (removing servers), sticky users on the removed server are disrupted. Fix: externalize session state to Redis/Memcached so any backend can serve any user. Stickiness becomes unnecessary.

2. Thundering Herd on Backend Recovery

A backend passes its health check and re-enters the pool. The LB immediately sends it a full share of traffic. If the backend needs time to warm up (JVM JIT, cache cold start), it gets overwhelmed and fails again — oscillating between healthy and unhealthy (flapping). Fix: use a slow start ramp so a newly healthy backend gradually receives traffic instead of taking a full share instantly. Envoy documents this behavior explicitly. [5]

3. The Load Balancer as SPOF

A single LB instance is a single point of failure. Fix: deploy multiple LB instances behind DNS round-robin, or use a cloud-managed LB configured across multiple Availability Zones. For self-managed Nginx/HAProxy, use keepalived with a floating VIP for active-passive failover between two LB nodes.

4. Connection Draining on Deploys

During a rolling deployment, you remove an old backend and add a new one. If you remove the old backend immediately, all in-flight requests to it fail. Fix: connection draining — mark the backend as "draining" (stop sending new requests, but let existing connections finish for up to N seconds). Different products expose this under names like drain, graceful shutdown, or AWS's deregistration delay. [4]

5. Uneven Backends (Heterogeneous Fleet)

If you mix c5.xlarge (4 vCPUs) and c5.4xlarge (16 vCPUs) instances behind a round-robin LB, the small instances get the same traffic as the big ones — and collapse under load. Fix: weighted round-robin with weights proportional to capacity, or use least-connections which naturally adapts.

Production War Story

A large SaaS provider experienced intermittent 502 errors after deploying a new version. The LB's health check was a simple TCP connect (port open = healthy). The new version had a bug that made it return 500 errors for ~10% of requests, but the port was always open, so the health check never caught it. After switching to an HTTP-level health check (GET /health expecting 200 + valid JSON), the buggy instances were automatically removed. The lesson: TCP health checks are not enough for HTTP services — always use deep, application-level health checks.

What If? — Interactive Exploration

Tweak Parameters
Impact

Tradeoff Cards

Round-robin vs least-connections

Round-robin: Even distribution, zero overhead, but ignores server load. A slow query on one server doesn't reduce its traffic share.
Least-connections: Adapts to real load, but requires connection tracking. Better for workloads with variable request duration (some requests take 5ms, others 500ms).

L4 vs L7 load balancing

L4: Faster (kernel-level packet forwarding), works for any TCP/UDP protocol, but can't make content-based routing decisions.
L7: Can route by URL/header/cookie, can terminate SSL, can cache responses, but adds ~1ms and must parse HTTP.

Adding more servers vs bigger servers

More servers (horizontal): Linear cost, near-unlimited scale, needs a load balancer, stateless design required.
Bigger servers (vertical): Simpler (no LB needed), but cost grows superlinearly, hard ceiling, single point of failure. Horizontal always wins long-term.

N+1 capacity planning

Improves: Survives one server failure without overloading the rest. If you have 4 servers and each runs at 75% capacity, losing one puts the remaining 3 at 100% — survivable.
Costs: You're paying for an "extra" server that only works during failures.
When worth it: Always. If you're running at 100% and any failure means an outage, you're one crash away from a bad day.

Toy Simulation: Algorithm Comparison

lb_sim.py — compare algorithms on skewed workload (macOS/Linux/WSL)
import random

def simulate(algo, n_servers, n_requests, request_costs):
    """Simulate load distribution. Returns per-server total load."""
    load = [0] * n_servers
    active = [0] * n_servers
    rr_idx = 0

    for cost in request_costs:
        if algo == "round_robin":
            server = rr_idx % n_servers
            rr_idx += 1
        elif algo == "random":
            server = random.randint(0, n_servers - 1)
        elif algo == "least_conn":
            server = min(range(n_servers), key=lambda i: active[i])
        elif algo == "power_of_two":
            a, b = random.sample(range(n_servers), 2)
            server = a if active[a] <= active[b] else b

        load[server] += cost
        active[server] += cost  # Simulate active connections proportional to cost

    return load

N_SERVERS = 5
N_REQUESTS = 10_000
# Skewed workload: most requests are cheap, a few are very expensive
costs = [random.choice([1,1,1,1,1,1,1,1,10,50]) for _ in range(N_REQUESTS)]

print(f"Simulating {N_REQUESTS} requests across {N_SERVERS} servers")
print(f"Workload: 80% cost=1, 10% cost=10, 10% cost=50 (skewed)\n")
print(f"{' Algorithm':-<22} {'Max Load':>10} {'Min Load':>10} {'Spread':>10}")
print("=" * 55)

for algo in ["round_robin", "random", "least_conn", "power_of_two"]:

    print(f"  {algo:<20} {mx:>10,} {mn:>10,} {spread:>10,}")

Interview Questions & Answers

Q1: What is a load balancer and why do you need one?

A load balancer distributes incoming traffic across multiple backend servers to prevent any single server from being overwhelmed. Three benefits: (1) Throughput — N servers ≈ N× capacity. (2) Availability — if one server dies, the LB routes around it (with health checks). (3) Flexibility — add/remove servers for scaling and zero-downtime deploys. Most production systems with more than one application server place some load-distribution layer in front of them.

Q2: What's the difference between L4 and L7 load balancing?

L4 operates on TCP/UDP packets — sees IP addresses and port numbers, routes based on those. Faster (kernel-level), works for any protocol. Examples: AWS NLB, IPVS, Maglev. L7 operates on HTTP requests — sees URLs, headers, cookies. Can do path-based routing, cookie stickiness, SSL termination, caching. Slower (HTTP parsing). Examples: AWS ALB, Nginx, Envoy. Default to L7 for HTTP. Use L4 for non-HTTP protocols or when latency is critical.

Q3: Compare round-robin, least-connections, and consistent hashing. When would you use each?

Round-robin: Simple, even distribution when requests have similar cost. Use for stateless services with uniform request patterns. Least-connections: Adaptive — sends to the server with fewest active connections. Best when request durations vary (some fast, some slow). Use as the default for most workloads. Consistent hashing: Same key always goes to the same server. Minimal disruption when servers are added/removed (only ~1/N keys re-map). Use when you need affinity (caching, sessions) and can't externalize state. Downside: can create hot spots if keys are skewed.

Q4: How do health checks work and what happens when a backend fails?

The LB periodically probes each backend (e.g., HTTP GET /health every 10s). If a backend fails N consecutive checks (typically 2-3), the LB marks it unhealthy and stops sending traffic. When the backend passes M consecutive checks (typically 2), it's marked healthy again. During the detection window (interval × fail threshold), some requests may hit the dead backend and fail. Mitigation: shorter intervals (5s), fewer required failures (2), and client-side retries. Critical: use application-level health checks (HTTP 200 + check dependencies), not just TCP connect — a server with an open port but crashing application will fool TCP checks.

Q5: How would you prevent the load balancer from being a single point of failure?

Three strategies: (1) Managed cloud LB (AWS ALB/NLB, GCP LB) — multi-AZ and managed by the cloud provider. This is the easiest approach. (2) Active-passive pair with keepalived — two Nginx/HAProxy instances share a floating VIP. If the primary dies, the passive claims the VIP in a few seconds. (3) DNS round-robin across multiple LBs — client resolves to one of several LB IPs. If one goes down, DNS health checks can remove it, though DNS TTL caching adds delay. Google's Maglev uses ECMP routing for N+1 redundancy across all LB machines.

Q6: A customer reports that some requests are slow but others are fast. The LB uses round-robin. What's happening?

Round-robin distributes requests evenly regardless of server load. If one backend is slower (degraded hardware, heavy background job, GC pauses, hot queries), it still gets the same share of requests as healthy backends. 1 out of N requests lands on the slow server → those users experience high latency. Diagnosis: check per-backend latency metrics (LB access logs, or the X-Backend header). Fix: switch to least-connections — the slow server accumulates active connections, so the LB naturally sends it fewer new requests. Or use least response time to route based on measured latency. Also investigate why the server is slow (it's a symptom, not the root cause).

Key Takeaways Cheat Sheet

The 10 Things You Must Know

1. Load balancers distribute traffic across servers to improve throughput, availability, and deployment flexibility.
2. L4 LBs route by IP/port (fast, any protocol). L7 LBs route by HTTP content (flexible, HTTP only). Default to L7 for web services.
3. Round-robin is simple but blind to load. Least-connections adapts. Consistent hashing provides affinity. Power-of-two choices is a strong low-overhead strategy in modern least-request implementations.
4. Health checks detect dead backends. Use HTTP-level checks (GET /health), not just TCP connect. Interval × fail threshold = detection time.
5. DNS load balancing is the simplest form (different IPs per query) but has no health awareness and suffers from client-side caching.
6. Sticky sessions (IP hash, cookies) break stateless design. Externalize session state to Redis instead.
7. Connection draining: stop new requests to a backend but let in-flight requests finish — essential for zero-downtime deploys.
8. The LB itself is a SPOF unless deployed redundantly. Use managed cloud LBs, active-passive pairs, or ECMP (Maglev-style).
9. Google Maglev: software LB on commodity servers, ECMP + consistent hashing, line-rate 10Gbps in the paper — a strong case for software-defined load balancing.
10. N+1 capacity planning: if you need N servers at peak, deploy N+1 so one failure doesn't cascade into overload.

MM

Mental Model — Load Balancer Under Uneven Load

Least-connections LB automatically routing around a slow server
Key Intuition

Load balancing improves throughput and resilience only if traffic is distributed well and unhealthy nodes are avoided. A round-robin LB with no health checks is barely better than no LB at all — it will happily send traffic to dead servers. The real value comes from the combination of an adaptive algorithm (least-connections, power-of-two) and reliable health checks that remove degraded backends before users notice.

• Describes a software L4 load balancer built on commodity servers, with ECMP at the routers spreading inbound traffic across many active Maglev instances instead of an active/passive pair.

• Defines Maglev hashing as a permutation-based fixed-size lookup table where each backend contributes a permutation of table slots and the fill process yields near-perfect balance.

• Shows the disruption bound on backend change: only entries owned by the removed backend need reassignment, so connection affinity survives pool churn much better than naive modulo hashing.

• Uses Direct Server Return and kernel-bypass style packet handling so the LB primarily processes inbound packets while backend responses bypass the LB on the return path.

[2] Mitzenmacher — "The Power of Two Random Choices" (1996/2001)

Michael Mitzenmacher. PhD Thesis, UC Berkeley, 1996. Survey in IEEE TPDS, 2001.
https://www.eecs.harvard.edu/~michaelm/postscripts/handbook2001.pdf

• Analyzes the classic balls-into-bins process and proves that comparing two random candidate bins dramatically reduces the maximum load relative to single-choice random assignment.

• The asymptotic result is the key technical point: maximum occupancy falls from Θ(log n / log log n) under one random choice to Θ(log log n) under two choices.

• This is why “pick two, choose the less loaded” is such an attractive engineering heuristic: it gets most of the benefit of global state with only constant-time sampling cost.

• Load balancers use the result operationally by sampling a small subset of backends and choosing the lower-load candidate rather than computing a global optimum on every request.
