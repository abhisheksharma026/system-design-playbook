01Key Terms

Man-in-the-Middle Attack (MITM)
An attack where an adversary secretly intercepts and potentially alters communication between two parties who believe they are communicating directly. On an unencrypted HTTP connection, an attacker on the same network (coffee shop WiFi, compromised router) can read passwords, session tokens, and all data in transit. HTTPS prevents MITM by encrypting the channel and authenticating the server's identity via certificates. Without HTTPS, ISPs can inject ads into pages, governments can surveil traffic, and attackers can steal credentials — all demonstrated in the wild.
Symmetric Encryption
Encryption where the same key is used to both encrypt and decrypt data. It is fast; AES-GCM and ChaCha20-Poly1305 are designed for bulk traffic, and hardware acceleration can push AES throughput into the gigabytes-per-second range. The challenge is key distribution: both parties need the same key before communication starts, but sending that key in plaintext gives it to the attacker too. TLS solves this by using a handshake to derive shared traffic keys without putting those keys directly on the wire.
Asymmetric Encryption (Public-Key)
Encryption using a mathematically linked key pair: a public key (shared openly) and a private key (kept secret). Data encrypted with the public key can only be decrypted with the private key, and a private-key signature can be verified with the matching public key. Public-key operations are much slower than symmetric encryption, so TLS uses them for authentication and key agreement, not bulk data transfer. Algorithms: RSA for signatures and legacy key transport, plus ECDSA/ECDH for smaller keys at comparable security levels.
AES (Advanced Encryption Standard)
The dominant symmetric encryption algorithm, standardized by NIST in 2001 after a 5-year international competition. Three key sizes: AES-128 (10 rounds), AES-192 (12 rounds), AES-256 (14 rounds). AES-256 is approved for protecting classified U.S. government information when used in approved systems. In TLS 1.3, AES is used in GCM (Galois/Counter Mode), an AEAD mode that provides both encryption and authentication. Modern CPUs often include AES acceleration, which makes bulk encryption fast enough that network and application work usually dominate.
HTTPS
HyperText Transfer Protocol Secure — HTTP layered on top of TLS. The browser connects to port 443 (instead of HTTP's port 80), performs a TLS handshake, then sends and receives HTTP requests/responses through the encrypted channel. The URL begins with https://. Chrome has marked all HTTP pages as "Not secure" since Chrome 68, and Google's 2025 Chrome data puts HTTPS main-frame navigations in the 95-99% range. HTTPS provides confidentiality, integrity, and server authentication.
TLS (Transport Layer Security)
A cryptographic protocol that provides secure communication over a network. TLS sits between the application layer (HTTP) and the transport layer (TCP). Current version: TLS 1.3 (RFC 8446, August 2018). TLS 1.3 normally completes a full handshake in one round trip, down from the common two-round-trip TLS 1.2 ECDHE flow, and supports 0-RTT for session resumption with replay caveats. TLS 1.0 and 1.1 were deprecated in 2021 (RFC 8996). "SSL" (Secure Sockets Layer) was TLS's predecessor (SSL 3.0 was the last version, deprecated in 2015 due to the POODLE attack). The terms "SSL" and "TLS" are often used interchangeably in casual conversation, but technically SSL is obsolete.
TLS Certificate
A digital document that binds a public key to one or more DNS names, issued and signed by a Certificate Authority (CA). The certificate contains the server's public key, subject alternative names (SANs), issuer, validity period, and signature metadata. When a browser connects to https://example.com, the server sends its certificate chain. The browser checks that the chain leads to a trusted root, the SAN covers the hostname, the certificate is currently valid, and browser-specific checks such as Certificate Transparency and revocation signals pass where applicable.
Certificate Authority (CA)
A trusted entity that issues and signs digital certificates. The browser/OS ships with a pre-installed trust store of root CAs. A CA verifies that the requester controls the domain, usually with an HTTP, DNS, or TLS-ALPN ACME challenge, before issuing a certificate. Major CAs include Let's Encrypt, DigiCert, Sectigo, and GlobalSign. The usual chain is root CA → intermediate CA → end-entity certificate. Browsers validate that chain during the TLS handshake.
TLS Handshake
The process by which a client and server establish a secure connection. In a typical TLS 1.3 full handshake: (1) Client sends ClientHello with supported cipher suites and a key share. (2) Server responds with ServerHello containing the chosen cipher suite, its key share, its certificate, and a signature proving it holds the private key. (3) Both sides derive the same traffic keys from the combined key shares. (4) Client sends Finished. This is normally one round trip. In a common TLS 1.2 ECDHE handshake, key exchange parameters were sent later, so the full handshake used two round trips.

02Why HTTPS Everywhere?

Confidentiality: Without HTTPS, every byte — passwords, credit card numbers, session cookies, personal messages — travels across the network in plaintext. Any device between client and server (routers, switches, WiFi access points, ISP equipment) can read everything. HTTPS encrypts all HTTP traffic.

Integrity: Without HTTPS, an attacker (or ISP) can modify the data in transit — injecting ads into web pages, altering form submissions, swapping download links with malware. TLS's AEAD encryption detects any tampering: if even one bit is changed, decryption fails.

Authentication: Without HTTPS, you have no proof you're talking to the real server. A DNS hijack or BGP route leak could redirect your requests to an attacker's server. The TLS certificate proves the server's identity — the browser verifies it was signed by a trusted CA for the correct domain.

The modern web assumes HTTPS: major browsers negotiate HTTP/2 over TLS, and powerful APIs such as Service Workers and Geolocation are restricted to secure contexts. Google Search has used HTTPS as a lightweight ranking signal since 2014. Chrome marks HTTP pages as "Not secure" today and plans to warn before loading public HTTP sites by default in 2026.

03Encryption Fundamentals

Symmetric: Fast, Shared Secret

// Conceptual symmetric encryption
key = "s3cr3t-256-bit-key"

ciphertext = AES_ENCRYPT(key, "Hello, World!")    // → "9f1a3c7b..."
plaintext  = AES_DECRYPT(key, "9f1a3c7b...")      // → "Hello, World!"

// Same key encrypts and decrypts
// Problem: how do client and server agree on the key securely?

Asymmetric: Slow, Key Pair

// Conceptual asymmetric encryption
(public_key, private_key) = GENERATE_RSA_KEYPAIR(2048)

// Encrypt with public key → only private key can decrypt
ciphertext = RSA_ENCRYPT(public_key, "secret message")
plaintext  = RSA_DECRYPT(private_key, ciphertext)  // → "secret message"

// Digital signature: sign with private key → anyone with public key can verify
signature = RSA_SIGN(private_key, "document hash")
valid     = RSA_VERIFY(public_key, "document hash", signature)  // → true

The Hybrid Approach (What TLS Actually Does)

TLS combines public-key cryptography with symmetric encryption. The handshake authenticates the server and derives shared traffic keys. After that, the record layer uses fast symmetric AEAD ciphers for application data.

Specifically in TLS 1.3: a certificate-based handshake normally uses ephemeral (EC)DHE to derive a shared secret. Both sides generate temporary key material, exchange public shares, and compute the same secret independently. That secret feeds the TLS key schedule, which produces traffic keys for the negotiated AEAD cipher such as AES-GCM or ChaCha20-Poly1305. Because the key exchange is ephemeral, stealing the server's long-term signing key later does not reveal old traffic keys.

Forward Secrecy

Without forward secrecy, static RSA key exchange in older TLS deployments had a bad failure mode: if an attacker recorded traffic and later stole the server's private key, they could decrypt those old sessions. With ephemeral Diffie-Hellman, each session uses temporary key material that is discarded after the session ends. The server's long-term private key authenticates the handshake; it does not directly decrypt old traffic. TLS 1.3 removes static RSA key exchange, so normal certificate-based HTTPS uses ephemeral key exchange by design.

TLS 1.3 sends the client key share immediately, so the server can complete key agreement and send the encrypted certificate flight in the same round trip.

Key improvement over TLS 1.2: In a common TLS 1.2 ECDHE handshake, the client sends ClientHello first, waits for the server's parameters and certificate, then sends its key exchange message. TLS 1.3 moves the client's key share into ClientHello, so the server can compute the shared secret immediately and encrypt the rest of its handshake flight. Result: one fewer network round trip on a full handshake. On a 100 ms path, that is roughly 100 ms saved before application data can flow.

05Certificates & Certificate Authorities

The Chain of Trust

Root CA (e.g., DigiCert Global Root G2)          ← pre-installed in browser/OS trust store
  └── Intermediate CA (e.g., DigiCert SHA2 Secure Server CA)
        └── End-Entity Certificate (e.g., *.example.com)   ← your server's certificate

The browser verifies the chain: your server's certificate is signed by the intermediate CA, whose certificate is signed by the root CA, which is in the browser's trust store. If any link in the chain is invalid, the connection fails.

Let's Encrypt & ACME Protocol

Let's Encrypt launched public certificate issuance in 2015 and made domain-validated certificates free and automatable through ACME. A typical flow is: (1) run an ACME client such as certbot, (2) request a certificate, (3) satisfy an HTTP, DNS, or TLS-ALPN challenge, (4) let the CA verify domain control, and (5) install the issued certificate and renewal job. By late 2025, Let's Encrypt reported serving more than 700 million websites and issuing ten million certificates on some days.

Short validity: Let's Encrypt's common server certificates have used 90-day lifetimes, and the CA has announced a move toward 64-day and then 45-day default lifetimes by 2028. Shorter lifetimes reduce the exposure window after key compromise and force certificate renewal to be treated as automation, not calendar work.

Certificate Transparency (CT)

A public, append-only log system for TLS certificates, originally specified in RFC 6962. Chrome requires publicly trusted TLS certificates issued after April 30, 2018 to be CT-compliant. The exact SCT count depends on delivery method and certificate lifetime, but the important design point is simple: a CA cannot quietly issue a publicly trusted certificate without leaving evidence that domain owners, browsers, and monitors can inspect.

$ openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem \

# Start an HTTPS server using the self-signed cert

server.socket = ctx.wrap_socket(server.socket, server_side=True)
print('HTTPS server on https://localhost:4443')
server.serve_forever()
"

# In another terminal — connect (accept self-signed cert with -k)
$ curl -k https://localhost:4443/

09Mini-Projects

Simple: Certificate Inspector

Goal: Write a Python script that takes a hostname, connects via TLS, extracts the certificate, and prints: subject, issuer, validity dates, days until expiry, and the complete certificate chain. Use the ssl and socket standard libraries. Bonus: alert if the certificate expires within 30 days.

Advanced: HTTPS Server with Let's Encrypt Simulation

Build a Python HTTPS server that generates its own CA (root + intermediate), signs an end-entity certificate, serves HTTPS with the full chain, and has a client that verifies the chain against the custom CA. This simulates the Let's Encrypt flow without actually using the ACME protocol — you'll understand the chain of trust from the inside.

10Production Examples

Let's Encrypt: HTTPS for Everyone

Before Let's Encrypt, many teams still treated certificates as annual manual work: generate a CSR, complete domain validation, install the certificate, and hope someone remembered renewal. Let's Encrypt made domain-validated TLS certificates free and automatable through ACME. By late 2025, it reported serving more than 700 million websites and issuing ten million certificates on some days. Its CA software, Boulder, is open source and written in Go.

Cloudflare: TLS Termination at Edge

Cloudflare terminates TLS at its global edge, so the browser usually handshakes with a nearby edge server instead of a distant origin. Cloudflare then maintains a separate TLS connection to the origin server. This can reduce handshake latency and lets the edge layer handle certificate management, cipher suite configuration, and TLS version negotiation centrally.

Keyless SSL: For financial institutions that can't store private keys outside their premises, Cloudflare offers Keyless SSL — the TLS handshake runs at Cloudflare's edge, but the private key operations (signing the handshake) are proxied to the customer's on-premises key server. The private key never leaves the customer's infrastructure.

Google: HTTPS + Certificate Transparency

Chrome requires CT compliance for publicly trusted TLS certificates issued after April 30, 2018. The number and source of Signed Certificate Timestamps depend on how the SCTs are delivered and on certificate lifetime; Chrome's policy checks both log state and log-operator diversity. The practical result is that public certificate issuance leaves evidence in logs that domain owners and monitors can inspect.

11Edge Cases & War Stories

Heartbleed (2014): A buffer over-read bug in OpenSSL's implementation of the TLS Heartbeat extension. An attacker could send a malformed heartbeat request and read up to 64 KB of server memory, potentially exposing private keys, session tokens, and user data. Contemporary scans reported hundreds of thousands of affected TLS endpoints. The fix was not just patching OpenSSL; teams also had to consider certificate reissue, key rotation, and session invalidation. Lesson: TLS security depends on implementation quality as much as the protocol design.

Certificate Expiry Outages: In 2020, Microsoft Teams had a multi-hour outage tied to an expired TLS certificate. Expiry is a common and preventable TLS incident: automate renewal, alert well before expiry, and test the renewal path before it matters. The fix is boring but important: production certificates should be managed by automation such as certbot, cert-manager, a cloud certificate manager, or a CA-integrated load balancer.

Mixed Content: An HTTPS page loads a JavaScript file over HTTP. The browser may block the HTTP resource, or worse, a network attacker may modify the script before it reaches the browser. Fix the source URLs to use HTTPS and add Content-Security-Policy: upgrade-insecure-requests while cleaning up old links.

TLS Termination Point Matters: Your load balancer terminates TLS and forwards plaintext HTTP to backend servers. If the internal network is compromised, an attacker can read all traffic between the LB and backends. Fix: use end-to-end TLS (re-encrypt between LB and backends) or mTLS (mutual TLS where backends also present certificates to the LB). In Kubernetes, service meshes like Istio/Linkerd handle this with automatic mTLS between all pods.

0-RTT Replay Attack: TLS 1.3's 0-RTT resumption lets a client send encrypted application data before the handshake completes. An attacker may be able to replay that early data, causing the server to process the request twice. For idempotent requests (GET), this is usually manageable. For non-idempotent requests (POST a payment), it is risky. Fix: disable early data for unsafe operations, or use replay protection with server-side tracking.

12Interview Q&A

Q1: Walk me through what happens when a browser connects to https://example.com.

Step 1 — DNS: Browser resolves example.com to an IP address.

Step 2 — TCP: Browser establishes a TCP connection to port 443 (SYN → SYN-ACK → ACK, 1 RTT).

Step 3 — TLS Handshake: Client sends ClientHello (supported ciphers, ECDHE key share). Server responds with ServerHello (chosen cipher, its key share), then sends its certificate and a CertificateVerify signature — all encrypted after the key share exchange. Both sides derive symmetric session keys from the ECDHE shared secret.

Step 4 — Certificate Verification: Browser checks that the certificate chains to a trusted root, the SAN covers the hostname, the certificate is within its validity window, and Chrome-specific CT requirements are satisfied for public certificates. Revocation handling varies by browser and deployment; OCSP stapling is one way to deliver revocation status.

Step 5 — HTTP Request: Browser sends the HTTP GET request through the encrypted TLS tunnel. Request and response data, including headers and cookies, is protected with the negotiated AEAD cipher such as AES-GCM or ChaCha20-Poly1305.

Total overhead: On an example path with a 50 ms RTT, TCP costs about one RTT and a full TLS 1.3 handshake adds one more RTT before application data. DNS may add more unless it is cached. With session resumption, repeat connections can reduce or avoid the TLS round trip, depending on whether early data is enabled.

Q2: Why does TLS use both symmetric and asymmetric encryption?

Performance: Public-key cryptography is expensive compared with symmetric AEAD ciphers. Using RSA or ECDSA-style operations for every byte of a response would be a poor design.

Key agreement: Symmetric encryption requires both parties to end up with the same traffic keys. Classic RSA key transport encrypted a premaster secret to the server's public key, but modern TLS normally uses ephemeral Diffie-Hellman: both sides exchange public shares and compute the same secret without sending the secret itself.

TLS's hybrid approach: Use public-key cryptography for authentication and key agreement. Feed the shared secret into the TLS key schedule. Use the resulting symmetric traffic keys for application data.

Q3: What is forward secrecy and why did TLS 1.3 remove static RSA key exchange?

Forward secrecy means that compromise of the server's long-term private key does not compromise past session keys. Even if an attacker records years of encrypted traffic and later steals the server's private key, they cannot decrypt the recorded sessions.

How: In normal certificate-based HTTPS, TLS 1.3 uses ephemeral ECDHE: both client and server generate temporary key shares for the connection. The shared secret and derived traffic keys are unique to that session. After the session ends, the ephemeral keys are destroyed. The server's long-term private key is used for signing, not for direct key transport.

Why TLS 1.3 changed the default: In TLS 1.2, static RSA key exchange was allowed: the client encrypted the premaster secret with the server's long-term RSA public key. If that RSA key was later compromised, recorded sessions using that key could be decrypted. TLS 1.3 removes static RSA key exchange and uses ephemeral key agreement for normal certificate-based HTTPS.

Q4: How would you handle TLS certificate management for 500 microservices?

External traffic (internet-facing): Use a reverse proxy (nginx, Envoy) or cloud load balancer as the TLS termination point. All external certificates are managed at this single point. Use Let's Encrypt + certbot for automated issuance and renewal. Store certificates in a secrets manager (Vault, AWS Secrets Manager). Monitor expiry with Prometheus + Alertmanager.

Internal traffic (service-to-service): Use a service mesh such as Istio or Linkerd for automatic mTLS. The mesh control plane or workload identity system acts as an internal CA, issuing short-lived certificates to workloads and rotating them without application code changes. This is common in Kubernetes environments where service identity matters.

Key principle: Production certificate work should be automated: generation, signing, installation, renewal, and revocation. Humans should own the policy and the break-glass runbook, not the normal renewal path.

Q5: What is a man-in-the-middle attack and how does TLS prevent it?

The attack: Attacker positions themselves between client and server (e.g., on the same WiFi network). For HTTP: the attacker reads all traffic in plaintext — passwords, sessions, data. They can also modify traffic — injecting malicious JavaScript, changing form submissions, redirecting to phishing pages.

TLS prevention — encryption: Data is encrypted with the negotiated AEAD cipher using traffic keys that only the client and server can derive. The attacker sees ciphertext.

TLS prevention — authentication: The server presents a certificate signed by a trusted CA. The attacker cannot forge a valid certificate for example.com because they don't have the CA's private key. If the attacker presents their own self-signed certificate, the browser shows a security warning ("Your connection is not private").

TLS prevention — integrity: Every TLS record includes an authentication tag (AEAD). If the attacker modifies any encrypted data in transit, the tag verification fails and the record is rejected. Tampering is detected immediately.

Q6: What is the performance overhead of HTTPS compared to HTTP?

Handshake latency: TLS 1.3 adds one network RTT to the first full connection after TCP. With 0-RTT resumption, a client may send early data immediately, but replay risk means servers should only accept it for safe operations. With HTTP/2 or HTTP/3, the handshake cost is amortized across many requests on the same connection.

CPU overhead: Modern CPUs accelerate AES and carry ChaCha20 efficiently, so bulk encryption is rarely the bottleneck. Handshake cost depends on the curve, certificate chain, signature algorithm, hardware, and whether session resumption is used.

Bandwidth overhead: TLS adds a small record header and authentication tag to each protected record. For normal web responses, this is usually tiny compared with payload size, images, JavaScript, and compression decisions.

Practical conclusion: Large providers run HTTPS at full scale. The work is real, but with current hardware, session resumption, and connection reuse, HTTPS is usually an operational and configuration concern rather than a raw throughput blocker.

13Actual System Design Interview

Scenario: Design the TLS infrastructure for an e-commerce platform serving 50 million users across 3 regions (US, EU, APAC). The platform handles payment data (PCI-DSS compliant), serves 100K HTTPS requests/second, and runs 200 microservices in Kubernetes.

Round 1: External TLS

Edge layer: Use a CDN such as Cloudflare or AWS CloudFront for TLS termination near users. The CDN manages public certificates automatically, for example a wildcard certificate for *.example.com via an ACME-capable CA or a commercial CA. Keeping the handshake close to the user reduces the cost of full handshakes and makes certificate policy easier to enforce.

TLS configuration: TLS 1.3 required (TLS 1.2 as fallback for older clients, no TLS 1.0/1.1). Cipher suites: TLS_AES_256_GCM_SHA384 and TLS_CHACHA20_POLY1305_SHA256 (for mobile). HSTS header with max-age=31536000; includeSubDomains; preload to prevent downgrade attacks. OCSP stapling enabled to avoid client-side revocation check latency.

PCI-DSS compliance: PCI DSS requires strong cryptography for cardholder data in transit and does not treat SSL or early TLS as strong cryptography. In practice, use TLS 1.2+ with approved cipher suites and prefer TLS 1.3 where clients support it. Store certificate private keys in managed key infrastructure or HSM-backed services when the risk profile requires it.

Round 2: Internal TLS (mTLS)

Service mesh: Deploy Istio or Linkerd across the Kubernetes clusters. The mesh identity system issues workload certificates and rotates them automatically. All service-to-service communication uses mTLS: both sides present and verify certificates before application traffic flows.

Zero-trust architecture: mTLS ensures that even if an attacker gains access to the internal network, they cannot eavesdrop on or impersonate services without a valid certificate. Combined with network policies (allow traffic only between specific services) and RBAC (restrict which services can call which endpoints), this implements defense in depth.

Round 3: Edge Cases

Interviewer: "A certificate is about to expire in production. The renewal automation failed. What's your incident response?"

Immediate: (1) Manually renew the certificate (certbot renew or reissue from CA). (2) Deploy the new certificate to all edge servers (CDN API or load balancer update). (3) Verify with openssl s_client that the new certificate is being served.

Prevention: (1) Monitor certificate expiry with Prometheus, cloud certificate inventory, or cert-manager metrics. Alert at 30 days, 14 days, and 3 days before expiry. (2) Use cert-manager in Kubernetes for automated ACME certificate management. (3) Keep a tested backup path for emergency reissue and deployment. (4) Prefer short-lived certificates because they force renewal automation to stay healthy.

16References & Further Reading

[1] RFC 8446 — The Transport Layer Security (TLS) Protocol Version 1.3 (Rescorla, 2018)

Eric Rescorla. IETF, August 2018.
https://datatracker.ietf.org/doc/html/rfc8446

• Defines TLS 1.3 and its secure-channel goals: authentication, confidentiality, and integrity. A normal full handshake completes in one round trip after TCP because the client can include a key share in ClientHello.

• Removes static RSA key transport, renegotiation, compression, and legacy non-AEAD cipher suites. RFC 8446 defines five TLS 1.3 cipher suites: AES-GCM, ChaCha20-Poly1305, and AES-CCM variants.

• For normal certificate-based HTTPS, the long-term certificate key authenticates the handshake while ephemeral (EC)DHE key agreement produces the shared secret. PSK modes exist too, so avoid the shorthand that every TLS 1.3 mode is automatically forward-secret.

• 0-RTT early data can reduce latency on resumed sessions, but it has replay risk. Servers need to disable it for unsafe operations or add replay defenses.

[2] RFC 6962 — Certificate Transparency (Laurie, Langley & Kasper, 2013)

Ben Laurie, Adam Langley, Emilia Kasper. IETF, June 2013.
https://datatracker.ietf.org/doc/html/rfc6962

• Defines Certificate Transparency logs, Signed Certificate Timestamps, and Merkle-tree proofs for auditing certificate issuance. CT does not replace CAs; it makes public certificate issuance observable.

• The log structure lets clients and monitors verify inclusion and detect inconsistent log views. Browser enforcement is a separate browser policy layered on top of the CT protocol.

[3] Chrome Certificate Transparency Policy

Chrome Certificate Transparency Team.
https://googlechrome.github.io/CertificateTransparency/ct_policy.html

• Chrome evaluates publicly trusted TLS certificates for CT compliance during validation, with enterprise-policy exceptions. SCT requirements depend on delivery method, certificate lifetime, log state, and log-operator diversity.

• This is the source for Chrome-specific wording. The page should say "CT-compliant" rather than hard-coding a fixed number of logs for every certificate.

[4] RFC 8996 — Deprecating TLS 1.0 and TLS 1.1

K. Moriarty, S. Farrell. IETF, March 2021.
https://datatracker.ietf.org/doc/html/rfc8996

• Deprecates TLS 1.0 and TLS 1.1 and updates older RFCs so new deployments do not negotiate those protocol versions.

• Good interview shorthand: support TLS 1.3 first, keep TLS 1.2 only when client compatibility requires it, and disable SSL, TLS 1.0, and TLS 1.1.

[5] Let's Encrypt — 2025 Scale and Certificate Lifetime Changes

Let's Encrypt / ISRG.
2025 year-end letter · certificate lifetime roadmap

• Let’s Encrypt reported serving more than 700 million websites and issuing ten million certificates on some days in late 2025.

• Its default certificate lifetime is moving from 90 days to 64 days in 2027 and 45 days in 2028, with an opt-in 45-day ACME profile in 2026. The operational lesson is clear: renewal must be automated.

[6] Google — HTTPS Adoption, Search, and Chrome Defaults

Google Search Central / Google Online Security Blog.
HTTPS as a ranking signal · Chrome 68 HTTP warning · HTTPS by default in Chrome

• Google Search started using HTTPS as a lightweight ranking signal in 2014. It was explicitly described as a small signal, not a replacement for content relevance.

• Chrome 68 marked HTTP pages as "Not secure" in 2018. Chrome's 2025 HTTPS-by-default plan says Chrome 154 will enable "Always Use Secure Connections" by default in October 2026, and Google's Chrome data puts HTTPS main-frame page loads in the 95-99% range after 2020.

[7] NIST SP 800-52 Rev. 2 and PCI SSC TLS FAQ

Kerry McKay, David Cooper. NIST, August 2019.
https://csrc.nist.gov/publications/detail/sp/800-52/rev-2/final

PCI Security Standards Council.
PCI SSC TLS FAQ

• NIST gives concrete TLS configuration guidance, including TLS 1.2 as the floor for federal systems in Rev. 2 and TLS 1.3 support where available.

• PCI SSC does not define one mandatory TLS version in its FAQ, but it does say SSL and early TLS are not strong cryptography. Payment systems should use current TLS configurations and keep cryptographic standards under review.
